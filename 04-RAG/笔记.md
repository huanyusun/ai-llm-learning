## 一、RAG 全流程总结

### 0. 为什么需要 RAG（动机）

大模型的四类缺陷（`knowledge/RAG/预备篇/第1章`）：
1. **知识陈旧** —— 训练数据有截止时间，不知道最新事实。
2. **幻觉（Hallucination）** —— 一本正经地编造。
3. **不掌握私有/领域数据** —— 企业内部知识、个人文档。
4. **不可追溯/不可引用** —— 答案来源不明。

RAG（Retrieval-Augmented Generation，检索增强生成）= 把"外部知识库"接入大模型：先检索相关片段，再让大模型"看着"片段作答，缓解以上问题。

**RAG vs 微调 vs 长上下文**（高频对比题）：
| 方案 | 适合 | 代价 |
|---|---|---|
| RAG | 知识频繁更新、需溯源、私有数据 | 需要检索系统，检索质量是天花板 |
| 微调 | 改变模型风格/能力/格式、稳定知识 | 重训成本高，知识更新要重训 |
| 长上下文 | 文档不太多、上下文窗口够 | 成本高、易"中间遗忘"、不可溯源 |

### 1. 经典六步流水线（必背）

> 对应 `mini_rag/main.py` 的 `build_index` + `ask`，对应 `knowledge/RAG/基础篇/第3章 3.1.1`。

```
原始知识
  │
  ▼ ① 加载读取 (load)        TXT/PDF/网页/数据库 → Document
  ▼ ② 分块 (Chunking)        Document → Node，控制块大小 + overlap
  ▼ ③ 嵌入 (Embedding)       Node 文本 → 定长向量
  ▼ ④ 向量存储与索引          向量入库（Chroma/Milvus），建 HNSW 等索引
  │
  ···（离线索引阶段结束，下面是查询阶段）···
  │
  ▼ ⑤ 检索 (Retrieve)        query → 向量 → 余弦相似 → top_k 个 Node
  ▼ ⑥ 生成 (Generate)        top_k 拼进 prompt（含防幻觉约束）→ LLM 生成
```

每一步的"最小知识"：

- **① 加载**：`Document` 不是和文件一一对应（PDF 按页、Markdown 按标题拆）。
- **② 分块**：块大小是 trade-off —— 大块语义完整但召回粗，小块召回精但可能破碎。
- **③ 嵌入**：构造索引和检索必须用**同一个**嵌入模型，否则向量空间不匹配。
- **④ 存储**：文本嵌入默认推荐 **cosine**（Chroma 默认是 l2，要手动 `hnsw:space=cosine`）。
- **⑤ 检索**：归一化向量点积 = 余弦相似度；top_k 典型取 3~5。
- **⑥ 生成**：prompt 里加"上下文不足就回答不知道，不要编造"约束（防幻觉）。

### 2. 关键概念速查（对应 `knowledge/RAG/基础篇`）

| 概念 | 一句话 | 出处 |
|---|---|---|
| Document / Node | Document=整篇，Node=切块后的最小检索单元；Document 继承自 TextNode | 第5章 5.1 |

| VectorStoreIndex | 自动完成嵌入+入库的最常用索引 | 第6章 6.3 |

## 二、优化点（RAG 进阶）

> 对应 `knowledge/RAG/高级篇/第8章`。朴素 RAG 的检索质量天花板，靠这些手段往上抬。

### 1. 检索前：查询转换（Query Transformation）

问题：用户的 query 和文档的语言/形式常有"语义鸿沟"，直接嵌入检索召回差。

- **简单查询转换**：让 LLM 把口语化/有错别字的问题重写成更标准的检索 query。
- **HyDE（Hypothetical Document Embeddings，假设性文档嵌入）**（重点）：
  - 思路：让 LLM 先针对问题生成一个"假设性答案"，再用这个假设答案去嵌入检索。
  - 为什么有效：**答案与答案在向量空间更接近**（query↔answer 有鸿沟，doc↔doc 更近）。
  - 陷阱：HyDE 转换结果放在 `query_bundle.custom_embedding_strs`，**只用于嵌入检索**，不替换原 query。
  - 效果：召回相关性明显提升，尤其事实性问答。
- **多步查询转换（MultiStep）**：复杂问题分多步，每步基于上一步推理提出子问题，分多次检索。关键参数 `index_summary` 告诉模型知识库范围。
- **子问题查询转换（SubQuestion）**：一次性把原问题拆成多个子问题，可按"可用工具"约束（更结构化，与多步的"基于推理"不同）。

### 2. 检索后：节点后处理器（Node Postprocessor）与 Rerank

问题：向量检索召回的 top_k 里常有"相关但不是最相关"的块，顺序不理想。

- **节点后处理器**：检索完后、送生成前，对结果做过滤/重排/替换。统一接口，可串多个。
  - 相似度过滤（低于阈值的丢）
  - 关键词过滤
  - 元数据替换（如 `SentenceWindowNodeParser` 的 window 注入）
  - 固定时间排序（时效性）
- **Rerank 重排**（重点）：用一个**更重的精排模型**（如 Cohere Rerank、开源 `bge-reranker-large`）对召回的候选二次打分排序。
  - 典型 pipeline：**向量检索召回较多候选（如 top 20）→ Rerank 精排 → 取 top 3~5 送生成**。
  - 为什么有效：向量检索是双塔近似（快但粗），Rerank 是交叉编码器（慢但准）。
  - 权衡：Rerank 模型推理慢/有成本，只对少量候选做。

### 3. 其他优化方向（连回第11章/第13章）

- **chunk 大小调优**：块大小是检索/生成质量的关键旋钮，需按数据集实测。
- **检索块与生成块分离**：检索用小块（精确召回），生成用大块/父块/窗口（上下文丰富）—— 对应 `SentenceWindowNodeParser` / `HierarchicalNodeParser`。
- **融合检索 / 递归检索**：关键词 + 向量混合；用 `IndexNode` 做多级检索。
- **评估驱动**：用 `faithfulness`（忠实度，是否基于上下文）/ `relevancy`（相关度）等指标量化检索与生成质量（第10章）。
- **前沿范式**：C-RAG（纠正式）、Self-RAG（自反思）、RAPTOR（层次摘要树）（第13章）。

### 优化点 mini 代码索引（可在 `main.py` 上扩展）

- 加 Rerank：在 `retrieve` 后对 hits 用一个重打分函数（真实场景接 `bge-reranker`）。
- 加 HyDE：在 `retrieve` 前先调 LLM 生成假设答案，用假设答案 `embed` 去检索。
- 加相似度阈值过滤：在 `ask` 里过滤 `score < threshold` 的 hit。

## 三、面试要点

### Q1：简述 RAG 的核心工作流程与必备模块。
**答**：六步——① 加载读取文档 → ② 分块（Chunking）→ ③ 嵌入（Embedding）为向量 → ④ 向量存储与索引 → ⑤ 检索（query 向量做语义相似检索，取 top_k）→ ⑥ 生成（top_k 作上下文拼进 prompt，含防幻觉约束，交 LLM 生成）。必备模块对应这六步。

### Q2：Chunking 为什么要 overlap？约束是什么？chunk_size 单位是什么？
**答**：overlap 让相邻块共享部分内容，避免边界处丢上下文、提升召回。约束 `0 <= overlap < chunk_size - 1`（按句/按 token 皆然），过大退化为完全重复。chunk_size 单位是 **token**（默认 tiktoken），是**上限**非精确值，且默认**包含元数据 token**，所以实际可用文本空间 < chunk_size。

### Q3：为什么文本嵌入用余弦相似度？归一化的作用？
**答**：文本嵌入关注"方向是否一致"而非绝对幅度，余弦相似度只看夹角。先对向量做 L2 归一化，则两向量的**点积就等于余弦相似度**，省一次除法，检索就是矩阵向量乘，可批量加速。（→ `mini_rag/main.py` 的 `embed` 末尾做 L2 归一，`query` 直接点积。）

### Q4：朴素 RAG 原型为什么不能上生产？至少 4 点。
**答**：① 索引与查询需异步分离而非顺序；② 缺可视化知识管理；③ 原始文档↔Chunk 缺对应、知识更新难同步；④ 缺外部系统 API 集成；⑤ 不支持多用户并发；⑥ 缺 UI 与历史记录持久化；⑦ 缺结果溯源；⑧ 鲁棒/可扩展/可维护性差。

### Q5：什么是 HyDE？为什么有效？有什么陷阱？
**答**：HyDE = 让 LLM 先针对问题生成"假设性答案文档"，再用它去嵌入检索。有效是因为**答案与答案在向量空间更接近**（query↔answer 有语义鸿沟）。陷阱：转换结果只用于嵌入检索（放在 `custom_embedding_strs`），**不替换原 query**；生成的假设文档本身可能不准，但对召回通常仍有提升。

### Q6：Rerank 是什么？为什么不直接用向量检索的排序？
**答**：向量检索是双塔近似（query 和 doc 各自编码再算相似），快但粗。Rerank 用交叉编码器（cross-encoder）对 query-doc 对联合编码打分，更准但慢。典型用法：向量检索**多召回**（top 20）→ Rerank **精排** → 取 top 3~5 送生成。

### Q7：Document 和 Node 的关系？为什么要分两层？
**答**：Document 继承自 TextNode（同根，都是"文本+元数据"容器）。Document=整篇，Node=切块后的最小检索单元。分两层是为了灵活控制检索粒度：Document 粒度大语义完整但召回粗，Node 粒度小召回精确。元数据会被 Node 自动继承。

### Q8：LlamaIndex 默认用什么模型/向量库？如何切到本地 Ollama？
**答**：默认 LLM=OpenAI GPT-3.5、嵌入=text-embedding-3-small、向量库=内存。切本地：通过全局 `Settings.llm = Ollama(model=...)`、`Settings.embed_model = OllamaEmbedding(model_name=...)`。陷阱：不配置就默认走 OpenAI，必须设 `OPENAI_API_KEY`。

### Q9：构造索引和检索时的嵌入模型必须一致吗？为什么？
**答**：必须一致。向量是"在某个嵌入模型的向量空间里的坐标"，换模型 = 换坐标系，旧向量在新空间里无意义，检索完全失效。同理，Chroma 的 `hnsw:space` 距离度量也要和嵌入模型推荐一致（文本嵌入一般用 cosine，不指定默认 l2 会排序失真）。

### Q10：列出至少 4 种 LlamaIndex CBEventType。
**答**：`CHUNKING`（分割）、`EMBEDDING`（嵌入）、`LLM`（大模型调用）、`RETRIEVE`（检索）、`SYNTHESIZE`（组装 prompt 并生成）、`QUERY`（查询引擎）、`NODE_PARSING`、`TREE`、`SUB_QUESTION`。

### Q11（开放）：怎么把 `mini_rag/main.py` 升级成真实 RAG？最少改动？
**答**：只动三个函数，骨架（六步编排）不变：
- `embed`：哈希词袋 → `sentence-transformers` 或 `ollama.embeddings`；
- `NumpyVectorStore` → `chromadb.PersistentClient`（`hnsw:space=cosine`），`query` → `collection.query(n_results=k)`；
- `generate_dummy` → `ollama.generate(model, prompt, stream=True)`。
这正是 `main.py` 与 `main_real.py` 并排存在的意义：**原理骨架与具体组件解耦**。
