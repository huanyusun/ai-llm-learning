# 综合项目总结：把所学串成一个产品

> 对应项目：`projects/智能知识助手/`
> 这份笔记回答两个问题：**（1）怎么把前面学的串成一个产品？（2）面试里这个项目能讲什么亮点？**

---

## 一、串成一个产品的思路

### 1.1 先想清楚"产品形态"，再倒推技术栈

学完 01–06 后，脑子里是一堆零件：Transformer、embedding、Prompt 技巧、RAG 六步、ReAct 循环、LoRA、量化……如果挨个做 demo，每个都很零散。**把它们串成产品的关键是先定一个产品形态**：

> "一个能多轮对话、按需查知识、必要时算数、答错会承认的知识助手。"

这个形态一确定，每个零件就有了**它在产品里的位置**：

```
产品需求              用到的零件              来自哪个模块
─────────────────────────────────────────────────────────
能"记住"上文          Memory + 指代消解        整合新增（基础工程能力）
能"查知识"            RAG 检索                 04-RAG
能"算数/调外部能力"    工具调用                 05-Agent
能"决定查不查/查什么" ReAct 推理循环           05-Agent + 03-Prompt(CoT)
能"说人话回答"        结构化 Prompt + 综合生成  03-Prompt
不会"瞎编"            防幻觉阈值 + 溯源         02-LLM基础(幻觉) + 04-RAG
底层"懂语义"          embedding + 余弦相似度   00-数学 + 01-Transformer 概念
```

**思路要点：产品形态定位置，零件填进去，而不是反过来。**

### 1.2 关键架构决策：把 RAG 降级成"工具"

新手做整合最容易犯的错是"RAG 一条线 + Agent 一条线，两条线并列"。更好的做法是：

> **RAG 是 Agent 工具箱里的一个工具。**

这样 ReAct 循环就成了唯一的"主控"：
- 事实问题 → Action: `检索[...]`
- 数学问题 → Action: `计算[...]`
- 都不需要 → 直接 `完成[...]`

好处：
1. 只有一条控制流（react_loop），代码清晰；
2. 加新工具（联网搜索、查数据库、调 API）只需在 `make_tools` 里注册一项，主循环不动；
3. 这正是 LangChain / OpenAI function calling 的设计哲学——**Agent 是壳，工具是插件**。

### 1.3 分层：骨架与实现分离

项目刻意把代码分成两层：

- **骨架层（不依赖任何外部服务）**：`react_loop`、`Memory`、`SYSTEM_PROMPT`、`compose_final_answer`、工具注册表。
- **实现层（可替换）**：`mock_llm_step`、`embed`、`NumpyVectorStore`。

骨架层用的是"思想"（ReAct 怎么循环、Prompt 怎么分层、记忆怎么管理），这些都是模型无关的；实现层是"具体的模型/库"。**换实现层不动的骨架，模拟版立刻变真实版。** 这是软件工程里"依赖反转"在 AI 应用上的直接体现，也是项目能在零依赖下跑通、又能平滑接真实模型的原因。

### 1.4 让"防幻觉"内建，而不是事后补

很多 demo 的防幻觉是"事后给答案加一句免责声明"。本项目把它做进了两层：
- **检索层**：`RELEVANCE_THRESHOLD` 阈值，低于即标 `is_relevant=False`；
- **生成层**：`compose_final_answer` 只采纳 relevant 片段，无片段则诚实拒答；
- **展示层**：每条答案带 `（来源：知识库#X）` 溯源标注。

三层联动，比单一兜底可靠得多。这也是真实生产 RAG 系统的标准做法。

## 二、面试可讲的项目亮点

### 亮点 1：从零搭了一个"RAG + Agent + 记忆"三位一体的助手

> 一句话：不是把三个 demo 拼起来，而是把 RAG 降级成 Agent 的一个工具，用 ReAct 循环统一调度，再叠加多轮记忆。

**可深挖的追问**：
- Q：为什么不让 RAG 单独跑？
  A：单独 RAG 是"必查"模式，对超纲/闲聊问题会硬塞不相关内容（幻觉来源）。放进 Agent 后，由 Agent 判断要不要查，能拒绝无意义检索。
- Q：Agent 怎么知道该用哪个工具？
  A：本模拟版用脚本化意图识别；真实版用 LLM 读工具描述（function calling / ReAct Prompt）自行决策。

### 亮点 2：纯 numpy 实现，对每个组件的原理都讲得清楚

> 面试官最爱问"你真的懂 RAG 吗"——本项目能从数学层答：

- **embedding**：词袋 + 特征哈希（hashing trick）把无限词表压到定维，L2 归一化后点积 = 余弦相似度，省一次除法。
- **检索**：`argpartition` 拿 top_k 是 O(N) 而非 O(N log N)，工程上常见的小优化。
- **为什么 toy 版语义弱**：纯统计、无上下文；真实版换 bge/e5 这类 Transformer 编码器后才有语义泛化。能讲清"骨架一致、区别在编码器"，说明理解到位。

### 亮点 3：把"防幻觉"做成了三层机制（见 1.4）

> 这是最能体现工程素养的点。面试问"如何缓解幻觉"，可以答：
> 1. 检索相关度阈值过滤；
> 2. 只让 LLM 基于检索到的上下文作答（Prompt 约束：上下文不足就说不知道）；
> 3. 输出带溯源标注，便于人工核查。
> 并说明各自解决幻觉的哪个环节（召回不足 / 生成编造 / 事后追责）。

### 亮点 4：多轮记忆 + 指代消解

> 体现"对话系统"思维，而不只是单轮 QA。
- 滑动窗口（max_turns）防止 Prompt 无限膨胀——这是真实场景必须考虑的成本/上下文长度问题。
- toy 指代消解（"它/这个"→上轮主题）演示了"上下文依赖"这件事的本质；真实版会让 LLM 自己消解。

### 亮点 5：骨架/实现分离，可平滑接真实模型

> 体现软件设计能力。面试问"你这个 demo 怎么上生产"，可以答：
> 替换三个点（LLM 调用、embedding 模型、向量库），骨架（ReAct 循环、记忆、Prompt、防幻觉）一行不改。这就是"依赖反转 / 面向接口编程"在 LLM 应用上的落地。

### 亮点 6：ReAct 的步数上限是有论文依据的

> 论文细节也能秀：ReAct 原论文（Yao et al. 2022, arXiv:2210.03629）在 HotpotQA 上把步数上限设 7、Fever 设 5，更多步不提升性能。本项目 `MAX_STEPS = 6`，且到上限会优雅回退（不崩、给兜底答案）——这叫"有依据的工程取舍"。

## 三、可能被追问的"坑"与你的回答

| 面试官追问 | 你的回答 |
|-----------|---------|
| toy embedding 分不出"晚饭"和"幻觉缓解"怎么办？ | 因为是字符 n-gram 统计，高频字会拉高相似度。生产用语义模型（bge）即可；本项目用 0.20 阈值在 toy 层做粗过滤，是"实现弱但有兜底"的设计。 |
| Agent 死循环怎么办？ | `MAX_STEPS` 上限 + 上限兜底答案；真实版再加"重复 Action 检测"和"工具调用频率限制"。 |
| 记忆无限增长怎么办？ | 滑动窗口（max_turns=3）；真实版还会做"记忆摘要压缩"（把老对话 summary 成一段）。 |
| 多工具冲突怎么办？ | Agent 自己决定顺序；必要时可串行（先检索确认事实，再计算）。 |
| 怎么评估这个系统？ | 1) 检索层：召回率/MRR；2) Agent 层：工具选择准确率、步数分布；3) 端到端：人工标注答案正确性 + 拒答率（防幻觉是否过度）。 |

## 四、一句话总结

> **学完 01–06 的标志，不是会背每个概念，而是能用一个产品形态把它们都"安顿"到正确的位置上。** 这个智能知识助手就是那个产品形态：Agent 当大脑、RAG 当知识工具、Prompt 当沟通规范、记忆当上下文、防幻觉当安全网——每个零件都在它该在的地方。
