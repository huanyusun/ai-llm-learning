## 一、设计一个智能客服系统

**目标**：7×24 回答用户咨询，覆盖 FAQ / 订单 / 售后；答不了的转人工；持续变好。

### 架构（分层，每层都能讲出"为什么"）

```
用户消息
   │
   ▼
[接入层] 会话管理、限流、敏感词、多轮上下文（Memory）
   │
   ▼
[意图识别 + 路由]  ← 【决策点1：先分流，别啥都扔给大模型，省成本降延迟】
   ├── 闲聊/简单 FAQ ──→ 规则/小模型/向量检索 FAQ 直答（快、便宜）
   ├── 知识问答        ──→ RAG（检索私有知识库）  → 04-RAG/mini_rag
   ├── 多步任务/查业务 ──→ Agent（工具调用：查订单/改地址/发工单）→ 05-Agent
   └── 都不靠谱/低置信 ──→ 人工兜底（human-in-the-loop）
   │
   ▼
[对话管理] 多轮记忆 + 指代消解（"它/这个"→上轮主题）→ projects/智能知识助手 的 Memory
   │
   ▼
[生成 + 防幻觉] 上下文不足就拒答/转人工；高危动作（退款/改密码）必须二次确认
   │
   ▼
[评估 + 日志] 忠实度/相关度打分、记录未解决 case → 喂给数据飞轮
```

### 关键决策（面试追问点）
- **先路由后大模型**：90% 的请求是简单 FAQ，直接规则/检索答，省钱省延迟；只有复杂的才进 Agent。这是成本控制的核心。
- **RAG vs 微调**：知识频繁变（活动/政策）→ RAG；固定话术/语气 → 微调。客服知识更新快，**默认 RAG**。（见二、RAG 优化）
- **防幻觉 + 溯源**：每条回复带"来源卡片"，答不了诚实转人工，别编。（04-RAG/eval_rag 的 faithfulness）
- **评估闭环**：在线用 faithfulness/relevancy 自动打分 + 抽样人工复核（eval_rag 思路）。
- **安全**：Prompt 注入防御（分隔符 + 角色标记 + 高危 human-in-the-loop，见 03-Prompt/笔记 §7）。

### 和本仓库的对应
| 客服组件 | 本仓库实现 |
|---|---|
| RAG 知识问答 | `04-RAG/mini_rag` + `main_advanced`（Rerank/HyDE） |
| Agent 工具调用 | `05-Agent/03-多工具agent` + `04-多Agent协作` |
| 多轮记忆 + 指代 | `projects/智能知识助手` 的 Memory |
| 评估 | `04-RAG/mini_rag/eval_rag` |
| 对齐/语气定制 | `06-微调与部署/05-最小SFT训练`（SFT 思路） |

## 二、如何优化一个已有的 RAG 系统（效果上不去怎么办）

**面试套路：先诊断，再对症下药。** 别上来就说"加 rerank"。

### 第 0 步：先量化（没有评估就没有优化）
- 先上评估：faithfulness（答得有无据）/ relevancy（答得切不切题）/ context_relevance（检索准不准）。
- 看是【检索】不行还是【生成】不行：检索召回的上下文里有没有正确答案？
  - 没有 → 检索层问题（召回率）。
  - 有但答错 → 生成层问题（遵循度/幻觉）。
- → 用 `04-RAG/mini_rag/eval_rag` 跑一遍，定位是哪一环。

### 七个优化杠杆（按"投入产出比"排序）

| # | 杠杆 | 解决什么 | 本仓库对应 |
|---|---|---|---|
| 1 | **检索→Rerank** | 双塔召回粗，cross-encoder 精排把最相关排前面 | `main_advanced.rerank` |
| 2 | **查询转换/HyDE** | 用户措辞与文档有鸿沟，改写后再检索 | `main_advanced.hyde_expand` |
| 3 | **chunk 大小调优** | 太大召回粗、太小破碎；检索块/生成块分离 | `mini_rag` split_by_sentences |
| 4 | **嵌入模型升级** | toy 哈希词袋→bge-m3 等语义模型 | `main_real.py` |
| 5 | **混合检索** | 关键词(BM25)+向量，互补 | （进阶，第11章） |
| 6 | **生成层：更好的 prompt + 防幻觉** | 约束"只基于上下文答，不足就拒答" | `mini_rag.build_prompt` |
| 7 | **评估驱动迭代** | 每改一处在评测集上 A/B + 回归 | `eval_rag` |

> 一句话答面试：**"先评估定位是检索还是生成的问题，再按性价比依次上 Rerank、HyDE、调 chunk、换嵌入模型；每步都靠评估指标回归，不凭感觉。"**

## 三、数据飞轮（data flywheel）

**目标**：让系统【越用越好】——线上数据回收 → 改进模型/知识 → 效果提升 → 吸引更多使用 → 更多数据。

### 闭环
```
        ┌─────────────────────────────────────────────────┐
        ▼                                                 │
  ┌──────────┐  真实问答   ┌──────────┐  人工/自动标注  ┌──────────┐
  │ 线上服务 │ ──────────→ │ 日志回收 │ ──────────────→ │ 数据沉淀 │
  │ (RAG/Agent)│            │ (query/  │                 │ (高质量  │
  └────┬─────┘              │  回答/   │                 │  对话/   │
       │                    │  反馈)   │                 │  纠错)   │
       │ 更新               └──────────┘                 └────┬─────┘
       │                                                   │
       │  ┌────────────────────────────────────────────────┘
       │  ▼
       │ 两种用法（按数据类型分）：
       │  ├─ 【知识型数据】→ 清洗后写回【知识库】→ RAG 直接用（立刻生效，无需训练）
       │  └─ 【行为/格式型数据】→ 做 SFT/对齐训练 → 模型更符合业务（06-微调与部署）
       ▼
  ┌──────────┐  评估通过才上线（eval_rag 把关）→ 效果提升 → 更多用户 → 更多数据 ↺
  │ 评估&上线 │
  └──────────┘
```

### 关键认知（面试点）
- **知识更新用 RAG，行为定制用微调**：飞轮里这两类数据走不同通道。新增"退货政策"→ 知识库（RAG）；希望客服更口语化/统一话术 → SFT。（06/笔记 §0 决策原则）
- **评估是闸门**：回收的数据训出来的模型必须过评估集（faithfulness/业务指标）才能上线，否则飞轮会"越转越烂"（数据污染、reward hacking）。
- **冷启动**：没有线上数据时，用人工标注的种子集 + 公开数据起步；飞轮转起来后逐步替换为真实数据。
- **闭环里的 RLHF/DPO**：用户"点赞/点踩"就是偏好数据 → 可做 DPO 对齐（06/笔记 §1.5），让回复更贴近用户期望。

## 四、三道题的共同答题框架（背下来）

1. **先澄清需求/约束**：用户量？延迟？准确率要求？预算？（别假设）
2. **先量化、再优化**：没有指标就没有方向（RAG 题 especially）。
3. **分层/分模块讲架构**：接入→路由→核心(RAG/Agent)→生成→评估→兜底，每层讲"为什么这么选"。
4. **讲权衡不讲黑科技**：RAG vs 微调、延迟 vs 准确率、成本 vs 效果，每个选择都给 trade-off。
5. **闭环意识**：任何系统都要有【评估 + 数据回收】，否则一次性的、不可迭代。

> 这套框架 + 本仓库已建的 RAG/Agent/评估/微调 项目，足以撑起一道 30 分钟的系统设计题。
