## 1. Agent 是什么

### 1.1 定义

**Agent（智能体）= 以 LLM 为大脑，能够感知环境、自主规划、调用工具、执行行动，并基于反馈循环迭代直到完成目标的系统。**

更工程化的定义（来自 Lilian Weng, OpenAI）：

> Agent = LLM（大脑）+ Planning（规划）+ Memory（记忆）+ Tool use（工具使用）+ Action（行动）

核心特征是 **自主性（autonomy）** 与 **闭环（closed-loop）**：它不是「问一句答一句」，而是「给我一个目标，我自己拆解、调用、修正、直到完成」。

### 1.2 Agent vs Chatbot vs RAG

| 维度 | Chatbot（聊天机器人） | RAG（检索增强生成） | Agent（智能体） |
|------|----------------------|--------------------|-----------------|
| 核心能力 | 单轮/多轮对话 | 检索知识 + 生成 | 自主规划 + 行动 + 反馈 |
| 是否有记忆 | 仅对话上下文 | 向量库（知识） | 短期+长期+工作记忆 |
| 是否调用工具 | 否 | 仅检索（算半个工具） | 是，多工具组合 |
| 是否有规划 | 否 | 否 | 是，可拆解多步 |
| 是否有行动 | 否（只输出文本） | 否（只输出文本） | 是（执行产生副作用：发邮件、写文件、调 API） |
| 是否闭环 | 否 | 否（一次检索一次生成） | 是（Observe → 修正 → 再行动） |
| 典型例子 | 早期客服机器人 | 文档问答 | AutoGPT、Devin、Claude Code |

**关键区别一句话：**
- Chatbot 是「嘴」，只会说；
- RAG 是「带书架的嘴」，会查资料再说；
- Agent 是「有手有脑的人」，会查、会想、会做、会改。

> 易混淆点：RAG 经常作为 Agent 的「一个工具」存在（Agent 调用检索工具）。两者不是互斥关系，而是包含关系。

### 1.3 Agent 的五大要素详解

**① LLM（大脑 / 推理引擎）**
- 负责理解意图、推理下一步、生成工具调用参数、综合最终答案。
- Agent 的能力上限由底层 LLM 决定（弱模型做不好 ReAct 推理）。

**② Planning（规划）**
- **任务拆解（Task Decomposition）**：把大目标拆成子目标。
- **子目标依赖与顺序**：哪些可以并行，哪些必须串行。
- 常见策略：CoT（链式思考）、ReAct、Plan-and-Execute、ToT、Reflexion。

**③ Memory（记忆）**
- 短期记忆：当前对话/任务的上下文窗口（context window）。
- 长期记忆：跨会话持久化，通常存向量库（用户偏好、历史事实）。
- 工作记忆：当前任务进行中的中间状态、变量、scratchpad。

**④ Tool Use（工具使用）**
- 搜索、计算器、代码解释器、数据库查询、外部 API、文件读写。
- Agent 通过工具突破 LLM 的三大限制：知识截止、无法计算、无法行动。

**⑤ Action（行动 / 执行）**
- 真正产生副作用的一步：调用 API 写入数据、执行代码改变环境、发送消息。
- 区分 Action 与 Thought：Thought 是「想」，Action 是「做并改变世界状态」。

### 1.4 Agent 的运行骨架（所有 Agent 框架的公约数）

```
User Goal
   │
   ▼
┌─────────┐
│  Plan   │ ← LLM 拆解目标
└────┬────┘
     ▼
┌─────────────────────────────────┐
│  Loop (until done or max steps):│
│   1. Thought（推理下一步）       │
│   2. Action（选择+调用工具）     │
│   3. Observation（读取结果）     │
│   4. Memory（更新记忆）          │
│   5. Reflect（是否完成/修正）    │
└────────────┬────────────────────┘
             ▼
        Final Answer
```

这个 Loop 就是后面 ReAct、Plan-and-Execute 等所有范式的「母体」。

## 2. Function Calling（函数调用 / 工具调用）

### 2.1 是什么

**Function Calling 是 LLM 厂商提供的一种原生能力：模型能够根据用户请求，输出一个结构化的「函数调用请求」（函数名 + JSON 参数），由外部代码执行该函数后把结果回传给模型。**

它本质是把「工具使用」这件事从 Prompt 工程下沉到了 **模型训练 + API 层**，让模型直接吐 JSON 而不是文本里抠参数。

### 2.2 为什么需要

在没有 Function Calling 之前，让 LLM 调用工具只能靠 Prompt：
```
请输出以下格式：Action: get_weather, Input: {"city":"北京"}
```
问题：
- 输出格式不稳定，常常漏字段、加废话、用错引号；
- 参数类型错误（把数字当字符串）；
- 解析依赖正则，脆弱；
- 厂商之间无标准。

Function Calling 通过 **微调 + 受限解码（constrained decoding）** 让模型直接生成保证合法的 JSON，从根本上解决「模型说人话 vs 程序要 JSON」的鸿沟。

### 2.3 原理机制

```
1. 开发者向 API 声明可用工具（JSON Schema 描述函数名/参数/描述）
        │
        ▼
2. 用户提问 → 模型推理是否需要工具
        │
        ├─ 不需要 → 直接返回文本
        │
        └─ 需要 → 返回 tool_calls（函数名 + 已校验的参数 JSON）
        │
        ▼
3. 开发者代码执行真实函数，得到结果
        │
        ▼
4. 把结果以 role=tool 的消息回传
        │
        ▼
5. 模型基于结果生成最终回答（或继续调用下一个工具）
```

**关键点：模型本身不执行函数，只生成调用意图；执行发生在你的代码里。** 这是安全和可控的基础。

### 2.4 OpenAI Function Calling 示例

```python
from openai import OpenAI
client = OpenAI()

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的当前天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名"},
                "unit": {"type": "string", "enum": ["c", "f"]}
            },
            "required": ["city"]
        }
    }
}]

resp = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "北京今天多少度？"}],
    tools=tools
)

# 模型返回 tool_calls，而非直接文本
tool_call = resp.choices[0].message.tool_calls[0]
# tool_call.function.name == "get_weather"
# tool_call.function.arguments == '{"city":"北京","unit":"c"}'

# 开发者执行真实函数
import json
args = json.loads(tool_call.function.arguments)
result = get_weather(**args)

# 把结果回传
resp2 = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "北京今天多少度？"},
        resp.choices[0].message,                       # assistant 的 tool_call
        {"role": "tool", "tool_call_id": tool_call.id, "content": str(result)}
    ],
    tools=tools
)
print(resp2.choices[0].message.content)  # "北京今天 25 摄氏度。"
```

### 2.5 各大模型实现差异

| 厂商 | 字段名 | 备注 |
|------|--------|------|
| OpenAI | `tools` / `tool_calls` | 行业事实标准，被广泛模仿 |
| Anthropic Claude | `tools` / `tool_use` block | 返回结构是 content block，而非顶层字段 |
| Google Gemini | `function_declarations` / `functionCall` | 参数 schema 略有差异 |
| Qwen / 通义 | 兼容 OpenAI 格式 | 阿里 DashScope 同时支持 ReAct prompt 兜底 |
| DeepSeek | 兼容 OpenAI 格式 | `deepseek-chat` 支持 function calling |
| 开源模型（Llama/Mistral） | 需训练或 prompt | 较小模型 function calling 能力弱 |

### 2.6 Function Calling vs ReAct（必考对比）

| 维度 | Function Calling | ReAct |
|------|-----------------|-------|
| 实现层级 | API/模型原生层 | Prompt 工程层 |
| 输出格式 | 结构化 JSON（厂商保证） | 自由文本，需正则解析 |
| 推理过程 | 模型内部，不可见（黑盒） | 显式 Thought，可见可调试 |
| 多步循环 | 需开发者写循环 | Prompt 里约定循环 |
| 可靠性 | 高（格式保证） | 中（依赖模型遵守 prompt） |
| 灵活性 | 受限于厂商 schema | 任意文本协议 |
| 调试性 | 较差（黑盒） | 好（有 Thought） |
| 模型要求 | 需支持 FC 的模型 | 任何强模型都可 |
| 关系 | 现代 ReAct 通常用 FC 实现 Action 步 | 是更上层的范式 |

**一句话：ReAct 是「思想/范式」，Function Calling 是「实现技术」。今天主流 Agent = ReAct 范式 + Function Calling 实现 + 框架循环。**

## 3. ReAct 范式

### 3.1 是什么

**ReAct = Reasoning + Acting（推理 + 行动）**，由 Yao et al. 2022 提出。核心思想：让 LLM 在每一步都先**推理（Thought）**再**行动（Action）**，然后**观察（Observation）**结果，三者交替循环，直到得出答案。

论文核心贡献：证明「让模型边想边做（交错 reasoning trace 和 action）」比「纯思考（CoT）」或「纯行动」效果都好。

### 3.2 为什么有效

- **Thought 的作用**：引导模型推理、跟踪进度、纠错、规划，减少幻觉；
- **Act 的作用**：引入外部信息（搜索/计算），突破知识截止与计算弱；
- **Obs 的作用**：把外部真实结果注入上下文，形成闭环反馈；
- 三者交错让模型「知道自己做了什么、看到了什么、下一步该做什么」。

### 3.3 Thought-Action-Observation 循环

```
Question: 美国的 GDP 是中国的多少倍？

Thought 1: 我需要先查美国和中国的 GDP 数据。
Action 1: Search["美国 2024 GDP"]
Observation 1: 美国 2024 年 GDP 约 28.7 万亿美元。

Thought 2: 现在需要中国的 GDP。
Action 2: Search["中国 2024 GDP"]
Observation 2: 中国 2024 年 GDP 约 18.5 万亿美元。

Thought 3: 28.7 / 18.5 ≈ 1.55，美国约为中国的 1.55 倍。
Action 3: Finish[美国 GDP 约为中国 1.55 倍]
```

### 3.4 完整 Prompt 模板（经典版）

```
尽可能回答以下问题。你可以使用以下工具：

[1] Search[q]：在搜索引擎查询 q，返回前 3 条结果摘要。
[2] Lookup[key]：在当前文档中查找关键词 key。
[3] Calculator[expr]：计算数学表达式 expr。
[4] Finish[answer]：当你得到最终答案时调用，参数是答案。

格式必须严格遵循：

Question: <输入问题>
Thought: <你的推理>
Action: <工具名>[<参数>]
Observation: <工具返回>
... (Thought/Action/Observation 可重复多次)
Thought: <最终推理>
Action: Finish[<最终答案>]

开始！

Question: {question}
```

**模型每生成到 `Action:` 就暂停（stop sequence）**，框架执行该 Action，把结果填到 `Observation:` 后面，再让模型继续生成下一个 `Thought:`。这就是 ReAct 的执行原理——**通过 stop sequence 实现「轮流」**。

### 3.5 执行流程（框架视角）

```
1. 拼接 Prompt（系统提示 + 工具说明 + 历史 + Question）
2. 调用 LLM，stop=["Observation:", "\nObservation"]
3. 解析输出，提取 Action 名和参数
   ├─ Action == Finish → 返回答案，结束
   └─ 其他 → 执行对应工具
4. 把 Observation 拼回 Prompt
5. 回到第 2 步
6. （保护）达到 max_iterations 强制结束
```

### 3.6 现代版 ReAct（基于 Function Calling）

今天不再用文本协议，而是用 FC 实现 ReAct 思想：

```python
while step < MAX_STEPS:
    msg = llm(messages, tools=tools)
    if msg.tool_calls:
        for call in msg.tool_calls:
            obs = dispatch(call)              # Action
            messages.append(tool_result(obs)) # Observation
        # 下一轮 LLM 会基于 obs 推理 → 隐式 Thought
    else:
        return msg.content  # 最终答案
    step += 1
```

注意：现代 ReAct 中 **Thought 被隐式地包含在「模型决定下一步调用什么」里**，除非显式要求模型输出 reasoning（如 OpenAI 的 reasoning models / Claude 的 thinking）。

### 3.7 优缺点

**优点：**
- 简单、可解释、易实现；
- 工具调用有据可依，减少幻觉；
- 推理过程可见，便于调试。

**缺点：**
- 依赖模型遵守格式（弱模型容易跑偏）；
- 容易陷入循环（重复同一 Action）；
- 每步都要把全部历史塞进上下文，**token 消耗线性增长**；
- 错误传播：一步 Observation 错，后续全错；
- 无全局规划，短视（贪心选下一步）。

## 4. Plan-and-Execute / Reflexion / Tree of Thoughts

ReAct 是「走一步看一步」（贪心、局部）。以下范式都在不同维度上改进它。

### 4.1 Plan-and-Execute（先规划后执行）

**思想：** 先用一个 Planner 模型一次性生成完整的多步计划，再用 Executor 逐步执行，必要时 Re-plan。

```
User Goal
   │
   ▼
Planner LLM → [Step1, Step2, Step3, Step4]   （全局规划）
   │
   ▼
Executor: 逐步执行 Step1 → 用 ReAct/工具完成
   │
   ▼
Re-planner: 根据执行结果，是否需要修改剩余计划？
   │
   ▼
... 直到所有 Step 完成 → 汇总答案
```

**代表：** LangChain 的 `PlanAndExecute`、LlamaIndex 的 `Planner`、BabyAGI 的任务创建思想。

**优点：** 全局视野，减少短视；多步任务效率高。
**缺点：** 计划可能脱离现实；Re-plan 代价高。

### 4.2 Reflexion（反思 / 自我纠错）

**思想：** Agent 完成一次尝试后，让一个「反思者」对整个过程进行**语言化的反思（verbal reinforcement）**，把教训写进记忆，下一次尝试时利用这些反思改进。

```
Attempt 1 → 失败（结果不对/异常）
   │
   ▼
Evaluator（评估这次尝试的结果）
   │
   ▼
Self-Reflection（生成"我哪里做错了"的文字反思）
   │
   ▼
存入 Episodic Memory（情节记忆）
   │
   ▼
Attempt 2：带着反思重新做 → 改进
   │ ... 重复直到成功或达上限
```

**核心创新：** 不更新权重（不像 RL），而是用「文本形式的经验」作为强化信号，等于给 Agent 加了「错题本」。

**优点：** 自我纠错能力强，适合编程、推理这类「有客观反馈」的任务。
**缺点：** 反思本身也可能幻觉（反思错了反而误导）；多轮尝试成本高。

### 4.3 Tree of Thoughts（ToT，思维树）

**思想：** 把推理过程组织成树，每步生成多个候选 Thought（分支），用评估函数给每个分支打分，进行 **BFS/DFS 搜索**，可回溯。

```
              问题
           /   |   \
        想A   想B   惡C        （多个候选）
        /\    /\    /\
      ...    ...    ...         （继续展开 + 评估剪枝）
        |
      最优路径 → 答案
```

**与 ReAct 区别：** ReAct 是单条链（线性），ToT 是树（可分支可回溯）。ToT 适合解谜、24点、创意写作这类「需要探索多条路径」的问题。

**在 Agent 中应用：** 用于复杂规划阶段（Planner 用 ToT 探索多种方案）；但成本极高（调用次数爆炸），生产中谨慎使用。

### 4.4 范式对比总表

| 范式 | 核心思想 | 决策方式 | 适用场景 | 成本 |
|------|---------|---------|---------|------|
| ReAct | 边想边做 | 贪心，单链 | 通用、可解释 | 中 |
| Plan-and-Execute | 先全局规划再执行 | 全局+局部结合 | 多步复杂任务 | 中 |
| Reflexion | 失败后反思重来 | 试错+改进 | 编程、有反馈任务 | 高（多次） |
| ToT | 树搜索+回溯 | 搜索 | 解谜、强探索 | 极高 |
| CoT | 纯推理链 | 无工具 | 纯推理题 | 低 |

## 5. Multi-Agent 多智能体

### 5.1 为什么需要多 Agent

单 Agent 的瓶颈：
- 单一 Prompt 难以兼顾多种能力（写代码 + 写文档 + 测试）；
- 上下文窗口有限，单 Agent 容易「上下文爆炸」；
- 角色混乱，一个 Agent 既要规划又要执行还要审查，容易顾此失彼。

Multi-Agent 的核心价值：**分工（ specialization）+ 对话（debate）+ 可扩展性**。

### 5.2 协作模式（必背 5 种）

**① 层级式（Hierarchical / Supervisor）**
```
        Supervisor（主管 Agent）
        /        |        \
   Worker A   Worker B   Worker C
```
Supervisor 拆任务、分发、汇总。代表：LangGraph 的 supervisor 模式。

**② 顺序式（Sequential / Pipeline）**
```
Researcher → Writer → Reviewer → Editor
```
流水线，每个 Agent 处理上一步输出。代表：CrewAI 的 process。

**③ 辩论式（Debate）**
```
   Agent A  ⇄  Agent B
   （多轮辩论）→ Judge Agent 裁决
```
通过对抗提升答案质量，适合事实核查、复杂决策。

**④ 角色扮演式（Role-Playing）**
如 MetaGPT 的「产品经理 → 架构师 → 工程师 → QA」软件公司模拟。

**⑤ 群聊式（Group Chat / Swarm）**
多个 Agent 在一个共享频道发言，由一个 Group Chat Manager 决定下一个谁说话。代表：AutoGen 的 GroupChat、OpenAI Swarm。

### 5.3 代表框架思想

**MetaGPT（SOP 驱动）**
- 思想：把**软件工程的标准作业流程（SOP）**编码进 Agent 协作。
- 角色固定：Product Manager、Architect、Project Manager、Engineer、QA。
- 每个 Agent 输入输出有严格的「结构化文档」（PRD、设计文档、API 规范）。
- 强调**可执行文档作为通信媒介**，减少幻觉。

**AutoGen（对话驱动）**
- 思想：Agent 之间通过**多轮对话**协作。
- 核心：`AssistantAgent`（LLM）+ `UserProxyAgent`（可执行代码/调工具的人形代理）。
- GroupChat 支持多 Agent 自由对话。
- 强项：**代码执行回路**（写代码 → 自动跑 → 报错 → 修改）。

**CrewAI（团队 + 角色 + 任务）**
- 三要素：**Crew（团队）+ Agent（成员+角色+目标+工具）+ Task（任务+期望产出）**。
- 支持 sequential 和 hierarchical 流程。
- 强调**易用性**和**角色人设**，上手最快。

### 5.4 Multi-Agent 的工程挑战

- **通信开销**：Agent 间传消息，token 消耗成倍增加；
- **协调复杂**：谁先说话、什么时候停、如何避免死循环；
- **错误放大**：一个 Agent 产出错误，下游全部受影响；
- **成本控制**：N 个 Agent × M 轮对话 = 高昂 API 账单；
- **调试困难**：难以定位是哪个 Agent 哪一步出了问题。

> 实践经验：**不要一上来就多 Agent**。先单 Agent + 工具跑通，遇到瓶颈再拆分。过度 Multi-Agent 是常见的过度工程。

## 6. 记忆机制

### 6.1 三层记忆模型（类比人类）

| 类型 | 类比 | 实现 | 生命周期 |
|------|------|------|---------|
| 短期记忆（Short-term） | 工作记忆 / 当前对话 | LLM 上下文窗口 | 单次会话 |
| 长期记忆（Long-term） | 经验 / 知识 | 向量数据库 | 跨会话持久 |
| 工作记忆（Working） | 草稿纸 / 临时变量 | scratchpad / 状态存储 | 单次任务 |

### 6.2 短期记忆：上下文管理

问题：上下文窗口有限（即使 200K 也会被长任务撑爆）。

策略：
- **滑窗（Sliding Window）**：保留最近 N 轮；
- **摘要压缩（Summarization）**：把旧对话压缩成摘要；
- **选择性保留**：保留工具调用结果摘要，丢弃原始长输出；
- **分页 / 卸载**：把早期内容存外部，需要时再检索回来（如 MemGPT）。

**MemGPT 的核心创新**：把操作系统的「虚拟内存」思想搬过来——LLM 上下文 = RAM，外部存储 = 磁盘，LLM 自己决定何时 page in / page out。

### 6.3 长期记忆：向量库

存储形式：把记忆文本 → embedding → 存向量库（Chroma / Pinecone / Qdrant / Weaviate）。

检索方式：
- **语义检索**：当前 query embedding → top-k 最相似记忆；
- **时间衰减**：近期记忆权重更高；
- **重要性评分**：像 Generative Agents（斯坦福小镇）那样，给每条记忆打重要性分。

典型流程：
```
新事件/对话 → 生成记忆条目 → embedding → 写入向量库
            ↓（检索时）
当前任务 → query embedding → 检索相关记忆 → 注入 Prompt
```

### 6.4 记忆管理（Reflection / 摘要 / 检索）

- **摘要（Summarization）**：定期把琐碎记忆压缩成高层总结；
- **反思（Reflection）**：从多条经验记忆中归纳出规则（Generative Agents 的核心）；
- **检索增强**：只在需要时召回相关记忆，避免上下文爆炸；
- **遗忘机制**：删除低重要性、过时、重复的记忆，控制库大小。

> Generative Agents（斯坦福「AI 小镇」论文）是记忆系统的经典参考：Memory Stream + Importance + Recency + Relevance 三因子检索 + Reflection 生成高层洞察。

## 7. 工具使用

### 7.1 常见工具类型

| 类型 | 例子 | 作用 |
|------|------|------|
| 搜索 | Google/Bing/Tavalli API | 获取最新信息 |
| 代码执行 | Python REPL / 沙箱 | 精确计算、数据处理 |
| 数据库 | SQL 查询 / Text2SQL | 查业务数据 |
| 外部 API | 天气/股价/地图 | 实时数据 |
| 文件操作 | 读写/创建 | 操作本地或工作区 |
| 浏览器 | Playwright/Puppeteer | 访问网页、点击 |
| 检索（RAG） | 向量检索 | 查私有知识库 |
| 子 Agent | 调用另一个 Agent | 任务委派 |

### 7.2 工具描述设计（关键工程点）

工具描述（给 LLM 看的）质量直接决定 Agent 会不会用、用得对。

**好的工具描述要做到：**

1. **函数名语义化**：`get_weather` 而非 `func1`；
2. **description 写清「做什么 + 何时用」**：不只说功能，还要说使用场景；
3. **参数有清晰 description 和约束**：type、enum、required；
4. **避免歧义和重叠**：两个工具职责要清晰，别让模型纠结选哪个；
5. **给出 example**：复杂参数可附示例；
6. **声明副作用**：只读 vs 写入（写入类工具要谨慎授权）。

**反例（差）：**
```json
{"name": "search", "parameters": {"q": "string"}}
```

**正例（好）：**
```json
{
  "name": "search_web",
  "description": "在互联网上搜索实时信息。当你需要查询模型训练数据之后发生的事件、最新价格、新闻、或你不确信的事实时使用。不要用于数学计算或你已确知的事实。",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "搜索关键词，尽量具体，如'2024 苹果财报'"},
      "max_results": {"type": "integer", "default": 5, "minimum": 1, "maximum": 20}
    },
    "required": ["query"]
  }
}
```

### 7.3 工具数量与选择

- 工具太多（>20）：模型选择困难，准确率下降（选择空间爆炸）；
- 工具太少：能力受限；
- 解决：**工具检索（Tool RAG）**——把工具也做向量索引，按任务动态召回相关工具子集（如 Toolformer / Gorilla 思想）。

### 7.4 安全：工具权限与确认

- **只读 vs 写入**：写操作（删文件、发邮件、转账）应有确认机制或权限分级；
- **沙箱**：代码执行必须在隔离沙箱，避免模型执行恶意代码；
- **审计日志**：记录每次工具调用的入参出参，便于追溯；
- **超时与限流**：防止 Agent 死循环调用拖垮外部系统。

## 8. MCP 协议（Model Context Protocol）

### 8.1 是什么

**MCP（Model Context Protocol，模型上下文协议）** 是 Anthropic 于 2024 年 11 月开源的**开放标准协议**，用于规范化「AI 应用（LLM）」与「外部数据源 / 工具」之间的连接。

一句话：**MCP 之于 AI 工具，就像 USB 之于硬件、LSP 之于编辑器——一个统一的插拔标准。**

### 8.2 解决什么问题（M before MCP）

在 MCP 之前，每个 AI 应用接外部工具都是「点对点」定制：
- 接 GitHub 要写 GitHub 适配；
- 接 Slack 要写 Slack 适配；
- 接本地数据库又要一套；
- **M 个应用 × N 个工具 = M×N 套集成**（组合爆炸）。

每个厂商格式还不同（OpenAI tools、Claude tools、Gemini……），工具开发者要重复造轮子。

**MCP 把它变成 M + N：** 应用实现一次 MCP Client，工具实现一次 MCP Server，二者通过标准协议互通。

```
M 个应用 ──┐                ┌── N 个工具（MCP Server）
          ├── MCP 协议 ────┤
M 个应用 ──┘                └── N 个工具
```

### 8.3 架构：Host / Client / Server

MCP 采用 **客户端-服务器架构**，三个核心角色：

| 组件 | 职责 | 例子 |
|------|------|------|
| **Host（宿主）** | 用户交互的 LLM 应用，管理整体 | Claude Desktop、Cursor、IDE |
| **Client（客户端）** | 在 Host 内部，与单个 Server 维护 **1:1** 连接 | Host 内嵌的连接器 |
| **Server（服务端）** | 暴露能力给 Client 的独立进程/服务 | GitHub MCP Server、本地文件 Server |

**关键关系：**
- 1 个 Host 可运行**多个 Client**，每个 Client 连**1 个 Server**；
- Host 是用户的入口（如 Claude Desktop），决定信任哪些 Server；
- Server 是能力提供方（可由任何人开发，类似插件生态）。

```
        ┌─────────────────────────────────┐
        │             Host                │
        │   (Claude Desktop / Cursor)     │
        │                                 │
        │   ┌────────┐  ┌────────┐  ┌────────┐
        │   │Client A│  │Client B│  │Client C│
        │   └───┬────┘  └───┬────┘  └───┬────┘
        └───────┼───────────┼───────────┼─────┘
                │           │           │
            ┌───▼───┐   ┌───▼───┐   ┌───▼───┐
            │Server │   │Server │   │Server │
            │GitHub │   │ Files │   │ Slack │
            └───────┘   └───────┘   └───────┘
```

### 8.4 MCP Server 暴露的三类原语

MCP Server 不是简单的「函数列表」，而是提供三类标准化能力：

1. **Tools（工具）** —— 可被 LLM 调用执行的函数（类似 Function Calling，但有副作用）。
   例：`create_issue`、`send_email`、`run_query`。

2. **Resources（资源）** —— 可被读取的数据/文件（只读，类似 GET）。
   例：`file:///project/main.py`、`db://schema`。

3. **Prompts（提示模板）** —— 预定义的 Prompt 模板，可复用。
   例：`code-review-prompt`、`summarize-prompt`。

这种区分让 Client 知道什么是「只读数据」、什么是「可执行操作」，便于权限控制。

### 8.5 通信机制

- **传输层（Transport）**：支持 **stdio**（本地子进程，标准输入输出）和 **HTTP + SSE / Streamable HTTP**（远程）；
- **消息格式**：基于 **JSON-RPC 2.0**；
- **能力协商**：Client 和 Server 握手时声明各自支持的能力（capability negotiation）；
- **双向**：Server 也可以主动通知 Client（如资源变化）。

### 8.6 一次 MCP 调用流程

```
1. Host 启动，加载配置的 Server 列表
2. 每个 Client 与对应 Server 建立 JSON-RPC 连接，协商能力
3. Host 汇总所有 Server 的 Tools/Resources/Prompts，转成 LLM 可用格式
4. 用户提问 → LLM 决定调用某 Server 的某 Tool
5. Host 通过 Client 向 Server 发 tools/call 请求
6. Server 执行，返回结果
7. 结果回传 LLM，继续推理
```

### 8.7 MCP vs 传统 Function Calling

| 维度 | 传统 Function Calling | MCP |
|------|----------------------|-----|
| 集成方式 | 应用内硬编码每个工具 | 标准协议，Server 即插即用 |
| 工具来源 | 开发者自己实现 | 生态共享（任何 MCP Server） |
| 跨应用复用 | 否（每个应用重写） | 是（一个 Server 多应用用） |
| 标准化 | 厂商各一套 | 开放协议 |
| 能力类型 | 只有 function | Tools + Resources + Prompts |
| 传输 | API 内部 | stdio / HTTP |

**关系：** MCP 不是替代 Function Calling，而是**标准化了「工具的供给端」**。Host 内部仍然可能把 MCP Tools 转成 OpenAI function calling 格式喂给 LLM。MCP 解决的是「工具从哪来、怎么连」，Function Calling 解决的是「LLM 怎么表达调用」。

### 8.8 MCP 的价值与生态

- **生态价值**：一个 MCP Server 写一次，所有支持 MCP 的 Host（Claude Desktop、Cursor、Windsurf、Zed、Continue 等）都能用；
- **本地能力**：通过 stdio Server，LLM 可以安全访问本地文件、本地数据库、本地工具链；
- **安全模型**：Host 控制信任边界，Server 隔离运行，用户可审批每个 Server 的权限。

## 9. 主流框架对比

### 9.1 LangChain Agent / LangGraph

- **定位**：最全的 LLM 应用框架，Agent 是其中一部分。
- **演进**：早期 `AgentExecutor`（基于 ReAct）→ 现在主推 **LangGraph**（基于状态图，可控、可可视化）。
- **特点**：工具丰富、生态最大；但抽象层多，新手易迷路。
- **LangGraph 优势**：把 Agent 建模为「状态机 / 图」，节点是函数，边是条件路由，支持循环、分支、人机回路、持久化检查点（checkpoint）。
- **适合**：需要精细控制 Agent 流程、构建复杂多 Agent 系统的团队。

### 9.2 LlamaIndex Agent

- **定位**：从「数据/RAG 框架」延伸到 Agent。
- **特点**：强项在**数据连接 + 检索**，Agent 与 RAG 深度整合。
- **核心**：`ReActAgent`、`FunctionAgent`、Workflow（事件驱动）。
- **适合**：以数据/知识为核心的 Agent 应用。

### 9.3 AutoGPT

- **定位**：早期「自主 Agent」开源项目（2023 爆款）。
- **特点**：给定一个高层目标，自主拆解、执行、循环。长期记忆用向量库。
- **局限**：可靠性差、容易跑偏、成本高，更像「概念验证」而非生产工具。
- **意义**：第一个让大众看到「自主 Agent」可能性，启发了整个领域。

### 9.4 MetaGPT

- **定位**：**多 Agent 协作**做软件开发（SOP 驱动）。
- **特点**：模拟软件公司角色（PM/架构师/工程师/QA），通过结构化文档协作。
- **强项**：从一句话需求生成完整小项目（代码 + 文档 + 测试）。
- **适合**：自动化软件开发、代码生成场景。

### 9.5 OpenAI Assistants API / Agents SDK

- **Assistants API**（已逐步被新版取代）：托管式 Assistant，内置 Code Interpreter、File Search、Function Calling。
- **Responses API + Agents SDK（2025）**：OpenAI 新一代方案，`Agents SDK` 是开源的轻量编排框架，支持 Handoff（Agent 间转交）、Guardrails、Tracing。
- **特点**：托管、简单、与 OpenAI 模型深度绑定。
- **适合**：纯 OpenAI 生态、想省事的应用。

### 9.6 框架对比总表

| 框架 | 核心范式 | 单/多 Agent | 模型绑定 | 上手难度 | 适合场景 |
|------|---------|------------|---------|---------|---------|
| LangGraph | 状态图 | 都支持 | 任意 | 中高 | 复杂可控 Agent 系统 |
| LlamaIndex | ReAct/Workflow | 主单 | 任意 | 中 | 数据驱动 Agent |
| AutoGPT | 自主循环 | 单 | 任意 | 低（但不可控） | 演示/实验 |
| MetaGPT | SOP 多角色 | 多 | 任意 | 中 | 软件开发自动化 |
| AutoGen | 对话驱动 | 多 | 任意 | 中 | 代码协作、辩论 |
| CrewAI | 角色+任务 | 多 | 任意 | 低 | 快速搭多 Agent 团队 |
| OpenAI Agents SDK | Handoff | 多 | OpenAI | 低 | OpenAI 生态 |

> 选型建议：复杂可控选 LangGraph；数据驱动选 LlamaIndex；快速搭多 Agent 选 CrewAI；纯 OpenAI 选 Agents SDK。

## 10. Agent 的局限

### 10.1 幻觉（Hallucination）

- Agent 不仅会「编造事实」，还会**编造工具调用**（调用不存在的函数、构造非法参数）；
- 长链条中，早期幻觉会被后续步骤「合理化」放大；
- 缓解：工具返回结构化数据、加校验、关键步骤人工确认。

### 10.2 循环（Infinite Loop）

- Agent 重复调用同一工具、在两个状态间来回切换；
- 原因：模型「卡住」、目标不可达却不会停止、记忆管理差；
- 缓解：`max_iterations`、循环检测（检测重复 Action）、强制 Finish。

### 10.3 成本（Cost）

- 多轮调用 + 长上下文 = token 爆炸；
- 一个任务可能调用 LLM 几十次，账单远超单次对话；
- 多 Agent 更是成倍增加；
- 缓解：上下文压缩、用小模型做简单步、缓存、限制步数。

### 10.4 可靠性（Reliability）

- 每一步都有失败概率，n 步串联后成功率指数下降（0.9^10 ≈ 35%）；
- 工具调用失败、API 超时、解析错误都会中断；
- 缓解：重试、降级、异常捕获、断点续跑（checkpoint）。

### 10.5 可控性与安全（Control & Safety）

- Agent 有「行动力」，错误行动会造成真实损失（删数据、发错邮件、错误交易）；
- 「目标误解」：Agent 用意想不到的方式达成目标（reward hacking）；
- 缓解：权限分级、人在回路（Human-in-the-loop）、沙箱、审计、只读优先。

### 10.6 评估困难（Evaluation）

- Agent 行为非确定（同一输入不同执行路径），难以回归测试；
- 评估维度多：最终答案质量、路径效率、工具使用正确性、成本；
- 缓解：轨迹评估（trajectory eval）、端到端任务集、人工评审。

### 10.7 延迟（Latency）

- 多轮串行调用，端到端延迟可达数十秒到分钟级；
- 不适合实时性要求高的场景；
- 缓解：并行化无关步骤、流式输出、缓存。

> 工程结论：**Agent 当前最大的瓶颈不是「能不能做」，而是「能不能稳定地做」。** 生产化 Agent 的工程量远大于 demo。

## 11. 面试考点

### Q1：Agent 和 RAG、Chatbot 的本质区别是什么？

**答：** 核心区别在「自主性」和「闭环行动」。
- Chatbot 只做多轮对话，无工具、无行动、无副作用；
- RAG 在对话基础上加了「检索」这个工具，本质仍是「检索 + 生成」一次性的，不闭环、不自主规划；
- Agent = LLM + 规划 + 记忆 + 工具 + 行动，能自主拆解目标、多步调用工具、根据反馈修正、产生真实副作用，形成闭环。
RAG 常作为 Agent 的一个工具存在，二者是包含而非互斥关系。

### Q2：Function Calling 的原理是什么？模型真的执行了函数吗？

**答：** Function Calling 让模型输出结构化的「函数调用意图」（函数名 + JSON 参数）。
**模型本身不执行函数**，只生成调用请求；真正的执行发生在开发者代码中。流程：声明工具 → 模型返回 tool_calls → 开发者执行 → 结果以 `role=tool` 回传 → 模型基于结果继续。
它通过微调 + 受限解码保证输出合法 JSON，解决了传统 Prompt 抠参数的脆弱性。

### Q3：Function Calling 和 ReAct 有什么区别和关系？

**答：**
- 层级不同：ReAct 是**范式/思想**（Thought-Action-Observation 交替），Function Calling 是**实现技术**（模型原生输出 JSON）。
- ReAct 早期用文本协议（正则解析），现代 ReAct 通常用 Function Calling 来实现 Action 步。
- ReAct 显式输出 Thought（可解释），FC 是黑盒；FC 格式可靠，ReAct 灵活但需模型守纪律。
**一句话：ReAct 是上层范式，Function Calling 是下层技术，现代 Agent = ReAct 思想 + FC 实现 + 框架循环。**

### Q4：详细描述 ReAct 的执行流程。

**答：** ReAct = Reasoning + Acting。每一步：Thought（推理下一步）→ Action（选工具并调用）→ Observation（读取工具结果）→ 回到 Thought，直到调用 Finish。
经典实现靠 stop sequence：模型生成到 `Action:` 就暂停，框架执行 Action 把结果填入 `Observation:`，模型继续生成下一个 Thought。循环直到 Finish 或达 max_iterations。
优点：可解释、减少幻觉、易调试。缺点：贪心短视、易循环、token 线性增长、依赖模型守格式。

### Q5：Plan-and-Execute 相比 ReAct 解决了什么问题？

**答：** ReAct 是「走一步看一步」（贪心、局部、短视），长任务易跑偏。Plan-and-Execute 先用 Planner 全局规划完整步骤，再逐步执行，必要时 Re-plan。
优点：全局视野、长任务更稳；缺点：计划可能脱离现实、Re-plan 代价高。本质是把「规划」和「执行」解耦，规划层看全局，执行层用 ReAct 落地。

### Q6：Reflexion 是怎么工作的？它和强化学习有什么异同？

**答：** Reflexion 让 Agent 在失败后生成**语言化的反思**（verbal reflection），把教训存入情节记忆，下次尝试时参考改进，循环到成功。
和 RL 的相同点：都是「试错 + 经验改进」的闭环。
不同点：**Reflexion 不更新模型权重**，而是用「文本形式的经验」作为上下文强化信号，等于给 Agent 一本「错题本」，无需训练、即插即用。代价是多轮重试成本高、反思本身也可能幻觉。

### Q7：Multi-Agent 有哪些协作模式？什么时候该用多 Agent？

**答：** 五种典型模式：
1. 层级式（Supervisor 分发汇总）；
2. 顺序式（流水线 Researcher→Writer→Reviewer）；
3. 辩论式（多 Agent 对抗 + Judge）；
4. 角色扮演式（MetaGPT 软件公司）；
5. 群聊式（GroupChat Manager 调度）。
**何时用多 Agent：** 单 Agent 上下文不够、需要明确分工、需要对抗审查时。
**何时不用：** 简单任务别过度工程，先单 Agent + 工具跑通。多 Agent 通信成本高、调试难、错误会放大。

### Q8：MCP 是什么？它解决了什么问题？架构是怎样的？（必考）

**答：** MCP（Model Context Protocol）是 Anthropic 2024 年开源的**开放标准协议**，规范化「AI 应用」与「外部数据源/工具」的连接。
**解决的问题：** 在 MCP 之前，M 个应用 × N 个工具要 M×N 套定制集成（组合爆炸），且厂商格式各异。MCP 把它降为 M + N：应用实现一次 MCP Client，工具实现一次 MCP Server，标准化互通。
**架构（Host / Client / Server）：**
- Host：用户交互的 LLM 应用（Claude Desktop、Cursor）；
- Client：Host 内部，与单个 Server 维持 1:1 连接；
- Server：暴露 Tools / Resources / Prompts 三类原语的独立进程。
1 个 Host 可跑多个 Client，每个连 1 个 Server。通信基于 JSON-RPC 2.0，传输支持 stdio（本地）和 HTTP/SSE（远程）。

### Q9：MCP 和 Function Calling 是什么关系？是替代吗？

**答：** 不是替代，是不同层面。
- Function Calling 解决「LLM 怎么表达工具调用意图」（模型 → JSON）；
- MCP 解决「工具从哪来、怎么标准化连接」（应用 ↔ 工具供给端）。
Host 内部仍可能把 MCP 的 Tools 转成 function calling 格式喂给 LLM。MCP 标准化的是工具供给生态（一个 Server 多应用复用），FC 是模型层调用机制。二者协同。

### Q10：Agent 的记忆机制有哪些？如何管理长任务的上下文？

**答：** 三层：
- 短期记忆：当前上下文窗口（对话历史、scratchpad）；
- 长期记忆：向量库，跨会话持久（用户偏好、历史事实）；
- 工作记忆：当前任务的中间状态变量。
长任务上下文管理策略：滑窗、摘要压缩、选择性保留、MemGPT 式分页卸载（外部存储 + 按需 page in）。长期记忆用向量检索按需召回，避免上下文爆炸。

### Q11：Agent 在生产环境有哪些主要局限？如何缓解？

**答：**
1. **幻觉**：会编造事实和非法工具调用 → 结构化校验、关键步人工确认；
2. **循环**：重复调用同一工具 → max_iterations、循环检测、强制 Finish；
3. **成本**：多轮 + 长上下文 token 爆炸 → 上下文压缩、小模型做简单步、缓存；
4. **可靠性**：n 步串联成功率指数下降 → 重试、降级、checkpoint 续跑；
5. **安全**：有行动力可造成真实损失 → 权限分级、人在回路、沙箱、只读优先；
6. **延迟**：串行多轮延迟高 → 并行无关步骤、流式输出；
7. **评估难**：行为非确定 → 轨迹评估 + 端到端任务集 + 人工评审。
核心结论：当前最大瓶颈是「稳定可靠」而非「能不能做」。

### Q12：设计一个工具描述时要注意什么？给个反例和正例。

**答：** 注意：函数名语义化、description 写清「做什么+何时用」、参数有 type/约束/required、避免工具职责重叠、声明副作用（只读/写入）。
反例：`{"name":"search","parameters":{"q":"string"}}`（无说明、无约束、名字模糊）。
正例：`search_web`，description 说明「查询训练数据之后的实时信息、新闻、不确定事实，不用于计算」，参数 query 带「尽量具体」的提示，max_results 限 1-20。
工具太多（>20）会降低选择准确率，可用「工具检索」动态召回子集。

## 12. 易错 / 陷阱

### 12.1 概念混淆

- ❌ 「Agent 就是带工具的 Chatbot」→ 漏了「自主规划 + 闭环行动」，RAG 也有工具但不闭环。
- ❌ 「Function Calling = ReAct」→ FC 是技术，ReAct 是范式，层级不同。
- ❌ 「MCP 替代了 Function Calling」→ MCP 标准化工具供给端，FC 是模型调用机制，二者协同。
- ❌ 「多 Agent 一定比单 Agent 强」→ 通信成本、错误放大、调试难，过度工程常见。

### 12.2 ReAct 实现陷阱

- **stop sequence 漏配**：没设 `Observation:` 为 stop，模型会自己「脑补」Observation（幻觉来源）。
- **解析脆弱**：正则解析 Action 易因模型输出微小变动而失败 → 优先用 Function Calling 而非文本协议。
- **未限步**：缺 `max_iterations`，模型可能死循环烧钱。
- **上下文不截断**：所有历史全塞，长任务必爆窗口 → 压缩/滑窗。

### 12.3 Function Calling 陷阱

- **相信模型一定给合法参数**：偶尔会缺 required 字段或类型错 → 代码层必须校验 + 容错。
- **工具描述太模糊**：模型不知道何时用 → 描述写清场景。
- **工具职责重叠**：两个工具都能干，模型纠结或乱选 → 职责清晰拆分。
- **副作用工具未授权**：模型擅自删数据/发邮件 → 写操作必须权限分级 + 确认。

### 12.4 Multi-Agent 陷阱

- **角色边界模糊**：Agent 之间职责重叠 → 越扯越乱；
- **无终止条件**：Agent 互相踢皮球无限循环 → 必须有 Supervisor 或步数上限；
- **消息爆炸**：每个 Agent 看到所有对话 → token 飞涨 → 控制消息可见性；
- **错误级联**：一个 Agent 错，下游全错 → 关键节点加审查/人工把关。

### 12.5 记忆陷阱

- **全量塞入**：把所有历史记忆塞 prompt → 上下文爆炸；
- **召回不准**：embedding 召回相关度低 → 误信息污染；
- **从不遗忘**：记忆库无限膨胀 → 需要重要性评分 + 遗忘机制；
- **隐私泄露**：长期记忆存了敏感信息跨会话调用 → 需脱敏和权限控制。

### 12.6 MCP 陷阱

- **权限过宽**：给 MCP Server 过大权限（如全盘文件访问）→ 风险高 → 最小权限；
- **信任任意 Server**：第三方 Server 可能恶意 → 用户审批 + 审计；
- **混淆 Tools 和 Resources**：把只读数据当 Tool 暴露 → 模型可能误执行；
- **以为 MCP = 更聪明**：MCP 只是连接标准，能力上限仍是底层 LLM。

### 12.7 通用工程陷阱

- **demo 跑通就上线**：Agent 非确定性强，10 次成功不代表可靠 → 大量轨迹测试；
- **只看最终答案**：路径可能低效/危险 → 评估轨迹；
- **忽略成本**：一个任务几十次调用 → 上线前估算 token 成本；
- **无降级方案**：LLM/API 挂了整个 Agent 瘫 → 重试 + 降级 + 兜底。

## 附录：核心范式速查

| 范式 | 一句话 | 关键词 |
|------|--------|--------|
| CoT | 让模型一步步想 | 链式推理，无工具 |
| ReAct | 边想边做边观察 | Thought-Action-Observation |
| Function Calling | 模型原生吐 JSON 调用 | 结构化、可靠 |
| Plan-and-Execute | 先规划全局再执行 | Planner + Executor |
| Reflexion | 失败反思重来 | 语言化经验，错题本 |
| ToT | 树搜索多分支 | BFS/DFS，可回溯 |
| Multi-Agent | 多角色协作 | 分工/辩论/层级 |
| MCP | 工具连接标准协议 | Host/Client/Server |

> 学习路径建议：先吃透 ReAct + Function Calling（地基）→ 理解记忆与规划（进阶）→ 掌握 Multi-Agent 与 MCP（生态）→ 用 LangGraph / CrewAI 落地一个真实 Agent（实践）。
