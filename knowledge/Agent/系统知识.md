## 一、Agent 的定义与本质

### 1.1 什么是 Agent

学术界对 Agent 的标准定义可追溯到 ReAct 论文中的形式化描述：在时间步 $t$，Agent 从环境接收观察 $o_t$，并按某种策略 $\pi$ 采取动作 $a_t$，其中上下文作为 Agent 的输入。ReAct 的核心思想是把动作空间扩展为 $A_t = \mathcal{A} \cup \mathcal{L}$，其中 $\mathcal{L}$ 是语言空间——即"思考（thought）"本身也被当作一种动作 [S1]。

工业界（LangChain 官方文档）对 Agent 的描述更工程化：Agent 由一个"模型循环（model loop）"驱动，循环外是 prompt、工具（tools）以及塑造行为的中间件（middleware）。LangChain 提供 `create_agent` 作为一个"最小化、高度可配置的框架（harness）" [S6]。

### 1.2 经典公式：Agent = LLM + 记忆 + 规划 + 工具

这是 Lilian Weng 提出的广为流传的拆解（虽然其博客本文未直接抓取，但被 ReAct/Reflexion 论文机制所印证）。各组件在抓取论文中的对应：

- **LLM（大脑）**：ReAct 用冻结的 PaLM-540B / GPT-3 作为"冻结大模型"被 few-shot 提示 [S1]；Reflexion 用 LLM 同时承担 Actor、Evaluator、Self-Reflection 三个角色 [S3]。
- **记忆（Memory）**：Reflexion 明确引入"情景记忆缓冲区（episodic memory buffer）"，把语言化反思存入长期记忆 [S3]；Reflexion 的 Actor 也带有一个 `mem` 组件提供额外上下文 [S3]。
- **规划（Planning）**：ReAct 的 thought 用于"分解任务目标、制定动作计划、跟踪进度、处理异常并调整计划" [S1]。
- **工具（Tools）**：Toolformer 证明 LLM 可自主学会调用计算器、QA、搜索引擎、翻译、日历等外部 API [S2]；ReAct 设计了 `Search/Lookup/Finish` 三种 Wikipedia API 动作 [S1]。

### 1.3 ReAct 论文对 Agent 协同的论述（关键原文）

ReAct 摘要原文指出 [S1]：
> "reasoning traces help the model induce, track, and update action plans as well as handle exceptions, while actions allow it to interface with and gather additional information from external sources such as knowledge bases or environments."

即：推理轨迹帮助模型归纳/跟踪/更新计划并处理异常；动作则让它与外部环境交互获取额外信息。二者交替（interleaved）产生"协同（synergy）"。

ReAct 在引言用"做饭"类比说明这种协同：在两个具体动作之间，人类会用语言推理来跟踪进度（"既然切完了，我该烧水"）、处理异常（"没有盐，就用酱油胡椒代替"）、意识到需要外部信息（"面团怎么做？搜一下"）[S1]。

## 二、ReAct：推理 + 行动的协同范式

### 2.1 设计动机

ReAct 指出 Chain-of-Thought（CoT）推理存在两大缺陷 [S1]：
1. CoT 是"静态黑盒"——模型仅用自身内部表示生成思考，不与外部世界绑定，无法"反应式推理"或更新知识；
2. 因此会导致**事实幻觉**和**误差传播**。

而纯动作（Act-only）方案缺乏推理，无法对复杂轨迹上下文做高层推理。ReAct 的解法是把两者交错。

### 2.2 ReAct 完整 Prompt 模板（来自论文附录 C，原文抓取）

ReAct 的 few-shot 示例中，每个轨迹由 `Thought-Action-Observation` 步骤组成。以下为论文附录中 HotpotQA/FEVER 的真实模板 [S1]：

```
Question/Claim: <输入问题或待验证声明>

Thought 1: <分解问题 / 决定搜索什么 / 注入常识>
Action 1: Search[<实体>]
Observation 1: <搜索引擎返回的前几句>

Thought 2: <从观察中提取关键信息 / 判断是否需继续搜索 / 推理>
Action 2: Lookup[<字符串>]   # 或 Search[新实体]
Observation 2: <结果>

...

Thought N: <综合以上信息得出结论>
Action N: Finish[<最终答案>]
```

论文给出的真实示例（FEVER 事实验证，抓取原文）[S1]：

```
Claim: Bermuda Triangle is a loosely-defined region in the Pacific Ocean.
Thought 1: I need to search Bermuda Triangle and find if it is in the Pacific Ocean.
Action 1: Search[Bermuda Triangle]
Observation 1: Reportedly haunted locations...
Thought 2: The observation says that it is in the western part of the North Atlantic Ocean,
           so it is not in the Pacific Ocean.
Action 2: Finish[REFUTES]
```

另一个展示"多轮反思 + 重新搜索"的例子 [S1]：

```
Claim: Soyuz was part of the American space program.
Thought 1: I need to search Soyuz and find if it was part of the American space program.
Action 1: Search[Soyuz]
Observation 1: Soyuz is a transliteration...
Thought 2: It does not mention the American space program, so I need to search "American
           space program" to find if it was part of it.
Action 2: Search[American space program]
Observation 2: The space policy...
Thought 3: It does not mention Soyuz, so I am not sure if this claim is true or not.
Action 3: Finish[NOT ENOUGH INFO]
```

### 2.3 ReAct 的动作空间设计

ReAct 为知识密集任务设计了 3 种简单 Wikipedia API 动作 [S1]：
1. `Search[entity]`：返回实体 wiki 页前 5 句，若不存在则返回最相似的 5 个实体；
2. `Lookup[string]`：模拟浏览器 Ctrl+F，返回页面中下一个含该字符串的句子；
3. `Finish[answer]`：结束任务并给出答案。

论文特意强调这个动作空间"比最先进的词法/神经检索器弱得多"，目的是模拟人类与 Wikipedia 的交互，强迫模型用语言显式推理来检索 [S1]。

### 2.4 ReAct 的执行流程（Reason to Act / Act to Reason）

```
┌─────────────────────────────────────────────────┐
│  Question  →  Thought  →  Action  →  Observation │
│       ↑                              │           │
│       └──────────────────────────────┘           │
│              (循环直到 Finish)                    │
└─────────────────────────────────────────────────┘
```

论文原文定义两种模式 [S1]：
- **推理为主的任务**（QA、事实验证）：交替生成 thought 和 action，轨迹由多个 thought-action-observation 步骤组成（dense thought）；
- **决策为主的任务**（ALFWorld、WebShop，动作多）：thought 只需稀疏出现在关键位置，让 LLM 自己决定 thought 和 action 的异步出现。

### 2.5 ReAct 的"思考"功能分类（论文枚举）

论文明确列出 useful thought 的多种类型 [S1]：
- 分解任务目标并制定动作计划；
- 注入与任务相关的常识知识；
- 从观察中提取重要部分；
- 跟踪进度并切换动作计划；
- 处理异常并调整动作计划；
- 引导搜索重写（"也许我该搜 X 而不是 Y"）；
- 综合最终答案。

### 2.6 ReAct 实验结论（抓取原文）

- HotpotQA / Fever：通过 Wikipedia API 交互，ReAct 克服了 CoT 的幻觉和误差传播 [S1]；
- ALFWorld / WebShop：1-2 shot 的 ReAct 分别比模仿学习/RL 方法在成功率上**绝对提升 34% 和 10%** [S1]；
- 最佳策略是 ReAct + CoT-SC 的混合：ReAct 失败时回退到 CoT-SC；CoT-SC 多数答案出现次数过低时回退到 ReAct [S1]；
- 步数上限：HotpotQA 设 7 步、Fever 设 5 步（更多步不提升性能）[S1]。

### 2.7 ReAct 的四大特性（论文原文 A-D）

[S1] 明确总结 ReAct 的四大特性：
- A) **直观易设计**：设计 prompt 就是让人在动作之上顺手写下思考，无需 ad-hoc 格式选择；
- B) **通用灵活**：灵活的思考空间和异步思考-动作格式适配多种任务；
- C) **高性能且稳健**：仅 1-6 个上下文示例即强于仅推理或仅动作的基线；
- D) **人类可控对齐**：人类可通过"编辑思考"实时纠正 Agent 行为（论文图 5 的 human-in-the-loop 实验）。

### 2.8 ReAct 的基线消融（论文 Setup，抓取原文）

ReAct 系统地消融自己的轨迹来构造基线 prompt [S1]：
- **Standard（标准提示）**：移除轨迹中所有 thought、action、observation；
- **CoT（思维链）**：移除 action 和 observation，只保留 reasoning-only 基线；还构造了 **CoT-SC（self-consistency）**：解码温度 0.7 采样 21 条 CoT 轨迹取多数票；
- **Act（仅动作）**：移除 thought，近似 WebGPT 与互联网交互的方式（但任务和动作空间不同，且用模仿/强化学习而非提示）。

### 2.9 ReAct + CoT-SC 混合回退（论文原文启发式）

论文给出了两条回退启发式 [S1]：
- **ReAct → CoT-SC**：当 ReAct 在给定步数内无法返回答案，回退到 CoT-SC（HotpotQA 7 步、Fever 5 步——更多步不提升性能，且正确答案中超过此步数的轨迹仅占 0.84% / 1.33%）；
- **CoT-SC → ReAct**：当 CoT-SC 的多数答案出现次数低于某阈值（即内部知识不够自信），回退到 ReAct。

> 设计哲学：ReAct 更"事实、接地（grounded）"，CoT 更擅长"推理结构"但易幻觉；二者互补。

### 2.10 ReAct 的 Human-in-the-loop 行为修正（附录 A.3）

论文附录 A.3 给出一个 ALFWorld 的人机协作实验 [S3→实为 S1]：ReAct 轨迹因一个"幻觉思考"（Act 17）而失败；人类仅编辑两个思考（删除 Act 17 的幻觉句、在 Act 23 加提示），ReAct 轨迹就改变行为并成功。论文指出这种"思考编辑"对 Act 和传统 RL 方法很难实现——人无法改模型参数，改几个动作也无法改变其余行为。ReAct 的思考空间允许人改写模型的"内部信念、推理风格"。

### 2.11 ReAct 跨模型泛化（附录 A.1）

论文在 PaLM-540B 之外补做了 GPT-3（text-davinci-002，贪心解码）实验 [S1]：
- HotpotQA（exact match）：PaLM-540B 29.4 vs GPT-3 30.8；
- ALFWorld（成功率%）：PaLM-540B 70.9 vs GPT-3 78.4。
GPT-3 一致优于 PaLM-540B，可能因其经过人类指令微调，证明 ReAct 跨模型有效。

## 三、Function Calling（函数调用）

### 3.1 定义与机制

Function Calling 是 LLM 厂商提供的原生能力：开发者用 JSON Schema 描述可调用的函数，模型在生成时输出结构化的工具调用（含函数名和参数），由宿主程序执行后再把结果回传给模型 [S8]。

每个工具的典型定义格式（OpenAI 风格）[S8]：
```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "Get weather for a given city",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {"type": "string"}
      },
      "required": ["city"]
    }
  }
}
```

LangChain 官方示例（抓取原文）也是这一模式 [S6]：
```python
def get_weather(city: str) -> str:
    """Get weather for a given city."""
    return f"It's always sunny in {city}!"

agent = create_agent(
    model="openai:gpt-5.5",
    tools=[get_weather],
    system_prompt="You are a helpful assistant",
)
```

### 3.2 Function Calling vs Toolformer vs ReAct 三条路线

这是 Agent 工具使用演化的三条路径，必须能区分（面试高频）：

| 路线 | 代表 | 学习方式 | 决策方 |
|------|------|----------|--------|
| **Function Calling** | OpenAI/Anthropic 原生 | 厂商在训练时内置 | 模型在推理时输出结构化 tool_call |
| **Toolformer** [S2] | Meta，自监督微调 | 模型自我标注数据集 + 过滤 + 微调 | 微调后的模型自主决定 |
| **ReAct** [S1] | Princeton/Google，纯 prompt | few-shot 提示，不更新权重 | 冻结模型 + 模板 |

### 3.3 关键工程细节（WebSearch 验证）[S8]

- **并行工具调用（parallel tool calls）**：多数模型默认开启，模型可在单轮内调用多个函数；用 `parallel_tool_calls: false` 可禁用。注意：使用内置工具（如 code_interpreter、file_search）时不能并行。
- **Structured Outputs + Function Calling**：在 function schema 中设 `strict: true`，模型生成的参数**保证**严格匹配给定 JSON Schema。
- **JSON mode vs Structured Outputs vs Function Calling** 是三个不同层次，需区分 [S8]。

## 四、Toolformer：自监督学会用工具

### 4.1 核心问题与目标

Toolformer 摘要 [S2]：
> "Language models ... struggle with basic functionality, such as arithmetic or factual lookup, where much simpler and smaller models excel."

LM 的固有局限：无法访问最新信息、易幻觉事实、低资源语言能力弱、缺乏精确数学能力、不感知时间推移 [S2]。Toolformer 的目标是让 LM 自学使用外部工具来"两全其美"。

### 4.2 三步自监督流程（论文 Figure 2，抓取原文）

Toolformer 的核心是一个**不依赖大量人工标注**的自监督 pipeline [S2]：

1. **采样 API 调用（Sampling API Calls）**：对每个 API 写一个 prompt，鼓励 LM 在示例上标注潜在 API 调用。对每个位置 $i$ 计算开始 API 调用的概率，按阈值 $k$ 保留候选位置，每个位置最多采样 $m$ 个 API 调用候选。

2. **执行 API 调用（Executing API Calls）**：执行所有候选调用得到结果（可涉及调用另一个神经网络、执行 Python 脚本、或用检索系统搜索）。

3. **过滤 API 调用（Filtering API Calls）**：用**加权交叉熵损失**作为过滤标准。关键判据（原文）[S2]：
   > "an API call is helpful ... if providing it with both the input and the output of this call makes it easier for the model to predict future tokens, compared to not receiving the API call at all, or receiving only its input."
   
   即：把 API 调用及其结果都作为前缀时，比"完全不调用"或"只调用不给结果"更能降低后续 token 的损失（差值至少为阈值 $\beta$），才保留该调用。

4. **模型微调（Model Finetuning）**：把过滤后的 API 调用与原文本交错，构成增强数据集 $\hat{D}$，用它微调原 LM。关键：$\hat{D}$ 除了插入的 API 调用外，文本与原数据 $D$ 完全一致，因此不损失通用性 [S2]。

### 4.3 API 调用的文本表示

论文用特殊 token 表示 API 调用，线性化为 [S2]：
```
e(<API>) = 名称(参数)
e(<API>, r) = 名称(参数) -> 结果
```
实践中用 `[`、`]`、`->` 这些已存在于词表的 token 序列来表示 `<API>`、`</API>`、`<|API|>`，避免修改 LM 词表 [S2]。

### 4.4 Toolformer 的五类工具（抓取原文 Table 1）

| API | 示例输入 | 示例输出 |
|-----|---------|---------|
| Question Answering | Where was the Knights of Columbus founded? | New Haven, Connecticut |
| Wikipedia Search | Fishing Reel Types | (BM25 检索片段) |
| Calculator | 27 + 4 * 2 | 35 |
| Calendar | (今天日期) | Today is Monday, January 30, 2023. |
| Machine Translation | sûreté nucléaire | nuclear safety |

- QA 工具：基于 Atlas（检索增强 LM，在 Natural Questions 上微调）[S2]；
- 计算器：仅支持四则运算，结果保留两位小数 [S2]；
- 搜索：BM25 检索器，索引 KILT 的 Wikipedia dump [S2]；
- 翻译：基于 600M 参数的 NLLB [S2]。

### 4.5 推理流程

推理时，正常解码直到模型生成 `->` token，表示它期望 API 响应；此时中断解码、调用相应 API 获得响应、插入响应和 `</API>` 后继续解码 [S2]。

### 4.6 结果

基于 6.7B 参数的 GPT-J 微调后，Toolformer 在零样本下游任务上远超更大的 GPT-3 及多个基线，且不损失核心语言建模能力 [S2]。

### 4.7 Toolformer 的两大设计目标（论文 Desiderata，抓取原文）

[S2] 明确列出 Toolformer 必须满足的两条：
1. **工具使用应以自监督方式学习，无需大量人工标注**——这不仅因标注成本，更因"人类觉得有用的"可能与"模型觉得有用的"不同；
2. **LM 不应损失任何通用性，应能自主决定何时、如何、用哪个工具**——区别于现有"仅限特定任务"的方案。

### 4.8 Toolformer 的 API 调用损失函数细节

论文定义加权交叉熵损失 [S2]：对位置 $i$ 的 API 调用 $c_i$ 及其响应 $r_i$，比较两种损失实例化：
- $L_{\text{with}}$：把 API 调用+结果都给模型作前缀时，后续 token 的加权损失；
- $L_{\text{without}} = \min(L_{\text{none}}, L_{\text{input-only}})$：即"完全不调用"与"调用但不给结果"两者损失的较小值。

过滤条件：$L_{\text{without}} - L_{\text{with}} \geq \beta$（即加入调用+结果使损失至少降低 β）。论文特别说明：因模型尚未在含 API 调用的样本上微调，把调用插在 $x$ 中间会打断行文、与预训练语料模式不符从而损害困惑度，所以过滤阶段把调用作为**前缀**而非中插 [S2]。

## 五、Reflexion：语言化反思强化

### 5.1 核心思想

Reflexion 提出一种"非权重更新的强化"——通过**语言反馈**而非梯度下降让 Agent 进步 [S3]。摘要原文：
> "Reflexion agents verbally reflect on task feedback signals, then maintain their own reflective text in an episodic memory buffer to induce better decision-making in subsequent trials."

即：Agent 把任务反馈信号语言化反思，存入情景记忆缓冲区，以改进后续试验的决策。

### 5.2 三个模块（论文 Figure 2 / Algorithm 1，抓取原文）

Reflexion 是一个三模型的模块化框架 [S3]：

1. **Actor（执行者）$L_a$**：基于 LLM，被提示生成文本和动作。可使用 CoT 或 ReAct 作为底层生成模型。带一个记忆组件 `mem` 提供额外上下文。
2. **Evaluator（评估者）$L_e$**：对 Actor 的输出打分。
   - 推理任务：基于精确匹配（EM）的奖励；
   - 决策任务：预定义启发式函数；
   - 也实验用另一个 LLM 实例作为 Evaluator（决策/编程任务）。
3. **Self-Reflection（自我反思模型）$L_{sr}$**：给定稀疏奖励信号（如成功/失败的二元状态）、当前轨迹、持久记忆 `mem`，生成"细致且具体"的反馈，存入记忆。

### 5.3 反思强化算法（Algorithm 1，抓取原文）

```
初始化 Actor, Evaluator, Self-Reflection
初始化策略, 情景记忆 mem
用 Actor 生成初始轨迹
用 Evaluator 评估
用 Self-Reflection 生成初始反思
while 未通过 且 未达最大试验次数:
    用 Actor（带 mem）生成新轨迹
    用 Evaluator 评估
    用 Self-Reflection 生成新反思
    把反思追加到 mem
    试验次数 +1
return 最终结果
```

### 5.4 反思示例（论文原文）

论文给出决策任务的反思示意 [S3]：当 Agent 收到失败信号，它可以推断某个具体动作 $a_t$ 导致了后续错误动作 $a_{t+1}, a_{t+2}$，然后"口述"它本应采取另一个动作 $a_t'$，并把这段经验存入记忆。后续试验中，Agent 在时刻 $t$ 利用过去经验选择 $a_t'$。

### 5.5 Reflexion vs 传统 RL 的四大优势（论文原文）

[S3] 明确列出对比传统 policy/value-based RL 的优势：
1. 轻量，不需要微调 LLM；
2. 允许更细致的反馈形式（如针对特定动作的修改），比标量/向量奖励更易做信用分配（credit assignment）；
3. 提供更显式、可解释的情景记忆；
4. 为后续试验的动作提供更明确的提示。

劣势 [S3]：依赖 LLM 自我评估能力（或启发式），且没有成功的形式化保证。

### 5.6 实验结果（抓取原文）

- HumanEval 编程基准：Reflexion 达到 **91% pass@1**，超过当时 SOTA 的 GPT-4（80%）[S3]；
- ALFWorld 决策：12 轮迭代内绝对提升 22% [S3]；
- HotPotQA 推理：提升 20% [S3]；
- HumanEval 编程：提升达 11% [S3]；
- 还引入 LeetcodeHardGym（40 道 LeetCode hard 题，19 种语言）[S3]。

### 5.7 Reflexion 与相关工作的差异表（论文原文对比）

论文用表格明确 Reflexion 相对相关工作的独特点 [S3]：

| 方法 | 自我 | 隐藏 | 决策 | 二元 | 记忆 |
|------|------|------|------|------|------|
| Self-Refine | ✓ | ✗ | ✗ | ✗ | ✗ |
| Beam search | ✓ | ✓ | ✓ | ✓ | ✗ |
| **Reflexion** | ✓ | ✓ | ✓ | ✓ | ✓ |

编程类对比 [S3]：

| 方法 | 测试执行 | 调试 | 自生成测试 | 多语言 | 自反思 |
|------|---------|------|-----------|--------|--------|
| AlphaCode | ✓ | ✗ | ✗ | ✓ | ✗ |
| CodeT | ✓ | ✗ | ✓ | ✗ | ✗ |
| Self-debugging | ✓ | ✓ | ✗ | ✗ | ✗ |
| CodeRL | ✓ | ✓ | ✗ | ✗ | ✗ |
| **Reflexion** | ✓ | ✓ | ✓ | ✓ | ✓ |

关键差异：AlphaCode/Self-debugging/CodeRL 虽能修简单 bug，但依赖 ground truth 测试用例（使 pass@1 失效），且不用自反思弥合"错误识别"与"实现改进"的鸿沟；CodeT 不访问隐藏测试但无自学习步骤。Reflexion 同时具备自反思 + 持久记忆 [S3]。

### 5.8 Reflexion 三种反馈信号来源

论文探索三种生成反思反馈的方式 [S3]：
1. **简单二元环境反馈**（成功/失败）；
2. **预定义启发式**（针对常见失败情形）；
3. **自我评估**——决策任务用 LLM 做二元分类，编程任务用自写单元测试。

所有实现中，评估信号被"放大"为可存入长期记忆的自然语言经验总结 [S3]。

## 六、记忆机制

### 6.1 短期 vs 长期记忆

- **短期记忆（工作记忆）**：即上下文窗口内的对话/轨迹。ReAct 的 thought-action-observation 序列就是工作记忆 [S1]。
- **长期记忆（情景记忆）**：Reflexion 的 `mem` 缓冲区是典型的长期记忆，把跨试验的反思文本持久化存储 [S3]。

### 6.2 Reflexion 的"语义梯度"记忆

Reflexion 把反思文本当作"语义梯度信号（semantic gradient）"，为 Agent 提供具体的改进方向 [S3]。这区别于传统 RL 的标量奖励——语言化反馈信息密度更高、可解释性更强。

### 6.3 记忆的取与舍

Reflexion 论文承认 LLM 自我评估能力是瓶颈：若反思本身错误（如归因错误），错误的经验会被写入记忆并在后续试验中误导 Agent。这是反思范式的根本风险 [S3]。

## 七、Multi-Agent（多智能体）

### 7.1 为何需要多 Agent

单 Agent 在长任务中易累积误差、上下文爆炸、角色混淆。多 Agent 通过角色分工、消息传递、协作/竞争提升鲁棒性。

### 7.2 主流框架架构对比（WebSearch 验证）[S8]

| 框架 | 核心架构范式 | 强项 |
|------|------------|------|
| **LangGraph** | 状态图（nodes + edges），最底层、最灵活 | 确定性 + agentic 工作流混合，最可控 |
| **AutoGen** | Agent 间**对话式工作流** | 自主代码生成、自我纠错 |
| **CrewAI** | 基于角色的"团队（crew）"，高层抽象 | 团队协作、易上手 |
| **OpenAI Swarm** | 轻量 Agent 间的**交接（handoff）** | 极简、模块化 |

LangChain 官方文档也给出了三档建议 [S6]：
- **Deep Agents**：batteries-included（自动上下文压缩、虚拟文件系统、子 Agent 派生），构建在 LangChain Agent 之上；
- **LangChain（`create_agent`）**：高度可定制的 harness；
- **LangGraph**：低层编排框架，用于确定性 + agentic 混合的高级需求；
- **LangSmith**：对以上任何框架构建的 Agent 做 trace、调试、评估。

### 7.3 多 Agent 协作模式

- **接力（handoff）**：Swarm 模式，一个 Agent 把控制权交给另一个 [S8]；
- **对话（conversation）**：AutoGen 模式，Agent 间多轮消息 [S8]；
- **图编排（graph）**：LangGraph 模式，显式状态转移 [S6][S8]；
- **角色分工（crew）**：CrewAI 模式，定义角色、任务、流程 [S8]。

## 九、MCP（Model Context Protocol）

### 9.1 MCP 是什么（官方原文）

MCP 官方介绍 [S4]：
> "MCP (Model Context Protocol) is an open-source standard for connecting AI applications to external systems."

比喻：**"MCP 就像 AI 应用的 USB-C 接口"**——USB-C 提供连接电子设备的标准化方式，MCP 提供连接 AI 应用与外部系统的标准化方式 [S4]。

用 MCP，AI 应用（如 Claude、ChatGPT）可连接数据源（本地文件、数据库）、工具（搜索引擎、计算器）和工作流（专用 prompt）[S4]。

### 9.2 为什么 MCP 重要（官方原文）

按生态角色划分 [S4]：
- **开发者**：减少构建/集成 AI 应用或 Agent 的开发时间和复杂度；
- **AI 应用/Agent**：获得数据源、工具、应用生态的访问能力；
- **终端用户**：得到更强大、可访问个人数据并代为行动的 AI 应用。

官方强调"build once, integrate everywhere"——Claude、ChatGPT、VS Code、Cursor、MCPJam 等都支持 MCP [S4]。

### 9.3 MCP 架构：Host / Client / Server（官方规范原文）

MCP 规范 [S5] 明确三层角色：
> "The protocol uses JSON-RPC 2.0 messages to establish communication between:
> - **Hosts**: LLM applications that initiate connections
> - **Clients**: Connectors within the host application
> - **Servers**: Services that provide context and capabilities"

MCP 借鉴了 **Language Server Protocol（LSP）** 的思想——LSP 标准化了在整个开发工具生态中添加编程语言支持的方式；MCP 类似地标准化了向 AI 应用生态集成额外上下文和工具的方式 [S5]。

### 9.4 MCP 基础协议特性（官方原文）

[S5] 列出 Base Protocol：
- **JSON-RPC 消息格式**；
- **有状态连接（Stateful connections）**；
- **服务器和客户端能力协商（capability negotiation）**。

附加工具（Additional Utilities）[S5]：
- Configuration（配置）
- Progress tracking（进度跟踪）
- Cancellation（取消）
- Error reporting（错误报告）
- Logging（日志）

### 9.5 MCP 核心原语（Primitives）—— 服务端 vs 客户端特性

这是 MCP 最关键的考点。规范明确区分 [S5]：

**Server 可向 Client 提供（Server Features）**：
- **Resources（资源）**：上下文和数据，供用户或 AI 模型使用；
- **Prompts（提示）**：模板化消息和工作流，供用户使用；
- **Tools（工具）**：供 AI 模型执行的函数。

**Client 可向 Server 提供（Client Features）**：
- **Sampling（采样）**：服务端发起的 agentic 行为和递归 LLM 交互；
- **Roots（根）**：服务端发起的对 URI 或文件系统边界的查询；
- **Elicitation（征询）**：服务端向用户请求额外信息。

> 易错点：Resources 是给"用户或模型读"的被动数据；Tools 是"模型主动执行"的函数；Prompts 是"模板化工作流"。三者职责不同，面试常考区分。

### 9.6 MCP 安全与信任原则（官方原文）

MCP 规范强调安全是首要考虑 [S5]：

- **用户同意与控制（User Consent and Control）**：用户必须明确同意并理解所有数据访问和操作；保留对共享数据和执行操作的控制。
- **数据隐私（Data Privacy）**：Host 暴露用户数据给 Server 前必须取得明确同意；未经同意不得将资源数据传向他处。
- **工具安全（Tool Safety）**：工具代表任意代码执行，必须谨慎对待；**工具行为描述（如 annotations）应被视为不可信（untrusted）**，除非来自可信 Server；Host 调用任何工具前必须取得明确用户同意。
- **LLM 采样控制（LLM Sampling Controls）**：用户必须明确批准任何 LLM 采样请求；用户应控制是否采样、实际发送的 prompt 等。

> 陷阱：工具的 description/annotation 不可信——MCP 官方明确警告，恶意 Server 可在工具描述里注入诱导模型执行危险操作的指令（即"工具描述里的 prompt injection"）。

### 9.7 MCP 在主流框架的支持

LangChain 官方文档明确支持 "Connect these docs to Claude, VSCode, and more via MCP" [S6]；LlamaIndex 文档侧边栏有专门的 "MCP / Model Context Protocol (MCP) / Using MCP Tools with LlamaIndex" 章节，并提供"把现有 LlamaIndex 工作流和工具转换为 MCP"的指南 [S7]。这印证 MCP 已成为跨框架标准。

### 9.8 MCP 客户端能力详解（官方原文）

Client 向 Server 提供的三种能力（常被忽略，是面试区分点）[S5]：

- **Sampling（采样）**：Server 发起的 agentic 行为和递归 LLM 交互——即 Server 可以反过来请求 Host 的 LLM 帮它生成内容（递归 agent 调用）。用户必须明确批准任何 sampling 请求，且应控制是否采样、实际发送的 prompt [S5]。
- **Roots（根）**：Server 发起的对 URI 或文件系统边界的查询——告诉 Server "你可以在哪些边界内操作"，避免越界访问。
- **Elicitation（征询）**：Server 发起的向用户请求额外信息——Server 运行中可向用户提问以补全所需输入。

> 设计哲学：MCP 是双向的。Server 不仅"提供"能力，也能"消费"Client/Host 的能力（LLM 推理、文件边界、用户输入），从而支持递归 agent 编排。

### 9.9 MCP 实现要点（Key Principles 原文）

规范列出实现者必须遵循的关键原则 [S5]：
- MCP 提供标准化方式让应用：(1) 与语言模型共享上下文信息；(2) 向 AI 系统暴露工具和能力；(3) 构建可组合的集成和工作流；
- 协议是**有状态**的（stateful connections），区别于无状态 REST；
- 规范基于 TypeScript schema（`schema.ts`），是"权威协议要求（authoritative protocol requirements）"；
- 规范使用 RFC2119 的关键词（MUST/SHOULD/MAY 等）解释义务级别 [S5]。

## 十一、Agent 的局限性与陷阱

### 11.1 论文/规范中明确指出的局限

- **ReAct 局限** [S1]：prompt 设定下推理和行动行为支持有限；微调能进一步提升但论文仅做了初步实验。
- **Toolformer 局限** [S2]：每个 API 仅需少量演示即可，但工具必须可文本化；且依赖基础 LM 的 in-context 能力来生成候选。
- **Reflexion 局限** [S3]：依赖 LLM 自我评估能力（或启发式）——若评估/反思错误会污染记忆；无成功的形式化保证。
- **MCP 安全局限** [S5]：工具描述不可信（prompt injection 风险）；任意代码执行需用户显式同意。

### 11.2 通用 Agent 痛点

- **误差累积（error propagation）**：长轨迹中早期错误传播放大（ReAct 摘要明确点名 CoT 的此问题）[S1]；
- **幻觉（hallucination）**：无外部锚定时模型编造事实 [S1]；
- **上下文窗口限制**：长任务超长轨迹无法塞入窗口（Deep Agents 的"自动上下文压缩"正是为此）[S6]；
- **信用分配（credit assignment）**：稀疏奖励下难以定位哪步出错（Reflexion 用语言化反思缓解）[S3]；
- **成本与延迟**：多轮工具调用导致 token 消耗和延迟激增。

### 11.3 论文揭示的"知识陈旧"问题（ReAct 附录 A.2）

ReAct 附录 A.2 记录了一个真实案例 [S1]：HotpotQA 中一道问酒店规模的问题，其原始标注已过时（酒店规模在 HotpotQA 构建后扩大了）。结果：
- Standard 和 CoT 因幻觉给错答案；
- Act 虽能访问真实网络，但因缺乏推理引导"如何为 QA 而交互"也失败；
- **只有 ReAct 能从互联网检索到最新信息并给出合理答案**。

启示：纯推理（CoT）会过时，纯动作（Act）不会用，推理+动作（ReAct）才能获取时效性知识。这对 RAG 与 Agent 的设计有直接指导意义 [S1]。

### 11.4 反思范式的根本风险（Reflexion 自述）

Reflexion 论文坦承 [S3]：反思范式依赖 LLM 的自我评估能力或外部启发式，且"没有成功的形式化保证"。具体表现：
- 若 Evaluator 误判（把错的判对），Actor 不会触发反思；
- 若 Self-Reflection 归因错误（把责任推给无关步骤），错误经验进入记忆；
- 错误记忆会在后续试验中持续误导，形成"反思污染"。
论文乐观预期"随着 LLM 能力提升此范式只会变好"，但当前仍是开放风险 [S3]。

### 11.5 工具描述注入攻击（MCP 视角的具象化）

结合 MCP 规范的安全警告 [S5] 与 ReAct/Toolformer 的工具机制：
- ReAct 的工具是 prompt 中描述的固定动作，攻击面相对小 [S1]；
- Toolformer 工具来自训练数据，攻击面在于训练时注入 [S2]；
- **MCP/Function Calling 攻击面最大**：运行时由任意 Server 提供工具描述，恶意描述可诱导模型执行危险操作（如"这个工具会删除文件，请优先调用"）。MCP 官方因此把"工具描述不可信"写入规范 [S5]。

## 十二、【面试考点】

### 高频概念题

1. **Agent 的四要素是什么？** → LLM + 记忆 + 规划 + 工具。各举论文对应 [S1][S2][S3]。

2. **ReAct 的核心创新？** → 把"思考"纳入动作空间 $\mathcal{A} \cup \mathcal{L}$，让推理与行动交替产生协同 [S1]。

3. **ReAct 与 CoT 的区别？** → CoT 是静态黑盒、不与外部绑定、易幻觉误差传播；ReAct 通过动作与外部环境交互获取信息 [S1]。

4. **写出 ReAct 的 Prompt 模板骨架。** → Thought N / Action N / Observation N 循环，详见第二节 2.2。

5. **Toolformer 如何不靠大量人工标注学会用工具？** → 三步自监督：采样候选 API 调用 → 执行 → 用"加权交叉熵损失降低"作为过滤标准 → 微调 [S2]。

6. **Toolformer 的过滤判据是什么？** → 给定 API 输入+输出作为前缀时，比"不调用"或"只调不给结果"更能降低后续 token 损失（差值 ≥ β）才保留 [S2]。

7. **Reflexion 与传统 RL 的区别？** → 不更新权重，用语言化反思存入情景记忆作"语义梯度"；轻量、可解释、无需微调 [S3]。

8. **Reflexion 的三个模型是什么？** → Actor、Evaluator、Self-Reflection [S3]。

9. **MCP 的三层角色？** → Host（发起连接的 LLM 应用）、Client（Host 内连接器）、Server（提供上下文和能力的服务）[S5]。

10. **MCP 服务端三大原语？** → Resources（数据/上下文）、Prompts（模板工作流）、Tools（可执行函数）[S5]。

11. **MCP 用什么消息格式？** → JSON-RPC 2.0 [S5]。

## 十三、【易错/陷阱】

1. **ReAct ≠ 只是"思考后调用工具"**：ReAct 的精髓是"思考"本身就是动作空间的一部分（语言空间 $\mathcal{L}$），且 thought 不产生外部观察 [S1]。

2. **Toolformer 的过滤不是"调用对不对"**：而是"调用及其结果是否降低后续 token 损失"——这是自监督信号，不是人工标签 [S2]。

3. **Reflexion 不更新模型权重**：所有"学习"都体现在情景记忆 `mem` 的文本积累上 [S3]。把它说成"用梯度更新"是错的。

4. **MCP 的 Tools 描述不可信**：官方规范明确警告 tool annotations 是 untrusted，存在工具描述内 prompt injection 风险 [S5]。

5. **MCP Resources ≠ Tools**：Resources 是被动数据（给模型读），Tools 是模型主动执行的函数。混淆是常见错误 [S5]。

6. **Sampling 是 Client 给 Server 的能力**：很多人以为 Server 只提供能力。实际上 Client 也可向 Server 提供 Sampling、Roots、Elicitation 三种能力 [S5]。

7. **并行 Function Calling 不能和内置工具一起用**：OpenAI 限制——用 code_interpreter/file_search 等内置工具时不能并行调用自定义函数 [S8]。

8. **CoT 的误差传播是 ReAct 要解决的**：不要把 CoT 当万能。ReAct 摘要明确批评 CoT 的幻觉与误差传播 [S1]。

9. **`strict: true` 保证的是参数结构**：Structured Outputs 保证函数参数严格匹配 JSON Schema，但不保证语义正确 [S8]。

10. **Agent ≠ RAG**：RAG 是检索增强生成（一次性检索注入上下文）；Agent 是循环决策（多轮工具调用 + 反思）。ReAct 是把检索变成"动作" [S1]。
