## 1.1 初识大模型

### 1.1.1 大模型时代：生成式 AI 应用的爆发

**是什么**
自 2023 年起，以 OpenAI ChatGPT 为代表的大模型引发了"百模大战 / 千模大战"，AI 进入 2.0 时代，并指向通用人工智能（AGI）。

**为什么大模型热潮远超区块链、元宇宙**
关键在于**应用层真正落地了价值与生产力**：底层是复杂的深度学习与神经网络，但在应用层以极度简洁的形式呈现给普通人（极低的使用门槛）。这一点是此前诸多"概念型"技术所不具备的。

**决策式 AI vs 生成式 AI**

| 维度 | 决策式 AI | 生成式 AI（大模型） |
|------|-----------|---------------------|
| 核心能力 | 发现隐藏模式、学习视觉/语言处理 | **推理与创造** |
| 输出形态 | 分类、预测、判别 | 文本 / 图 / 音视频 / 结构化信息 |
| 典型产物 | 标签、分数 | AIGC（AI Generated Content，人工智能生成内容） |

> AIGC：无论是自然语言文本还是结构化信息，凡由大模型生成的内容统称为 AIGC。涵盖文生文的 LLM、文生图/文生视频的扩散模型（Diffusion）等。

### 1.1.2 大模型应用的持续进化

大模型的价值必须在真实应用场景中体现，覆盖 To C（个人）与 To B（企业）两端。

- **To C 个人应用**：对话机器人、精确搜索、翻译、文档写作、辅助阅读、虚拟角色扮演、多媒体创作、设计助理、代码辅助生成等。形态上分独立工具与嵌入式 AI Copilot。
- **To B 企业级应用**：智能客服、在线咨询、智能营销、交互式数据分析、智能企业搜索、RPA（机器人流程自动化）增强等。受限于更严苛的业务环境与工程化要求，很多仍处原型/验证阶段。

**AI 能力输出的多种形式**
1. 通过 API 将 AI 能力植入现有软件与流程（AI 赋能）。
2. 独立的 AI 原生应用与工具（更主流）。

**应用的进化方向**：从最初版本的简单对话机器人，演进为具备**自我规划与记忆、使用外部工具/插件、自我反省与纠错**能力的 **AI Agent / AI Assistant**。

> RAG 的双重定位：既可以是一种独立的应用形态，也可以是开发更复杂 AI 智能体时依赖的一种设计范式/架构。

### 1.1.3 大模型是无所不能的吗

大模型在 NLP 上带来革命性提升，但同时存在**天然、甚至"很难根治"的缺陷**。这正是 RAG 等优化方案诞生的根本动因。

#### 缺陷一：知识的时效性问题

- **机制**：大模型是海量参数的神经网络，预训练/微调耗费极大财力物力时间，迭代周期短则几天、长则几个月；商业通用大模型还需安全测试与风险评估。
- **结果**：存在**知识滞后**，无法回答超出训练知识时间点之后的问题。

#### 缺陷二：输出难以解释的"黑盒子"问题

- **机制**：除 Prompt 外，使用者无须也无法观察内部推理、决策与输出过程。
- **影响**：降低使用门槛，但给深层应用与调试带来麻烦——除了改 Prompt 和几个推理参数，大部分时候要靠"运气"（随机性大）。
- **前沿研究**：Anthropic 2023 年报告 *Towards Monosemanticity: Decomposing Language Models With Dictionary Learning*，在可解释性上取得进展，但更大规模模型的可解释性仍面临技术、方法、工具上的极大挑战。

#### 缺陷三：输出的不确定性（随机性）问题

**机制（根因）**：大模型本质上是**基于所学知识统计规律的概率输出模型**，是非线性模型。它学到的是基于概率分布的多个可能输出，而非"上一个是 X 下一个就是 Y"的明确规则——因此即使上下文相同，也可能在不同时间选择不同的下一个 token。

**两面性**：
- 创意生成、自媒体内容创作场景中，多样性恰恰是优点。
- 智能家居精确意图理解、开发调试需要一致性输出场景中，则是挑战。

**控制手段（无法完全消除）**：

| 参数 | 作用 | 说明 |
|------|------|------|
| `temperature` | 在一定程度上控制随机性 | **无法完全消除随机性** |
| `seed` | OpenAI 后续引入 | 在相同输入前提下尽量产生**可重现**的输出 |

#### 缺陷四：著名的"幻觉"（Hallucination）问题

**是什么**：大模型生成内容或回答问题时，结果不完全正确甚至错误——"一本正经地胡说八道"。表现为事实错误陈述/编造、错误复杂推理、复杂语境下处理能力不足。

**四大成因**：

1. **训练知识存在偏差**：海量训练知识可能含错误、过时、偏见信息，被学习后在未来输出中重现。
2. **过度泛化推理**：从大量语料学到的普遍规律被错误套用到特定场景。
3. **理解存在局限性**：并未真正"理解"深层含义，不具备人类常识与经验。
4. **缺乏特定行业与垂直领域知识**：通用大模型像掌握大量通用知识的优秀学生，但不是某垂直领域专家；面对高复杂度领域问题或企业私有知识问题时会编造信息。

**为什么"幻觉"是企业级应用最大拦路虎**：企业对准确性与可靠性要求更高——可以接受文案不一致，但无法接受给客户介绍产品时胡编乱造、把错误财务分析呈现给决策层、错误理解客户意图（差之毫厘谬以千里）。

> **关键结论**：本书主题——**RAG 正是为尽可能解决大模型实际应用中的问题（特别是幻觉）而诞生，是该领域最重要的优化方案。**

## 1.2 了解 RAG

### 1.2.1 为什么需要 RAG

**基本思想（一句话）**：将传统生成式大模型与**实时信息检索技术**相结合，为大模型补充来自外部的相关数据与上下文，帮助其生成更丰富、更准确、更可靠的内容——让大模型在生成时依赖**实时与个性化**的数据/知识，而不只依赖训练知识。

**三个诞生动机（对应三大收益）**：

| 动机 | 现状痛点 | RAG 的收益 |
|------|----------|------------|
| 知识覆盖度 | 特定领域复杂任务受限于训练知识覆盖度 | 借助额外信息检索，利用更广泛深入的知识 |
| 实时性 | 静态语言模型不能反映最新信息 | 结合实时检索，访问最新数据，提供实时准确内容 |
| 计算效率 | 扩大参数规模意味着更高成本与更长训练时间 | 通过检索引入相关信息，更高效利用计算资源，短期内提升输出能力 |

**经典类比（重要，面试常考）**：
- 大模型 = 经过大量训练的**优秀学生**。
- 大模型响应 = **考试**。
- 学生遇到没掌握的知识 → 编造答案（幻觉）。
- **RAG = 考试时临时给他的一本参考书**，要求参考作答 → 遇到相关知识时得分大幅提高。

### 1.2.2 一个简单的 RAG 场景

**场景**：开发在线自助产品咨询工具，客户用自然语言交互式问答，如"请介绍贵公司这款产品与 ×× 产品的不同之处"，嵌入官网。

**直接用通用大模型的失败**：大模型不具备企业最新产品知识，无法回答（有时会编造）。

**用 RAG 思想的完整闭环**：

```
[企业私有知识库] 
       │ ① 检索相关补充知识（一段文字）
       ▼
[组装到 Prompt] ── ② 要求大模型基于提供的知识回答
       │
       ▼
[大模型] ── ③ "吸收"外部知识 + 结合自身知识 → 推理给出答案
```

**重要澄清（避免误解 RAG）**：
- RAG 本质上**是一种借助"外挂"的提示工程**，但**绝不仅限于此**。
- 实际应用涉及众多技术细节与挑战：
  - 自然语言问题千变万化，**从哪里检索**对应外部知识？
  - 用**怎样的索引**来查询？
  - 如何**确保补充的外部知识是回答该问题最需要的**？
- 类比延伸：考试知识点是"一元二次方程"，却给学生一本《微积分》——于事无补。**检索的精确性是 RAG 的核心难点之一。**

## 1.3 RAG 应用的技术架构

### 1.3.1 RAG 应用的经典架构与流程

一个最小粒度的 RAG 应用，从整体上分为**数据索引（Indexing）**与**数据查询（Query）**两大阶段。

```
┌─────────────── 数据索引阶段（离线/构建期）────────────────┐
│                                                          │
│  原始知识  → ① 加载   → ② 分割   → ③ 嵌入   → ④ 索引     │
│  (多模态)    Loading    Splitting   Embedding  Indexing   │
│                                                          │
└──────────────────────────┬───────────────────────────────┘
                           │ 持久化到向量库等
                           ▼
┌─────────────── 数据查询阶段（在线/运行期）────────────────┐
│                                                          │
│  用户问题 → [检索前处理] → ⑤ 检索 → [检索后处理] → ⑥ 生成  │
│            Pre-Retrieval  Retrieval  Post-Retrieval Gen   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 1. 数据索引阶段（Indexing）

**核心目标**：准备可检索的内容。

> **检索方式对比**：传统计算机检索多用**基于关键词**的精确/模糊匹配（搜索引擎、关系数据库）；RAG 中最常见的是**基于向量的语义检索**——计算相似度排序，输出最相关的前 K 个数据块（**top_K**）。因此**向量存储索引**是 RAG 最常见的索引形式。

**向量的本质**：将文本/图像/音频等复杂信息转换为高维空间中的点，每个维度代表一种特征/属性；向量保留了词汇的语义关系（相似词在向量空间距离近），可做语义相似度计算与聚类。把信息转成向量的模型叫**嵌入模型（Embedding Model）**。语义检索 = 计算查询词与已有信息向量的相似度（如**余弦相似度**）。

**索引阶段四个关键子阶段**：

| 子阶段 | 作用 | 关键问题/说明 |
|--------|------|---------------|
| ① 加载（Loading） | 连接与读取不同形式/模态知识（结构化、半结构化、非结构化、互联网/企业内部、文档/问答对） | 需支持多源多格式 |
| ② 分割（Splitting） | 把较大知识内容（Word/PDF/Excel/网页/数据库表）切分成知识块（**Chunk**）做索引 | 核心问题：**块切多大最合适？用什么标记段落结尾？** |
| ③ 嵌入（Embedding） | 把分割后的 Chunk 转成高维向量（如 1024 维） | 需借助商业或开源**嵌入模型**，如 `text-embedding-3-small` |
| ④ 索引（Indexing） | 将向量持久化到内存或磁盘 | 实际建议用**向量数据库（向量库）**，提供检索算法与管理接口；高级 RAG 还可建**知识图谱索引、关键词表索引**等 |

#### 2. 数据查询阶段（Query）

数据索引准备完成后，**两大核心阶段是检索与生成**：

| 子阶段 | 作用 |
|--------|------|
| ⑤ 检索（Retrieval） | 借助数据索引从存储库中检索相关知识块，按相关性排序，作为参考上下文 |
| ⑥ 生成（Generation / 合成） | 大模型（本地部署或 API 远程）根据相关知识块 + 用户原始查询 + 精心设计的 Prompt，生成并输出结果 |

**两个新增的优化阶段**（随 RAG 范式演进而引入）：

| 阶段 | 位置 | 典型任务 | 目的 |
|------|------|----------|------|
| 检索前处理（Pre-Retrieval） | 检索之前 | 查询转换、查询扩充、检索路由 | 为检索与检索后处理做准备，提高召回精确度与生成质量 |
| 检索后处理（Post-Retrieval） | 检索之后 | 用更专业排序模型/算法**重排序（Rerank）**、过滤不符合条件的块 | 让最需要、最合规的块处于上下文最前端，提升输出质量 |

### 1.3.2 RAG 应用面临的挑战

传统 RAG 应用面临的五大挑战（也是催生新架构与优化方法的动力）：

| # | 挑战 | 说明 |
|---|------|------|
| 1 | **检索召回的精确度** | 检索出的外部知识块要足够精确全面才能保障生成质量；自然语言复杂 + 检索技术模糊 → 噪声、无用、矛盾信息会影响生成 |
| 2 | **大模型自身对抗干扰的能力** | 检索上下文中的干扰/多余/矛盾信息，需大模型能推理、识别、区分并按 Prompt 输出；大模型能力是影响最终生成质量的重要因素 |
| 3 | **上下文窗口的限制** | 大模型有输入输出 token 上限；大规模知识库检索更多块可能打破窗口限制导致失败；**如何在窗口内尽可能多地携带知识块**是常见问题 |
| 4 | **RAG 与微调的选择** | 微调把垂直知识变成训练语料、把优化提前到使用前、简化架构；如何选择/配合/协调以最大化输出能力，是常被纠结的问题 |
| 5 | **响应性能问题** | RAG 增加了处理步骤，复杂 RAG 范式（多次迭代优化）虽输出更优但与端到端响应性能下降矛盾；**对延迟敏感场景如何兼顾质量与延迟** |

### 1.3.3 RAG 应用架构的演进

**RAG 发展简史**：

1. **2020 年**：Meta（原 Facebook）技术团队在论文 *Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks* 中正式提出 RAG，为大型预训练语言模型提供"**非参数化**"（训练知识以外的）信息与记忆。
2. **2022 年 ChatGPT 之前**：RAG 关注度不高，部分研究集中在提高外部知识检索与效率；ChatGPT 后迎来拐点，大量研究转向用 RAG 提升大模型可控性、解决幻觉、优化推理生成。
3. **GPT-4、Gemini Pro 之后**：RAG 在更多场景应用，传统范式问题凸显，涌现大量优化（RAG + 微调结合、顺序流程引入迭代实现自我反省、针对各阶段模块的深度优化算法）。

**同济大学智能自主系统上海研究所等团队 2024 年报告 *Retrieval-Augmented Generation for Large Language Models: A Survey*，将 RAG 演进分为三阶段**：

| 阶段 | 特征 | 模块/流程 |
|------|------|-----------|
| **① Naive RAG**（朴素/经典 RAG） | 最早的经典思想；传统基础顺序流程；只保留最简单的 Node 且**顺序式执行** | 索引 → 检索 → 生成（3 大模块） |
| **② Advanced RAG**（高级 RAG） | 在 Naive RAG 基础上增强索引/检索/生成，**特别是检索阶段** | 增加**检索前处理 + 检索后处理** |
| **③ Modular RAG**（模块化 RAG） | 超越链式顺序式范式；**更灵活、自由、高度扩展** | 见下 |

**Modular RAG 的核心思想（三级抽象）**：

```
模块类（Module Class）  ── 代表一个核心流程，如"预检索"
    └─ 模块（Module）   ── 核心流程中的功能模块，如"查询转换"
        └─ 算法（Algorithm） ── 模块的一种实现方法，如查询转换的：
                                  · 普通重写
                                  · 后退式重写（Sub-question / Step-back）
                                  · HyDE 重写（Hypothetical Document Embeddings）
```

- 模块与算法**不再有固定选择与顺序流程**，由使用者根据场景灵活组合，构造适合自己的 RAG 工作流。
- **好处**：极强扩展性（更多模块/算法可不断加入）+ 灵活性（灵活组合）。
- 本书第 13 章会介绍如何通过灵活组合实现复杂工作流以达成特定优化目标。

## 1.4 关于 RAG 的两个话题

### 1.4.1 RAG 与微调的选择

#### 大模型训练的阶段（以 GPT 为例）

**1. 预训练阶段（Pre-training）**
- 最复杂阶段；GPT-4 级别需成千上万 GPU，在**海量无标记数据**上训练数月，占用绝大部分时间。
- 输出叫**基座模型（Base Model）**：有的会发布（如开源 Llama），有的不发布（如 GPT-4）。
- **关键认知**：基座模型可直接用，但**不是"回答问题"的模型，而是"补全文档"的模型**——必须假装输出一个文档让它补全（如提示"下面是一首赞美祖国的诗歌："），而不能直接要求"写一首赞美祖国的诗歌"。
- 要变成交互式 AI 助手 → 进入**微调**。

**2. 微调阶段（Fine-tuning）**
- 宏观上包含：**受监督微调（SFT）**、**奖励模型 + RLHF（基于人类反馈的强化学习）**。
- 在少量（相对预训练）已标注数据上再次训练与强化学习，使模型适应特定场景/下游任务。

**微调的五大用途**：
1. 强化某方面应用能力（如情感检测）。
2. 适应特定使用场景（如对话场景输出无害安全内容）。
3. 适应特定知识领域（如医疗/法律术语或语义）。
4. 适应可标注数据相对稀缺的任务。
5. 适应特定语言输出要求（如某场景的语言风格）。

**微调特点**：
- 算力要求与成本相比预训练大大降低，对企业相对可行（但仍高于 RAG）。
- 是相对专业的技术任务，涉及深度学习架构/参数/算法，多种技术：**全量微调、Prompt Tuning、Prefix Tuning、P-tuning** 等，资源/成本/数据/效果各异。
- 工具/框架/平台：OpenAI 在线微调 API、**DeepSpeed**（重量级大模型并行训练框架）等。

**微调数据集准备的四大挑战**（垂直领域尤为突出）：
1. 数据从哪采集、如何确保专业性与有效性。
2. 对多形态数据如何清洗与归一。
3. 怎么标注提示/输入/输出。
4. 处理老化数据（知识过期后如何反馈到大模型）。

#### RAG vs 微调：经典类比

| | 类比 |
|---|---|
| **RAG** | 考试时给他某领域参考书，要求**现学现用**给出答案 |
| **微调** | 考试前一天对他**辅导**，使他成为某领域专家，再参加考试 |

#### RAG 与微调对比（表 1-1）

| 维度 | RAG | 微调 |
|------|-----|------|
| **优点** | 1. 使用灵活，可随时调整 Prompt 获得期望输出<br>2. 技术上更简单<br>3. 可输入知识增强 Prompt 让大模型**立即**适应领域知识<br>4. **无额外训练成本** | 1. 大模型自身拥有特定知识输出能力，或适应特定输出格式<br>2. 对下游应用更友好，特定任务中使用更简单<br>3. 可节约推理阶段 token，**推理成本更低** |
| **缺点** | 1. 易受限于上下文窗口大小<br>2. 输入本地知识增强 Prompt 在实现上下文连续对话时较困难<br>3. 大模型输出不确定性在高准确性场景会增加失败概率<br>4. 输入较长上下文 Prompt 会带来较高推理成本<br>5. 模型迭代后可能需重新调整 Prompt | 1. 非开箱即用<br>2. 需额外数据准备/标注/清洗成本 + 必要算力与训练成本<br>3. 需足够技术专家（ML 专家、数据专家）<br>4. 微调**无法阻止幻觉**，过度微调甚至可能导致某些能力下降<br>5. 模型迭代周期长，对实时性要求高的知识不适用 |

#### 选择标准

**更适合考虑微调的场景（不考虑成本前提下）**：
1. 需注入较大数据量且相对稳定、迭代周期较长的领域知识；需形成相对通用的领域大模型对外服务/运营。
2. 执行需极高准确率的关键任务，且其他手段无法满足——通过高效/全量微调提高精度，如**医疗诊断**。
3. 采用提示工程、RAG 等技术后，仍无法达到指令理解准确、输出稳定或其他业务目标。

> **其余很多场景优先考虑 RAG**；条件允许时，**两者融合应用**或许是未来更佳选择。

### 1.4.2 RAG 与具有理解超长上下文能力的大模型

**背景**：大模型上下文窗口（Context Window）以惊人速度增大——从早期 4K~8K，到 Claude 2 的 200K，再到 Claude 3 / Gemini 1.5 号称的 **1M**。
- 注：上下文窗口中 **K = 1000，M = 1,000,000**，单位是 **token**。

**争议话题**：Gemini 1.5 技术报告展示"**大海捞针**"（在超长上下文中精确检索特定位置的事实性知识）能力后——**如果未来能把几百个文档塞进对话窗口，且大模型能在其中检索事实，还有必要做外部索引与检索（RAG）吗？**

**为何不能简单替代（七个反思）**：

1. 1M 上下文窗口是否真的够？如果还不够呢？
2. 反复输入如此长上下文，时间、网络与推理成本上升值得吗？
3. 企业级应用中大量实时变化、形式多样的知识，如何每次都输入大模型？
4. 大模型"大海捞针"能力可靠吗？与"针"的**数量和位置**有关系吗？
5. 知识密集型任务不仅是简单事实问答，大模型能否应对？
6. 过长上下文是否让应用的跟踪、调试、评估更困难？
7. 相对 RAG 在应用层的精确控制，把知识都输入大模型如何确保**知识安全性**？

**实测结论（重要）**：受限于主流大模型依赖的 **Transformer 底层架构基础原理**，当前大模型理解超长上下文的能力**并不像宣传的那么出色**。使用开源工具 **LLMTest_NeedleInAHaystack** 测试发现，能否准确检索事实并完成推理，至少依赖：

| 因素 | 影响 |
|------|------|
| 上下文长度 | 输入 2K 与 128K 时精确检索能力不同 |
| 相关知识块在上下文中的位置 | 文档中位置会影响检索结果 |
| 完成任务依赖的相关知识块数量 | 检索 1 个 vs 10 个的准确率不同 |

> **结论**：即使完全不考虑成本与响应速度，依赖超长上下文大模型准确检索与推理完成知识密集型任务也**不太现实**。

**正确姿势**：RAG 与超长上下文大模型**相互结合、取长补短**：
- 大模型为 RAG 提供优化基础（未来可以考虑以较大文档为单位而非较小知识块作为检索召回基础）。
- RAG 提供大模型所不具备的灵活性。
- 在**简洁性与灵活性之间取得平衡**，或许是当下最合适的方案。

## 【面试考点】

> 以下是本章高频面试题及参考答案要点。

### Q1：什么是 RAG？为什么需要 RAG？

**要点**：
- RAG（Retrieval-Augmented Generation，检索增强生成）= 将生成式大模型与实时信息检索结合，为大模型补充外部相关数据/上下文，使其生成更丰富、准确、可靠内容。
- 三大动机：① 扩展特定领域知识覆盖度；② 提供实时性（访问最新数据）；③ 高效利用计算资源（相比扩大参数规模）。
- 类比：考试时给学生一本参考书现学现用。
- 主要目标：缓解大模型"幻觉"，提升企业级应用的准确性与可靠性。

### Q2：大模型有哪些固有缺陷？

**要点（四类）**：
1. 知识时效性（训练周期长导致滞后）。
2. 黑盒子（不可解释）。
3. 输出不确定性（本质是概率输出模型，temperature 只能调节不能消除）。
4. 幻觉（训练数据偏差、过度泛化、理解局限、缺乏垂直知识）。

### Q3：请描述 RAG 的经典架构与完整流程。

**要点**：
- 两大阶段：**数据索引（Indexing）** + **数据查询（Query）**。
- 索引阶段：加载 → 分割（Chunk）→ 嵌入（Embedding）→ 索引（持久化到向量库）。
- 查询阶段核心：检索（Retrieval）→ 生成（Generation）。
- 优化阶段：检索前处理（查询转换/扩充/路由）+ 检索后处理（Rerank 重排序、过滤）。
- 关键概念：top_K、Chunk、嵌入模型、向量存储索引、余弦相似度、语义检索 vs 关键词检索。

### Q4：什么是大模型的"幻觉"？产生原因有哪些？如何缓解？

**要点**：
- 幻觉：大模型生成不完全正确甚至错误内容（一本正经地胡说八道）。
- 四大原因：训练数据偏差、过度泛化、理解局限、缺乏垂直/私有知识。
- 缓解手段：RAG（注入外部可信知识）、Prompt 工程、temperature/seed 调节、微调（注意微调无法根治幻觉）、模型架构与训练数据质量提升、可解释性研究。

### Q5：RAG 与微调如何选择？

**要点**：
- RAG 优势：灵活、技术简单、立即适应新知识、无训练成本；劣势：受上下文窗口限制、长 Prompt 推理成本高、不确定性。
- 微调优势：模型自身具备能力、对下游友好、推理 token 更省；劣势：非开箱即用、需数据/算力/专家、无法根治幻觉、迭代周期长。
- 选微调场景：稳定大领域知识、极高准确率关键任务（如医疗诊断）、提示/RAG 后仍不达标。
- 其余优先 RAG；条件允许两者融合。
- 类比：RAG = 考试给参考书现学现用；微调 = 考前一天辅导成专家。

### Q6：有了超长上下文（如 1M）大模型，还需要 RAG 吗？

**要点**：
- 仍需要。原因：成本/延迟、知识实时性与多样性、检索准确率随上下文长度/位置/针数量变化（Transformer 架构限制）、调试评估困难、知识安全性。
- 实测工具 LLMTest_NeedleInAHaystack 证明超长上下文检索能力被高估。
- 正确做法：RAG + 超长上下文大模型**结合取长补短**。

### Q7：RAG 架构演进的三个阶段是什么？

**要点**：Naive RAG（朴素，索引-检索-生成顺序流）→ Advanced RAG（增加检索前/后处理）→ Modular RAG（模块类-模块-算法三级抽象，灵活组合非固定流程）。

## 【易错 / 陷阱】

> 实际开发与认知中容易踩的坑。

1. **把 RAG 等同于简单 Prompt 工程/知识外挂**
   - 陷阱：低估检索精确性的难度。检索给错知识（如考一元二次方程却给《微积分》）比不给还糟。
   - 正解：RAG 核心难点在检索召回质量，不是 Prompt 拼接本身。

2. **temperature 设为 0 就以为输出完全确定**
   - 陷阱：temperature 只能降低随机性，**无法完全消除**。需要可重现输出还要配合 `seed`。

3. **认为基座模型可以直接回答问题**
   - 陷阱：基座模型是"补全文档"模型，不是"回答问题"模型；直接问会得到续写而非回答。
   - 正解：需经 SFT + RLHF 微调后才成为对话助手。

4. **认为微调能解决幻觉**
   - 陷阱：微调无法阻止幻觉，**过度微调甚至导致某些能力下降**。

5. **盲目相信大模型超长上下文的"大海捞针"宣传**
   - 陷阱：检索准确率与上下文长度、知识块位置、知识块数量都相关；测试不等于生产可靠。

6. **忽视上下文窗口限制**
   - 陷阱：检索过多知识块塞入 Prompt 会超窗口导致失败。开发时需在窗口内最大化有效知识携带。

7. **RAG 与微调非此即彼**
   - 陷阱：很多场景两者融合效果更佳；不是二选一。

8. **混淆"决策式 AI"与"生成式 AI"**
   - 陷阱：前者擅长判别/预测，后者擅长推理/创造；AIGC 特指生成式产出。

9. **忽视响应延迟**
   - 陷阱：复杂 RAG（多次迭代优化）输出更优但延迟上升；对延迟敏感的企业场景必须权衡质量与性能。
