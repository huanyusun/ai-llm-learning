## 2.1 开发 RAG 应用的两种方式

当前两种主要开发方式：**① 低代码开发平台**；**② 主流大模型应用开发框架**。

### 2.1.1 使用低代码开发平台

**代表平台**：
- 商业大模型公司平台：百度 **AppBuilder**、阿里云**百炼**、字节跳动 **Coze**。
- 独立平台（含开源）：**FastGPT**、**Dify** 等。

**四大特点**：

| # | 特点 | 说明 |
|---|------|------|
| 1 | 使用简单便捷 | 可视化应用定义与配置，快速开发 |
| 2 | 完整开发工具栈 | 含数据集/知识库管理、应用测试、Web UI / API 发布工具 |
| 3 | 支持 RAG 工作流设计 | 可视化拖曳 + 少量编码，低代码编排 |
| 4 | 灵活选择模型 | 可灵活搭配嵌入模型与大模型 |

**便捷性与灵活性是矛盾体** —— 低代码平台有五大局限：

1. **安全合规挑战**：通常需在线开发部署；RAG 应用是知识密集型，敏感领域/企业面临数据安全合规问题。
2. **通用性 vs 个性化**：面向通用需求设计，企业级个性化需求时捉襟见肘。
3. **工程要求与范式灵活度不足**：企业级应用有更高工程要求与多样业务需求，关键场景需灵活 RAG 范式/流程，通用平台可能无能为力。
4. **形态局限**：低代码产出的主要是知识查询/对话类应用，但这不是 RAG 全部。例：**Vanna** 框架用 RAG 思想向 Prompt 注入检索出的数据结构描述信息，优化 **Text-to-SQL**（文本转关系数据库 SQL）输出准确性——这类场景无法直接用低代码平台开发。
5. **屏蔽技术细节**：降低工作量的同时屏蔽底层原理，给后期优化与开发者能力提升增加难度。

> **适用场景判断**：低代码平台快速、简单、部署方便、基本无后期维护，但缺乏灵活性与个性化，安全性面临挑战——**更适合面向个人与小微企业的知识型应用/AI 智能体**；面向 B 端的主流 RAG 应用需要更灵活的技术。

### 2.1.2 使用大模型应用开发框架

**主流开源框架**：LangChain 系列、LlamaIndex、微软 AutoGen。基本采用"开源 + 少量商业服务"模式。

**五大特点**：

1. **易用的组件化/模块化 + 完善文档**：成熟框架经长期迭代与大量开发者使用，功能/性能/扩展性/第三方支持/文档/社区完善。
2. **屏蔽底层技术细节**：对基础设施访问细节抽象，提供更易用的上层接口（大模型通信、文档处理、复杂底层算法）。
3. **预置大量可重用封装组件**：搭积木式组装，不重新造轮子；内置先进范式与算法支持。具体到 RAG，内置大量模块化 RAG 运算符与算法，支持 RAG 工作流编排。例：
   - **LangChain 的 LangGraph**
   - **LlamaIndex 的 Query Pipeline**
   - 二者都采用**图（Graph）思想**构造灵活复杂的工作流。
4. **极好的灵活性与扩展性**：可派生定制组件；内置对第三方支持（不同大模型、数据源、嵌入模型、向量库、第三方 API 工具）。
5. **覆盖软件全生命周期**：不仅提供开发组件，还提供帮助应用投产的工程化平台/工具（如 LlamaIndex 内置**评估组件**），助原型快速过渡到生产应用。

#### 1. 了解 LangChain 框架

**定位**：著名的大模型应用开发框架；基础开发库完全开源免费，可从调用简单大模型到开发复杂 RAG/Agent。本书完稿时最新版本 **0.2.x**。

**LangChain 组成**：

| 库 | 职责 |
|----|------|
| **LangChain-Core** | 基础抽象 + **LCEL（LangChain 表达语言）** |
| **LangChain-Community** | 组件 + 大量第三方集成模块，常封装为合作伙伴模块（如 `LangChain-openai`） |
| **LangChain** | 构成应用架构的上层核心组件：Chains、Agents 等 |
| **Templates** | 官方常见应用模板 |
| **LangServe** | 将 LangChain 链部署为 **REST API** 的平台 |
| **LangSmith** | 在线平台：跟踪调试、测试、评估、监控大模型应用 |
| **LangGraph**（后期推出） | 构造更灵活复杂的 RAG/Agent 工作流 |

**评价**：
- 优点：功能强大，几乎涵盖所有大模型相关应用类型，能对接几乎所有第三方技术/服务，提供强大 LCEL 与 Chain 组件。
- 缺点：**学习门槛相对较高**，组件结构庞大复杂，版本快速迭代导致**部分存在过度封装或冗余设计之嫌**。

#### 2. 了解 LlamaIndex 框架

**定位**：连接客户数据与大模型、开发大模型应用的开源免费框架；简单、灵活、强大。本书完稿时最新版本 **0.10.x**（本书开发采用版本）。

**整体结构**：

| 部分 | 说明 |
|------|------|
| **Core Framework** | 核心框架，实现大量可扩展的基础组件与工具组件 |
| **Integrations** | Core 之上由官方/第三方提供的扩展组件（数据加载器、LLM、嵌入模型、向量库、Agent 工具等） |
| **Templates** | 更上层的应用模板，可基于其修改开发 |
| **Eval Datasets** | 评估数据集，通常两部分：① 开发 RAG 的**原始知识数据集**；② 评估 RAG 的**测试数据集** |

> **LlamaHub**：LlamaIndex 官方维护的存储库/网络平台，发布、共享、使用各类组件与工具包（核心框架以外的扩展组件、模板、数据集都在此）。

**LlamaIndex 四大特点**：

1. **面向生产级/企业级**：基于成熟 RAG 范式提供易用、可靠、鲁棒的集成组件（数据加载、向量库访问、索引构造、模型访问）。
2. **灵活、可扩展、可定制化**：可派生扩展实现定制组件。
3. **最适合开发 RAG 应用**：相对 LangChain，最初主要面向 RAG；内置大量模块化 RAG 优化组件与算法，使用比 LangChain 简单。
4. **内置强大的 Data Agent 开发功能**：RAG 以数据查询为主，LlamaIndex 允许在此基础上开发更强大的**数据智能体**（智能规划与使用外部工具进行数据操控）；最新版推出 **Query Pipeline**（基于图结构编排 RAG 工作流）。

**与 LangChain 对比**：LlamaIndex 学习与使用门槛更低，更侧重 RAG 应用开发 → **本书选定 LlamaIndex 作为基础框架**。

## 2.2 RAG 应用开发环境准备

本节搭建一个**可完全本地化运行与使用**的开发环境。

### 2.2.1 硬件环境建议

若希望开发阶段所有工作在一台电脑完成（主程序 + 向量库 + 本地大模型 + 可能的关系数据库 + 前端 Web 应用），建议配置：

| 维度 | 建议配置 |
|------|----------|
| 操作系统 | **MacOS（推荐）** / Windows 10 / Linux |
| 内存 | 不低于 **16GB**；运行本地大模型则需 **32GB 甚至更大**（按模型大小） |
| GPU | 需本地大模型推理时，按模型大小配置独立显卡 |
| CPU | 最新 Intel CPU 或 **Apple M 系列芯片** |
| 磁盘 | 足够的高速磁盘空间 |
| 网络 | 高速以太网（下载大模型 / 访问大模型 API） |

### 2.2.2 基础大模型

基础大模型是 RAG 的核心与"智慧大脑"。两种使用方式：**① API 访问云端商业大模型**；**② 部署本地开源大模型**。

#### 1. 通过 API 访问云端商业大模型

**通用形态**：所有商业服务商都提供开放 API（**SDK** 与 **HTTP API** 两种形式），有详细文档；协议大同小异，使用简单（大部分在线模型只需一个 **API Key**）。

**API 申请流程（以阿里云通义千问/百炼平台为例，其他厂商类似）**：
1. 登录阿里云**百炼大模型服务平台**。
2. 点击"登录"（先注册），建议手机号 + 验证码登录。
3. 成功后点击左侧导航"**模型广场**"。
4. 点击"**去开通**"，同意协议 → "确认开通"即开通大模型调用服务（新用户可能获得免费体验额度）。
5. 点击右上"账号管理" → "**API-KEY**" → "**创建新的 API-KEY**"获得 API Key。
6. 点击右上"**产品文档**"查看模型介绍、速率限制、API 文档、计费方式等。

> **务必保存 API Key**；免费额度用完后确保账号有足够余额。
> 后续会用 OpenAI GPT 系列模型 API，建议通过**微软 Azure 云服务平台**开通 OpenAI API 服务，审核后付费使用。

**多模型切换痛点与解决（重要工程技巧）**：
- 痛点：商业大模型多，各厂家 API 协议与 SDK 不统一通用；同一厂家升级也可能版本不兼容。上层应用切换不同商业大模型面临较多适配与调试。
- 技巧：**构造对上层应用的统一 API 平台**，简单配置即可自动适配多个不同商业大模型 API 协议——前端切换大模型通常只需改模型名称，无需改代码；还可实现多通道**流量与负载均衡**提高可用性。
- **推荐开源项目：ONE-API**：把后端配置的不同大模型 API 全部"OpenAI 化"（统一成 OpenAI API 协议），极大方便前端对不同大模型的灵活调用与切换。

#### 2. 基于 Ollama 部署本地开源大模型

**Ollama 定位**：大模型运行与推理的管理工具；使用简单，**可利用普通 CPU 推理**（也支持 GPU）。安装简单。

**1）安装 Ollama**
1. 进入 Ollama 官网，选择对应操作系统。
2. 下载安装包，本机找到后双击按引导完成安装（会安装一个命令行工具）。
3. 终端测试版本号：

```bash
ollama --version
```

**2）拉取与运行大模型**
1. 登录 Ollama 官网，查看当前支持的全部大模型。
2. 选择模型（如阿里通义千问 `qwen` / `qwen2` 系列），查看各参数模型介绍与运行命令，按硬件选择（如选 `14b`）。
3. 拷贝运行命令，终端粘贴执行，自动拉取到本地运行（有进度提示），拉取完成后自动启动推理服务，可在命令提示符下交互式对话。
4. 退出：输入 `/bye` 退出 Ollama 命令行。
5. 重新启动对话测试：再次执行运行命令。

```bash
# 示例：拉取并运行通义千问 14b
ollama run qwen:14b

# 退出交互
/bye
```

**3）使用 API 服务模式**

应用开发需用 API 调用而非交互命令行（交互仅用于测试）。

```bash
# 方式一：直接进入服务模式（MacOS 上 ollama run 启动大模型会自动进入服务模式）
ollama serve

# 方式二：修改默认端口（默认 11434，与其他服务冲突时）
OLLAMA_HOST=:11435 ollama serve
```

**验证 API 服务器**（采用 Ollama 兼容 OpenAI API 协议的接口测试）：

```bash
curl http://localhost:11434/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "model": "qwen:14b",
        "messages": [
            {
                "role": "system",
                "content": "You are a helpful assistant."
            },
            {
                "role": "user",
                "content": "你好呀！"
            }
        ]
    }'
```

> 至此获得三类基础 API 服务：通义千问（在线）、OpenAI（在线）、Ollama 本地系列开源大模型（可随时下载新模型切换）。

### 2.2.3 嵌入模型

**为什么需要**：RAG 中需用嵌入模型（Embedding Model）实现知识块向量化，从而实现基于语义相似度的检索；**嵌入模型的向量生成质量对检索准确性有重要影响**。

**两类**：
- 商业大模型基座的向量模型（如 OpenAI Embedding 系列）。
- 开源向量模型（如智源研究院 **BGE Embedding**）。

#### 1. 通过 API 访问云端嵌入模型

商业平台嵌入模型接口一般**无须单独申请**，直接用基础大模型服务的 API Key 即可。

**通义千问可用的嵌入模型（表 2-1）**：

| 模型中文名 | 模型英文名 | 向量维度 | 单次请求文本最大行数 | 单行最大输入字符长度 | 支持语种 |
|------------|-----------|----------|----------------------|----------------------|----------|
| 通用文本向量 | text-embedding-v1 | 1536 | 25 | 2048 | 汉、英、西、法、葡、印尼 |
| 通用文本向量 | text-embedding-async-v1 | 1536 | 100,000 | 2048 | 汉、英、西、法、葡、印尼 |
| 通用文本向量 | text-embedding-v2 | 1536 | 25 | 2048 | 汉、英、西、法、葡、印尼、日、韩、德、俄 |
| 通用文本向量 | text-embedding-async-v2 | 1536 | 100,000 | 2048 | 汉、英、西、法、葡、印尼、日、韩、德、俄 |

> 关键参数：**向量维度（1536）**、**单行最大输入字符长度（2048）**、**同步 vs 异步（async 版本单次请求行数大幅提升至 10 万）**。

#### 2. 基于 Ollama 部署本地嵌入模型

Ollama 也可部署开源嵌入模型提供向量生成服务。

```bash
# 拉取中文向量模型
ollama pull milkey/dmeta-embedding-zh:f16

# 启动服务
ollama serve
```

测试向量生成：

```bash
curl http://localhost:11434/api/embeddings -d '{
   "model": "all-minilm",
   "prompt": "Here is an article about llamas..."
}'
```

#### 3. 基于 TEI 部署本地嵌入模型

**TEI（Text Embeddings Inference）**：Hugging Face 推出的**嵌入模型部署工具**（注意 TEI 本身不是模型，是类似 Ollama 的部署工具），也支持部署 **Rerank 排序模型**（RAG 中可能需要的模型）。功能比 Ollama 更强大。

**部署示例：以本地方式部署开源嵌入模型 `bge-large-zh-v1.5` 的向量服务 API**。

安装过程（MacOS 为例）：

```bash
# 1. 安装 rust（出现错误需修改 .bash_profile 权限）
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# 2. 从 GitHub 下载 TEI 代码（项目名 text-embeddings-inference）
git clone https://github.com/huggingface/text-embeddings-inference.git

# 3. 进入目录，安装 metal（Intel 芯片则把 metal 改为 mkl）
cd text-embeddings-inference
cargo install --path router -F metal

# 4. 启动向量服务
#    （Linux 可能需安装 gcc/openssl：sudo apt-get install libssl-dev gcc -y）
model=BAAI/bge-large-zh-v1.5
text-embeddings-router --model-id $model --port 8080
```

**验证**：模型服务用 **FastAPI** 暴露 API，可直接浏览器访问 `http://localhost:8080/docs/` 查看当前 TEI 模型服务 API（Swagger UI）。对 `embed`（生成文本向量）API 测试：点击该 API → "Try it out" → 修改 Request Body → "Execute"，正常则应看到大量向量生成。

### 2.2.4 Python 虚拟运行环境

**开发语言**：Python（简单易用 + 强大开发库）。
**环境工具**：**Miniconda**（小巧好用，方便管理虚拟环境）。

#### 1. 安装 Miniconda（MacOS 为例）

1. 访问 Miniconda 官网下载中心。
2. 按硬件与操作系统选择下载（MacOS 选 "64-bit"，按 CPU 选 Intel x86 或 Apple M 系列）。
3. 下载 `.pkg` 安装包或 `.sh` 安装脚本。
   - `.pkg`：双击按屏幕引导安装。
   - `.sh`：进入下载目录运行脚本，按提示安装。

#### 2. 验证安装

```bash
conda --version
python --version
```

#### 3. 搭建虚拟运行环境

**为什么要虚拟环境**：为不同项目搭建不同版本 Python 环境，达到**环境隔离**，减少项目间 Python 库潜在版本冲突。

```bash
# 搭建名为 rag 的虚拟运行环境，采用默认 Python 版本
conda create -n rag python=3.12.1

# 激活名为 rag 的虚拟运行环境
conda activate rag
```

> 建议在虚拟环境中安装 Python 第三方库，且**优先使用 `conda install` 命令**安装。

### 2.2.6 向量库

**为什么需要向量库**：需持久化存储并使用嵌入模型生成的文本向量；向量库提供快速的基于相似度的语义检索功能，是**检索阶段的重要基础设施**。开发 RAG 应用首先需考虑向量库的技术选择与环境安装。

**向量库的三种类型**：

| 类型 | 特点 | 代表 |
|------|------|------|
| 基于内存的嵌入式向量库 | 仅用于测试或原型应用 | **FAISS** |
| 本地管理的向量库 | 部署在本地自管理；可细分为嵌入式使用与 Client/Server 模式 | **Chroma**、**Milvus**、**PostgreSQL（+pgvector 插件）** |
| 云端管理的向量库 | 数据托管在云服务商，通过控制台与 API 访问 | 腾讯 **Tencent Cloud VectorDB**、**Pinecone** |

本书选用两款开源向量库：**Chroma** 与 **PostgreSQL（+pgvector）**。

#### 1. 安装 Chroma

**Chroma 定位**：强大、高性能的开源**嵌入式**向量库；提供方便接口存储与检索向量和相关元数据；提供 Python 与 JavaScript SDK。

```bash
pip install chromadb
```

**两种使用方式**：

**1）嵌入式使用方式**（直接 SDK 访问）

```python
# 测试 Chroma
import chromadb
client = chromadb.PersistentClient(path="./chroma_db")
client.heartbeat()
```

> 返回正常（一串纳秒数字）即工作正常。

**2）Client/Server 模式**（需启动 Server）

```bash
# 启动 Server
chroma run --path ./chroma_db
```

正常启动提示（服务端口为本机 8000）：

```
Running Chroma
Saving data to: ./chroma_db
Connect to chroma at: http://localhost:8000
```

客户端测试代码：

```python
# 测试 Chroma，Client/Server 模式
import chromadb
chroma_client = chromadb.HttpClient(host='localhost', port=8000)
chroma_client.heartbeat()
```

## 【面试考点】

> 本章高频面试题及参考答案要点。

### Q1：开发 RAG 应用有哪两种主要方式？如何选择？

**要点**：
- **低代码平台**（AppBuilder/百炼/Coze/FastGPT/Dify）：可视化快速开发，含完整工具栈，适合个人/小微企业知识型应用；局限：在线部署安全合规挑战、个性化不足、形态局限（主要是查询/对话）、屏蔽底层细节。
- **大模型应用开发框架**（LangChain/LlamaIndex/AutoGen）：组件化/模块化、屏蔽底层细节、预置可重用组件、强扩展性、覆盖全生命周期；适合企业级生产级 RAG。
- 选择依据：是否企业级、是否需灵活 RAG 范式、数据安全要求、个性化需求程度。

### Q2：LangChain 与 LlamaIndex 有何区别？为什么本书选 LlamaIndex？

**要点**：
- **LangChain**：生态最全（Core/Community/LangChain/Templates/LangServe/LangSmith/LangGraph），覆盖几乎所有大模型应用；缺点是学习门槛高、组件庞大、部分过度封装。
- **LlamaIndex**：面向生产级/企业级，最初专注 RAG，内置大量 RAG 优化组件与算法，使用更简单；含 LlamaHub、Templates、Eval Datasets、Data Agent、Query Pipeline。
- 选 LlamaIndex：门槛更低、更侧重 RAG、内置 RAG 优化算法完善。

### Q3：如何在本地部署开源大模型用于 RAG？

**要点**：
- 工具：**Ollama**（CPU 也能推理，支持 GPU）。
- 流程：`ollama --version` 验证 → `ollama run qwen:14b` 拉取运行 → `ollama serve` 进 API 服务模式（默认端口 11434，可用 `OLLAMA_HOST` 改端口）。
- API 调用：Ollama 兼容 OpenAI API 协议，可用 `http://localhost:11434/v1/chat/completions` 调用。

### Q4：如何解决多商业大模型 API 协议不统一、切换困难的问题？

**要点**：
- 构造对上层应用的**统一 API 平台**，简单配置自动适配多个商业大模型 API 协议——前端切换只改模型名，不改代码；还可多通道流量/负载均衡提高可用性。
- 推荐开源项目 **ONE-API**：把后端不同大模型 API 统一成 OpenAI API 协议（"OpenAI 化"）。

### Q5：常见向量库有哪些类型？如何选择？

**要点**：
- 三类：① 内存嵌入式（FAISS，仅测试/原型）；② 本地管理（Chroma/Milvus/PostgreSQL+pgvector，分嵌入式与 C/S 模式）；③ 云端管理（腾讯 VectorDB/Pinecone）。
- 选择依据：是否生产、数据量、是否需元数据过滤、运维成本、是否已有 PostgreSQL（有则 pgvector 性价比高）。

### Q6：什么是 TEI？它和 Ollama 有何区别？

**要点**：
- **TEI（Text Embeddings Inference）**是 Hugging Face 推出的**嵌入模型部署工具**（不是模型本身），也支持部署 Rerank 排序模型；功能比 Ollama 更强大，适合对嵌入质量/性能要求高的场景。
- 区别：Ollama 通用（LLM + 嵌入），TEI 专注嵌入/排序部署；TEI 用 FastAPI 暴露 API（`/docs/` 可查 Swagger）。

### Q7：为什么用 Miniconda 而不是系统 Python？

**要点**：环境隔离——为不同项目搭建不同版本 Python 虚拟环境，减少项目间库版本冲突；优先用 `conda install` 安装第三方库。

## 【易错 / 陷阱】

> 实际搭建与开发中容易踩的坑。

1. **API Key 泄漏**
   - 陷阱：把 API Key 硬编码进代码或提交到 Git。
   - 正解：用环境变量/`.env` 文件管理，加入 `.gitignore`；额度用完确保账号余额。

2. **Ollama 端口冲突**
   - 陷阱：默认端口 11434 被占用导致服务起不来。
   - 正解：用 `OLLAMA_HOST=:11435 ollama serve` 改端口。

3. **误以为 `ollama run` 只是命令行对话**
   - 陷阱：开发时还用交互命令行而非 API。
   - 正解：应用开发用 `ollama serve` 的 API 服务模式；交互仅用于测试。

4. **MacOS 上 Ollama 端口与 `ollama run` 行为混淆**
   - 陷阱：MacOS 上 `ollama run` 启动大模型会**自动进入服务模式**，不了解这点会重复启动。
   - 正解：理解 MacOS 特殊行为，避免端口/进程冲突。

5. **嵌入模型向量维度与向量库 schema 不匹配**
   - 陷阱：嵌入模型输出 1536 维，但建表时 `vector(3)` 或向量库 collection 维度设错，导致写入/检索失败。
   - 正解：建表/collection 维度必须与嵌入模型输出维度严格一致（通义千问 v1/v2 = 1536）。

6. **嵌入模型单行输入超长**
   - 陷阱：单行输入字符超过 2048 上限被截断或报错。
   - 正解：分块（Chunk）时控制大小，或用 async 版本处理大批量。

7. **TEI 在 Intel 芯片上误用 metal**
   - 陷阱：Intel 芯片机器上 `cargo install --path router -F metal` 编译失败。
   - 正解：Intel 芯片把 `metal` 改为 `mkl`；Linux 需先装 `libssl-dev gcc`。

8. **pgvector 安装前未装 Xcode Command Line Tools**
   - 陷阱：`make` 报错。
   - 正解：先 `xcode-select --install`。

9. **PostgreSQL 环境变量未生效**
   - 陷阱：`psql` 命令找不到。
   - 正解：`echo 'export PATH=...' >> ~/.zshrc && source ~/.zshrc`；注意 Apple Silicon 是 `/opt/homebrew/opt/...`，Intel 是 `/usr/local/...`。

10. **Chroma 嵌入式 vs C/S 模式混用**
    - 陷阱：用 `PersistentClient` 写入后用 `HttpClient` 读取，或反之，导致数据"丢失"。
    - 正解：两种模式数据存储位置/访问方式不同，明确选择一种。

11. **conda 与 pip 混装导致依赖冲突**
    - 陷阱：同一库 conda 装一份、pip 装一份，版本打架。
    - 正解：优先 `conda install`，conda 没有的再用 `pip install`。

12. **低代码平台用于企业级生产**
    - 陷阱：用低代码平台做企业级敏感数据应用，面临安全合规与个性化不足。
    - 正解：企业级主流 RAG 应用走开发框架路线。

13. **基座模型 / 微调模型 / API 模型混淆**
    - 陷阱：误把基座模型当对话助手用（续写而非回答）。
    - 正解：RAG 中生成阶段用经过 SFT+RLHF 的对话/指令模型，或商业 API 模型。
