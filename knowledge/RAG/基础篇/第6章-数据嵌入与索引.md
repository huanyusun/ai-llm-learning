# 第6章 数据嵌入与索引

> 本章节覆盖 RAG 的"增强（Augment）"准备阶段：理解嵌入与向量、向量存储（VectorStore）、向量存储索引（VectorStoreIndex）的 3 种构造方式与内部原理，以及文档摘要索引、对象索引、知识图谱索引、树索引、关键词表索引等 6 类索引。所有 API 以 LlamaIndex 框架为主。

---

## 0. 本章定位

数据加载与分割后形成大量知识块（Node），下一步是为这些 Node **构造索引**，以便能快速检索。向量存储索引是最常见形式（基于嵌入模型 + 向量存储实现语义检索），但并非唯一最佳形式——某些场景向量表现欠佳，因此本章也介绍其他常用索引。

---

## 6.1 理解嵌入与向量

**嵌入（Embedding）**：生成能表示文本内容及语义的多维浮点数（向量）。最大特点：不是一一对应的编码转换，而是**携带语义信息**——语义相近的文本，向量在数学上也相近，哪怕表面用词不同。这决定了向量可用于**语义检索**（按语义而非关键词召回）。

嵌入过程需要**嵌入模型**。不同模型的性能、维数、成本不同，可远程（OpenAI）或本地（TEI）部署。

**关键约束：构造索引与检索时必须使用同一个嵌入模型**，否则向量空间不匹配。

LlamaIndex 中生成向量的方式有几种，但实际开发中**大多是框架自动完成**，很少显式调用。

### 6.1.1 直接用模型生成向量

```python
# 批量接口
embeddings = embedded_model.get_text_embedding_batch(
    [node.get_content(metadata_mode=MetadataMode.EMBED) for node in nodes],
    show_progress=True
)
# 把向量填到 Node 的 embedding 字段
for node, embedding in zip(nodes, embeddings):
    node.embedding = embedding
```

要点：
- 每个 Node 都有 `embedding` 字段（在 BaseNode 中定义），是构造向量存储索引必需的属性
- 生成向量的内容要用 `get_content(metadata_mode=MetadataMode.EMBED)`，**不要直接用 `node.text`**——前者会按设置带入部分元数据

### 6.1.2 借助转换器生成向量

所有嵌入模型组件都实现了转换器接口（`__call__`），上面的代码可简化为一行：

```python
nodes = embedded_model(nodes)
```

在摄取管道中只需把嵌入模型当转换器插入：

```python
pipeline = IngestionPipeline(
    transformations=[
        splitter,
        embedded_model      # 自动生成向量
    ]
)
nodes = pipeline.run(documents=docs, show_progress=True)
```

## 6.2 向量存储（VectorStore）

向量存储用于存储向量并构造向量索引，类似关系库为快速检索构造索引。可以是：
- 轻量级嵌入式（FAISS）
- 客户端/服务器模式的本地或云向量库（Chroma、Milvus 等）

LlamaIndex 支持多种向量存储，都实现标准接口（增删 Node、语义检索等），但底层差异大（内存/本机/云端），使用时需查对应文档。所有向量存储都会实现 `add`（添加）、`query`（检索）、`delete`（删除）等基础接口。

### 6.2.1 简单向量存储（SimpleVectorStore）

基于内存的基础向量库。**不显式设置时框架默认用它**。仅建议用于测试/原型，不用于生产。

#### 1. 存储向量

```python
embedded_model = OllamaEmbedding(model_name="...", embed_batch_size=50)
Settings.embed_model = embedded_model

docs = [Document(text="...", metadata={"title": "..."}, doc_id="doc1"), ...]
splitter = SentenceSplitter(chunk_size=100, chunk_overlap=0)
nodes = splitter.get_nodes_from_documents(docs)

nodes = embedded_model(nodes)         # 生成向量

simple_vectorstore = SimpleVectorStore()
simple_vectorstore.add(nodes)         # 存储
```

`SimpleVectorStore` 内部存储：各 Node 的嵌入向量、Node id 与原 Document id 的对应关系、Node 元数据。

#### 2. 语义检索

```python
result = simple_vectorstore.query(
    VectorStoreQuery(
        query_embedding=embedded_model.get_text_embedding('什么是文心一言'),
        similarity_top_k=1
    )
)
# VectorStoreQueryResult(nodes=None, similarities=[0.805...], ids=['a6e7fe8f-...'])
```

**关键陷阱：SimpleVectorStore 的检索结果 `nodes=None`**——它只存 id 和向量，不存原 Node 完整内容。检索后需根据 id 与 node_id 对应关系重建 Node。第三方库（如 Chroma）会同时存完整 Node，更方便。

#### 3. 持久化

```python
simple_vectorstore.persist()          # 默认存到 ./storage
# 从本地加载
simple_vectorstore = SimpleVectorStore.from_persist_path('./storage/vector_store.json')
```

### 6.2.2 第三方向量存储（以 Chroma 为例）

`ChromaVectorStore` 是对 Chroma 官方 SDK 的封装。

#### 1. 存储向量

```python
from llama_index.vector_stores.chroma import ChromaVectorStore

# Server 模式
chroma = chromadb.HttpClient(host="localhost", port=8000)
collection = chroma.get_or_create_collection(name="vectorstore")

vector_store = ChromaVectorStore(chroma_collection=collection)
ids = vector_store.add(nodes)
print(f'{len(ids)} nodes ingested into vector store')
```

**stores_text 字段（重要）**：表示向量库是否直接存储 Node 文本内容。
- `stores_text=True`（如 Chroma）：检索时可直接重建完整 Node 返回
- `stores_text=False`（如 SimpleVectorStore）：需根据 id/node_id 映射重建 Node，增加复杂度

```python
# 查询记录条数（用官方接口）
count_result = collection.count()
```

#### 2. 向量检索

```python
result = vector_store.query(
    VectorStoreQuery(
        query_embedding=embedded_model.get_text_embedding('什么是语言模型'),
        similarity_top_k=1
    )
)
# 与 SimpleVectorStore 接口一致，但 nodes 字段含完整 Node（因为 Chroma 存了文本）
```

#### 3. 持久化

由底层 Chroma 保证：
- Client/Server 模式：数据在 Server 端持久化
- 嵌入模式：构造时提供 `persist_dir` 参数

## 6.3 向量存储索引（VectorStoreIndex）

**索引**：帮助在大量数据中查询目标数据的软件功能（由特定数据结构的存储与检索算法实现）。向量存储索引的目的：通过向量快速语义检索出相关 Node，把 Node 内容作为上下文交给大模型生成。

LlamaIndex 中向量的索引与语义检索能力在底层由 VectorStore + 向量库决定；为简化上层使用，封装了 `VectorStoreIndex`。

### 6.3.1 方法一：用向量存储构造（最底层）

前置：已完成加载、分割、嵌入、入库。

```python
# 加载 + 分割
docs = [Document(text="...", metadata={"title":"..."}, doc_id="doc1"), ...]
splitter = SentenceSplitter(chunk_size=100, chunk_overlap=0)
nodes = splitter.get_nodes_from_documents(docs)

# 嵌入 + 入库
nodes = embedded_model(nodes)
vector_store.add(nodes)

# 用向量存储构造索引对象
index = VectorStoreIndex.from_vector_store(vector_store)

# 测试
query_engine = index.as_query_engine()
response = query_engine.query("什么是速达飞行者")
```

工作原理：VectorStoreIndex 借助引用的 vector_store 做 `query` 检索，根据返回结果获得/重建 Node 列表，组装为大模型上下文。

### 6.3.2 方法二：用 Node 列表构造（省略嵌入+入库）

```python
# 以下两步被 VectorStoreIndex 自动完成：
# nodes = embedded_model(nodes)
# vector_store.add(nodes)

index = VectorStoreIndex(nodes)

query_engine = index.as_query_engine()
response = query_engine.query("什么是速达飞行者")
```

构造时发现 Node 无向量 → 用默认嵌入模型生成 → 自动存入默认向量库 → 返回索引对象。

#### 自定义嵌入模型与向量存储

```python
# 默认嵌入模型从 Settings.embed_model 取
# 默认向量存储是 SimpleVectorStore（打印 index.__dict__ 可见）

# 切换为 Chroma：
vector_store = ChromaVectorStore(chroma_collection=collection)
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex(nodes, storage_context=storage_context)
```

### 6.3.3 方法三：用文档直接构造（最高层封装）

```python
docs = [Document(text="...", doc_id="doc1"), Document(text="...", doc_id="doc2")]
vector_index = VectorStoreIndex.from_documents(docs)   # 一行！

query_engine = vector_index.as_query_engine()
response = query_engine.query("请解释什么是区块链技术")
print(response)
```

`from_documents` 自动完成：文档分割 → 向量构造 → 向量存储。

#### 个性化设置（在最高层封装下控制底层细节）

**1. 模型**：通过全局 `Settings` 设置大模型与嵌入模型。

**2. 底层向量存储**：

```python
storage_context = StorageContext.from_defaults(vector_store=vector_store)
vector_index = VectorStoreIndex.from_documents(docs, storage_context=storage_context)
```

**3. 文本分割器与元数据抽取器（transformations 参数）**：

```python
storage_context = StorageContext.from_defaults(docstore=docstore, vector_store=vector_store)

mySplitter = SentenceWindowNodeParser(window_size=2)
myExtractor = TitleExtractor()

vector_index = VectorStoreIndex.from_documents(
    docs,
    storage_context=storage_context,
    transformations=[mySplitter, myExtractor]
)

# 验证：查询出的 Node 元数据应含 document_title（TitleExtractor 生成）和 window（SentenceWindowNodeParser 生成）
nodes = vector_index._vector_store.query(
    VectorStoreQuery(query_str='速达飞行者？', similarity_top_k=1)
).nodes
pprint.pprint(nodes[0])
```

### 6.3.4 深入理解 VectorStoreIndex 的内部原理

三种构造方法本质相同，区别只是框架帮你做了多少工作。以 `from_documents` 为例：

```python
"""from_documents 核心逻辑"""
# 1. storage_context：传入或默认
storage_context = storage_context or StorageContext.from_defaults()

# 2. transformations：传入或默认（默认 SentenceSplitter）
transformations = transformations or transformations_from_settings_or_context(Settings, service_context)

# 3. 运行转换器做数据摄取，生成 Node（run_transformations 就是摄取管道用的方法）
nodes = run_transformations(documents, transformations, show_progress=show_progress, **kwargs)

# 4. 用 Node 构造索引对象（回到方法二）
return cls(nodes=nodes, storage_context=storage_context, ...)
```

方法二（`__init__`）核心逻辑：

```python
self._storage_context = storage_context or StorageContext.from_defaults()
self._vector_store = self._storage_context.vector_store
with self._callback_manager.as_trace("index_construction"):
    if index_struct is None:
        nodes = nodes or []
        index_struct = self.build_index_from_nodes(nodes + objects)
    self._index_struct = index_struct
```

`build_index_from_nodes` → `_add_nodes_to_index`（关键）：

```python
for nodes_batch in iter_batch(nodes, self._insert_batch_size):
    # 生成嵌入向量（如果还没有）
    nodes_batch = self._get_node_with_embedding(nodes_batch, show_progress)
    # 存入向量库
    new_ids = self._vector_store.add(nodes_batch, **insert_kwargs)
```

`_get_node_with_embedding` + `vector_store.add` 正是方法一中需要手动完成的工作——所以三种方法最终在底层统一。

**结论**：`VectorStoreIndex` 是底层依赖向量库的索引封装类，提供向量检索的高层接口，可用向量存储/Node 列表/文档三种方式构造。

## 6.4 更多索引类型

向量存储索引虽最重要，但很多场景需要结合其他索引实现更强检索能力。

### 6.4.1 文档摘要索引（DocumentSummaryIndex）

**原理**：借助大模型把传入 Node 列表生成**文档摘要（Summary）Node**，嵌入后存入向量库，提供摘要 Node 的向量检索能力，同时提供从摘要 Node 到原始 Document/基础 Node 的查询能力。

**与向量存储索引最大区别**：不直接对基础 Node 做语义检索，而是在**文档摘要层**检索，再映射到基础 Node。

```python
docs = [Document(text="...", metadata={"title":"..."}, doc_id="doc1"), ...]
doc_summary_index = DocumentSummaryIndex.from_documents(docs)

# 查询 Document 级摘要（默认包含总结 + 该文档能回答的假设性问题）
pprint.pprint(doc_summary_index.get_document_summary("doc1"))
```

个性化：

```python
storage_context = StorageContext.from_defaults(vector_store=vector_store)

doc_summary_index = DocumentSummaryIndex.from_documents(
    docs,
    storage_context=storage_context,
    summary_query="用中文描述所给文本的主要内容，同时描述这段文本可以回答的一些问题。"
)
```

可调参数：`storage_context`（向量库）、`summary_query`（生成摘要的 Prompt）、`llm`（生成摘要的大模型）、`transformations`（数据摄取转换器）。

### 6.4.2 对象索引（ObjectIndex）

**原理**：对**任意 Python 对象**（不仅是文本）做索引。底层依赖其他索引（通常是 VectorStoreIndex）实现 Node 的嵌入与检索，并通过 `ObjectNodeMapping` 维护"对象 ↔ Node"映射。

**典型应用场景**：
1. **Text-to-SQL**：大库 schema 过大，用对象索引在执行时只检索必要的 schema，避免上下文溢出
2. **Agent 工具检索**：在大量工具中只检索必要的一组交给 Agent 使用

```python
# 构造一些普通对象
obj1 = {"name": "小米", "cpu": "骁龙", "battery": "5000mAh", "display": "6.67 英寸"}
obj2 = ["iPhone", "小米", "华为", "三星"]
obj3 = (['A','B','C'], [100,200,300])
obj4 = "大模型是一种基于自然语言处理技术的生成式 AI 模型!"
objs = [obj1, obj2, obj3, obj4]

# 对象 → Node 映射
obj_node_mapping = SimpleObjectNodeMapping.from_objects(objs)
nodes = obj_node_mapping.to_nodes(objs)

# 构造对象索引（底层用 VectorStoreIndex）
storage_context = StorageContext.from_defaults(vector_store=vector_store)
object_index = ObjectIndex(
    index=VectorStoreIndex(nodes=nodes, storage_context=storage_context),
    object_node_mapping=obj_node_mapping,
)

# 检索测试
object_retriever = object_index.as_retriever(similarity_top_k=1)
results = object_retriever.retrieve("小米手机")
```

#### ObjectNodeMapping 的两个核心作用

1. 将普通对象转换为可被索引检索的 Node
2. 检索时根据 Node 找到并返回普通对象

`SimpleObjectNodeMapping` 实现：

```python
def to_nodes(self, objs):
    return [self.to_node(obj) for obj in objs]
def to_node(self, obj):
    return TextNode(text=str(obj))      # 通过 str() 转字符串构造 TextNode

def __init__(self, objs=None):
    objs = objs or []
    for obj in objs:
        self.validate_object(obj)
    # 保存 hash(str(obj)) -> obj 的映射，检索时据此反查
    self._objs = {hash(str(obj)): obj for obj in objs}
```

更简洁的构造方式（等价）：

```python
object_index = ObjectIndex.from_objects(
    objs,
    index_cls=VectorStoreIndex,
    storage_context=storage_context
)
```

### 6.4.3 知识图谱索引（PropertyGraphIndex）

**知识图谱**：基于图结构的语义知识库，描述实体、概念及关系，存入图数据库。常用"实体-关系-实体"三元组表达。

**图与图数据库基础**：
- 图 = Node（对象）+ 边（关系）
- 图数据库用 Node、边、属性表示数据，适合高度连接数据，多跳关系查询比关系库高效
- 常见图库：Neo4j、Amazon Neptune、OrientDB、TigerGraph

**知识图谱索引特点**：完全不同于向量存储索引，**底层不依赖其他索引**，是相对独立的索引类型。底层用 `PropertyGraphStore`（旧版 `GraphStore`）存储，与 VectorStore 对应。

**查询流程**：对自然语言做关键词解析 → 在知识图谱中检索相关实体与关系（可结合原始文本）→ 交给大模型生成。

```python
from llama_index.core.graph_stores import SimplePropertyGraphStore

documents = SimpleDirectoryReader(input_files=["../../data/graph.txt"]).load_data()

# 内存图存储
property_graph_store = SimplePropertyGraphStore()
storage_context = StorageContext.from_defaults(property_graph_store=property_graph_store)

# 构造知识图谱索引（带本地化持久化）
if not os.path.exists("./storage/graph_store"):
    index = PropertyGraphIndex.from_documents(
        documents,
        storage_context=storage_context
    )
    index.storage_context.persist(persist_dir="./storage/graph_store")
else:
    print('Loading graph index...')
    index = load_index_from_storage(
        StorageContext.from_defaults(persist_dir="./storage/graph_store")
    )

# 查询引擎
query_engine = index.as_query_engine(include_text=True, similarity_top_k=2)
response = query_engine.query("介绍一下西京的城市信息吧")
```

#### 与 VectorStoreIndex 的本质区别

| 维度 | VectorStoreIndex | PropertyGraphIndex |
|------|------------------|---------------------|
| 底层存储 | VectorStore（如 Chroma） | PropertyGraphStore（如 SimplePropertyGraphStore/Neo4j） |
| 处理过程 | 给每个 Node 生成向量，存内容+向量+元数据 | 借大模型提取**实体与关系（三元组）**，存到图存储 |
| 定制提取 | - | `kg_extractors` 参数 |

查看构造的三元组：

```python
graph = index.property_graph_store.graph
pprint.pprint(graph.triplets)   # 实体-关系-实体 三元组列表
```

三元组用于生成图谱的 Node 与 Relationship，检索时通过算法查找相关实体与关系作为上下文（`include_text=True` 时还携带源文本）。

生产环境推荐用 Neo4j 等成熟图数据库，而非内存版 SimplePropertyGraphStore。

### 6.4.4 树索引（TreeIndex）

**原理**：对输入 Node 列表用大模型生成汇总/摘要形成上级 Node（Parent Node），递归自底向上形成索引树（仅当叶子 Node 超过一定数量才构造上级 Node，由参数控制）。检索时根据不同检索模式从树中返回根 Node 或叶子 Node。

```python
splitter = SentenceSplitter(chunk_size=200, chunk_overlap=0)
nodes = splitter.get_nodes_from_documents(documents)

# 构造树索引
index = TreeIndex(nodes, num_children=2)
print_attrs(index.index_struct)
```

内部结构：保存根 Node、叶子 Node、父子映射 `node_id_to_children_ids`（每个 Node 对应的叶子 Node），迭代形成索引树。

检索时默认根据输入问题检索相关叶子 Node，但可指定检索方式（向量相似度 vs 大模型判断）、检索根/叶子 Node、返回数量等（详见第 7 章）。

### 6.4.5 关键词表索引（KeywordTableIndex）

**原理**：从每个 Node 提取关键词，建立"关键词 → Node 集合"映射作为索引结构。检索时提取输入问题关键词，查找最相关 Node。

**3 种关键词提取方法**：

| 组件 | 提取方法 | 中文适配 |
|------|---------|----------|
| `KeywordTableIndex` | 大模型智能提取 | 支持 |
| `SimpleKeywordTableIndex` | 正则（按空格分词） | **仅英文** |
| `RAKEKeywordTableIndex` | RAKE 轻量库 | 视库支持 |

```python
nodes = SentenceSplitter.get_nodes_from_documents(documents)

# 大模型智能提取关键词
index = KeywordTableIndex(nodes)
print_attrs(index.index_struct)

query_engine = index.as_query_engine()
response = query_engine.query("文心一言的主要应用场景有哪些？")
```

内部结构：每个关键词（如"文心一言""大语言模型"）映射到一个 Node id 集合。检索时先提取问题关键词，再按关键词查 Node，用合适算法排序后输出指定数量。

## 【面试考点】

**Q1：什么是嵌入？为什么嵌入向量能用于语义检索？嵌入模型在索引和检索时有什么约束？**
A：嵌入是生成能表示文本内容与语义的多维浮点向量。语义相近的文本，向量在数学上也相近（哪怕表面用词不同），所以可按语义而非关键词召回。**约束：构造索引与检索时必须用同一个嵌入模型**，否则向量空间不匹配，检索完全失效。

**Q2：SimpleVectorStore 和 ChromaVectorStore 的关键区别是什么？**
A：核心区别在 `stores_text` 字段。SimpleVectorStore（stores_text=False）只存 id+向量+元数据，检索结果 `nodes=None`，需根据 id/node_id 映射重建 Node。ChromaVectorStore（stores_text=True）同时存 Node 完整内容，检索可直接重建完整 Node。SimpleVectorStore 基于内存仅适合测试，生产用 Chroma/Milvus 等。

**Q3：VectorStoreIndex 有哪三种构造方式？它们的关系是什么？**
A：① `from_vector_store(vector_store)`：用已存好向量的向量库构造；② `VectorStoreIndex(nodes)`：传 Node 列表，框架自动嵌入+入库；③ `from_documents(docs)`：传 Document，框架自动分割+嵌入+入库。三者本质相同，区别只是框架帮你做了多少工作——`from_documents` 内部 `run_transformations` 后回到方法二，方法二的 `_add_nodes_to_index` 内部 `_get_node_with_embedding` + `vector_store.add` 正是方法一手动做的事。

**Q4：如何自定义 VectorStoreIndex 的嵌入模型、向量存储、分割器？**
A：嵌入模型通过全局 `Settings.embed_model` 设置；向量存储通过 `storage_context=StorageContext.from_defaults(vector_store=...)` 传入；分割器/抽取器通过 `transformations=[...]` 参数传入（这些转换器在构造前自动运行做数据摄取）。

**Q5：6 种索引分别解决什么问题？如何选型？**
A：
- **VectorStoreIndex**：通用语义检索，最常用
- **DocumentSummaryIndex**：文档级摘要检索，适合"找哪篇文档"而非"找哪段"
- **ObjectIndex**：索引任意 Python 对象（非文本），用于 Text-to-SQL schema 检索、Agent 工具检索
- **PropertyGraphIndex**：知识图谱，独立底层（PropertyGraphStore），适合复杂关系/多跳查询
- **TreeIndex**：分层摘要，适合长文档总结性问答
- **KeywordTableIndex**：关键词精确匹配，作为向量检索的补充

**Q6：知识图谱索引和向量存储索引的本质区别？**
A：① 底层存储不同：VectorStore vs PropertyGraphStore；② 数据处理不同：向量索引给每个 Node 生成向量，图谱索引借大模型提取**实体-关系-实体三元组**存入图；③ 图谱索引不依赖其他索引（独立），对象索引依赖其他索引；④ 查询方式：向量索引靠向量相似度，图谱索引靠关键词解析+图谱遍历（可选 Text-to-Cypher/向量/关键词三种检索模式）。

**Q7：ObjectIndex 如何实现对任意对象的索引？**
A：依赖底层索引（通常是 VectorStoreIndex）+ `ObjectNodeMapping`。映射做两件事：① `to_node(obj) = TextNode(text=str(obj))` 把对象序列化为文本 Node；② 用 `{hash(str(obj)): obj}` dict 保存映射。检索时向量索引召回 Node → 取文本 → 算 hash → 反查返回原对象。

**Q8：stores_text=False 的向量库会带来什么问题？**
A：检索结果不含完整 Node 内容（`nodes=None`），需要额外维护索引 id 与原始 node_id 的对应关系来重建 Node，增加构造索引与检索的复杂度。所以生产中倾向选 stores_text=True 的库。

## 【易错 / 陷阱】

1. **嵌入模型不一致 = 灾难**：构造索引和检索必须用同一个嵌入模型。换嵌入模型必须重建整个索引。

2. **生成向量要用 `get_content(metadata_mode=MetadataMode.EMBED)`，不要用 `node.text`**：前者会按 excluded_embed_metadata_keys 等设置带入部分元数据，直接用 text 会丢失元数据增强。

3. **SimpleVectorStore 检索结果 nodes=None**：不要假设所有向量库检索都返回完整 Node，stores_text=False 的库需要重建。生产环境优先选 Chroma/Milvus。

4. **不显式设置向量存储时默认 SimpleVectorStore**：`from_documents(docs)` 一行代码看似方便，但底层用的是内存向量库，重启即丢。生产必须显式传 `storage_context`。

5. **三种 VectorStoreIndex 构造方式不等价于三种索引类型**：它们是同一个 VectorStoreIndex 的三种构造便捷程度，不是不同的索引。区别只在框架代劳多少。

6. **transformations 参数在 from_documents 中才能影响分割**：用 Node 列表或 vector_store 构造时，分割已经做完了，传 transformations 无意义。

7. **知识图谱索引的提取结果不确定**：三元组由大模型抽取，每次输出可能不同（大模型不确定性），不要在测试中断言三元组内容固定。

8. **SimpleKeywordTableIndex 不支持中文**：按空格分词。中文场景必须用 KeywordTableIndex（大模型）或确认 RAKE 库支持。

9. **ObjectIndex 不是独立索引**：它依赖底层 VectorStoreIndex 等，构造时必须传入底层索引或用 `from_objects(index_cls=...)` 指定。

10. **DocumentSummaryIndex 检索的是摘要 Node 不是基础 Node**：返回的是摘要 Node 对应的基础 Node，不要混淆检索粒度。

11. **chunk_size 在 from_documents 中默认含元数据**：若元数据很大，实际文本空间被压缩，可能导致块比预期小很多。

12. **图谱索引 `include_text=True` 会携带源文本**：上下文会变长，注意 token 成本与窗口溢出。
