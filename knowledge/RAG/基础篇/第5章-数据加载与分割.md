# 第5章 数据加载与分割

> 本章节覆盖 RAG 应用的数据准备阶段：理解 Document/Node 两个核心数据抽象、用数据连接器（Reader）从多种来源加载数据、用各种分割器（Splitter/Parser）把 Document 切成 Node、用摄取管道（IngestionPipeline）串接多个转换器。所有 API 以 LlamaIndex 框架为主。

---

## 0. 本章在 RAG 流程中的位置

数据加载与分割是 RAG 应用的第一个阶段，负责把"原始知识"转化为后续可被索引、检索、生成的内存对象。典型数据源：

1. 本地文档：TXT、PDF、Office、Email、图片
2. 数据库中的结构化数据
3. 通过 URL 访问的网页/资源
4. 云存储文档（OSS/S3 等）
5. 公开接口的网络数据（社交媒体等）

输出物：`Document` 列表 → `Node` 列表 → 后续嵌入与索引。

---

## 5.1 理解两个核心概念：Document 与 Node

### 5.1.1 什么是 Document 与 Node

LlamaIndex 数据层的两个基础抽象：

- **Document（文档）**：一个通用数据容器（注意：不一定对应物理文件 File）。可装来自文档、数据库、应用系统的数据，主要存储**文本内容**与**元数据**。可手工构造，也可由数据连接器自动生成。
- **Node（节点）**：可理解为 Document 分割后的一个"块"（Chunk），可以是文本块、图像块。Node 包含元数据和相关 Node 信息。从 Document 分割出的 Node 会**继承** Document 的元数据。

#### 继承关系（重要：Document 本质上就是 Node）

```python
class BaseNode(BaseComponent):
    """Base node Object. Generic abstract interface for retrievable nodes"""

class TextNode(BaseNode):
    ......

class Document(TextNode):
    ......

# 继承链：BaseComponent -> BaseNode -> TextNode -> Document
```

Document 不是独立类型，它继承自 TextNode。二者本质同根同源，都是"存储文本内容 + 元数据的容器"。区别只是语义角色：

- Document：表示从数据源加载的"整篇"内容
- Node：表示从 Document 中切出的"块"

#### 四种 Node 类型

| 类型 | 对应类 | 说明 |
|------|--------|------|
| TEXT | `TextNode` | 基础类型，存储文本与元数据的容器 |
| IMAGE | `ImageNode` | 继承 TextNode，额外存储 base64 图像、图片 URL |
| DOCUMENT | `Document` | 继承 TextNode，包装从数据源加载的文档 |
| INDEX | `IndexNode` | 继承 TextNode，存储对其他对象（Node/Index/Retriever）的引用，用于递归检索 |

`BaseNode` 中定义了所有 Node 类型共有的属性与抽象接口，更具体的 Node 类型实现其中接口。

### 5.1.2 深入理解 Document 与 Node

```python
from llama_index.core.schema import Document
import pprint

doc = Document(

pprint.pprint(doc.dict())
```

打印出的内部结构（换成 TextNode 打印结果完全一致，因为 Document 本质就是 TextNode）：


#### Document/Node 内部重要属性

| 属性 | 说明 |
|------|------|
| `text` | 最基本属性，保存文本内容 |
| `metadata` | 元数据（描述数据的数据） |
| `id_` | 唯一 ID，可自设或构造时自动生成 |
| `relationships` | 保存相关文档/Node 信息（前后关系、父子关系） |
| `embedding` | 嵌入模型生成的向量 `List[float]`，用于构造向量索引 |
| `start_char_idx` / `end_char_idx` | 数据在原始文档中的起止位置 |

### 5.1.3 深入理解 Node 对象的元数据

**元数据（metadata）= 描述数据的数据**：原始文档、创建时间、数据库表名、创建者等。元数据特点：

1. 类型是 `dict`，由多个 key/value 组成
2. value 通常建议为字符串或数值
3. 可自行设置，也有的会被框架自动生成
4. Document 生成的 Node 会**自动携带**源 Document 的元数据
5. 元数据会和文本一起被输入嵌入模型或大模型，帮助模型控制输出

#### 元数据 → 字符串的组合机制

由于元数据会和文本一起送给模型，需要把 dict 转字符串。通过以下属性控制：

| 属性 | 作用 | 默认值 |
|------|------|--------|
| `metadata_seperator` | 多个 key/value 对之间的连接符 | `\n` |
| `metadata_template` | 每个 key/value 的转换格式 | `{key}: {value}` |
| `text_template` | 元数据字符串与文本内容的组合方式 | `{metadata_str}\n\n{content}` |
| `excluded_embed_metadata_keys` | 送入**嵌入模型**时排除的 key 列表 | `[]` |
| `excluded_llm_metadata_keys` | 送入**大模型**时排除的 key 列表 | `[]` |

#### get_content 与 MetadataMode（4 种元数据输出模式）

```python
doc4 = Document(
    text="百度是一家中国的搜索引擎公司。",
    metadata={
        "file_name": "test.txt",
        "category": "technology",
        "author": "random person",
    },
    excluded_llm_metadata_keys=["file_name"],
    excluded_embed_metadata_keys=["file_name", 'author'],
    metadata_seperator=" | ",
    metadata_template="{key}=>{value}",
    text_template="Metadata: {metadata_str}\n-----\nContent: {content}",
)

print(doc4.get_content(metadata_mode=MetadataMode.ALL))
print(doc4.get_content(metadata_mode=MetadataMode.EMBED))
print(doc4.get_content(metadata_mode=MetadataMode.LLM))
print(doc4.get_content(metadata_mode=MetadataMode.NONE))
```

```python
class MetadataMode(str, Enum):
    ALL = "all"      # 输出全部元数据
    EMBED = "embed"  # 输出嵌入模型看到的元数据
    LLM = "llm"      # 输出大模型看到的元数据
    NONE = "none"    # 不输出元数据
```

EMBED 模式下被排除了 `file_name`、`author`，只保留 `category`；LLM 模式下只排除了 `file_name`，保留 `category`、`author`。这个机制允许对嵌入和生成分别裁剪元数据。

### 5.1.5 生成 Node 对象

#### 1. 直接生成（与 Document 几乎一致）

```python
texts = ["This is a chunk1", "This is a chunk2"]
nodes = [TextNode(text=text) for text in texts]
```

#### 2. 用 Document 生成（通过分割器）

```python
docs = [Document(text='AIGC 是...\n AIGC 的发展得益于...')]

parser = TokenTextSplitter(chunk_size=100, chunk_overlap=0, separator="\n")
nodes = parser.get_nodes_from_documents(docs)

for i, node in enumerate(nodes):
    print(f"Node {i}: {node.text}")
```

#### 3. Node 对象之间的关系（5 种基础关系类型）

| 关系类型 | 含义 |
|---------|------|
| `SOURCE` | 源 Node（如 Document 对象本身） |
| `PREVIOUS` | 上一个 Node（同一 Document 分割出的） |
| `NEXT` | 下一个 Node |
| `PARENT` | 父 Node |
| `CHILD` | 子 Node |

关系信息存储在 `relationships` 属性中（dict，key 是 NodeRelationship，value 是 `RelatedNodeInfo`）。手工设置示例：

```python
nodes[0].relationships[NodeRelationship.NEXT] = \
    RelatedNodeInfo(node_id=nodes[1].node_id)
```

框架自动建立关系的典型场景：Document 分割成多个 Node 时，自动填充 SOURCE（指向源 Document）、PREVIOUS/NEXT。首节点没有 PREVIOUS，末节点没有 NEXT。

### 5.1.6 元数据的生成与抽取

#### 1. 设置与自动生成

```python
# 手工设置
doc1 = Document(text="...", metadata={"file_name": "test.txt", "category": "technology"})

# 框架自动生成（路径、文件名、类型、大小、日期等）

# {'file_path', 'file_name', 'file_type', 'file_size', 'creation_date', 'last_modified_date'}

# Node 自动继承 Document 的元数据

```

#### 2. 元数据抽取器（MetadataExtractor）

借助大模型/自定义算法生成 Node 中与内容相关的额外元数据（摘要、标题、实体、假设性问题）。目的：提供原始内容以外的语义/参考信息，提升检索与生成能力。

**1）SummaryExtractor（摘要抽取器）**

```python
from llama_index.core.extractors import SummaryExtractor

summary_extractor = SummaryExtractor(
    llm=llm,
    show_progress=False,
    prompt_template="请生成以下内容的中文摘要：{context_str}\n 摘要:",
    metadata_mode=MetadataMode.NONE
)
print(summary_extractor.extract(docs))
# [{'section_summary': '摘要：本文介绍了...'}]
```

参数：`llm`（大模型）、`prompt_template`（Prompt 模板）、`metadata_mode`（输入模式，此处 NONE 表示调用时不携带已有元数据）。

**2）QuestionsAnsweredExtractor（问答抽取器）**：生成 Node 内容能回答的假设性问题列表。输出 key 为 `questions_this_excerpt_can_answer`。

**3）TitleExtractor（标题抽取器）**：给 Node 生成标题（key 为 `document_title`）。重要特性：**标题是文档级别的**，同一 `ref_doc_id` 的多个 Node 输出的标题相同。

```python
parser = TokenTextSplitter(chunk_size=300, chunk_overlap=0, separator="\n")
nodes = parser.get_nodes_from_documents(docs)
title_extractor = TitleExtractor(llm=llm, metadata_mode=MetadataMode.NONE)
print(title_extractor.extract(nodes))
# 把抽取结果回写到 Node 元数据
titles = extractor.extract(nodes)
for idx, node in enumerate(nodes):
    node.metadata.update(titles[idx])
```

### 5.1.7 初步了解 IndexNode 类型

`IndexNode` 在 TextNode 基础上扩展，可携带指向其他对象的引用。两种指向方式：

- 通过 `index_id` 属性保存其他对象的 id（如 node id）
- 通过 `obj` 属性保存其他对象的引用（如检索器）

可指向的对象类型：Node、Index、Retriever、QueryEngine。

**核心作用**：检索出 IndexNode 时，根据其 index_id/obj 进行**递归检索**。典型用途：
1. 根据摘要 Node 找到完整内容 Node
2. 根据一级 Node 找到二级检索需要的检索器或查询引擎

（IndexNode + 递归检索在高级篇深入）

## 5.2 数据加载

**数据连接器（Reader/Loader）**：加载不同来源、不同接口协议、不同格式的数据，提取出包含内容与元数据的 Document 对象。

### 5.2.1 从本地目录中加载

`SimpleDirectoryReader` 是最简单的本地加载组件，自动按扩展名检测类型。

#### 支持的常见类型

`.csv` / `.docx` / `.ipynb` / `.jpeg` `.jpg` / `.md` / `.mp3` `.mp4` / `.pdf` / `.png` / `.ppt` `.pptx`

#### 关键参数（表 5-1）

| 参数 | 类型 | 说明 |
|------|------|------|
| `input_dir` | str | 加载的目录路径 |
| `input_files` | List | 加载的文档列表（覆盖 input_dir、exclude） |
| `exclude` | List | 排除的文档列表（支持通配符） |
| `exclude_hidden` | bool | 是否排除隐藏文档 |
| `encoding` | str | 编码（默认 utf-8） |
| `recursive` | bool | 是否加载子目录（默认 False） |
| `filename_as_id` | bool | 是否用源文档名作 doc_id |
| `required_exts` | Optional[List[str]] | 限制加载的扩展名 |
| `file_extractor` | Optional[Dict[str, BaseReader]] | 按扩展名指定底层读取器 |
| `num_files_limit` | Optional[int] | 处理文档最大数量 |
| `file_metadata` | Optional[Callable[[str], Dict]] | 根据文档名生成元数据的函数 |
| `raise_on_error` | bool | 读取错误时是否抛异常 |
| `fs` | Optional[AbstractFileSystem] | 文档系统（本地或网络） |

#### 1. 快速加载不同类型文档

```python
def print_docs(docs: list[Document]):
    print('Count of documents:', len(docs))
    for index, doc in enumerate(docs):
        print("-----")
        print(f"Document {index}")
        print(doc.get_content(metadata_mode=MetadataMode.ALL))
```

#### 3. 自定义文档阅读器（继承 BaseReader，实现 load_data）

场景：`.psql` 文档存储 SQL，加载时执行 SQL，把结果作为 Document 内容。

```python
class PSQLReader(BaseReader):
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)

    def load_data(self, file: Path, extra_info=None) -> List[Document]:
        with open(file) as f:
            content = f.read()
        result = execute_sql_and_return_results(content)   # 执行 SQL
        metadata = {'file_suffix': 'SQL'}
        if extra_info:
            metadata = {**metadata, **extra_info}
        return [Document(text=result, metadata=metadata)]
```

```python
import psycopg2
def execute_sql_and_return_results(sql: str) -> str:
    conn = psycopg2.connect(host="localhost", user="postgres",
                            password="******", database="postgres")
    cur = conn.cursor()
    cur.execute(sql)
    results = [str(result) for result in cur]
    conn.close()
    return "\n".join(results)
```

使用：`SimpleDirectoryReader(input_files=['9-test.psql'], file_extractor={".psql": PSQLReader()})`。

#### 2. 加载数据库数据（DatabaseReader）

```python
from llama_index.readers.database import DatabaseReader

db = DatabaseReader(
    scheme="postgresql", host="localhost", port="5432",
    user="postgres", password="*****", dbname="postgres"
)
docs = db.load_data(query="select * from questions")
```

源码要点（基于 sqlalchemy）：

```python
class DatabaseReader(BaseReader):
    def load_data(self, query: str) -> List[Document]:
        documents = []
        with self.sql_database.engine.connect() as connection:
            result = connection.execute(text(query))
            for item in result.fetchall():
                doc_str = ", ".join(
                    f"{col}: {entry}" for col, entry in zip(result.keys(), item)
                )
                documents.append(Document(text=doc_str))
        return documents
```

每行 → 一个 Document，字段拼成 `"col: value, col: value"` 形式。

#### 3. 自定义网络数据加载器（示例：阿里云 OSS）

```python
class AliOSSReader(BaseReader):
    def __init__(self, bucket_name, access_key_id=None,
                 access_key_secret=None, endpoint=None, *args, **kwargs):
        access_key_id = access_key_id or os.getenv('ACCESS_KEY_ID')
        # ... 初始化 oss2.Auth + oss2.Bucket
        self.bucket = oss2.Bucket(auth, endpoint, bucket_name)

    def load_data(self, object_names: List[str]) -> List[Document]:
        documents = []
        for object_pattern in object_names:
            for object_info in oss2.ObjectIterator(self.bucket):
                if fnmatch.fnmatch(object_info.key, object_pattern):   # 支持通配符
                    content = self.bucket.get_object(object_info.key).read()
                    documents.append(
                        Document(text=content, metadata={"file_name": object_info.key})
                    )
        return documents
```

## 5.3 数据分割

**分割器（Splitter）/ 解析器（Parser）**：把 Document 解析与分割成更小粒度的 Node，作为后续索引与检索的基本单元。最常见的是文本分割器（TextSplitter）。

**为什么不直接用 Document 做检索单位？** 较大的块语义完备但召回精确性低（类比关系库把所有数据存一条记录），影响生成质量。Document 层与 Node 层分离，提供更大灵活性。

### 5.3.1 如何使用数据分割器

输入：Document 列表；输出：Node 列表（每个 Node 是 Document 的子块，自动继承元数据）。

#### 1. 独立使用

```python
from llama_index.core.node_parser import SentenceSplitter

splitter = SentenceSplitter()
nodes = splitter.get_nodes_from_documents([Document(text="This is a test.\n haha!")])
```

继承链（SentenceSplitter 为例）：所有分割器继承自 `NodeParser`，后者实现 `get_nodes_from_documents` 的基本逻辑（构造 Node 间关系、继承元数据），具体分割逻辑由内部 `_split_text` 完成。

#### 2. 插入其他组件自动调用

```python
index = VectorStoreIndex.from_documents(
    documents,
    transformations=[SentenceSplitter(chunk_size=1024, chunk_overlap=20)],
)
```

### 5.3.2 常见的数据分割器

#### 1. 文本分割的 4 种底层方法

```python
from llama_index.core.node_parser.text.utils import (
    split_by_char,
    split_by_regex,
    split_by_sentence_tokenizer,
    split_by_sep,
)
```

| 方法 | 原理 | 中文适配 |
|------|------|----------|
| `split_by_sep(sep)` | 按指定分隔符分割（如 `\n`） | 好 |
| `split_by_sentence_tokenizer()` | 基于 nltk 无监督句子分割 | **不支持中文**（需训练数据） |
| `split_by_regex(pattern)` | 正则提取句子 | **适合中文** |
| `split_by_char()` | 逐字符分割 | 最粗暴，丢失语义，一般不用 |

```python
# split_by_sentence_tokenizer 内部实现
import nltk
tokenizer = nltk.tokenize.PunktSentenceTokenizer()
def split(text):
    spans = list(tokenizer.span_tokenize(text))
    sentences = []
    for i, span in enumerate(spans):
        start = span[0]
        end = spans[i+1][0] if i < len(spans)-1 else len(text)
        sentences.append(text[start:end])
    return sentences
```

框架在切完后还做：构造 Node 的位置属性、构造/继承元数据、生成 `relationships`。

#### 2. chunk_size 与 chunk_overlap（影响 RAG 优化的关键参数）

**4 个关键认知：**

1. **基于 token 数量计算，不是字符串长度**。默认用 OpenAI 的 `tiktoken`。

   ```python
   import tiktoken
   enc = tiktoken.encoding_for_model("gpt-3.5-turbo")
   print(len(enc.encode('Google 公司是一家搜索引擎公司。')))   # 12（tokens）
   print(len('Google 公司是一家搜索引擎公司。'))                # 18（字符）
   ```

2. **只是上限参数，不是绝对等值**。无法保证块大小精确等于该值。

3. **默认 chunk_size 包含元数据大小**。实际文本可用空间会比 chunk_size 小（要扣元数据的 token）。

4. **会尽量让块接近 chunk_size**。切完后执行合并（Merge）算法：

   ```python
   def _merge(self, splits: List[str], chunk_size: int) -> List[str]:
       chunks, cur_chunk, cur_len = [], [], 0
       for split in splits:
           split_len = len(self._tokenizer(split))
           if cur_len + split_len > chunk_size:
               chunk = "".join(cur_chunk).strip()
               if chunk:
                   chunks.append(chunk)
               # 保留重叠：弹出前面元素，直到 cur_len <= chunk_overlap 且不再超
               while cur_len > self.chunk_overlap or cur_len + split_len > chunk_size:
                   first_chunk = cur_chunk.pop(0)
                   cur_len -= len(self._tokenizer(first_chunk))
           cur_chunk.append(split)
           cur_len += split_len
       chunk = "".join(cur_chunk).strip()
       if chunk:
           chunks.append(chunk)
       return chunks
   ```

#### 3. TokenTextSplitter（基于分隔符的最基础分割器）

```python
splitter = TokenTextSplitter(
    chunk_size=50,
    chunk_overlap=0,
    separator="\n",
    backup_separators=["。"]
)
nodes = splitter.get_nodes_from_documents(docs)
```

**分割优先级**（依次尝试，直到每块都 < chunk_size）：
1. 基于 `separator` 分割
2. 基于 `backup_separators` 分割
3. 基于字符分割

主要参数：`chunk_size`、`chunk_overlap`、`separator`、`backup_separators`。

#### 4. SentenceSplitter（基于段落+句子，最常用）

```python
def my_chunking_tokenizer_fn(text: str):
    sentence_delimiters = re.compile(u'[。！？]')
    sentences = sentence_delimiters.split(text)
    return [s.strip() for s in sentences if s]

node_parser = SentenceSplitter(
    chunk_size=50,
    chunk_overlap=0,
    paragraph_separator="***",                  # 默认 \n\n\n
    chunking_tokenizer_fn=my_chunking_tokenizer_fn,
    secondary_chunking_regex="[^,.;。？！]+[,.;。？！]?",
    separator="\n"
)
nodes = node_parser.get_nodes_from_documents(docs)
```

**分割优先级（5 级，逐级降级）**：
1. 按段落分隔符 `paragraph_separator`（默认 `\n\n\n`）
2. 按 `chunking_tokenizer_fn` 分句（默认 nltk）
3. 按正则 `secondary_chunking_regex`（默认 `[^,.;。？！]+[,.;。？！]?`）
4. 按 `separator` 分句（默认空格）
5. 按字符（基本用不到）

如果上一级就能切成 < chunk_size 的块，就不会执行下一级。

#### 5. SentenceWindowNodeParser（句子窗口）

按句子分割，**把"周围窗口内的句子"带入元数据**（key 为 `window`），窗口大小由 `window_size` 控制。

```python
splitter = SentenceWindowNodeParser(
    window_size=2,
    sentence_splitter=my_chunking_tokenizer_fn
)
```

使用要点（关键）：
1. window 内容**默认对嵌入模型和大模型不可见**
2. **不建议**把 window 输入嵌入模型（嵌入只针对本 Node 文本，利于语义细分，提高检索精确度）
3. **建议**把 Node 内容替换成 window 内容送大模型生成（提供更多上下文，提升生成质量）

#### 6. HierarchicalNodeParser（多层 Node）

在多个粒度（不同 chunk_size）上解析，生成具备**层次关系**的 Node（自动维护父子关系）。检索出小粒度 Node 后可自动找到父 Node 替换，提供更丰富上下文。

```python
from llama_index.core.node_parser import HierarchicalNodeParser

node_parser = HierarchicalNodeParser.from_defaults(
    chunk_sizes=[2048, 100, 50]
)
nodes = node_parser.get_nodes_from_documents(docs)
# chunk_size=2048: 1 个；=100: 2 个；=50: 4 个；共 7 个
# Relationships 中保存父子 Node 信息
```

#### 7. SemanticSplitterNodeParser（基于语义与向量）

不基于分隔符/正则/固定块大小，而是**借助嵌入模型识别句子间语义相似度并合并**，确保语义相关的句子进入同一 Node，最大化保证 Node 内容的语义相关性与独立性。无需 `chunk_size` 参数。

```python
splitter = SemanticSplitterNodeParser(
    breakpoint_percentile_threshold=85,        # 阈值，越大块越少，越小块越多
    sentence_splitter=my_chunking_tokenizer_fn,
    embed_model=embed_model                    # 计算向量距离必须用嵌入模型
)
nodes = splitter.get_nodes_from_documents(docs)
```

参数：
- `breakpoint_percentile_threshold`：上下文句子向量距离阈值，距离大于阈值就分割
- `sentence_splitter`：分句方法（默认 nltk）
- `embed_model`：嵌入模型

特点：Node 数量不固定，由语义相关性决定；缺点是大文本分割慢（要算向量）。

#### 8. 分割特殊文本

```python
# Markdown
markdown_parser = MarkdownNodeParser()
# HTML
html_parser = HTMLNodeParser()
# JSON
json_parser = JSONNodeParser()
# 源代码
code_parser = CodeSplitter(language="python")
```

这些分割器会按预置规则切分（如 Markdown 自动识别标题）。`SimpleFileNodeParser` 是更上层封装，可统一处理 Markdown/HTML/JSON。

## 5.4 数据摄取管道

### 5.4.1 什么是 IngestionPipeline

把"加载 → 分割 → 抽取 → 嵌入 → 存储"抽象成一条数据处理流水线。基本思想：**插入多个转换器（Transformation），实现自动化连续处理，无须对中间过程干预**。

每个转换器：输入 Node 列表 → 处理 → 输出新 Node 列表 → 送入下一个转换器。中间结果**会被缓存**，节约重复处理时间。

```python
from llama_index.core.ingestion import IngestionPipeline, IngestionCache

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=500, chunk_overlap=0),
        TitleExtractor(llm=llm, show_progress=False)
    ]
)
nodes = pipeline.run(documents=docs)
```

等价的手工实现：

```python
splitter = SentenceSplitter(chunk_size=500, chunk_overlap=0)
nodes = splitter.get_nodes_from_documents(docs)

extractor = TitleExtractor(llm=llm, show_progress=False)
titles = extractor.extract(nodes)
for idx, node in enumerate(nodes):
    node.metadata.update(titles[idx])
```

摄取管道本质是对其他数据处理组件使用过程的抽象，让代码更优雅。

### 5.4.2 转换器（TransformComponent）

```python
class TransformComponent(BaseComponent):
    @abstractmethod
    def __call__(self, nodes: List["BaseNode"], **kwargs) -> List["BaseNode"]:
        ...

    async def acall(self, nodes, **kwargs) -> List["BaseNode"]:
        return self.__call__(nodes, **kwargs)
```

`__call__` 输入 Node 列表、输出 Node 列表，保证多个转换器可串接。符合该接口的组件：

| 组件 | 作用 |
|------|------|
| **NodeParser** | 数据分割器（TextSplitter、CodeParser 等），Document → Node 列表 |
| **MetadataExtractor** | 元数据抽取器（TitleExtractor、SummaryExtractor 等） |
| **Embedding Model** | 嵌入模型，生成 Node 文本向量 |

NodeParser 实现 `__call__`：直接调 `get_nodes_from_documents`。

```python
class NodeParser(TransformComponent, ABC):
    def __call__(self, nodes, **kwargs):
        return self.get_nodes_from_documents(nodes, **kwargs)
```

嵌入模型实现 `__call__`：批量生成向量并填回 `node.embedding`。

```python
class BaseEmbedding(TransformComponent):
    def __call__(self, nodes, **kwargs):
        embeddings = self.get_text_embedding_batch(
            [node.get_content(metadata_mode=MetadataMode.EMBED) for node in nodes],
            **kwargs,
        )
        for node, embedding in zip(nodes, embeddings):
            node.embedding = embedding
        return nodes
```

### 5.4.3 自定义转换器

```python
from llama_index.core.schema import TransformComponent
import re

class TextCleaner(TransformComponent):

        for node in nodes:
            node.text = re.sub(r"[^一-龥A-Za-z0-9，。？！“”‘’；：【】《》（）\[\]\"\'\.\,\?\!\:\;\(\)\n\r]", "", node.text)
        return nodes

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=500, chunk_overlap=0),
        TextCleaner(),   # 自定义转换器

    ]
)

```

#### 4. 文档存储与管理（重复排除）

## 5.5 完整认识数据加载阶段（核心要点回顾）

1. **数据连接器（Reader）**：不同来源文档 → Document 对象
2. **数据分割器（Parser）**：Document → 更小粒度 Node
3. **元数据**：Node 的重要信息，可自行设置或用抽取器自动抽取
4. **数据摄取管道**：组装多个转换器，实现对 Node 的自动化连续处理（分割器、抽取器、嵌入模型都可作转换器）
5. **缓存与持久化**：管道有良好缓存设计与持久化机制，支持并行处理

## 【面试考点】

**Q1：Document 和 Node 的关系是什么？为什么 LlamaIndex 要分两层？**
A：Document 继承自 TextNode（同根同源，都是"文本+元数据"容器）。Document 表示从数据源加载的整篇内容，Node 表示从 Document 切出的块。分两层是为了灵活控制检索粒度——Document 粒度大、语义完备但召回精确性低（类比关系库把所有数据放一条记录）；Node 粒度小、召回精确。两层分离让检索与生成在粒度上更可控。

**Q2：元数据（metadata）在 RAG 中起什么作用？excluded_embed_metadata_keys 和 excluded_llm_metadata_keys 有何区别？**
A：元数据会和文本一起送入嵌入模型/大模型，帮助模型控制输出（如文件名、作者、类别）。`get_content(metadata_mode=...)` 提供了 ALL/EMBED/LLM/NONE 四种模式。`excluded_embed_metadata_keys` 控制送入**嵌入模型**时排除的 key（影响检索时的向量），`excluded_llm_metadata_keys` 控制送入**生成大模型**时排除的 key（影响生成）。可以对检索和生成分别裁剪元数据。

**Q3：chunk_size 和 chunk_overlap 单位是什么？为什么是上限而不是精确值？**
A：单位是 **token 数**（不是字符长度），默认用 tiktoken 计算，且默认包含元数据的 token 大小。是上限因为：分词不是逐字符、chunk_size 还要扣除元数据、分割后会做 Merge 算法尽量逼近但不能保证精确。chunk_overlap 通过保留前一块尾部实现块间重叠，避免语义在切分边界被截断。

**Q4：SentenceSplitter 的分割优先级是怎样的？如何适配中文？**
A：5 级降级：段落分隔符（默认 `\n\n\n`）→ `chunking_tokenizer_fn` 分句（默认 nltk）→ `secondary_chunking_regex`（默认 `[^,.;。？！]+[,.;。？！]?`）→ `separator`（默认空格）→ 字符。上级能切成 < chunk_size 就不执行下级。适配中文：用 `secondary_chunking_regex` 用中文标点正则；或自定义 `chunking_tokenizer_fn`（nltk 默认不支持中文）。

**Q5：SemanticSplitterNodeParser 和普通文本分割器的本质区别？**
A：语义分割器不基于分隔符/正则/固定块大小，而是借助嵌入模型计算句子间向量距离，距离超过 `breakpoint_percentile_threshold` 就切分，保证语义相关的句子进同一 Node。无需 chunk_size，块数量由语义阈值决定。缺点：要算向量，大文本分割慢。

**Q6：IngestionPipeline 的转换器接口是什么？哪些组件可以做转换器？**
A：转换器继承 `TransformComponent`，实现 `__call__(nodes) -> List[BaseNode]`（输入输出都是 Node 列表，可串接），并可选实现 `acall` 异步版本。NodeParser、MetadataExtractor、Embedding Model 都符合该接口。管道还支持 vector_store（处理完自动入库）、docstore（去重）、num_workers（并行）。

**Q7：SentenceWindowNodeParser 和 HierarchicalNodeParser 各解决什么问题？**
A：都为了"在不改变检索精确度的前提下，给生成提供更多上下文"。SentenceWindowNodeParser 把窗口内句子放入 `window` 元数据，**检索用单句**（精确）、**生成用窗口**（上下文丰富），window 默认对模型不可见需手动注入。HierarchicalNodeParser 在多个 chunk_size 粒度上解析，自动维护父子关系，检索到小 Node 后可上溯父 Node 替换。

**Q8：如何自定义一个数据连接器/阅读器？**
A：继承 `BaseReader`，实现 `load_data(self, ...) -> List[Document]`，在其中读取数据源、生成 `Document(text=..., metadata=...)`。然后通过 `SimpleDirectoryReader(file_extractor={".ext": YourReader()})` 注入，或直接实例化调用。自定义转换器则继承 `TransformComponent` 实现 `__call__`。

## 【易错 / 陷阱】

1. **Document 与源文件并非一一对应**：PDF 默认按页生成多个 Document；Markdown 默认在标题处分割。不要假设 1 文件 = 1 Document。

2. **图片默认不做 OCR，也不做 base64**：要 OCR 需 `ImageReader(parse_text=True)`，要 base64 需 `keep_image=True`，否则元数据只有路径/时间等。

3. **`split_by_sentence_tokenizer`（nltk）不支持中文**：中文场景务必换 `secondary_chunking_regex` 或自定义 `chunking_tokenizer_fn`。

4. **chunk_size 包含元数据 token**：实际可用文本空间 < chunk_size。若元数据很大会被压缩。

5. **chunk_size/chunk_overlap 是上限不是精确值**：不要在代码里断言"块大小等于 chunk_size"。

6. **SentenceWindowNodeParser 的 window 元数据默认对模型不可见**：以为加 window 就能自动增强生成是误解，必须手动把生成上下文替换为 window 内容（或在 Node 后处理器中替换）。同时不要把 window 输入嵌入模型，否则降低检索精确度。

7. **TitleExtractor 抽取的标题是文档级别**：同一 `ref_doc_id` 的多个 Node 会得到相同标题，不要按 Node 维度去重或对比。

8. **元数据 value 推荐字符串/数值**：放复杂对象可能导致序列化/嵌入异常。

9. **num_workers 不是越大越好**：受 CPU/内存、模型并发限制；`if __name__ == '__main__'` 下要 `freeze_support()`（多进程）。

10. **SimpleVectorStore 不存储完整 Node 内容**：检索结果 `nodes=None`，只存 id+向量。stores_text=False 的库需额外映射重建 Node。生产建议用 Chroma/Milvus 等（stores_text=True）。

11. **构造索引时的嵌入模型必须与检索时一致**：否则向量空间不匹配，检索完全失效。

12. **去重依赖文档 hash**：docstore 去重判断的是文档内容 hash，仅修改元数据但不改文本可能被判为未变化。
