> 定位：从"原型"到"生产"，把组件拼装成可上线的端到端全栈 RAG 系统。

## 12.1 对生产型 RAG 应用的主要考量

原型应用只考虑"能用"，生产应用要考虑"好用、稳定、可扩展、安全"。四大核心需求：

### 1) 知识库索引的构造与使用分离
典型生产中，索引构造与使用**不在一段上下文顺序执行**，不能每次需要检索时都重建索引。知识文档的导入/维护/管理与索引的构造/优化，往往需要一个**独立管理后台**完成，并与前端应用同步协作。

### 2) 前端应用与后端服务分离
端到端 Web 应用要求前后端分离。Agent / 查询引擎 / 对话引擎的能力需通过**服务 API** 发布给前端调用，而非在管理后台直接对话。

### 3) 多用户 / 多租户使用需求
- 多用户：对话引擎独立、记忆窗口独立、索引分离。
- 多租户 SaaS：资源隔离（不同租户知识库、不同缓存与本地存储）。

### 4) 企业级非功能性需求
- 流式响应提升体验
- 缓存 + 本地存储提效
- 选择合适大模型与向量库
- 优化 Prompt/流程/模型选型降本
- 合适 API 框架与流量控制优化并发体验
- 详细调试与跟踪信息便于诊断

## 12.2 端到端的企业级 RAG 应用架构（5 层）

> 以"企业内共享智能知识问答助手"为例（前端自然语言搜索/问答，后端管理知识/索引/API/日志）。完整架构见图 12-1。

### 12.2.1 数据存储层

需持久化的数据：
1. **原始知识文档与数据**：内部文档、图片、视频、关系数据库结构化数据、网页数据、API 数据、用户上传文档等。
2. **向量**：嵌入模型生成，建议借专业向量库（Milvus、Chroma）实现持久化/检索/备份恢复。
3. **应用运行数据**：鉴权管理、用户账号、查询对话数据、系统日志、知识导入中间数据等。

存储形式：
- 关系数据库（结构化信息、管理数据、元数据）
- 向量库（知识块向量，语义检索）
- 文档（原始知识文档）
- 其他（图数据库存知识图谱等）

通过专有管理接口提供给上层。

### 12.2.2 AI 模型层

- **大模型**：核心引擎，用于查询转换、响应生成、语义路由。
- **嵌入模型**：知识→高维向量。
- **其他模型**：如 Rerank 重排模型。

通过 API 开放使用接口；在 LangChain / LlamaIndex 中无需关心 API 差异，用框架封装的 LLM 组件即可。

### 12.2.3 RAG 工作流与 API 模块（核心模块）

端到端 Web 形态下常不作为独立运行程序，而是以 **API 形式**提供服务。能力：

1. **检索前处理**：从接收查询到索引检索前的处理（查询转换、语义路由等），提升检索精度与生成质量。（参考 8.1 节）
2. **检索**：借助索引用合适算法/流程/范式召回相关上下文块。（参考第7章、11.4 节）
3. **检索后处理**：对检索块做融合、去重、重排等，组装进 Prompt 给生成器。（参考 8.2 节、11.4 节）
4. **生成**：接收相关上下文 + 问题，用合适响应生成算法生成结果。（参考第7、9章）
5. **API 发布**：借 Web 框架（FastAPI、Flask）把上述能力发布为 HTTP API，提供安全与流控。常见 API：对话接口、历史对话查询接口、文档上传接口、知识库管理接口等。

### 12.2.4 前端应用模块

用 Vue/React 构建，底层 HTML/JS/CSS。典型 ChatBot UI（图 12-2）。核心功能：
1. **账号管理**：注册、绑定、登录、改密、注销。
2. **对话与会话管理**：通过 API 与后端交互实现连续对话，管理 Session/上下文/历史/归档/备份/删除；可按业务模块化（知识对话、交互式数据查询、自定义文档对话）。
3. **ChatBot 管理**：不同风格/类型/能力 ChatBot 的构造、配置、删除。
4. **配置项**：UI 风格、对话保存时间、本地目录、大模型选择、Prompt 自定义等。

### 12.2.5 后台管理模块

针对不同使用对象在不同 RAG 阶段做应用分离；把原型中的硬编码转为可视化管理。后台模块分 4 类：

#### 1) 索引与知识库管理模块
索引准备是前端运行前提。管理员用它完成知识文档的读取/加载/索引，以及辅助处理（假设性问题生成、摘要生成）。核心功能：知识库创建、导入、索引、备份；辅助：检索测试、模型配置、重复筛选。

#### 2) 应用与 API 管理模块
核心是提供 RAG 流程的灵活配置：
- 多应用构建与配置
- API 发布、鉴权与日志管理（端点与 Key 维护、日志查询监控）
- 应用测试（模拟测试、跟踪、调试）
- 可视化 RAG 工作流编排（拖曳组件、链接配置、嵌入式代码）

#### 3) 模型管理模块
管理大模型、嵌入模型、Rerank 模型的调用入口与参数。模型分在线 API 模型与本地大模型；复杂场景可能多模型（不同阶段用不同来源模型）。功能：连通性测试、单位成本配置、底层日志跟踪。

#### 4) 系统管理模块
账号管理、安全权限、系统日志、存储层目录管理、数据库连接与参数配置等。

## 12.3 端到端的全栈 RAG 应用案例

### 12.3.1 简单的全栈 RAG 查询应用

#### 1. 技术栈
- 本地 Ollama 大模型 Qwen2
- OpenAI 嵌入模型 text-embedding-3-small
- Chroma 向量库
- LlamaIndex 框架
- FastAPI（API Server）
- React + TypeScript（前端 UI）

> 演示全栈前后端开发与交互；不考虑多用户账号管理、数据隔离、关系数据库、并发控制。

#### 2. 后端模块之 Index Server（索引服务）

##### 整体模块设计（4 个功能）
- `initialize_index`：加载或初始化索引（动态插入实现增量更新 + 每次更新后持久化；启动时直接加载已构造索引）。
- `insert_into_index`：文档检验、读取、分割并插入已有索引。
- `query_index`：构造查询引擎，输入问题返回结果（默认检索器+生成器）。
- `get_documents_list`：查询已上传并构造索引的文档列表（减少重复上传）。

##### 索引的共享访问设计
索引是**可写**的（用户上传文档），涉及并发：
- 隔离不同使用者索引（按 token 分配不同存储空间）
- 多用户共享索引（需顺序访问，加 lock）

##### 独立 Index Server 模块设计
通过独立可远程调用的 Index Server 控制顺序访问。好处：
- API Server 与 Index Server 功能解耦，便于分别管理/调试/调优。
- 灵活分布式部署，可部署在不同物理设备。

> 借 Python 的 `multiprocessing.managers` 模块实现。

##### 模块实现

```python
import os, pickle, chromadb
from multiprocessing import Lock
from multiprocessing.managers import BaseManager
from llama_index.core import Settings, SimpleDirectoryReader, VectorStoreIndex, StorageContext, load_index_from_storage
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.llms.ollama import Ollama

# 准备模型
llm_ollama = Ollama(model='qwen2')
embedded_model_openai = OpenAIEmbedding(model_name="text-embedding-3-small", embed_batch_size=50)
Settings.llm = llm_ollama
Settings.embed_model = embedded_model_openai

# 全局共享数据
index = None
stored_docs = {}
lock = Lock()                                  # 线程安全
index_name = "./saved_index"
pkl_name = "stored_documents.pkl"
SERVER_PORT = 5602

# 初始化索引（不对前端开放）
def initialize_index():
    global index, stored_docs
    chroma = chromadb.HttpClient(host="localhost", port=8000)
    collection = chroma.get_or_create_collection(name="chat_docs_collection")
    vector_store = ChromaVectorStore(chroma_collection=collection)
    with lock:
        if os.path.exists(index_name):
            storage_context = StorageContext.from_defaults(persist_dir=index_name, vector_store=vector_store)
            index = load_index_from_storage(storage_context=storage_context)
        else:
            storage_context = StorageContext.from_defaults(vector_store=vector_store)
            index = VectorStoreIndex([], storage_context=storage_context)
            index.storage_context.persist(persist_dir=index_name)
        if os.path.exists(pkl_name):
            with open(pkl_name, "rb") as f:
                stored_docs = pickle.load(f)

# 查询
def query_index(query_text):
    global index
    response = index.as_query_engine().query(query_text)
    return response

# 插入文档
def insert_into_index(doc_file_path, doc_id=None):
    global index, stored_docs
    document = SimpleDirectoryReader(input_files=[doc_file_path]).load_data()[0]
    if doc_id is not None:
        document.doc_id = doc_id
    with lock:
        index.insert(document)
        index.storage_context.persist(persist_dir=index_name)
        stored_docs[document.doc_id] = document.text[0:200]   # 简化只存前 200 字
        with open(pkl_name, "wb") as f:
            pickle.dump(stored_docs, f)

# 查询文档列表
def get_documents_list():
    global stored_docs
    documents_list = [{"id": doc_id, "text": doc_text} for doc_id, doc_text in stored_docs.items()]
    return documents_list

# 启动分布式 server
if __name__ == "__main__":
    print("initializing index...")
    initialize_index()
    print(f'Create server on port {SERVER_PORT}...')
    manager = BaseManager(('', SERVER_PORT), b'password')
    manager.register('query_index', query_index)
    manager.register('insert_into_index', insert_into_index)
    manager.register('get_documents_list', get_documents_list)
    server = manager.get_server()
    print("server started...")
    server.serve_forever()
```

**关键点**：
- `with lock` 保证多用户对共享 index/stored_docs 的互斥顺序访问（Index Server 被远程调用时产生多个并发线程）。
- 简化代码只读 `load_data()[0]`，PDF 等多 Document 需循环处理且注意 doc_id 唯一性。
- Index Server 与 API Server 分离部署时，需保证 Index Server 能访问 `doc_file_path`（如共享网络存储 + 自定义阅读器）。
- `serve_forever` 监听到调用会起新线程处理，处理完自动结束。

#### 3. 后端模块之 API Server（FastAPI）

FastAPI：现代高性能 Web 框架，自动文档生成（Pydantic 验证）、异步高并发。

##### API 设计（表 12-1）

| API | 类型 | 端点 | 参数 | 请求体 |
| --- | --- | --- | --- | --- |
| 测试 | GET | / | 无 | 无 |
| 问题查询 | GET | /query | text：输入问题 | 无 |
| 文档上传 | POST | /uploadFile | filename_as_doc_id | file：上传文档 |
| 文档查询 | GET | /getDocuments | 无 | 无 |

##### API 实现

```python
import os, uvicorn
from multiprocessing.managers import BaseManager
from werkzeug.utils import secure_filename
from fastapi import FastAPI, Request, UploadFile, File
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()
# 启用 CORS
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])

# 连接 Index Server（端口/密码须一致）
manager = BaseManager(('', 5602), b'password')
manager.register('query_index')
manager.register('insert_into_index')
manager.register('get_documents_list')
manager.connect()

@app.get("/")
def home():
    return "Hello, World!"

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request, exc):
    return JSONResponse(content="Invalid request parameters", status_code=400)

# 问题查询接口
@app.get("/query/")
def query_index(request: Request, query_text: str):
    global manager
    if query_text is None:
        return JSONResponse(content="No text found...", status_code=400)
    response = manager.query_index(query_text)._getvalue()
    response_json = {
        "text": str(response),
        "sources": [{"text": str(x.text), "similarity": round(x.score, 2), "doc_id": str(x.id_)}
                    for x in response.source_nodes]
    }
    return JSONResponse(content=response_json, status_code=200)

# 文档上传接口
@app.post("/uploadFile")
async def upload_file(request: Request, file: UploadFile = File(...), filename_as_doc_id: bool = False):
    global manager
    try:
        contents = await file.read()
        filepath = os.path.join('documents', file.filename)
        with open(filepath, "wb") as f:
            f.write(contents)
        if filename_as_doc_id:
            manager.insert_into_index(filepath, doc_id=file.filename)
        else:
            manager.insert_into_index(filepath)
    except Exception as e:
        if os.path.exists(filepath):
            os.remove(filepath)
        return JSONResponse(content="Error: {}".format(str(e)), status_code=500)
    if os.path.exists(filepath):
        os.remove(filepath)             # 删除临时文件
    return JSONResponse(content="File inserted!", status_code=200)

# 文档查询接口
@app.get("/getDocuments")
def get_documents(request: Request):
    document_list = manager.get_documents_list()._getvalue()
    return JSONResponse(content=document_list, status_code=200)

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=5601)
```

- 远程调用与本地函数调用一致（如 `manager.query_index(...)`），需用 `_getvalue` 取返回值。
- `UploadFile` 是 FastAPI 处理上传文件的类型。
- FastAPI 自动生成交互式 API 文档：`http://localhost:5601/docs`，可在 `/docs` 直接 Try it out 测试。

```typescript
export type ResponseSources = { text: string; doc_id: string; similarity: number; };
export type QueryResponse   = { text: string; sources: ResponseSources[]; };

const queryIndex = async (query: string): Promise<QueryResponse> => {
  const queryURL = new URL('http://localhost:5601/query?');
  queryURL.searchParams.append('query_text', query);
  const response = await fetch(queryURL, { mode: 'cors' });
  if (!response.ok) return { text: 'Error in query', sources: [] };
  return (await response.json()) as QueryResponse;
};
export default queryIndex;
```

#### 5. 小结

端到端全栈与单模块的核心区别：
- 分离前端 UI / 后端 API / 知识库索引服务
- 分布式设计（独立索引服务器）
- 持久化存储 + 动态插入索引（非一次性生成）
- 多用户并发（索引顺序化操作，加锁）

可继续扩展：多用户独立知识库/索引、多轮对话引擎、上传到网络共享存储、独立后台管理。

### 12.3.2 基于多文档 Agent 的端到端对话应用

> 在 11.3.3 多文档 Agentic RAG 基础上升级，支持复杂提问（事实性→总结性、单文档→跨文档）+ 连续对话 + 流式输出 + 多用户多租户。

#### 1. 基本架构（图 12-11）

3 个模块：
- **后端 Agent 模块**：核心，给多文档构造索引/查询引擎，以查询引擎为工具创建上层 Agent。通过模块导入 API 模块。
- **后端 API 模块**：前端直接访问的 API，与 Agent 对话。
- **前端 UI 应用**：支持连续对话的 ChatBot，实现流式响应。

目录结构：
- `api/`：面向前端的 API 服务代码
- `app/`：后端数据查询引擎与 Agent 代码
- `data/`：原始知识文档
- `main.py`：后端主程序入口

#### 2. 后端 Agent 模块

##### 2) 创建 Top Agent（带工具检索功能）

```python
def _create_top_agent(doc_agents: Dict, callback_manager: CallbackManager):
    all_tools = []
    for city in doc_agents.keys():
        city_summary = f"这部分包含了有关{city}的城市信息. 如果需要回答有关{city}的任务问题，请使用这个工具.\n"
        doc_tool = QueryEngineTool(
            query_engine=doc_agents[city],
            metadata=ToolMetadata(name=f"tool_{city}", description=city_summary))
        all_tools.append(doc_tool)

    tool_mapping = SimpleToolNodeMapping.from_objects(all_tools)
    if not os.path.exists(f"{STOR_DIR}/top"):
        storage_context = StorageContext.from_defaults()
        obj_index = ObjectIndex.from_objects(all_tools, tool_mapping, VectorStoreIndex, storage_context=storage_context)
        storage_context.persist(persist_dir=f"{STOR_DIR}/top")
    else:
        storage_context = StorageContext.from_defaults(persist_dir=f"{STOR_DIR}/top")
        index = load_index_from_storage(storage_context)
        obj_index = ObjectIndex(index, tool_mapping)

    top_agent = ReActAgent.from_tools(
        tool_retriever=obj_index.as_retriever(similarity_top_k=3),
        verbose=True,
        system_prompt="你是一个被设计来回答关于一组给定城市查询的助手...",
        callback_manager=callback_manager)
    return top_agent
```

##### 3) 基础流式版本（StreamingResponse + SSE）

**SSE（Server-Sent Events）**：基于 HTTP 从服务端向客户端单向推送实时数据，无须客户端询问，适用于流式响应、大文档推送。

```python
@r.post("")
async def chat(data: _ChatData):
    # ... 校验与消息构造（省略）
    chat_result = agent.stream_chat(lastMessage.content, messages)   # stream_chat 而非 chat

    def event_generator():
        for token in chat_result.response_gen:    # response_gen 是流数据生成器
            yield convert_sse(token)
    return StreamingResponse(event_generator(), media_type="text/event-stream")

def convert_sse(obj: str | dict):
    return "data: {}\n\n".format(json.dumps(obj, ensure_ascii=False))   # SSE 要以 "data:" 开头
```

##### 4) 升级流式版本（事件跟踪 + 异步队列）

> 普通流式只在最后生成阶段输出，看不到 Agent 中间过程。升级版用 LlamaIndex 内置**事件跟踪与派发机制**（参考 3.2 节），通过自定义事件处理器把中间事件推到异步队列，由专门线程从队列取数据实时推送（图 12-19）。优势：复用已有机制，异步减少侵入性。

**步骤**：
1. 自定义事件处理器，把关注事件推到异步队列。
2. 创建 Agent 时传入自定义事件管理器。
3. API 调用时访问队列实时生成响应。

**(1) 自定义事件处理器**（继承 `BaseCallbackHandler`）：

```python
class EventObject(BaseModel):
    type: str
    payload: dict

class StreamingCallbackHandler(BaseCallbackHandler):
    def __init__(self, queue: Queue) -> None:
        super().__init__([], [])
        self._queue = queue

    def on_event_start(self, event_type, payload=None, event_id="", parent_id="", **kwargs):
        # 跟踪工具使用（ReAct）/ Function Call（OpenAI Agent）
        if event_type == CBEventType.FUNCTION_CALL:
            self._queue.put(EventObject(
                type="function_call",
                payload={"arguments_str": str(payload["function_call"]), "tool_str": str(payload["tool"].name)}))

    def on_event_end(self, event_type, payload=None, event_id="", **kwargs):
        if event_type == CBEventType.FUNCTION_CALL:
            self._queue.put(EventObject(
                type="function_call_response",
                payload={"response": payload["function_call_response"]}))
        elif event_type == CBEventType.AGENT_STEP:     # Agent 每步响应
            self._queue.put(payload["response"])

    @property
    def queue(self): return self._queue
    def start_trace(self, trace_id=None): pass
    def end_trace(self, trace_id=None, trace_map=None): pass
```

关注两类事件：
- `FUNCTION_CALL`：触发工具使用/Function Call 时派发，跟踪 Agent 对工具（RAG 查询引擎）的输入输出。
- `AGENT_STEP`：Agent 每个执行步骤结束时派发，跟踪每步详细信息与最终输出（通常在获工具输出后借大模型完成）。

**(2) 使用自定义事件管理器**：

```python
def get_agent():
    queue = Queue()
    handler = StreamingCallbackHandler(queue)
    callback_manager = CallbackManager([handler])
    doc_agents = _create_doc_agents(callback_manager)
    top_agent = _create_top_agent(doc_agents, callback_manager)
    return top_agent
```

**(3) API 实现（异步读队列）**：

```python
@r.post("")
async def chat(data: _ChatData):
    agent = get_cached_agent(user_id)
    # ... 校验与消息构造（省略）

    # Agent 执行放新线程，主线程从共享队列读
    thread = Thread(target=agent.stream_chat, args=(lastMessage.content, messages))
    thread.start()

    def event_generator():
        queue = agent.callback_manager.handlers[0].queue
        while True:
            next_item = queue.get(True, 60.0)
            if isinstance(next_item, EventObject):       # FUNCTION_CALL 事件
                yield convert_sse(dict(next_item))
            elif isinstance(next_item, StreamingAgentChatResponse):   # AGENT_STEP 事件
                response = cast(StreamingAgentChatResponse, next_item)
                for token in response.response_gen:      # 迭代流式响应
                    yield convert_sse(token)
                break
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

#### 5. 未来的优化（服务端记忆）

当前靠客户端每次携带完整对话历史（`messages: List[_Message]`）实现多轮对话，缺点：
- 历史不持久化（依赖浏览器本地存储）
- 对话变长后传输越来越多，降低接口性能

优化方向：
1. 增加用户管理，服务端（Postgres/MySQL）保存对话历史与明细。
2. API 增加安全机制（token 识别用户），从数据库/缓存（Redis）取历史，或用 Agent 自身记忆能力缓存。
3. 客户端每次只携带本次问题，简化接口。

按用户缓存 Agent（利用 Agent 自身记忆，而非每次 `get_agent` 重建）：

## 【面试考点】

### Q1：生产型 RAG 应用的四大核心考量？
**要点**：①索引构造与使用分离（独立管理后台）；②前后端服务分离（API 发布）；③多用户/多租户（对话/记忆/索引隔离、租户资源隔离）；④非功能性需求（流式、缓存、模型/向量库选型、降本、流控、调试跟踪）。

### Q2：端到端企业级 RAG 的 5 层架构？
**要点**：数据存储层（关系库/向量库/文档/图库）、AI 模型层（LLM/嵌入/Rerank）、RAG 工作流与 API 模块（检索前/检索/检索后/生成/API 发布）、前端应用模块（账号/对话会话/ChatBot/配置）、后台管理模块（索引知识库/应用API/模型/系统）。

### Q3：为什么要把 Index Server 独立出来？
**要点**：①API Server 与 Index Server 功能解耦，便于分别管理/调试/调优；②灵活分布式部署，可部署在不同物理设备；③独立模块中控制顺序访问共享索引（加锁），多用户并发安全。

### Q4：`multiprocessing.managers.BaseManager` 怎么实现远程调用？
**要点**：构造 `BaseManager((host, port), authkey)` → `register('func_name', func)` 注册函数 → `get_server().serve_forever()` 监听。客户端 `register('func_name')` + `connect()` 后用 `manager.func_name(...)._getvalue()` 调用（`_getvalue` 取真实返回值）。

### Q5：FastAPI 的优势？
**要点**：高性能、Pydantic 自动数据校验、异步高并发、自动生成交互式 API 文档（`/docs`，可 Try it out）、CORS 中间件、StreamingResponse 流式响应、APIRouter 模块化路由组织。

### Q6：流式响应的实现方式？
**要点**：FastAPI `StreamingResponse` + 生成器函数（`yield` 逐步产生 token）+ SSE（`text/event-stream`，数据以 `data:` 开头）。LlamaIndex 中 Agent 的 `stream_chat` 返回结果用 `response_gen` 迭代流数据。

### Q7：升级版流式怎么跟踪 Agent 中间过程？
**要点**：用 LlamaIndex 内置事件跟踪与派发机制，自定义 `BaseCallbackHandler`，在 `FUNCTION_CALL` 与 `AGENT_STEP` 事件触发时把信息推到异步队列；Agent 执行放新线程，主线程从共享队列读队列对象，按类型（EventObject / StreamingAgentChatResponse）分别处理推送客户端。优势是复用机制、异步低侵入。

### Q8：多用户对话怎么实现？为什么要服务端记忆？
**要点**：原型靠客户端每次携带完整消息历史（兼容大模型对话 API）。生产应加用户管理 + 服务端持久化历史（Postgres/Redis）+ token 鉴权 + Agent 自身记忆/缓存（如 `lru_cache` 按 user_id 缓存 Agent），客户端只传本次问题，简化接口、提性能。

### Q9：Tool Agent 用 ReActAgent 还是 OpenAIAgent？
**要点**：ReActAgent 通用不依赖函数调用；OpenAIAgent 需大模型支持函数调用（更高效）。Top Agent 文档多时务必加 ObjectIndex 工具检索器（`tool_retriever`）缩小选择范围。

### Q10：APIRouter 与直接在 app 上定义路由的区别？
**要点**：APIRouter 可在大型系统中对大量路由模块化组织管理，便于扩展；通过 `app.include_router(router, prefix="...")` 包含进主程序。

## 【易错 / 陷阱】

1. **Index Server 与 API Server 端口/密码必须一致**：`BaseManager` 连接时端口与 authkey 不一致会异常。
2. **远程返回值必须 `_getvalue()`**：直接用返回值会是代理对象，不是真实数据。
3. **共享索引必须加 lock**：Index Server 被多用户远程调用产生多并发线程，对全局 index/stored_docs 访问不加锁会冲突异常。
4. **PDF 多 Document 上传**：简化代码只读 `load_data()[0]`，PDF 可能解析为多个 Document，需循环处理且保证 doc_id 唯一。
5. **Index Server 与 API Server 分离部署时**：必须保证 Index Server 能访问 `doc_file_path`（共享存储 + 自定义阅读器）。
6. **CORS 必须启用**：前后端分离时跨源请求会被浏览器拦截，需 `CORSMiddleware`。
7. **`get_agent` 返回的 Agent 无状态**：非流式版本必须让客户端携带完整对话历史；最后一条消息 role 必须是 user 否则抛异常。
8. **SSE 数据格式**：必须以 `data:` 开头并以 `\n\n` 结尾，否则客户端无法解析；中文需 `ensure_ascii=False`。
9. **流式升级版必须 `logits_all=True` 与 `logprobs` 参数**：但本场景是事件跟踪，主要确保 `CallbackManager` 正确注入；线程间队列用阻塞 `get(True, 60.0)` 设超时避免死锁。
10. **`include_router` 必须在主程序**：APIRouter 单独定义后，忘记 `include_router` 则路由不生效。
11. **`stream_chat` vs `chat`**：流式必须用 `stream_chat`，返回结果的 `response_gen` 才是流生成器。
12. **Agent 缓存按 user_id**：`lru_cache` 缓存 Agent 时 key 必须是 user_id，否则不同用户串话；maxsize 要结合内存与并发用户数。
13. **临时上传文件要删除**：上传接口处理完务必删除临时文件，否则磁盘泄漏。
