## 0. 为什么需要"企业级优化"

10 分钟就能搭出一个原型，但要做成 **Production Ready** 的系统很难。企业场景的四大痛点：

1. **数据复杂**：多形态、多来源、多特点的数据难以用简单一致方式处理。
2. **海量文档**：成百上千份文档带来更高的精确检索要求。
3. **业务复杂**：任务形态不止事实性问答（还有摘要、对比、跨文档、复合任务）。
4. **工程化高**：对响应时间、准确性、合规性、并发、成本等有更高要求。

本章围绕 **5 大优化方向**展开：
- 知识块大小（chunk_size）
- 检索块与生成块分离（4 种策略）
- 大文档集知识库的检索（3 种方案）
- 高级检索方法（融合检索 / 递归检索）

## 11.1 选择合适的知识块大小

### 11.1.1 为什么 chunk_size 很重要

`chunk_size` 是把原始文档切分成 Chunk（LlamaIndex 中即 **Node**）的大小参数，也是后续从向量库检索单位，直接影响检索与响应质量。

| chunk_size | 优点 | 风险 |
| --- | --- | --- |
| **越小** | 粒度小、语义精确 | 上下文少；top_K 小时重要信息可能不在顶部 |
| **越大** | 上下文完整 | 语义不精确；性能下降；上下文窗口溢出；token 成本升高（超大上下文模型可缓解） |
| **与任务相关** | 事实性问答：少量特定块；摘要/对比/总结：更大块甚至全量块 | 一刀切难以适配所有任务 |

本质：在**不牺牲性能的前提下尽量捕获最重要信息**的"平衡"问题。

> 企业级最佳实践：用**评估框架 + 评估数据集**对多个 chunk_size 做指标评估，再做选择（如 RAGAS 或 LlamaIndex 的 Evaluation 模块）。

### 11.1.2 评估知识块大小（实操）

**数据集**：LlamaHub 上的 `MiniTruthfulQADataset`，用 `download_llama_dataset` 或 `llamaindex-cli` 下载到本地 data 目录。

**考察指标**：忠实度（Faithfulness）、正确性（Correctness）、相关性（Relevancy），同时跟踪响应延迟。

#### 1) 评估主程序

```python
vector_store = ChromaVectorStore(chroma_collection=collection)

# 知识文档
eval_documents = SimpleDirectoryReader(
    "../../data/MiniTruthfulQADataset/source_files/").load_data()

# 评估数据集
eval_questions = LabelledRagDataset.from_json(
    "../../data/MiniTruthfulQADataset/rag_dataset.json")

# 评估组件
faithfulness = FaithfulnessEvaluator()
relevancy    = RelevancyEvaluator()
correctness  = CorrectnessEvaluator()

for chunk_size in [128, 1024, 2048]:
    avg_time, avg_faithfulness, avg_relevancy, avg_correctness = \
        evaluate_response_time_and_accuracy(chunk_size)
    print(f"Chunk size {chunk_size} ...")
```

#### 2) 评估方法 `evaluate_response_time_and_accuracy`

```python
def evaluate_response_time_and_accuracy(chunk_size, num_questions=100):
    total_response_time = total_faithfulness = 0
    total_relevancy = total_correctness = 0

    # 用 chunk_size 切分 Node
    node_parser = SentenceSplitter(chunk_size=chunk_size, chunk_overlap=0)
    nodes = node_parser.get_nodes_from_documents(eval_documents, show_progress=True)
    storage_context = StorageContext.from_defaults(vector_store=vector_store)
    vector_index = VectorStoreIndex(nodes, storage_context=storage_context)
    query_engine = vector_index.as_query_engine(top_K=3)

    for index, question in enumerate(eval_questions.examples[:num_questions]):
        start_time = time.time()
        response_vector = query_engine.query(question.query)
        elapsed_time = time.time() - start_time

        # 三大评估
        faithfulness_result = faithfulness.evaluate_response(
            query=question.query, response=response_vector).score
        relevancy_result = relevancy.evaluate_response(
            query=question.query, response=response_vector).score
        # 正确性评估需提供参考答案
        correctness_result = correctness.evaluate_response(
            query=question.query, response=response_vector,
            reference=question.reference_answer).score

        total_response_time += elapsed_time
        total_faithfulness  += faithfulness_result
        total_relevancy     += relevancy_result
        total_correctness   += correctness_result

    n = num_questions
    return (total_response_time/n, total_faithfulness/n,
            total_relevancy/n, total_correctness/n)
```

> 评估消耗 token 较多，建议测试时用本地大模型。最终对比各 chunk_size 的统计结果做最优选择。

## 11.2 分离检索阶段的知识块与生成阶段的知识块

### 11.2.1 为什么需要分离

单一 chunk_size 难以两全：
1. 一个块可能含生成所需的详细信息，但关键词噪声导致**检索偏差**。
2. 问答型应用：问题简单适合精确检索少量块，但**生成时需要更多上下文**才能完整作答。

**核心思想**：给大模型输入的块 ≠ 直接检索出的 top-K 块。用**小块提升检索精度，扩展到大块保证生成上下文丰富**。

### 11.2.2 常见的 4 种分离策略

经典 RAG 中检索块 = 生成块（图 11-2）。下面 4 种策略打破这一等式。

---

#### 策略 1：从问题扩充到完整的问答对

**适用**：原始知识是大量结构化问答对。只对**问题**做向量嵌入，检索出问题后，把**答案**一起喂给大模型生成（图 11-3）。

预处理技巧：
- 用大模型生成更多相似问法，提高语义召回。
- 用大模型生成文档的假设性问答对，模拟问答格式。

**实现要点（LlamaIndex Node 灵活输出）**：问题作为 Node 内容嵌入与生成；答案放 `metadata`；嵌入时排除答案，生成时取回。

```python
def read_csv_file(file_path):
    nodes = []
    with open(file_path, 'r') as file:
        csv_reader = csv.reader(file)
        for row in csv_reader:
            question, answer = row[0], row[1]
            node = TextNode(text=question, metadata={"answer": answer})
            # 嵌入时不带入答案
            node.excluded_embed_metadata_keys = ["answer"]
            node.text_template = "{content}\n{metadata_str}\n"
            nodes.append(node)
    return nodes

# 观察两种模式下的内容差异
for node in nodes:
    print(node.get_content(metadata_mode=MetadataMode.EMBED))  # 仅 question
    print(node.get_content(metadata_mode=MetadataMode.LLM))     # question + answer
```

#### 策略 2：扩充知识块所在的上下文窗口

**思想**：检索出小块后，把指定窗口大小内的前后相邻块也输入大模型（图 11-4）。窗口=3 时，读出前后各 3 块一起送入。小块精确召回，大窗口保证上下文。

**实现**：`SentenceWindowNodeParser`（第5章）在 Node 元数据中存扩展后的窗口，再用节点后处理器动态替换。

```python

```

#### 策略 3：从知识摘要扩充到知识内容

**思想**：构造两类块——摘要块 + 内容块。**检索基于摘要**，命中摘要块后通过关联拿到对应内容块用于生成（图 11-5）。适合原始知识细节多、问题输入相对概括单一的场景。

**实现**：`DocumentSummaryIndex`。构造时借大模型生成 Document 级摘要，构造摘要 Node 嵌入索引；检索时先找到摘要 Node，再按 `doc_id` 找到该 Document 的**全部**原始 Node 用于生成。

> 注意：`DocumentSummaryIndex` 检索出的原始 Node 是该 Document 的全部 Node，**不做二次检索**；若需二次检索，要用递归检索（见 11.4）。

两种摘要检索模式：
- `embed` 模式：输入问题与摘要向量相似度检索。
- `llm` 模式：让大模型判断问题与摘要的相关性。

---

#### 策略 4：从小知识块扩充到大知识块（多层分割嵌入）

**思想**：同一知识在多个粒度（多个 chunk_size）上分割嵌入，多层检索。如 chunk_size = 128 与 512 两层，保存大小块之间的关系。检索出小块后按关系合并成大块送入模型（图 11-6）。好处：召回知识更丰富、更连续。

**两个关键组件**：
- `HierarchicalNodeParser`：分层 Node 解析器，按多个 chunk_size 自动解析多层 Node。
- `AutoMergingRetriever`：自动合并检索器，基于分层解析器把检索出的小 Node 合并成大 Node。

```python
reader = SimpleDirectoryReader(input_files=["../../data/citys/南京市.txt"])
documents = reader.load_data()

# 3 个粒度分层解析
node_parser = HierarchicalNodeParser.from_defaults(
    chunk_sizes=[2048, 512, 128], chunk_overlap=0)
nodes = node_parser.get_nodes_from_documents(documents)

# 叶子/根 Node 查询
# from llama_index.core.node_parser import get_leaf_nodes, get_root_nodes
# leaf_nodes = get_leaf_nodes(nodes); root_nodes = get_root_nodes(nodes)

vector_store = ChromaVectorStore(chroma_collection=collection)

# 关键：把所有 Node 全部放入 docstore（合并时需更大粒度 Node）
docstore = SimpleDocumentStore()
docstore.add_documents(nodes)

# 只在叶子 Node 层构造向量索引
storage_context = StorageContext.from_defaults(
    vector_store=vector_store, docstore=docstore)
leaf_index = VectorStoreIndex(nodes=leaf_nodes, storage_context=storage_context)

# 叶子层检索器
leaf_retriever = leaf_index.as_retriever(similarity_top_k=1)

# 在叶子检索器之上构造自动合并检索器
retriever = AutoMergingRetriever(
    leaf_retriever, storage_context,
    verbose=True, simple_ratio_thresh=0.1)
```

**实现关键点**：
1. 传多个 chunk_size 生成多层 Node。
2. `DocumentStore` 存所有 Node（区别于 VectorStore，用于存原始 Node、检索后重建）。索引只基于叶子 Node 构造，合并时需从 docstore 取大粒度 Node。
3. 自动合并**不是简单替换**，而是判断小 Node 在父 Node 中的占比，达到阈值才合并并删除子 Node。默认阈值 50%。例：2048 大小父 Node 有 16 个 128 子 Node，命中 8+ 个则合并（书例为 `simple_ratio_thresh=0.1` 仅为演示）。

## 11.3 优化对大文档集知识库的检索

> 企业可能有几百份不同来源/类型文档。简单 text-split + top-k 会产生精度不足、知识相互干扰。**核心问题在检索阶段**。一个重要优化方法是大文档集下"分层"过滤与检索。

### 11.3.1 方案一：元数据过滤 + 向量检索

#### 1. 原理与架构（图 11-9）

构造向量库时按文档信息给每个块打**元数据**（地区、类型等）。检索流程：
1. 大模型推理输入问题的元数据。
2. 借向量库元数据过滤定位到部分文档的块。
3. 向量检索进一步定位 top-K。

**优点**：借助向量库能力，快速自动过滤检索。
**缺点**：
- 需设计元数据标识。
- 大模型推理元数据有不确定性。
- 元数据只能精确匹配，不能语义检索。
- 需借助向量库。

#### 2. 实现：`VectorIndexAutoRetriever`

**作用**：根据自然语言问题，借大模型 + 元数据描述信息（`VectorStoreInfo`）推理出一组元数据过滤器，再用向量库元数据过滤 + 语义检索。

##### 步骤 1：定义生成元数据的方法

三种情形：
- 数据结构化（分类问答）→ 直接构造 Node 并设元数据。
- 普通文档 → 加载后设 Document 元数据。
- 其他 → 大模型提取元数据（如下）。

```python
catalog_prompt_temp = """\
你是一个聪明的内容分类器。请把我的内容归类到以下类别之一：
-------
基本 / 历史 / 经济 / 文化 / 交通 / 旅游 / 其他
-------
我的内容是：{text}
直接输出类别，不要有多余说明。
"""
catalog_prompt = PromptTemplate(catalog_prompt_temp)

def get_catalog(text: str):
    return llm.predict(catalog_prompt, text=text)

# 摄取管道转换器：把元数据插入 Node
class MetadataRicher(TransformComponent):
    def __call__(self, nodes, **kwargs):
        for node in nodes:
            node.metadata["catalog"] = get_catalog(node.text)
        return nodes
```

##### 步骤 2：嵌入并存入向量库

```python
city_docs = SimpleDirectoryReader(input_files=["../../data/citys/南京市.txt"]).load_data()
```

### 11.3.2 方案二：摘要检索 + 内容检索

#### 1. 原理与架构（图 11-11）

吸收"小块到大块"思想：对每个文档做摘要，在**摘要级与内容级**分别嵌入向量存储并做好关联。检索流程：
1. 摘要级检索 → 相关摘要块。
2. 按摘要块关联对原始内容块检索。
3. 递归检索出原始内容块找到关联上下文。

**优点**：两个级别都可做语义检索。
**缺点**：
- 需大模型生成摘要块，麻烦且增本。
- 摘要语义匹配错则后续上下文无效。

#### 2. 实现：基于 `IndexNode` 的两级递归检索

核心：在文档的**摘要 Node（IndexNode）中放入二级检索器的引用**，被检索后自动递归检索内容 Node。

##### 步骤 1：准备原始文档（设固定 index_id）

```python
city_docs = SimpleDirectoryReader(input_files=[
    "../../data/citys/南京市.txt", "../../data/citys/北京市.txt",
    "../../data/citys/上海市.txt", "../../data/citys/广州市.txt"]).load_data()

for doc in city_docs:
    doc.metadata['index_id'] = os.path.splitext(doc.metadata["file_name"])[0]
```

##### 步骤 2：生成摘要（持久化）

```python
def generate_docs_summary():
    def save_summary_txt(doc_id, summary_txt): ...
    def load_summary_txt(doc_id): ...

    def generate_summary_txt(doc):
        summary_index = SummaryIndex.from_documents([doc])
        query_engine = summary_index.as_query_engine(
            llm=llm_dash, response_mode="tree_summarize")
        summary_txt = str(query_engine.query("请用中文生成摘要"))
        save_summary_txt(doc.metadata["index_id"], summary_txt)
        return summary_txt

    for doc in city_docs:
        doc_id = doc.metadata["index_id"]
        summary_txt = load_summary_txt(doc_id)
        if summary_txt is None:
            summary_txt = generate_summary_txt(doc)
        # 摘要在原始 Node 中不参与嵌入/生成
        doc.metadata["summary_text"] = summary_txt
        doc.excluded_embed_metadata_keys = ["summary_text"]
        doc.excluded_llm_metadata_keys   = ["summary_text"]
```

##### 步骤 3：构造原始文档索引（二级索引）

普通向量索引构造 + 持久化（省略，与 11.3.1 类似），返回 `docs_index`。

##### 步骤 4：构造摘要 Node（IndexNode，含二级检索器引用）

```python
def create_doc_index_node(doc):
    summary_txt = doc.metadata["summary_text"]
    filters = MetadataFilters(filters=[
        MetadataFilter(key="index_id", operator=FilterOperator.EQ,
                       value=doc.metadata["index_id"]),
    ])
    # 关键：obj = 二级检索器（递归检索时被调用）
    index_node = IndexNode(
        index_id=doc.metadata["index_id"],
        text=summary_txt,
        metadata=doc.metadata,
        obj=docs_index.as_retriever(filters=filters))
    return index_node
```

> `MetadataFilters` 是元数据自动化检索的底层组件，二级检索时用于区分对应城市的原始文档（因 docs_index 是所有城市文档的统一索引）。也可给每个文档单独构造检索器，则不需要过滤器。

##### 步骤 5：构造摘要索引（最终使用的索引）

```python
def create_summary_index():

    vector_store = ChromaVectorStore(chroma_collection=summary_collection)
    index_nodes = [create_doc_index_node(doc) for doc in city_docs]
    # 注意：摘要 Node 用 objects 参数
    storage_context = StorageContext.from_defaults(vector_store=vector_store)
    vector_index = VectorStoreIndex(objects=index_nodes, storage_context=storage_context)
    return vector_index
```

### 11.3.3 方案三：多文档 Agentic RAG

#### 1. 原理与架构（图 11-13）

适用场景（经典 RAG 无法胜任）：
1. 文档事实性问答（什么是大模型？）
2. 基于摘要/总结回答（×文档主要讲了什么？）
3. 跨文档/知识库问答（Self-RAG 与 C-RAG 区别？）
4. 结合其他工具复合应用（从×文档提取产品介绍发给×客户）

**架构**：两级 Agent 配合 + 底层 RAG 查询引擎：
- **Tool Agent**（知识 Agent）：每个文档/知识库一个，能用一个或多个 RAG 查询引擎回答问题。
- **Top Agent**（语义路由 Agent）：在多个 Tool Agent 之上，借推理能力使用后端 Tool Agent 完成查询任务。

**优势**：
- 灵活扩展（Tool Agent 可加更多工具：整理、摘要、数据分析、API 获取外部知识）。
- 多 Tool Agent 可协作（对比汇总多个文档）。
**缺点**：实现复杂；对大模型推理能力要求高。

#### 2. 实现

##### 步骤 1：准备原始文档

```python
names = ['c-rag', 'self-rag', 'kg-rag']
files = ['../../data/c-rag.pdf', '../../data/self-rag.pdf', '../../data/kg-rag.pdf']
```

##### 步骤 2：创建单文档 Tool Agent（含两个工具）

每个文档构造两个索引与查询引擎：向量索引（事实性问题）+ 摘要索引（总结类问题）。

## 11.4 使用高级检索方法

> 检索是 RAG 最重要的阶段，召回率与精确性决定生成质量。影响因素：原始知识形式/质量、输入问题、索引类型、嵌入/大模型、检索算法、排序算法。本章讲基础索引检索以外的复杂检索技巧。

### 11.4.1 融合检索（Fusion Retrieval）

**定义**：多维度检索方法，通过多个不同检索方法检索，对结果用 **RRF 算法**（或其他算法）重排序后输出。可组合多个不同输入问题或不同类型索引的检索结果，弥补单索引精确性不足（图 11-17）。

**RRF（Reciprocal Rank Fusion，倒数排名融合）**：将多个搜索结果的排名组合成单一统一排名，增加最相关文档出现在最终排名顶部的机会。

#### 1. 用原生代码实现融合检索（5 步）

步骤：①查询转换 ②构造两检索器 ③重排序 ④自定义融合检索器 ⑤主程序测试。

##### 1) 查询转换（非必需）

```python
# 异步并行检索
async def run_queries(queries, retrievers):
    tasks = []
    for query in queries:
        for i, retriever in enumerate(retrievers):
            tasks.append(retriever.aretrieve(query))
    task_results = await tqdm.gather(*tasks)
    results_dict = {}
    for i, (query, query_result) in enumerate(zip(queries, task_results)):
        results_dict[(query, i)] = query_result
    return results_dict
```

##### 3) 重排序（RRF 通用算法）

```python
def rerank_results(results_dict, similarity_top_k: int = 3):
    k = 60.0
    fused_scores = {}
    text_to_node = {}
    for nodes_with_scores in results_dict.values():
        for rank, node_with_score in enumerate(
            sorted(nodes_with_scores, key=lambda x: x.score or 0.0, reverse=True)
        ):
            text = node_with_score.node.get_content()
            text_to_node[text] = node_with_score
            if text not in fused_scores:
                fused_scores[text] = 0.0
            fused_scores[text] += 1.0 / (rank + k)   # RRF 核心
    reranked_results = dict(
        sorted(fused_scores.items(), key=lambda x: x[1], reverse=True))
    reranked_nodes = []
    for text, score in reranked_results.items():
        reranked_nodes.append(text_to_node[text])
        reranked_nodes[-1].score = score
    return reranked_nodes[:similarity_top_k]
```

##### 4) 自定义融合检索器（继承 BaseRetriever，实现 `_retrieve`）

```python
class FusionRetriever(BaseRetriever):
    def __init__(self, retrievers: List[BaseRetriever], similarity_top_k: int = 3) -> None:
        self._retrievers = retrievers
        self._similarity_top_k = similarity_top_k
        super().__init__()

    def _retrieve(self, query_bundle: QueryBundle) -> List[NodeWithScore]:
        querys = rewrite_query(query_bundle.query_str, num=3)
        results_dict = asyncio.run(run_queries(querys, self._retrievers))
        final_results = rerank_results(results_dict, similarity_top_k=self._similarity_top_k)
        return final_results
```

### 11.4.2 递归检索（Recursive Retrieval）

> 融合检索：多索引 + 重排弥补单索引偏差。递归检索：**分层检索**，通过递归形式完成。

#### 1. 递归检索的原理（图 11-19/11-20）

类比找书：基本过滤 → 翻简介定位少量书 → 目录+翻阅找文字。在不同层次构造检索 Node 与索引（如摘要层/详细内容层），通过 Node 间链接关系，每次检索自动递归查询，直至递归结束条件。

**实现方法**：在需递归检索的 Node（LlamaIndex 的 Node）上保存指向下一层递归调用组件的引用，被检索时通过引用递归调用。被深度检索出的块替代最初检索出的块成为大模型上下文。

**引用类型（4 种）**：
1. **指向其他 Node 的引用**：递归调用被引用 Node 返回内容，无真正"检索"过程，用于明确层次映射：
   - 小块→大块引用
   - 摘要块→内容块引用
   - 假设性问题块→内容块引用
2. **RAG 查询引擎**：递归调用查询引擎，获答案构造成 Node 返回。
3. **Data Agent**：递归调用 Agent 获答案返回。
4. **复杂 RAG 范式的查询管道**：递归调用查询管道获答案返回。

支持保存外部对象引用的 Node 类型是 **`IndexNode`**。`RecursiveRetriever` 从 root Node（root_id 指定）开始检索；若检索出的 Node 是 IndexNode，就找到其指向的对象（依赖三个字典：`retriever_dict`、`query_engine_dict`、`node_dict`）。

## 【面试考点】

### Q1：chunk_size 怎么选？为什么不能一刀切？
**要点**：小块语义精确但上下文少；大块上下文完整但语义不精确、token 成本高；事实性问答偏好小块、摘要/对比偏好大块。企业级应**用评估数据集 + 评估框架**（RAGAS / LlamaIndex Evaluation）对多个 chunk_size 跑忠实度/正确性/相关性 + 延迟，按结果选优。

### Q2：检索块和生成块为什么要分离？有哪几种分离策略？
**要点**：单一 chunk_size 难以两全检索精度与生成上下文丰富度。4 种策略：
1. 问题→问答对（Node metadata 分离 embed/llm 内容）
2. 上下文窗口扩充（SentenceWindowNodeParser + MetadataReplacementPostProcessor）
3. 摘要→内容（DocumentSummaryIndex，支持 embed/llm 两种摘要检索模式）
4. 小块→大块（HierarchicalNodeParser + AutoMergingRetriever，按 simple_ratio_thresh 阈值合并）

### Q3：AutoMergingRetriever 的合并机制？
**要点**：基于分层 Node + DocumentStore；只在叶子层构造向量索引；检索出叶子 Node 后，按子 Node 在父 Node 中的占比达到阈值（默认 50%）才合并输出父 Node 并删除子 Node；合并所需的大粒度 Node 从 DocumentStore 取。

### Q4：大文档集检索有哪 3 种方案？
**要点**：
1. 元数据过滤 + 向量检索（VectorIndexAutoRetriever + VectorStoreInfo）。
2. 摘要检索 + 内容检索（IndexNode 内嵌二级检索器，递归检索）。
3. 多文档 Agentic RAG（两级 Agent：Tool Agent + Top Agent，文档多时加 ObjectIndex 工具检索器）。

### Q5：IndexNode 在递归检索中的作用？
**要点**：IndexNode 是支持保存外部对象引用的 Node 类型，`index_id`/`obj` 字段可指向其他 Node / 检索器 / 查询引擎 / Agent。被检索时自动递归调用引用对象。RecursiveRetriever 通过 `retriever_dict`/`query_engine_dict`/`node_dict` 三个字典找到被指向对象。

### Q6：融合检索与递归检索的区别？
**要点**：融合检索是"横向"——多索引/多查询并行检索 + RRF 重排；递归检索是"纵向"——分层检索，通过 Node 间引用链自动递归到底层。两者可结合。

### Q7：RRF 算法公式？
**要点**：`fused_score(text) = Σ 1/(rank + k)`，其中 k 通常取 60，rank 为该 text 在某检索结果中的排名（从 0 起）。某文档在多个检索器中排名靠前，得分累加越高。

### Q8：QueryFusionRetriever 支持哪些重排算法？
**要点**：`reciprocal_rerank`(RRF)、`relative_score`、`dist_based_score`、`simple`。

### Q9：路由查询引擎 vs Agent 的区别？
**要点**：路由查询引擎多数时候只做选工具 + 转发问题，不多次迭代；Agent 会观察工具返回结果，可能多次迭代调用多个工具完成任务。

## 【易错 / 陷阱】

1. **chunk_size 不是越小越好也不是越大越好**：要在评估指标指导下选，盲目追求小块会丢上下文、盲目大块会语义模糊+成本爆炸。
2. **`DocumentSummaryIndex` 不做二次检索**：检索出某 Document 的全部原始 Node。需要二次检索得用递归检索（11.3.2 的 IndexNode 方案）。
3. **AutoMergingRetriever 必须配 DocumentStore**：索引只在叶子层建，合并需要更大粒度 Node 必须从 docstore 取，否则合并失败。
4. **`simple_ratio_thresh` 默认 50%**：书例设 0.1 仅为演示效果，生产中阈值过小会过度合并、过大则形同虚设。
5. **`VectorIndexAutoRetriever` 依赖向量库的元数据过滤能力**：元数据只能精确匹配，无法语义检索；且大模型推理元数据有不确定性。
6. **IndexNode 的 `obj` 引用类型必须与字典匹配**：指向检索器→`retriever_dict`；指向查询引擎/Agent→`query_engine_dict`；指向 Node→`node_dict`。传错字典递归会失败。
7. **递归检索器 root_id 必须在对应字典中存在**：否则启动即报错。
8. **多文档 Agentic RAG 工具过多易出错**：文档量大时务必加 ObjectIndex 工具检索器缩小选择范围，否则 Top Agent 推理准确率下降。
9. **PDF 等多 Document 文档上传**：简化代码只读 `load_data()[0]`，PDF 可能解析为多个 Document，需循环处理且注意 doc_id 唯一性。
10. **node_to_df 表格清洗**：`UnstructuredElementNodeParser` 生成的表格 Node 前面带摘要，不能直接 read_csv，需先大模型清洗。
11. **`metadata_mode` 三态**：`DEFAULT`/`EMBED`/`LLM`，分离检索/生成内容时务必用 `get_content(metadata_mode=...)` 验证，避免答案泄漏到嵌入向量。
