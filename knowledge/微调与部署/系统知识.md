## 1. 为什么微调

### 1.1 是什么

微调（Fine-tuning, FT）= 在一个已经预训练好的基座模型（base model）参数上，用「领域数据 / 任务数据 / 偏好数据」继续训练，让模型把通用能力「收敛」到具体场景。

直觉：预训练给了模型「世界知识 + 语言能力」，但它不知道你想要的回答风格、领域术语、输出格式、价值取向。微调就是把这些「场景特异性」蒸馏进权重里。

### 1.2 微调 vs RAG vs Prompt 工程的决策

三者不是互斥，而是叠加层。决策核心是「你要改的是模型行为，还是模型知识，还是模型风格」。

| 维度 | Prompt 工程 | RAG（检索增强） | 微调（FT） |
|---|---|---|---|
| 改的是什么 | 输入指令 | 输入上下文（外挂知识） | 模型权重 |
| 适合的问题 | 任务结构、输出格式、少样本引导 | 事实型问答、私有知识、时效性数据 | 风格 / 语气 / 领域术语 / 稳定输出格式 / 复杂推理模式 |
| 数据需求 | 0，写 prompt 即可 | 文档库 | 几百到几十万条标注样本 |
| 成本 | 极低 | 中（向量库 + 检索链路） | 高（GPU + 数据标注 + 评估） |
| 更新成本 | 改一行 prompt | 增删文档即可 | 重新训练 |
| 知识来源 | 参数内 | 参数外（显式可追溯） | 参数内 |
| 典型失败 | 长上下文挤占、指令遵循不稳 | 检索失败 / 噪声召回 | 灾难性遗忘、过拟合 |

**决策树（面试可直接背）**：
1. 先试 Prompt 工程 + Few-shot。够用就停。
2. 如果问题本质是「模型不知道某事实 / 知识时效 / 私有文档」→ **RAG**。
3. 如果问题是「模型回答风格不对 / 不愿按固定 JSON 输出 / 领域术语理解错 / 推理链条不够好」→ **微调（先 SFT，再考虑对齐）**。
4. 很多时候是 **RAG + 微调组合**：微调让模型会用工具 + 风格稳定，RAG 提供实时事实。

**口诀**：知识用 RAG，行为用微调，引导用 Prompt。三者分层叠加，不是替代。

### 1.3 什么时候不该微调
- 数据 < 几百条：不足以稳定地改变行为，会过拟合。
- 任务是纯知识问答：微调会把知识写进权重，更新困难，RAG 更合适。
- 基座模型已经能做好，只是没调好 prompt：先穷尽 prompt。
- 没有评估集：没法判断微调是否有效，等于盲改。

## 2. 微调类型

### 2.1 全参微调（Full Fine-Tuning, Full FT）

**是什么**：更新模型所有参数（W 全部参与梯度下降）。

**优点**：理论上效果上限最高；能学到最深的行为变化。

**缺点**：
- 显存爆炸。优化器状态（Adam 的 m、v 各占一份 fp32）+ 梯度 + 副本，使训练显存 ≈ 模型参数量的 **16–20 倍**。一个 7B 模型全参微调大约要 100GB+ 显存。
- 每个任务都要存一份完整模型权重（部署时多份模型）。
- 容易灾难性遗忘。

**显存拆解（全参 FT，Adam + fp16 混合精度，单位 = 参数量）**：
```
模型权重(fp16)    : 2 × N
梯度(fp16)        : 2 × N
Adam m, v (fp32)  : 4 × N + 4 × N
主权重 fp32 副本   : 4 × N
─────────────────────────
合计 ≈ 16 × N  bytes
```
N=7B → ≈ 112 GB。这就是为什么全参微调 7B 单卡（24/48G）跑不动。

### 2.2 PEFT（Parameter-Efficient Fine-Tuning，参数高效微调）

**核心思想**：冻结绝大部分预训练参数，只新增 / 训练极少量的「额外参数」（通常 < 1% 总参数），让这些少量参数承载任务适配。

**为什么有效（直觉）**：研究表明，大模型的权重更新存在「低本征秩」——即任务适配只需要在一个低维子空间内调整就够了，没必要动全部参数。

**PEFT 主要家族**：

| 方法 | 思路 | 新增参数比例 |
|---|---|---|
| **LoRA / QLoRA** | 低秩矩阵旁路（重点，见 §3） | ~0.1%–1% |
| **Adapter** | 在 Transformer 层间插入小 MLP 模块 | ~1%–5% |
| **Prefix / Prompt Tuning** | 在输入前加可训练的虚拟 token（前缀向量） | <0.1% |
| **P-Tuning v2** | 每层都加 prefix，效果接近全参 | ~0.1% |

**PEFT 的工程价值**：
- 显存大幅下降（只训练旁路参数，基座冻结可量化）。
- 一个基座 + 多个 LoRA 适配器，热切换不同任务，部署成本极低。
- 缓解灾难性遗忘（基座权重不动）。

**Adapter vs LoRA 关键区别**：Adapter 在推理时引入额外计算层（延迟增加）；LoRA 训练完可把 BA 合并回 W₀，**推理零额外开销**。这是 LoRA 成为事实标准的重要原因。

## 3. LoRA 与 QLoRA（重点）

### 3.1 LoRA 是什么

LoRA（Low-Rank Adaptation）= 在冻结的原权重 W₀ 旁边加一个「低秩更新」ΔW = B·A，只训练 A 和 B。

### 3.2 原理机制（公式直觉）

原始前向：`h = W₀ x`，W₀ ∈ R^(d×k)。

LoRA 把更新约束为低秩：
```
h = W₀ x + ΔW x = W₀ x + B A x
```
其中：
- `A ∈ R^(r×k)`，`B ∈ R^(d×r)`，秩 `r << min(d,k)`（典型 r=8/16/32/64）。
- 初始化：`A ~ N(0, σ²)`（高斯随机），`B = 0`（零初始化）。
  - 关键：训练开始时 `BA = 0`，模型输出与原模型完全一致，不破坏基座。
- 训练时只更新 A、B，W₀ 冻结。

**参数量对比**（以一个 d=k=4096 的方阵为例，r=8）：
```
全参  : d × k             = 4096² ≈ 16.7M 参数
LoRA  : r×k + d×r = r(d+k) = 8 × 8192 ≈ 65K 参数
压缩比 ≈ 256×
```

### 3.3 为什么省显存（重点直觉）

1. **训练参数少**：只有 A、B 参与反向传播，优化器状态（Adam 的 m/v）只对 A、B 分配。7B 模型全参 FT 的优化器状态要 ~56GB，LoRA 只需几百 MB。
2. **基座冻结 → 可卸载 / 量化**：W₀ 不需要梯度、不需要优化器状态，甚至可以量化到 4bit 存（这就是 QLoRA）。
3. **梯度只对旁路回传**：激活值和梯度的存储大幅减少。

**显存直觉**：LoRA 把「训练时的显存」从「参数量级别」降到「可训练参数量级别」，而后者通常只有原模型的 0.1%。

### 3.4 rank r 和 α（scaling）怎么选

- **r（秩）**：决定 ΔW 的表达能力。
  - r 越大 → 表达能力越强，但参数增多、过拟合风险上升。
  - 经验：简单任务 / 风格调整 r=4~8；复杂领域适配 r=16~64；r=128 已接近全参效果但失去 PEFT 优势。
  - 实践中 r=16 是非常常用的默认值。
- **α（缩放系数）**：`ΔW 的实际贡献 = (α/r) · BA`。
  - α 控制「旁路更新相对原权重的强度」。常见做法：固定 α=2r（即 α/r=2），然后只调 r。
  - 直觉：α/r 起到「学习率缩放」的作用。

**选 r 的决策**：先 r=8/16 跑基线，如果欠拟合（loss 下不去 / eval 不收敛）增大 r；如果过拟合（train ↓ 但 eval ↑）减小 r 或加 dropout（LoRA dropout 典型 0.05~0.1）。

**为什么低秩假设成立（理论直觉）**：Aghajanyan 等人的研究（"Intrinsic Dimensionality Explains the Effectiveness of Language Model Pre-Fine-Tuning"）发现，预训练模型的微调更新本征维度远小于参数空间——也就是说 ΔW 实际上生活在一个低维子空间里。用一个秩为 r 的分解 `BA` 去近似 ΔW，只要 r ≥ 本征维度，就能几乎无损地捕获任务适配所需的方向。这就是 LoRA「r=8 就够用」的理论支撑。

**挂载哪些模块（target modules）**：
- 早期实践只挂 attention 的 q/v proj。
- 现在共识：**attention 的 q/k/v/o proj + MLP 的 gate/up/down proj 都挂**效果最好（参数仍极少但覆盖更全）。
- 只挂 q/v 与全挂的差距可达数个百分点，是常见踩坑点。

### 3.5 训练完合并

LoRA 的一大优势：推理时可把 `BA` 加回 `W₀` 得到 `W = W₀ + BA`，**零额外推理开销**。
```
W_merged = W₀ + (α/r) · B A
```
也可不合并，保留基座 + 多个 LoRA 适配器，运行时动态加载（多任务热切换）。

### 3.6 QLoRA（重点中的重点）

**是什么**：QLoRA = Qu**a**ntized LoRA = 把冻结的基座 W₀ 量化到 **4bit** 存储，只在上面训练 fp16 的 LoRA 旁路。

**意义**：单张 48GB GPU 就能微调 65B 模型（原论文标题级别的能力）。

**QLoRA 三大创新**：

1. **NF4（NormalFloat 4-bit）数据类型**
   - 普通量化（均匀 INT4）假设权重均匀分布，但实际预训练权重近似**正态分布**。
   - NF4 把正态分布的 CDF 分成 16 个等概率分位点，得到 16 个「最优」4bit 值。对正态分布的权重信息利用率比均匀 INT4 高。
   - 直觉：把 16 个码字放在权重密集的地方（中间），稀疏的尾部少放，减少量化误差。

2. **Double Quantization（双重量化）**
   - 4bit 量化时每个 block（如 64 个权重）需要一个 scaling constant（通常 fp32）。
   - 这些常量本身也占显存（平均每参数 ~0.5 bit）。
   - 双重量化 = 把这些 scaling constant **再量化一次**到 8bit fp。
   - 净效果：平均每参数再省 ~0.37 bit（4bit → 3.63bit 有效），对大模型是几 GB 的节省。

3. **Paged Optimizers（分页优化器）**
   - 利用 NVIDIA 统一内存（CPU↔GPU），在优化器状态分配出现显存峰值时自动把部分页换出到 CPU 内存，避免 OOM。
   - 解决的是「瞬时显存尖峰导致训练崩」的工程问题。

**前向时的反量化**：计算时把 4bit 的 W₀ 反量化回 fp16/bf16 做矩阵乘，梯度只对 LoRA 旁路回传。反量化是流式进行的，不持久化，所以不占额外大显存。

**QLoRA 显存直觉**：基座从 16bit/参数压到 ~4bit/参数，直接降到 1/4。对 7B 模型，权重从 14GB 降到 ~3.5GB，单张消费级显卡就能微调。

## 4. SFT 监督微调

### 4.1 是什么

SFT（Supervised Fine-Tuning）= 用「指令-回答」对的标注数据做监督学习，让基座模型学会「听指令、按格式回答」。这是把「续写机器」变成「对话助手」的第一步。

### 4.2 指令数据格式

**Alpaca 格式**（经典，单轮）：
```json
{
  "instruction": "把下面句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today."
}
```

**ChatML 格式**（现代多轮对话，OpenAI 风格，被 Qwen / Yi / 很多开源模型采用）：
```
<|im_start|>user
今天天气怎样？<|im_end|>
<|im_start|>assistant
很好，晴朗。<|im_end|>
```

**关键工程点**：
- **只对 assistant 部分计算 loss**，user / system 部分的 label 设为 -100（PyTorch 忽略索引）。否则模型学着「生成用户的问题」，浪费容量并污染行为。
- **特殊 token 要与基座模型的 tokenizer 对齐**（如 `<|im_start|>`）。用错分词模板是 SFT 失败的常见原因——模型要么不收敛，要么推理时输出乱码。
- 多轮对话要把历史拼成长序列，仍是只对当前轮 response 算 loss。

### 4.3 数据准备

- **质量 >> 数量**。LIMA 论文证明 1000 条高质量数据可以微调出很强的对话能力；LIMA 的核心论点就是「对齐是风格 / 格式问题，少量高质量样本足矣」。
- **去重、去噪、去有毒**。重复样本会让模型记住而非泛化。
- **多样性**：覆盖任务类型、指令长度、输出格式。同质化数据导致行为坍缩。
- **长度分布**：要有长样本，否则模型学不会长链推理。
- **模板增强**：同一意图用多种表达方式，提升鲁棒性。

### 4.4 过拟合（重点）

**症状**：
- train loss 持续下降，eval loss 在某点后开始上升。
- 模型「背答案」：测试时对训练集相似问题答得极好，稍变样就不行。
- 输出长度 / 风格坍缩（如一律回答得很短或很啰嗦）。

**原因**：epoch 过多 / 数据太少 / 数据同质 / 学习率过大。

**对策**：
- epoch 控制在 1–3（SFT 通常 2–3 epoch 足够，多轮对话有时 1 epoch 就过拟合）。
- 早停（监控 eval loss）。
- LoRA 自带正则（参数少本身就是正则），全参 FT 更易过拟合。
- 加 dropout、加权重衰减。

## 5. 对齐：RLHF / DPO / GRPO（重点）

### 5.1 为什么需要对齐

SFT 后的模型会「按格式回答」，但可能：胡说八道（幻觉）、有害、不安全、回答啰嗦或不符合人类偏好。**对齐（Alignment）** 阶段把「人类偏好」注入模型——不只是答案对，还要答案好。

对齐的本质是：相对偏好数据 `(prompt, chosen_response, rejected_response)`，让模型更倾向 chosen。

### 5.2 RLHF（Reinforcement Learning from Human Feedback）

经典三阶段流程（InstructGPT / ChatGPT 路线）：

**阶段一：SFT**（同上，得到初始策略）。

**阶段二：训练奖励模型（Reward Model, RM）**
- 让人工标注员对同一 prompt 的多个回答排序。
- 用 (chosen, rejected) 对训练 RM：`loss = -log σ(r(x,y_chosen) - r(x,y_rejected))`。
- RM 输出一个标量奖励 r(x,y)，越大越好。
- 直觉：RM 学会「什么回答人类更喜欢」。

**阶段三：PPO 强化学习**
- 目标：最大化奖励 `E[r(x,y)]`，同时用 KL 散度约束策略 π 不要离 SFT 模型 π_ref 太远。
- 目标函数：`max E[r(x,y)] - β · KL(π || π_ref)`。
  - KL 项防止模型为刷分而崩坏（reward hacking），保持流畅性。
- PPO 需要 **4 个模型同时驻留显存**：当前策略 π、参考策略 π_ref、奖励模型 RM、Critic 价值模型（估 baseline）。
  - 这就是 RLHF 极贵、工程极复杂的根源。

**RLHF 的痛点**：
- 4 模型显存爆炸。
- PPO 超参敏感、训练不稳定。
- 训练慢、需要大量工程。
- RM 学到的奖励可能被钻空子（reward hacking）。

### 5.3 DPO（Direct Preference Optimization，重点）

**核心洞察**：可以**不训练奖励模型**，直接用偏好数据优化策略。DPO 把「奖励最大化 + KL 约束」的最优解形式代入，推导出一个直接的损失函数。

**理论桥梁（为什么能去掉 RM）**：
RLHF 的目标是 `max E[r(x,y)] - β·KL(π||π_ref)`。这个目标在数学上有闭式最优解：
```
π*(y|x) ∝ π_ref(y|x) · exp(r(x,y)/β)
```
反解出奖励：`r(x,y) = β·log(π*(y|x)/π_ref(y|x)) + const`。
也就是说：**最优策略 π 与参考策略 π_ref 的对数似然比，本身就是奖励的等价表示**。把这一项代回 Bradley-Terry 偏好模型 `P(y_w ≻ y_l) = σ(r(y_w) - r(y_l))`，就得到 DPO 损失——绕过了显式训练 RM 的步骤。

**DPO 损失（核心公式）**：
```
L_DPO = -log σ( β·log(π(y_w|x)/π_ref(y_w|x)) - β·log(π(y_l|x)/π_ref(y_l|x)) )
```
其中 y_w = chosen（偏好回答），y_l = rejected，β 是温度（控制偏离参考模型的强度，类似 KL 系数）。

**直觉解读**：
- 不需要显式 RM：`log(π/π_ref)` 这一项本身就是隐式奖励。
- 损失鼓励：增大 chosen 相对参考模型的似然比，减小 rejected 的似然比。
- 把 RL 问题转化为**监督分类问题**（二分类：chosen vs rejected）。

**DPO 相比 RLHF 的优势**：
| 维度 | RLHF (PPO) | DPO |
|---|---|---|
| 需要训练 RM | 是 | 否 |
| 需要在线采样 | 是（on-policy RL） | 否 |
| 需要 Critic | 是 | 否 |
| 同时驻留模型数 | 4（π, π_ref, RM, critic） | 2（π, π_ref） |
| 训练稳定性 | 差（PPO 超参敏感） | 好（监督学习） |
| 工程复杂度 | 极高 | 低 |
| 训练速度 | 慢 | 快 |
| 效果 | 上限略高 | 接近 RLHF，多数场景够用 |

**DPO 的代价 / 注意点**：
- DPO 是 **off-policy**（用固定偏好数据），不像 PPO 在线探索。如果偏好数据分布与当前模型差距大，效果打折。
- DPO 可能让模型「既降低 rejected 概率，也降低 chosen 概率」（只要相对差变大即可）——这是 DPO 已知缺陷，衍生出 IPO / SimPO / KTO 等改进。
- β 太大 → 行为不变；β 太小 → 偏离基座、退化。

### 5.4 GRPO（Group Relative Policy Optimization）

**是什么**：DeepSeek 提出的 RL 算法，DeepSeek-R1 / DeepSeek-Math 用它做推理对齐，效果惊艳。是 PPO 的高效替代。

**核心创新：去掉 Critic**。
- PPO 需要一个 Critic 网络来估计每个状态的 baseline（价值函数），用于计算 advantage `A = R - V(s)`，目的是降低梯度方差。
- GRPO 的做法：对同一个 prompt，**采样一组（group）G 个回答**，用组内奖励的**均值**作为 baseline：
```
A_i = (R_i - mean(R_1..R_G)) / std(R_1..R_G)
```
- 不需要单独的 Critic 模型，省一半显存、省一次前向。

**为什么对推理任务特别有效**：
- 推理任务（数学、代码）有**可验证的奖励**（答案对错可判定），不需要训 RM，直接用规则奖励。
- GRPO + 可验证奖励 = DeepSeek-R1 涌现「Aha moment / 长链反思」的关键配方。

**三者一句话对比**：
- **PPO**：经典 RLHF，效果好但贵（4 模型、critic、在线采样）。
- **DPO**：把 RL 变成监督学习，最简单、最便宜，off-policy。
- **GRPO**：保留 RL 的在线探索（适合可验证奖励的推理），但去掉 critic 更高效。DeepSeek-R1 的秘密武器。

**决策**：通用对齐 / 没有可验证奖励 → DPO；强推理 / 数学代码、有规则奖励 → GRPO；预算充足要上限 → PPO（现在很少这么干了）。

## 6. 微调实践

### 6.1 数据量

- SFT：最少几百条高质量；几千到几万是常见区间；超过 ~10万条边际收益快速递减。
- DPO：几千到几万偏好对足够。
- 经验法则：先小数据（500–1k）验证 pipeline 跑通，再扩量。

### 6.2 关键超参经验值

| 超参 | LoRA SFT | 全参 SFT | DPO |
|---|---|---|---|
| 学习率 | 1e-4 ~ 2e-4 | 1e-5 ~ 5e-5 | 5e-7 ~ 5e-6 |
| Epoch | 2–3 | 1–2 | 1–2 |
| Batch size（有效） | 32–128（梯度累积） | 64–256 | 同左 |
| Warmup ratio | 3%–10% | 3%–10% | 同 |
| LoRA r / α | r=16, α=32 | — | r=16, α=32 |
| LoRA target | q/k/v/o proj + gate/up/down proj | — | 同 |

**为什么 LoRA 学习率比全参大一个数量级**：LoRA 只训旁路参数，且参数是新初始化的，需要更大步长；全参微调的是已收敛的基座，步长太大会破坏预训练。

### 6.3 loss 观察

- **train loss 单调下降 + eval loss 同步下降** = 正常。
- **train ↓ 但 eval ↑** = 过拟合，停。
- **loss 平稳不降** = 学习率太小 / 数据问题 / 模板对不上。
- **loss 突然 NaN/爆炸** = 学习率过大 / 数值不稳（fp16 → 换 bf16 / 梯度裁剪）。
- **DPO loss**：关注 reward margin（chosen reward - rejected reward），应稳步上升；若 chosen/rejected reward 同时下降是已知缺陷信号。

### 6.4 灾难性遗忘（Catastrophic Forgetting）

**是什么**：微调让模型在目标任务变强，但通用能力（常识、其他任务）大幅退化。

**原因**：模型容量有限，新数据的梯度覆盖了旧知识在权重中的编码。

**对策**：
- 用 PEFT（LoRA）——基座冻结，天然缓解。
- 混入通用数据（replay / mix）——微调数据里掺一部分原始通用数据。
- 降低 epoch、降低学习率。
- 任务算子不要过窄（避免行为坍缩）。

## 7. 量化（重点）

### 7.1 是什么

量化 = 把模型权重（默认 fp16/bf16，16bit）压缩到更低精度（INT8 / INT4），降低显存和提升推理速度，代价是少量精度损失。

**为什么有效**：
- 显存：7B fp16 = 14GB；INT4 = 3.5GB。直接决定能不能在消费级显卡 / Mac / 手机上跑。
- 速度：低精度算术更快、内存带宽占用更小（推理往往是内存带宽受限 memory-bound）。

### 7.2 量化分类

- **按阶段**：PTQ（Post-Training Quantization，训练后量化，主流）vs QAT（Quantization-Aware Training，训练时模拟量化，效果更好但贵，少见）。
- **按对象**：weight-only（只量化权重，激活仍 fp16，主流 LLM 量化都是这类）vs weight+activation（W8A8 等）。

### 7.3 INT8 / INT4 基础

- **INT8**：每个权重用 8bit 表示（256 个值），通常 absmax 或对称量化。精度损失极小（<1%）。
- **INT4**：每个权重 4bit（16 个值），显存降到 1/4，但量化误差显著，需要更聪明的算法（见下）。

**RTN（Round-To-Nearest）**：最朴素的量化，直接把 fp16 映射到最近的 INT 值。简单但误差大，是更高级方法的 baseline。

### 7.4 GPTQ

**是什么**：基于二阶（Hessian）信息的训练后量化。逐层量化，每量化一个权重就用 Hessian 信息补偿其余权重，使整体输出误差最小。

**机制直觉**：
1. 对每层做一次前向，拿到 Hessian 近似 `H = X^T X`（X 是校准集激活）。
2. 逐列量化权重，量化误差通过 Hessian 反向传播到剩余未量化权重上做补偿。
3. 用一个小的校准集（128~1024 样本）即可。

**特点**：
- 4bit 精度很好（误差小）。
- **量化过程慢**（要算 Hessian），但量化完推理快。
- 配合 ExLlama v2 / Marlin kernel 推理极快。

### 7.5 AWQ（Activation-aware Weight Quantization）

**核心洞察**：不是所有权重都同等重要。少数「显著（salient）」权重对模型输出影响大，应保护它们。

**机制直觉**：
1. 用一个校准集跑一遍，观察每层激活的幅度。
2. 找出对应**大激活的权重通道**（这些通道的精度对输出影响最大）。
3. 对这些显著通道做 **scaling 保护**：通过等价的缩放变换，让显著权重的量化误差更小。
4. 显著比例通常只需保护 ~1% 的权重就能大幅减少精度损失。

**AWQ vs GPTQ（面试高频对比）**：

| 维度 | GPTQ | AWQ |
|---|---|---|
| 方法论 | 优化驱动（Hessian 补偿） | 观察驱动（激活幅度选显著通道） |
| 量化速度 | 慢（Hessian 计算） | 快（统计激活） |
| 4bit 精度 | 很好 | 常略好于 GPTQ（生成/推理任务） |
| 核心假设 | 所有权重都重要，用优化补偿误差 | 权重不等重要，保护关键的 |
| 激活处理 | weight-only | weight-only，但选权重是 activation-aware |

### 7.6 GGUF / llama.cpp

**GGUF**：llama.cpp 项目的文件格式（前身 GGML）。一个文件打包模型权重 + 元数据 + tokenizer，跨平台加载。

**特点**：
- 专为 **CPU / 消费级硬件 / Mac（Metal）** 优化。
- 支持多种 k-quant 级别（`Q4_K_M`、`Q5_K_M`、`Q8_0` 等），K 表示「重要性分层量化」——关键层用更高精度。
- `Q4_K_M` 是最常见的「甜点」档：4bit 为主，关键张量用 5/6bit，精度损失小、显存省。
- 适合本地部署、CPU 推理、嵌入式、Mac 用户。

**llama.cpp**：C++ 实现的高效推理引擎，支持上述 GGUF，可 CPU/GPU/Metal 混合推理。

### 7.7 精度损失直觉

| 量化 | 相对 fp16 显存 | 典型精度损失（perplexity / 任务分） |
|---|---|---|
| INT8 | 1/2 | <1%，几乎无感 |
| INT4（GPTQ/AWQ） | 1/4 | 1%–5%，可接受 |
| INT4（RTN 朴素） | 1/4 | 显著退化，不可用 |
| INT3 | 1/5+ | 明显退化，谨慎 |

**经验法则**：4bit（GPTQ / AWQ / GGUF Q4_K_M）是当前「性价比甜点」，绝大多数场景精度损失可接受；8bit 接近无损但省一半显存；3bit 以下需要权衡。

### 7.8 量化的本质权衡

量化误差 = 把连续值压到离散码字的信息损失。损失分两类：
- **权重量化误差**：直接来自权重舍入。
- **激活量化误差**：激活有离群值（outliers），少数大值会主导 scale，导致大多数小值精度变差——这是 W8A8（连激活一起量化）困难的根源。SmoothQuant 等方法就是处理激活离群值。

> 重点：LLM 主流量化都是 **weight-only**（激活保持 fp16），因为激活离群值难量化，而权重量化误差可控。

## 8. 推理优化（重点）

### 8.1 推理的两个阶段

- **Prefill（预填充）**：处理输入 prompt，并行计算所有 token 的 KV，计算密集（compute-bound）。
- **Decode（解码）**：逐 token 自回归生成，每步只算 1 个 token，内存带宽密集（memory-bound）。
- 两个阶段瓶颈不同，优化手段不同。

### 8.2 KV Cache（必考）

**是什么**：自回归生成时，每生成一个 token，attention 需要 attending 到所有历史 token 的 Key/Value。如果每步都重算历史的 K/V 太浪费。KV Cache = 把历史 token 的 K、V 缓存下来，逐步追加。

**为什么必须**：没有 KV cache，生成长度为 N 的序列是 O(N²) 计算；有 KV cache，每步只算新 token 对历史的 attention，是 O(N)。

**代价**：显存随序列长度 × batch 线性增长。对长上下文（128k），KV cache 显存常超过模型权重本身——这就是 KV cache 优化的核心动机。

**显存公式直觉**：
```
KV cache 显存 ≈ 2 (K+V) × num_layers × num_heads × head_dim × seq_len × batch × dtype_bytes
```
长上下文 + 大 batch 时这会爆炸。

**实例估算（Llama-2 7B，fp16）**：
- num_layers=32，num_heads=32，head_dim=128，seq_len=2048，batch=1。
- KV cache = 2 × 32 × 32 × 128 × 2048 × 1 × 2 bytes ≈ 1 GB / 单请求。
- batch=64 → 64GB。所以并发是 KV cache 优化的生死线。

**KV cache 的衍生优化**：
- **KV cache 量化**（fp16 → INT8/INT4）：直接砍 KV 显存。
- **Prefix caching / prompt caching**：相同前缀（system prompt、few-shot）的 KV 复用，省重复 prefill。
- **KV cache offloading**：KV 存 CPU 内存或 SSD，按需取回（用容量换带宽，长上下文服务常用）。
- **滑动窗口 / KV 淘汰**：丢弃老 token 的 KV（适合流式、对话历史超长场景）。

### 8.3 Flash Attention（重点）

**是什么**：一种 **IO-aware 的精确 attention** 算法（不是近似！结果和标准 attention 数学等价）。FlashAttention v1/v2/v3。

**为什么快（核心直觉）**：
- 现代 GPU 计算快、但显存带宽（HBM↔SRAM）慢。GPU 是 **memory-bound**，不是 compute-bound。
- 标准 attention：要把 N×N 的注意力矩阵写回 HBM，再读出来做 softmax，再写回……大量 HBM 读写。
- Flash Attention：
  1. **Tiling（分块）**：把 Q、K、V 切成小块，加载到片上高速 SRAM。
  2. **Kernel fusion（算子融合）**：Q·K^T → softmax → ·V 全部在一个 kernel 里完成，**中间结果不写回 HBM**（N×N 矩阵从不物化）。
  3. 用分块 softmax 的数值技巧（在线 softmax / rescaling）保证结果精确。

**效果**：
- HBM 访问减少 ~9 倍。
- 训练速度提升 2–4 倍，显存从 O(N²) 降到 O(N)（attention 矩阵不物化）。
- **结果完全精确**，无精度损失，这是它取代标准 attention 的根本原因。

**v2 改进**：更好的并行（沿序列长度切分到不同线程块）和任务划分，提升 GPU 利用率。v3 针对 H100 进一步优化。

**直觉总结**：Flash Attention 快不是因为算得更少（FLOP 其实略多），而是因为**读写 HBM 的次数大幅减少**。

### 8.4 Paged Attention（vLLM 的核心）

**问题**：传统 KV cache 为每个请求预分配「最大可能长度」的连续显存，导致严重碎片化和浪费（实际生成长度未知，常预留过长）。

**类比**：操作系统的虚拟内存分页。

**机制**：
- 把 KV cache 切成固定大小的 **block（块，如每块存 16 个 token 的 KV）**。
- 用 block table（块表）做逻辑→物理映射。
- 按需分配块，不预留；请求结束释放块。
- 同一 prompt 的不同 token 的 KV 可以分散在不连续的物理块里。

**效果**：
- KV cache 显存浪费几乎为零。
- 可接纳更多并发请求，吞吐大幅提升（vLLM 论文报告 2–4 倍吞吐提升）。

### 8.5 Continuous Batching（连续批处理）

**问题**：静态批处理（static batching）必须等同一批所有请求都生成完才接受新请求。短请求要等长请求，GPU 大量空闲。

**机制**：
- 在 **每次 decode step（iteration-level）** 检查：完成的请求弹出 batch，等待的新请求插入 batch。
- 不同请求可同时在不同位置生成，互不阻塞。
- GPU 始终满载。

**对比**：
- static batching：批级调度，吞吐受最慢请求拖累。
- continuous batching：迭代级调度，吞吐最大化。

Paged Attention（解决内存）+ Continuous Batching（解决调度）= vLLM 高吞吐的两大支柱。

### 8.6 vLLM

**是什么**：当前最主流的高性能开源 LLM 推理服务引擎，组合了 Paged Attention + Continuous Batching + 高效 CUDA kernel。

**特点**：
- 极高吞吐（适合生产 serving）。
- 兼容 OpenAI API。
- 支持张量并行、流水并行、量化模型加载。
- 活跃生态，已支持大量模型。

**适用场景**：生产环境高并发推理服务、需要 OpenAI 兼容 API、对吞吐敏感。

### 8.7 其他推理优化

**TensorRT-LLM**：
- NVIDIA 官方，基于 TensorRT。
- 极致优化（kernel fusion、量化、in-flight batching）。
- 在 NVIDIA 硬件上性能顶尖，但编译 / 部署复杂、闭源生态绑定。

**Speculative Decoding（投机解码）**：
- 用一个**小模型（draft model）**快速生成 k 个候选 token。
- 大模型（target）**一次前向并行验证**这 k 个 token。
- 接受小模型猜对的前缀，剩余从小模型重算。
- 直觉：大模型逐 token 生成是 memory-bound（每次只算 1 个 token，浪费算力）。投机解码让大模型一次验证多个 token，把 memory-bound 变 compute-bound，**无损精度下 2–3 倍加速**。
- 变体：自投机（用同一模型的不同头）、Medusa（多头并行预测）。
- **关键点**：draft model 必须与 target 分布接近，否则接受率低、反而变慢；接受率是投机解码的性能命脉。

**Prefix Caching / Prompt Caching**：相同前缀（system prompt、few-shot）的 KV 缓存复用，省重复计算。

### 8.8 推理优化小结表

| 技术 | 解决的问题 | 阶段 |
|---|---|---|
| KV Cache | 避免重复计算历史 K/V | decode |
| Flash Attention | attention 的 HBM 读写瓶颈 | prefill + decode |
| Paged Attention | KV cache 显存碎片 / 浪费 | decode |
| Continuous Batching | 静态批处理 GPU 空闲 | serving 调度 |
| Speculative Decoding | decode 的 memory-bound | decode 加速 |
| Prefix Caching | 重复前缀重复计算 | prefill |

## 9. 部署方案对比

### 9.1 Ollama

- **是什么**：本地一键运行 LLM 的桌面/CLI 工具，封装 llama.cpp。
- **特点**：极简安装、自动量化（GGUF）、模型库丰富、API 简单。
- **适用**：本地开发、个人使用、Mac、原型验证、隐私敏感的本地推理。
- **不适用**：高并发生产服务。

### 9.2 vLLM

- **特点**：高吞吐、PagedAttention、continuous batching、OpenAI 兼容 API、量化支持。
- **适用**：生产级高并发推理服务、GPU 集群。
- **不适用**：CPU / 边缘 / 没有合适 GPU 的环境。

### 9.3 TGI（Text Generation Inference，HuggingFace）

- **特点**：HF 官方推理服务器，支持大量模型、量化、流式输出。
- **适用**：HF 生态、Docker 部署、与 HF 模型紧密集成。
- 现在很多场景下吞吐被 vLLM / SGLang 超过，但仍被广泛使用。

### 9.4 Triton Inference Server（NVIDIA）

- **是什么**：通用模型推理服务器，支持多框架（TensorRT / PyTorch / ONNX / vLLM 作为 backend）。
- **特点**：企业级、多模型管理、监控完善、与 NVIDIA 栈深度集成。
- **适用**：大型企业多模型统一服务、需要精细化运维和监控。

### 9.5 选型决策

| 场景 | 推荐 |
|---|---|
| 本地 / 个人 / Mac | Ollama |
| 生产高并发 GPU 服务 | vLLM（首选）/ SGLang |
| 极致 NVIDIA 性能 | TensorRT-LLM + Triton |
| HF 生态 / 快速 Docker | TGI |
| 边缘 / CPU / 嵌入式 | llama.cpp + GGUF |

## 10. 监控与评估

### 10.0 端到端微调流水线（实战速查）

把前面的知识串成一条可执行流水线，面试时讲「我做过的一次微调」可直接套：

```
1. 数据
   - 收集 → 清洗去重 → 模板化（严格对齐基座 chat template）
   - 只对 response 部分算 loss（label=-100 屏蔽 prompt）
   - 切 train/eval（eval 必须有，否则无法判断过拟合）

2. 基座选择
   - 通用场景：Llama / Qwen / Mistral 系列指令版
   - 中文场景：Qwen / Yi / GLM
   - 优先选已经 SFT 过的 Instruct 版，少走弯路

3. 微调方法
   - 默认 QLoRA（NF4 + LoRA r=16 α=32，挂 q/k/v/o + gate/up/down proj）
   - 学习率 1e-4~2e-4，epoch 2-3，bf16，梯度累积凑有效 batch 32~128
   - warmup 3-5%，cosine 衰减

4. 训练监控
   - train loss / eval loss 双曲线，eval 上升即早停
   - 梯度范数、学习率曲线
   - 显存峰值（防 OOM）

5. 评估
   - 领域 eval（任务分）+ 通用 eval（防遗忘）
   - LLM-judge + 人工抽检
   - 金标准回归集

6. 对齐（可选）
   - 收集偏好对 → DPO（β=0.1，lr=5e-7）
   - 强推理场景 → GRPO + 可验证奖励

7. 部署
   - 量化（INT4 GPTQ/AWQ 或 GGUF Q4_K_M）
   - 推理引擎（生产 vLLM / 本地 Ollama）
   - 监控 TTFT/TPOT/吞吐/GPU 利用率

8. 持续迭代
   - 在线 A/B + 反馈数据回流 → 下一轮 SFT/DPO
```

### 10.1 关键指标

**性能指标**：
- **TTFT（Time To First Token）**：首 token 延迟，反映 prefill 速度。交互式应用的关键。
- **TPOT（Time Per Output Token）/ ITL（Inter-Token Latency）**：每 token 生成延迟，反映 decode 速度。
- **Throughput（吞吐）**：tokens/sec（总量）或 requests/sec。serving 关注整体吞吐，单用户关注延迟。
- **吞吐 vs 延迟权衡**：高并发提升吞吐但拉高单请求延迟（队列等待），需按 SLA 取舍。

**资源指标**：
- **GPU 利用率 / 显存占用**：持续满载说明调度好；显存看 KV cache 占比。
- **KV cache 利用率**：vLLM 等会报告，反映并发能力。
- **显存峰值**：警惕 prefill 长序列导致的尖峰 OOM。

### 10.2 A/B 评估

- **离线评估**：在固定 benchmark（MMLU / HumanEval / 自建领域 eval）上跑分。
- **在线 A/B**：把流量分到新旧模型，用人工偏好 / 业务指标（点击、采纳率、人工评分）判断。
- **LLM-as-judge**：用强模型（GPT-4 级）给两个模型输出打分，节省人工。
- **回归测试**：微调后必须跑通用能力回归（防灾难性遗忘），不能只看目标任务涨点。
- **金标准集**：维护一批人工标注的 golden cases，每次模型迭代必跑。

## 11. 面试考点

### Q1：LoRA 的原理？为什么省显存？

**答**：LoRA 假设权重更新 ΔW 是低秩的，把它分解为 `ΔW = BA`，其中 B∈R^(d×r)、A∈R^(r×k)，秩 r 远小于 min(d,k)。冻结原权重 W₀，只训练 A、B。初始化 B=0、A 高斯，保证训练开始时 BA=0 不破坏基座。省显存的原因：(1) 只有 A、B 参与反向传播，优化器状态（Adam 的 m/v）只分配给旁路参数；(2) 基座冻结不需要梯度和优化器状态，可量化存储（QLoRA 进一步压到 4bit）；(3) 参数量从 O(d·k) 降到 O(r(d+k))，通常 <1%。推理时 BA 可合并回 W₀，零额外开销。

### Q2：LoRA 的 rank r 和 α 怎么选？

**答**：r 决定旁路表达能力，r 越大越强但越易过拟合、参数越多。简单风格任务 r=4~8，复杂领域适配 r=16~64。默认 r=16。α 是缩放系数，实际更新贡献为 `(α/r)·BA`，常固定 α=2r 只调 r。欠拟合增 r，过拟合减 r 或加 LoRA dropout(0.05~0.1)。

### Q3：QLoRA 的三大创新？为什么能单卡微调 65B？

**答**：(1) **NF4**：基于正态分布分位点的 4bit 数据类型，对真实权重分布的信息利用率高于均匀 INT4；(2) **Double Quantization**：把量化用的 scaling constant 再量化到 8bit，平均每参数再省 ~0.37 bit；(3) **Paged Optimizers**：用 NVIDIA 统一内存在显存尖峰时把优化器状态分页换出到 CPU，防 OOM。基座从 16bit 压到 ~4bit，显存降到 1/4，只训练 fp16 的 LoRA 旁路，所以单 48GB 卡能微调 65B。

### Q4：RLHF（PPO）的流程？为什么贵？

**答**：三阶段：SFT → 训练奖励模型 RM（用 (chosen, rejected) 偏好对，loss=-logσ(r_chosen - r_rejected)）→ PPO 强化学习（最大化 E[r(x,y)] - β·KL(π||π_ref)）。贵在 PPO 阶段需要 **4 个模型同时驻留**：当前策略 π、参考策略 π_ref、RM、Critic；且需要在线采样、超参敏感、训练不稳定。

### Q5：DPO 相比 RLHF 的优势？核心公式？

**答**：DPO 把 RL 推导成一个直接监督损失，**不需要训练 RM、不需要在线采样、不需要 Critic**，同时驻留模型从 4 个降到 2 个（π、π_ref）。损失：`L = -logσ( β·log(π(y_w|x)/π_ref(y_w|x)) - β·log(π(y_l|x)/π_ref(y_l|x)) )`。直觉：`log(π/π_ref)` 是隐式奖励，最大化 chosen 相对参考模型的似然比、最小化 rejected 的。训练稳定、工程简单、速度快，多数场景效果接近 RLHF。代价是 off-policy、偏好数据分布漂移时效果打折。

### Q6：GRPO 和 PPO 的区别？为什么 DeepSeek 用它？

**答**：GRPO 去掉了 Critic。PPO 用 Critic 估 baseline 算 advantage 降方差；GRPO 对同一 prompt 采样一组 G 个回答，用组内奖励均值/标准差做归一化得 advantage `A_i=(R_i-mean)/std`，不需要 Critic 模型，省一半显存。DeepSeek-R1 用它是因为推理任务（数学/代码）有**可验证奖励**（答案对错可判），GRPO + 可验证奖励 = 涌现长链反思的关键。

### Q7：量化 INT8 / INT4 的精度损失？为什么主流是 weight-only？

**答**：INT8 损失 <1% 几乎无感；INT4 用 GPTQ/AWQ 损失 1–5% 可接受，朴素 RTN 显著退化不可用。主流 weight-only（只量化权重，激活保持 fp16）是因为**激活有离群值（outliers）**，少数大值主导 scale 导致大多数小值精度变差，激活难量化；而权重量化误差可控。W8A8（连激活量化）需要 SmoothQuant 等处理离群值。

### Q8：AWQ 和 GPTQ 的区别？

**答**：GPTQ 是**优化驱动**：逐层用 Hessian（二阶）信息补偿量化误差，4bit 精度好但量化慢。AWQ 是**观察驱动**：用校准集看激活幅度，找出 ~1% 的「显著」权重通道做 scaling 保护，量化快、4bit 精度常略好于 GPTQ。核心哲学差异：GPTQ 假设所有权重都重要用优化补偿，AWQ 假设权重不等重要只保护关键的。

### Q9：KV Cache 是什么？为什么需要？

**答**：自回归生成时每步 attention 都要 attending 历史所有 token 的 K/V。KV Cache 把历史 K/V 缓存复用、逐步追加，避免重复计算。没有它生成是 O(N²)，有了是 O(N)。代价是显存随 seq_len × batch 线性增长，长上下文时 KV cache 显存常超过模型权重本身，是后续优化（Paged Attention、量化 KV、prefix cache）的核心动机。

### Q10：vLLM 为什么吞吐高？Paged Attention + Continuous Batching 各解决什么？

**答**：(1) **Paged Attention** 解决 KV cache 显存问题：把 KV cache 切成固定 block 按需分配（类比 OS 虚拟内存分页），消除预留连续显存的碎片和浪费，几乎零浪费，可接纳更多并发。(2) **Continuous Batching** 解决调度问题：每次 decode step 弹出完成请求、插入等待请求（迭代级调度），GPU 始终满载，不像静态批处理被最慢请求拖累。两者组合 + FlashAttention 等 kernel 使 vLLM 吞吐 2–4 倍于朴素实现。

### Q11：Flash Attention 为什么快？它是近似吗？

**答**：不是近似，结果是**精确**的。快是因为它 **IO-aware**：GPU 是 memory-bound（HBM 带宽瓶颈），标准 attention 要把 N×N 注意力矩阵反复写回 HBM。Flash Attention 用 tiling 把 Q/K/V 切块载入片上 SRAM，在一个 kernel 里融合 Q·K^T → softmax → ·V，中间 N×N 矩阵从不物化，HBM 访问减少 ~9 倍。FLOP 没减少（甚至略多），但读写次数大幅减少，所以快、且显存从 O(N²) 降到 O(N)。

### Q12：微调 vs RAG vs Prompt 怎么选？灾难性遗忘怎么处理？

**答**：先 Prompt 工程，不够再上 RAG（解决知识/时效/私有文档），最后微调（改风格/格式/领域行为/推理能力）。知识用 RAG、行为用微调、引导用 Prompt，常组合。灾难性遗忘对策：(1) 用 LoRA（基座冻结天然缓解）；(2) 数据混入通用数据 replay；(3) 降 epoch、降学习率；(4) 微调后跑通用能力回归测试。

## 12. 易错点与陷阱

### 12.1 数据 / 模板类
- **只对 response 算 loss**：忘记把 prompt 部分的 label 设 -100，模型学着生成用户输入，浪费容量、污染行为。最常见 SFT bug。
- **chat template 对不上**：用 Alpaca 模板训 ChatML 模型（或反之），推理输出乱码 / 不收敛。必须严格匹配基座模型的 tokenizer 模板。
- **数据同质化**：大量雷同样本导致行为坍缩（输出长度/风格趋同）。要保证多样性。
- **质量被数量淹没**：几万低质数据 < 几百高质数据（LIMA 教训）。

### 12.2 训练类
- **LoRA 学习率用全参的量级（1e-5）**：会几乎不学习。LoRA 要 1e-4~2e-4。
- **LoRA target modules 选不全**：只挂 q/v proj 不够，要挂 q/k/v/o + gate/up/down proj（即 attention + MLP 都挂），效果差很多。
- **epoch 过多过拟合**：SFT 通常 2–3 epoch 够，多轮对话有时 1 epoch 就过拟合。务必监控 eval loss 早停。
- **fp16 训练不稳**：大模型用 fp16 容易 NaN，优先 bf16；必要时梯度裁剪。
- **混淆 train loss 和 eval 效果**：train loss 漂亮不代表模型好用，必须跑领域 eval + 人工抽检。

### 12.3 对齐类
- **DPO 的 β 选错**：β 太大行为不变、太小退化崩溃。需要调（典型 0.1~0.5）。
- **DPO 数据 chosen/rejected 质量差**：偏好差距不明显（都差不多）学不到信号；差距太大模型只学「别学坏的」。
- **reward hacking**：RLHF/PPO 中模型钻 RM 的空子（如刷长度、重复讨好话术），需 KL 约束 + 多维奖励。
- **以为 DPO 一定比 RLHF 好**：DPO 是 off-policy，分布漂移时不如在线 RL；强推理 + 可验证奖励场景 GRPO 更合适。

### 12.4 量化 / 推理类
- **量化后忘记重测精度**：INT4 在某些任务（数学、长链推理）退化明显，必须按目标任务回归测。
- **GGUF vs GPTQ 混用**：GGUF 是 llama.cpp 生态，GPTQ/AWQ 是 GPU 生态，工具链不通用，选错装不上。
- **KV cache OOM**：长上下文 + 大 batch 时 KV cache 显存爆炸，需限制并发 / 量化 KV / 调小 max_seq_len。
- **混淆吞吐和延迟**：高吞吐配置（大 batch）可能拉高单请求 TTFT，按业务 SLA 取舍。
- **以为 Flash Attention 是近似**：它是精确的，可放心替代标准 attention（除非要修改 attention 中间值）。
- **投机解码以为一定无损**：标准 speculative decoding 数学上无损，但小模型选不好接受率低反而变慢；需要 draft model 与 target 分布接近。
- **vLLM 量化模型加载方式错**：vLLM 加载 GPTQ/AWQ/llama.cpp 量化模型各有特定参数，配错会 fallback 到 fp16（不报错但没省显存）。

### 12.5 评估 / 部署类
- **只看 benchmark 涨点不看真实体验**：MMLU 涨 2 分不代表用户满意，要有业务/人工/LLM-judge 评估。
- **灾难性遗忘没测**：微调后只测目标任务，没测通用能力退化，上线后翻车。
- **没有回归测试集**：模型迭代没有金标准 case，无法判断是变好还是变差。
