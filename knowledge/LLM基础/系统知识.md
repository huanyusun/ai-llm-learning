## 1. 什么是 LLM · 参数量 · Scaling Law

### 1.1 定义（来自 [S1] §2.1 原文）

> "Typically, large language models (LLMs) refer to Transformer language models that contain **hundreds of billions (or more) of parameters**, which are trained on massive text data, such as GPT-3, PaLM, Galactica, and LLaMA."（[S1] §2.1）

要点（均来自 [S1] §2.1 原文）：
- LLM 建立在 Transformer 架构之上，由多头注意力层堆叠成很深的网络（multi-head attention layers are stacked in a very deep neural network）（[S1] §2.1）。
- LLM 与小模型采用**相似的 Transformer 架构与预训练目标（如语言建模）**，区别在于把"模型规模/数据规模/总算力"扩展了若干数量级（[S1] §2.1）。
- 论中明确指出：**文献中对 LLM 的最小参数规模没有正式共识**，因为模型能力也与数据规模和总算力相关；本综述取较宽松定义，主要讨论 **>10B** 的模型（脚注 4）（[S1] §2.1）。
- 重要提醒：**LLM 不一定就比小 PLM 更强**，涌现能力也可能不出现于某些 LLM（脚注 1）（[S1] §2.1）。

【面试考点】"LLM 和 PLM 的区别？"——架构和目标可以一样（都是 Transformer + 语言建模），核心区别是**规模扩展（model/data/compute）**带来质变（涌现能力）（[S1] §2.1）。

【易错/陷阱】不要回答"LLM 就是参数多的模型"。准确说法：LLM 是参数量（通常数百亿以上）、数据量、算力同时大幅扩展的 Transformer 语言模型，且因为扩展而表现出小模型没有的涌现能力（[S1] §2.1）。

### 1.2 Scaling Law（核心，来自 [S1] §2.1 全文公式）

研究动机：既然扩展能大幅提升能力（[26,55,56]），就需要**定量刻画扩展效应**（establish a quantitative approach to characterizing the scaling effect）（[S1] §2.1）。论文介绍了两条代表性 Scaling Law：

**(A) KM Scaling Law（Kaplan et al., OpenAI, 2020）**
- 首次提出用**幂律（power-law）**建模模型性能与三个主要因素的关系：模型规模 N、数据规模 D、训练算力 C（[S1] §2.1）。
- 在给定算力预算 c 下，给出三条基本公式（原文式 1，量纲分别为非嵌入参数数、训练 token 数、所需算力）（[S1] §2.1）。
- 实验范围：模型规模 768M~1.5B 非嵌入参数；前提假设是分析某一因子时**不应被另外两个因子瓶颈**（[S1] §2.1）。

**(B) Chinchilla Scaling Law（Hoffmann et al., Google DeepMind）**
- 提出**算力最优训练（compute-optimal training）**的另一种 Scaling Law 形式（[S1] §2.1）。
- 实验更严格：模型规模 70M~16B，数据规模 5B~500B token（[S1] §2.1）。
- 拟合公式（原文式 2，抓取原文逐字）：

  `L(N, D) = E + A/N^α + B/D^β`

  其中 `E=1.69, A=406.4, B=410.7, α=0.34, β=0.28`（[S1] §2.1）。
- 在约束 `C ≈ 6ND` 下优化，得到最优分配（原文式 3）：

  `N_opt(C) = G·(C/6)^a,  D_opt(C) = G⁻¹·(C/6)^b`，其中 `a=α/(α+β), b=β/(α+β)`（[S1] §2.1）。

**两条律的关键差别（[S1] §2.1 原文总结，必记）**：
> "given an increase in compute budget, the **KM scaling law favors a larger budget allocation in model size** than the data size, while the **Chinchilla scaling law argues that the two sizes should be increased in equal scales**, i.e., having similar values for a and b."（[S1] §2.1）

【面试考点】"KM vs Chinchilla 区别？"——KM 主张算力增长时多投模型参数（数据偏少）；Chinchilla 主张**模型规模和数据规模等比例增长**。实践含义：早期 GPT-3 等模型按 KM 偏好"参数远多于数据"，导致训练不充分；Chinchilla 后业界普遍**提高 token 数**（[S1] §2.1）。

**(C) Predictable Scaling（可预测扩展）**
- 用小模型性能**可靠外推大模型性能**，指导训练（[S1] §2.1）。
- 收益：大模型上无法穷举训练 trick，可用小模型先验证（[S1] §2.1）。GPT-4 报告强调其引入了 predictable scaling 机制，能在训练中用很小比例算力准确预测最终性能（[S1] §2.3 GPT-4 段）。

【易错/陷阱】
- Scaling Law 不是"参数越多 loss 越低"这么简单——**要同时匹配数据量**，否则（按 KM）训练不充分。Chinchilla 论文实验显示，许多当时模型（如 GPT-3、Gopher）**显著"参数过剩、数据不足"**（[S1] §2.1）。
- `C ≈ 6ND` 是估算算力的经验公式（每个参数每个 token 约 6 FLOPs），面试常考（[S1] §2.1）。

## 2. 训练三阶段：预训练 → SFT → RLHF/DPO

综述明确把 LLM 技术拆为四大方面：**pre-training（预训练）、adaptation tuning（适配微调）、utilization（使用）、capacity evaluation（能力评估）**（[S1] 引言）。本节覆盖前两个。

### 2.1 阶段一：预训练（Pre-training，来自 [S1] §4）

**目标**：在海量文本上做语言建模，习得"语言理解 + 世界知识"。GPT-3 即用 1750 亿参数的自回归模型在大量语料上预训练（[S4] 摘要）。

**数据来源（[S1] §4.1.1 节选）**：综述把预训练语料分为多类，关键几类：
- 通用自然语言文本（网页、书籍、对话等）。
- **代码（Code）**：综述特别指出——在大量代码语料上训练能显著提升程序合成质量；并且有研究推测"在代码上训练可能是复杂推理能力（如思维链 CoT）的来源之一"（原文：a recent study also speculates that training on code might be a source of complex reasoning abilities (e.g., chain-of-thought ability)）（[S1] §4.1.1）。
- 科学/领域语料（含数学符号、蛋白质序列等，需特殊 tokenization 与预处理）（[S1] §4.1.1）。
- 代码语料两类来源：Stack Exchange 类问答社区、GitHub 等公开软件仓库（含注释/文档）（[S1] §4.1.1）。

**数据预处理（[S1] §4.1.2）**：
- 预处理对构建预训练语料至关重要，需去除噪声、冗余、无关、潜在有毒数据（"removing noisy, redundant, irrelevant, and potentially toxic data"）（[S1] §4.1.2）。
- 综述提到开源工具 **Data-Juicer**：提供 50+ 处理算子与工具（[S1] §4.1.2）。

【面试考点】"为什么很多模型要混代码数据预训练？"——①直接提升代码能力；②被推测能带来复杂推理/CoT 能力（[S1] §4.1.1）。

### 2.1.1 预训练目标（语言建模，来自 [S1] §2.1 + [S4]）

- 综述指出 LLM 与小模型采用**相似的预训练目标（如语言建模 language modeling）**，差异主要在规模（[S1] §2.1）。
- GPT-3 [S4] 是**自回归（autoregressive）语言模型**：给定前文 x_<i，预测下一个 token x_i，即最大化 `P(x_i | x_<i)`，等价于最小化负对数似然（[S4] 摘要：an autoregressive language model with 175 billion parameters）。
- 主流 Decoder-only LLM 几乎都用**因果语言建模（Causal LM）**；Encoder-Decoder（如 T5）则用 span-corruption/去噪目标（[S1] §3 大表 T5 备注）。
- "涌现的 few-shot 能力"不来自任务微调，而来自大规模 LM 预训练——GPT-3 在所有任务上**不做任何梯度更新或微调**，任务与样例都通过文本交互指定（[S4] 摘要：GPT-3 is applied without any gradient updates or fine-tuning, with tasks and few-shot demonstrations specified purely via text interaction）。

【面试考点】"GPT 类模型预训练目标？"——因果/自回归语言建模（next-token prediction）；"为什么 next-token 能学出推理？"——大规模 + 代码 + 复杂语料的统计规律（[S1] §4.1.1 推测 + [S4]）。

【易错/陷阱】
- 预训练目标 ≠ 微调目标。预训练是"无监督语言建模"，对齐阶段才引入人类偏好（[S2]/[S3]）。
- 不要混淆 Causal LM（GPT）与 Masked LM（BERT，[S1] §3 提到 BERT 用 WordPiece）：前者自回归生成，后者双向编码填空。

### 2.1.2 架构：Decoder-only Transformer（来自 [S1] §2.1 + [S4]）

- LLM 建立在 Transformer 之上，由**多头注意力层堆叠成很深的网络**（[S1] §2.1 原文）。
- GPT 系采用 **Decoder-only**：堆叠 N 个"掩码自注意力 + 前馈网络（FFN）"块，掩码保证自回归（[S4]）。
- 关键超参：层数 n_layers、隐藏维 d_model、头数 n_heads、词表 V、上下文窗口——共同决定参数量与算力（[S1] §2.1，`C≈6ND`）。

### 2.1.3 MoE（混合专家，缺口标记，[WS] + 待补一手）

综述时间窗内 MoE 未成主流，仅占位（一手待补）：
- **稀疏 MoE**：每个 token 只激活少数"专家"FFN，总参数大但单次计算小（[WS]）。
- 代表：Mixtral 8×7B、DeepSeek-V3 等（[WS]）。
- 与 §4.6 的 MLA 同属"大模型推理效率"主线（[WS]）。

### 2.2 阶段二：监督微调 SFT（Supervised Fine-Tuning / Instruction Tuning，来自 [S1] §5 + [S2]）

**是什么**：用"指令-回答"对（人工书写的示范 demonstration）做监督微调，让模型学会**按指令格式输出**。

**与 RLHF 的关系（[S1] §2.3 原文，重要）**：
> "the wording of 'instruction tuning' has seldom been used in OpenAI's paper and documentation, which is substituted by **supervised fine-tuning on human demonstrations (i.e., the first step of the RLHF algorithm [66])**."（[S1] §2.3）

即：在 InstructGPT 的语境里，**SFT 就是 RLHF 三阶段算法的第一步**（[S1] §2.3）。

**InstructGPT 的 SFT 细节（[S2] §3）**：
- 起点是 labeler 书写的 prompts + OpenAI API 提交的 prompts，收集 labeler 对"期望行为"的示范数据集（[S2] 摘要）。
- 用这些示范**监督微调 GPT-3**（[S2] 摘要）。
- 三种模型规模：1.3B / 6B / 175B，**架构都是 GPT-3**（[S2] §3，we train three model sizes (1.3B, 6B, and 175B parameters), and all of our models use the GPT-3 architecture）。

【面试考点】"Instruction Tuning 和 SFT 是一回事吗？"——概念上等价；InstructGPT 论文里 SFT 就是 RLHF 第一步（[S1] §2.3）。区别于"在具体任务数据集上 fine-tune"的传统 fine-tuning：SFT 用的是**多种指令格式**的示范，目标是泛化到未见指令。

### 2.3 阶段三：RLHF（来自 [S2] 全文 + [S1] §2.3 历史）

#### 2.3.1 为什么需要 RLHF（[S2] 摘要 + §1，原文逐句）

> "Making language models bigger does not inherently make them better at following a user's intent. For example, large language models can generate outputs that are **untruthful, toxic, or simply not helpful** to the user. In other words, these models are **not aligned** with their users."（[S2] 摘要）

要点（[S2] 摘要/§1）：
- **把模型做大并不能让它更好地遵循用户意图**——这是对"只靠 Scaling"的根本反驳。
- 问题表现：不真实（untruthful）、有毒（toxic）、对用户没帮助（not helpful）。
- 本质：模型**未对齐（not aligned）**用户。
- 解法：用**人类反馈微调**来对齐（aligning language models with user intent by fine-tuning with human feedback）（[S2] 摘要）。

#### 2.3.2 InstructGPT 三阶段算法（[S2] Fig.2 原文图注，必背）

InstructGPT（[S2]）正式建立了**三阶段 RLHF 算法**（[S1] §2.3 原文：InstructGPT was proposed in January 2022 to improve the GPT-3 model for human alignment, which formally established a three-stage reinforcement learning from human feedback (RLHF) algorithm）。

图注原文（[S2] Fig.2）：
> "A diagram illustrating the **three steps** of our method: **(1) supervised fine-tuning (SFT), (2) reward model (RM) training, and (3) reinforcement learning via proximal policy optimization (PPO)** on this reward model."

三步（[S2] Fig.2 + §3）：
1. **SFT**：监督微调（见 §2.2）。
2. **RM 训练**：在 Step 2，让模型对同一 prompt 生成 A~D 多个样本，由 labeler 排序（原文：in Step 2, boxes A-D are samples from our models that get ranked by labelers），训练一个奖励模型拟合人类偏好（[S2] Fig.2/§3）。
3. **PPO 强化学习**：用 PPO 在 RM 上做 RL 微调，最大化奖励同时不偏离原始模型（[S2] Fig.2）。

补充：
- **PPO（Proximal Policy Optimization）** 是 2017 年 OpenAI 提出的 RL 算法，现已成为"从人类偏好中学习"的基础 RL 算法（[S1] §2.3 原文：the paper of the Proximal Policy Optimization (PPO) was published in July 2017, which now has been the foundational RL algorithm for learning from human preferences）。
- 历史脉络（[S1] §2.3）：对齐研究可追溯到 2017 年 OpenAI 博客"learning from human preferences"→ 同年 PPO → 2020 年用这些 RL 算法基于人类偏好改进 GPT-2 → 2022 年 InstructGPT 正式确立三阶段 RLHF。
- RLHF 的核心价值（[S1] §2.3）：除提升指令遵循外，RLHF 对**减少有害/有毒内容生成尤为关键**，是 LLM 安全部署的关键（"the RLHF algorithm is particularly useful to mitigate the issues of generating harm or toxic content for LLMs, which is key to the safe deployment of LLMs in practice"）。

#### 2.3.3 InstructGPT 的关键实验结论（[S2] §1，全部来自抓取原文，必背数据）

> "outputs from the **1.3B parameter InstructGPT** model are preferred to outputs from the **175B GPT-3**, despite having **over 100x fewer parameters**."（[S2] §1）

具体（[S2] §1 原文）：
- **1.3B InstructGPT 输出优于 175B GPT-3**（参数量差 100 倍）——这是"对齐 > 堆参数"的最强证据。
- 175B InstructGPT 在 **85%±3%** 的情况下优于 175B GPT-3；在 **71%±4%** 的情况下优于 few-shot 175B GPT-3。
- **真实性提升**：在 TruthfulQA 等上更 truthful；幻觉率显著下降（见 §6）。
- **毒性下降**：在 RealToxicityPrompts 上，被要求"respectful"时，InstructGPT 比少约 **25%** 的有毒输出（[S2] §1，InstructGPT models generate about 25% fewer toxic outputs than GPT-3 when prompted to be respectful）。
- **泛化**：hold-out（未参与训练）的 labeler 也以接近相同比率偏好 InstructGPT（[S2] §1，holds for held-out labelers）。
- **代价/回归**：RLHF 在公开 NLP 数据集上会有性能回归（performance regression），但可通过修改 RLHF 流程尽量减小（[S2] §1，minimize performance regressions on public NLP datasets by modifying our RLHF fine-tuning procedure）。
- **重要洞察**：公开 NLP 数据集**不能反映语言模型的真实使用方式**（[S2] §1，public NLP datasets are not reflective of how our language models are used）。

【面试考点】
- "为什么大模型还不够，要 RLHF？"——大不等于对齐；RLHF 解决 untruthful/toxic/unhelpful 三类不对齐问题（[S2] 摘要）。
- "InstructGPT 的标志性结论？"——1.3B 对齐模型胜过 175B 原始模型，对齐比堆参数更划算（[S2] §1）。

### 2.4 阶段三替代：DPO（Direct Preference Optimization，来自 [S3] 全文）

#### 2.4.1 为什么需要 DPO（[S3] 摘要 + §1，原文逐句）

> "RLHF is a **complex and often unstable** procedure, first fitting a reward model that reflects the human preferences, and then fine-tuning the large unsupervised LM using reinforcement learning to maximize this estimated reward **without drifting too far from the original model**."（[S3] 摘要）

要点（[S3] 摘要/§1）：
- RLHF 复杂且不稳定（complex and often unstable）。
- 痛点：要先训 RM，再用 RL 微调大模型去最大化这个"估计出来的"奖励，同时要约束**不偏离原模型太远**（KL 约束）。
- DPO 的做法：引入一种**奖励模型的新参数化**，使得**对应的最优策略可以用闭式解（closed form）直接解出**，从而把标准 RLHF 问题简化为**一个简单的分类损失**（[S3] 摘要：enables extraction of the corresponding optimal policy in closed form, allowing us to solve the standard RLHF problem with only a simple classification loss）。

#### 2.4.2 DPO 算法特性（[S3] 摘要，必背）

- **稳定（stable）、高性能（performant）、计算轻量（computationally lightweight）**。
- 微调时**无需从 LM 采样**（eliminating the need for sampling from the LM during fine-tuning）——省掉 PPO 在线采样的开销。
- **无需大量超参调优**（without performing significant hyperparameter tuning）。
- 实验（[S3] 摘要原文）：
  - 在**情感控制**任务上**超过** PPO-based RLHF（exceeds PPO-based RLHF in ability to control sentiment of generations）。
  - 在**摘要、单轮对话**上质量**持平或更好**（matches or improves response quality in summarization and single-turn dialogue）。
  - 实现与训练都**显著更简单**（substantially simpler to implement and train）。

#### 2.4.3 DPO 的核心思想（[S3] §1 + 标题）

标题即核心论点：**"Your Language Model is Secretly a Reward Model"**（你的语言模型本身就是一个奖励模型）（[S3]）。
- 传统 RLHF：策略 π 与奖励 r 解耦，要分别训。
- DPO：利用 RLHF 的 KL 约束最优解，把奖励用策略本身表达（reward = β·log(π/π_ref) + 常数），从而**用偏好数据直接训策略**，绕过显式 RM 与 RL（[S3] §1/§3）。
- DPO 损失本质是一个**类 logistic 的偏好分类损失**（dpo.txt 中 "logistic" 关键词所在推导段，[S3] §3）。

#### 2.4.4 DPO 推导链（[S3] §1/§3，逻辑串联，便于面试口述）

标准 RLHF 的目标（[S3] §1）：最大化学到的奖励 r，同时用 KL 约束不让策略偏离参考模型 π_ref：

```
max_π  E_{x~D, y~π}[r(x,y)]  -  β · D_KL( π(·|x) || π_ref(·|x) )
```

DPO 的三步关键推导（[S3] §3）：
1. **闭式最优策略**：上述 KL 约束优化问题有解析解，最优策略可写为
   `π*(y|x) = (1/Z(x)) · π_ref(y|x) · exp(r(x,y)/β)`（[S3] §3）。
2. **奖励用策略反解**：把上式对 r 求解，得到
   `r(x,y) = β · log( π*(y|x) / π_ref(y|x) ) + β·log Z(x)`（[S3] §3）。
   —— 这正是标题"Your LM is Secretly a Reward Model"的数学含义：**任意策略本身（相对参考模型的对数比）就等价于一个奖励函数**。
3. **代入偏好模型得 DPO loss**：用 Bradley-Terry 偏好模型（人类偏好 y_w > y_l 的概率 = σ(r(x,y_w) − r(x,y_l))）代入第 2 步的 r 表达式，配分函数 Z(x) 在差值中相消，得到**无需显式 RM、无需 RL 采样**的 DPO 损失（[S3] §3）：
   ```
   L_DPO = - E[ log σ( β · log( π(y_w|x)/π_ref(y_w|x) ) − β · log( π(y_l|x)/π_ref(y_l|x) ) ) ]
   ```
   形态就是一个**带 sigmoid 的偏好二分类交叉熵**（[S3] §3）。

【面试考点】
- "DPO 推导的三步？"——①KL 约束 RLHF 目标有闭式最优策略；②反解出 r = β·log(π/π_ref)+常数；③代入 Bradley-Terry 偏好模型，Z(x) 相消，得到纯分类损失（[S3] §3）。
- "为什么 DPO 不需要采样？"——因为最终损失只用到 π 与 π_ref 在 (y_w, y_l) 上的**对数概率比**，是离线监督信号，不需要像 PPO 那样在线 rollout（[S3] §3）。
- "β 控制什么？"——KL 强度：β 大 → 离 π_ref 近（保守）；β 小 → 偏好学习更激进（[S3] §3）。

#### 2.4.5 RLHF vs DPO 对比表（综合 [S2]/[S3]）

| 维度 | RLHF（[S2]） | DPO（[S3]） |
|---|---|---|
| 步骤 | SFT → RM → PPO（3 步） | SFT → DPO（2 步，无显式 RM） |
| 是否需在线采样 | 是（PPO rollout） | 否 |
| 显式奖励模型 | 需要单独训 | 不需要（策略即奖励） |
| 稳定性 | 复杂、常不稳（[S3] 摘要） | 稳定（[S3] 摘要） |
| 超参敏感度 | 高（PPO 多超参） | 低（[S3] 摘要：without significant hyperparameter tuning） |
| 探索能力 | 强（在线 RL） | 弱（离线） |
| InstructGPT/ChatGPT 用 | 是（[S2]） | 否（DPO 是后续简化） |

【易错/陷阱】
- DPO **不是**"无奖励模型"——它仍然有奖励的概念，只是把奖励用策略参数化、闭式消掉了显式 RM 训练步骤（[S3] 标题/§3）。
- DPO 与 SFT 不冲突：通常 pipeline 是 **SFT → DPO**，DPO 在 SFT 模型基础上对齐偏好。
- DPO 对**偏好数据质量**极敏感，标注噪声会直接放大；对 π_ref 选择敏感（[WS]）。

### 2.5 三阶段总览图（面试可直接画）

```
原始文本语料
   │  (语言建模目标)
   ▼
[预训练 Pre-training]  ──→  Base Model（懂语言、有知识，但不听指令）
   │  (指令-示范数据)
   ▼
[SFT / Instruction Tuning]  ──→  会按指令格式回答的模型
   │  (人类偏好排序数据)
   ▼
[RLHF (RM+PPO)  或  DPO]  ──→  对齐人类偏好的对话模型（更安全、更有用、更真实）
```
（三阶段划分见 [S1] §2.3 / [S2] Fig.2；DPO 见 [S3]）

## 3. Tokenizer：BPE / WordPiece / Unigram（来自 [S1] §3.2 全文）

### 3.1 为什么需要 Tokenizer

模型不直接读字符/词，而是读 token。综述原文（[S1] §3.2）：
> "including Byte-Pair Encoding tokenization, WordPiece tokenization and Unigram tokenization. HuggingFace has maintained an excellent online NLP course on tokenizer ... we refer the beginners to this course."（[S1] §3.2）

三种主流子词（subword）tokenization 方法（[S1] §3.2）。

### 3.2 Byte-Pair Encoding (BPE)（[S1] §3.2）

- **起源**：1994 年作为通用**数据压缩算法**提出，后适配到 NLP 做 tokenization（originally proposed as a general data compression algorithm in 1994, and then adapted to NLP for tokenization）（[S1] §3.2）。
- **算法**：从一组基本符号（如字母表、边界字符）开始，**迭代地合并语料中频繁共现的相邻 token 对**（iteratively combine frequent pairs of two consecutive tokens in the corpus as new tokens，称为 merge）（[S1] §3.2）。
- **选择标准**：基于两个连续 token 的**共现频率**，选频率最高的一对合并；重复直到达到预设词表大小（[S1] §3.2）。
- **Byte-level BPE**：把**字节**作为基本符号做 merge，提升多语种（尤其含非 ASCII 字符文本）的 tokenization 质量（[S1] §3.2）。
- **代表模型**：GPT-2、BART、LLaMA（Representative language models with this tokenization approach include GPT-2, BART, and LLaMA）（[S1] §3.2）。

### 3.3 WordPiece（[S1] §3.2）

- **起源**：Google 内部子词 tokenization 算法，最初为**语音搜索系统**开发（originally proposed by Google in developing voice search systems）（[S1] §3.2）。
- 2016 年用于神经机器翻译，2018 年被 **BERT** 采用为词 tokenizer（adopted as the word tokenizer for BERT in 2018）（[S1] §3.2）。
- **与 BPE 的区别**：思想类似（都是迭代合并相邻 token），但**合并选择标准不同**——WordPiece 先训一个语言模型给所有可能配对打分，每次合并选**让语言模型似然最大化**的配对（takes a slightly different selection criterion for the merge）（[S1] §3.2）。

### 3.4 Unigram（[S1] §3.2）

综述列为第三种代表方法（与 BPE、WordPiece 并列）（[S1] §3.2）。代表模型如 SentencePiece 工具链（常用于多语/中日韩）。

### 3.5 三者对比表（综合 [S1] §3.2）

| 方法 | 合并选择标准 | 词表构建方向 | 代表模型 | 来源 |
|---|---|---|---|---|
| **BPE** | 相邻 token **共现频率**最高 | 自底向上合并 | GPT-2 / BART / LLaMA | [S1] §3.2 |
| **WordPiece** | 使**语言模型似然最大化**的配对 | 自底向上合并 | BERT | [S1] §3.2 |
| **Unigram** | 基于概率**删除/保留**子词，最大化语料似然 | 自顶向下裁剪 | SentencePiece（多语） | [S1] §3.2 |

字节级变体（[S1] §3.2）：
- **Byte-level BPE**：以字节为基本符号，解决多语/非 ASCII（含中文、emoji、罕见字符）问题；GPT-2/BART/LLaMA 采用（[S1] §3.2）。

【面试考点】
- "BPE 和 WordPiece 区别？"——都迭代合并，但 BPE 用**频率**选 pair，WordPiece 用**语言模型似然增益**选 pair（[S1] §3.2）。
- "为什么 LLaMA/GPT 用 Byte-level BPE？"——以字节为基础符号，能覆盖任意字符（含中文/emoji/罕见字符），避免 OOV（[S1] §3.2）。
- "中文为什么费 token？"——中文一个字常被切成多个 byte/子词 token，导致同样语义中文 token 数更多、成本更高（[S1] §3.2 推论）。

【易错/陷阱】
- tokenizer 的词表是**训练前固定**的，训练后不能改（除非重训）。
- 不同模型 tokenizer 不通用：把 GPT-2 的 token id 喂给 LLaMA 是错的。
- "merge 次数 = 词表大小"——BPE 词表大小 = 初始基本符号数 + merge 次数。

## 4. Token / Embedding / 上下文窗口

### 4.1 Token 与 Embedding

- 文本经 tokenizer → 一串 token id；token id 经**嵌入矩阵（embedding matrix）** → 向量（embedding），再进入 Transformer（[S1] §2.1 架构基础）。
- 词表大小 V 决定嵌入矩阵规模（V × d_model），是参数量的组成部分（综述脚注提到 KM 律用"非嵌入参数数"度量 N，[S1] §2.1）。

### 4.2 上下文窗口（Context Window，来自 [S4] §2 原文）

GPT-3 论文对 few-shot 的定义直接揭示了上下文窗口的作用（[S4] §2）：
> "(a) 'few-shot learning', or **in-context learning where we allow as many demonstrations as will fit into the model's context window (typically 10 to 100)**, (b) 'one-shot learning', where we allow only one demonstration, and (c) 'zero-shot' learning, where no demonstrations are allowed and only an instruction in natural language is given to the model."（[S4] §2）

要点（[S4] §2）：
- **few-shot = 把尽量多的示范塞进上下文窗口**（典型 10~100 条）。
- **one-shot = 只给 1 条示范**。
- **zero-shot = 不给示范，只给自然语言指令**。
- GPT-3 还提到原则上也能做传统 fine-tuning，但 GPT-3 论文留作未来工作（[S4] §2）。

### 4.3 上下文学习的"内循环"机制（来自 [S4] Fig.1.1 图注，原文）

> "During unsupervised pre-training, a language model develops a broad set of skills and pattern recognition abilities. It then uses these abilities at inference time to rapidly adapt to or recognize the desired task. We use the term '**in-context learning**' to describe the **inner loop of this process, which occurs within the forward-pass upon each sequence**."（[S4] Fig.1.1 图注）

关键点（[S4] Fig.1.1）：
- ICL 不是梯度更新，而是**发生在单次前向传播内**的"内循环"。
- 预训练让模型获得"广泛技能 + 模式识别能力"，推理时用它**快速适配**到目标任务。
- 这解释了为什么 GPT-3 能 few-shot 而 GPT-2 不行：规模带来的模式识别能力（[S1] §2.1 涌现能力）。

### 4.4 上下文窗口的工程制约（[WS] + [S1] §6.1 末）

- 上下文窗口上限受**位置编码（position encoding）** 与 **KV Cache 显存** 双重制约（见 §8）。
- GPT-3 时代典型窗口约 2048 token（gpt3.txt 中 "2048" 关键词）（[S4]）；后续 GPT-4 Turbo 扩到 **128k token**（[S1] §2.3 GPT-4 段：long context window up to 128k tokens）。

### 4.5 位置编码（综述与 GPT-3 未深入，缺口标记，[WS] + 待补一手）

综述 [S1] 与 GPT-3 [S4] 把 Transformer 架构作为既定前提（[S1] §2.1：multi-head attention layers are stacked in a very deep neural network），未展开位置编码细节。此处仅作框架性梳理，一手原文待补（见末尾缺口）：
- **绝对位置编码**（如原版 Transformer/GPT-2 的可学习或正弦编码）：每个位置一个固定向量，**不可外推**超出训练长度（这是 GPT-3 窗口受限 2048 的根因之一）（[WS]）。
- **RoPE（旋转位置编码，arXiv:2104.09864，待抓）**：用旋转矩阵把**相对位置**编码进 Q/K，支持一定程度外推，LLaMA 系采用（[WS]）。
- **ALiBi（arXiv:2108.12409，待抓）**：在注意力分数上按距离加线性偏置，外推能力强（[WS]）。

【面试考点】"为什么早期模型上下文窗口小？"——绝对位置编码不可外推 + KV Cache 显存 + 训练长度限制三重制约（[WS]）。RoPE/ALiBi/NTK 等是为外推而设计（[WS]，一手待补）。

### 4.6 注意力效率变体（GQA/MQA/MLA，缺口标记，[WS] + 待补一手）

综述时间窗（2023 上）未覆盖这些显存/推理优化结构，仅作占位，一手原文待补：
- **MHA（Multi-Head Attention）**：标准多头，每头独立 K/V。
- **MQA（Multi-Query Attention）**：所有头共享 1 组 K/V，省 KV Cache 显存（[WS]）。
- **GQA（Grouped-Query Attention）**：折中，多个头共享一组 K/V，Llama 2/3 采用（[WS]）。
- **MLA（Multi-head Latent Attention）**：DeepSeek 提出，低秩压缩 KV，进一步省显存（[WS]，一手待补 DeepSeek 技术报告）。

> 关键联系：MQA/GQA/MLA 都是为了**减小 KV Cache**（见 §8）以提升推理效率（[WS]）。

【面试考点】
- "few-shot/one-shot/zero-shot 区别？"——见 [S4] §2 定义，核心是上下文里塞几条示范。
- "ICL 是怎么工作的？"——预训练习得广泛技能，推理时在前向传播内快速适配，无梯度更新（[S4] Fig.1.1）。

【易错/陷阱】
- ICL **不是 fine-tuning**——不更新参数，纯靠上下文（[S4] Fig.1.1）。
- "上下文窗口越大越好"是片面的——窗口越大，KV Cache 显存越大、注意力计算越多、还可能"中间遗忘"（lost in the middle）。

## 5. 采样策略：Top-k / Top-p / Temperature（来自 [S1] §6.1 全文公式）

### 5.1 问题

自回归生成时，模型输出的是词表上的**概率分布**，下一个 token 怎么选？综述 §6.1 系统介绍 decoding 策略（[S1] §6.1）。

### 5.2 Temperature Sampling（[S1] §6.1 原文公式 10）

公式（抓取原文逐字）：
```
P(x_j | x_<i) = exp(l_j / t) / Σ_{j'} exp(l_{j'} / t)
```
其中 l_j 是每个词的 logit，**t 是温度系数**（[S1] §6.1）。

行为（[S1] §6.1 原文）：
- **降低 t** → 增大选中高概率词的机会、降低低概率词的机会（reducing the temperature t increases the chance of selecting words with high probabilities while decreases the chances of selecting words with low probabilities）。
- **t = 1** → 默认随机采样（default random sampling）。
- **t → 0** → 等价于**贪心搜索（greedy search）**。
- **t → ∞** → 退化为**均匀采样（uniform sampling）**。

### 5.3 Top-k Sampling（[S1] §6.1）

> "top-k sampling directly **truncates the tokens with lower probability** and only samples from the tokens with the **top k highest probabilities**."（[S1] §6.1）

- 直接截断低概率词，只在**概率最高的前 k 个**里采样。
- 例子（原文）：top-5 在 "coffee, water, tea, rice, chai" 这 5 个里按重缩放概率采样（[S1] §6.1）。

### 5.4 Top-p Sampling / Nucleus Sampling（[S1] §6.1）

> "top-p sampling (a.k.a. **nucleus sampling**) is proposed by **sampling from the smallest set having a cumulative probability above (or equal to p**)."（[S1] §6.1）

- 从词表按概率**降序**逐个加入，直到**累计概率 ≥ p**，在这个最小集合里采样（[S1] §6.1）。
- 与 Top-k 区别：Top-k 的 k 是**固定数量**，Top-p 的集合大小是**自适应的**——分布尖锐时集合小（保守），分布平坦时集合大（多样）。

### 5.5 其他进阶采样（[S1] §6.1）

综述还提到（[S1] §6.1）：
- **η-sampling**：在 top-p 基础上引入基于概率分布的**动态阈值**，进一步改进。
- **对比搜索（contrastive search）**、**典型采样（typical sampling）**：提升生成连贯性。
- **对比解码（contrastive decoding）**：大模型倾向给重要词更高概率，用一大一小两个模型（如 OPT-13B vs OPT-125M）比较 log-likelihood 差，按 delta 采样以放大重要词。
- **DoLa**：把对比思想扩展到**同一模型不同层**的 logits 对比（更高层给重要词更多权重）（[S1] §6.1）。

【面试考点】
- "Temperature 高/低什么效果？"——高温趋近均匀（多样、随机、可能乱），低温趋近贪心（确定、保守、可能重复）；t→0 退化为 greedy（[S1] §6.1）。
- "Top-k vs Top-p？"——Top-k 固定候选数；Top-p（nucleus）按累计概率自适应候选数，更鲁棒（[S1] §6.1）。
- "生产里怎么配？"——常见 Temperature∈[0,2]，Top-p≈0.9，或 Top-k=40~50；RAG/事实问答用低温度，创意写作用高温度。

【易错/陷阱】
- **t=0 不是"无采样"，而是贪心**（每步取 argmax），仍可能陷入重复（[S1] §6.1）。
- Temperature 在 logits 上**除以 t**（不是乘）；高温是 t>1，低温是 t<1（注意有些 API 用 1/t 约定，方向相反，看文档）（[S1] §6.1）。
- Top-p=1 不等于贪心——它仍采样，只是候选几乎全词表。

## 6. 幻觉（Hallucination）：成因与缓解

### 6.1 定义与表现

- 幻觉：模型生成**看似合理但与事实/输入不符**的内容。
- InstructGPT 用具体数据刻画它（[S2] §1，必背）：
> "InstructGPT models **make up information not present in the input about half as often** as GPT-3 (a **21% vs. 41% hallucination rate**, respectively)."（[S2] §1）

即：**InstructGPT 幻觉率 21%，GPT-3 是 41%，对齐让幻觉率大约减半**（[S2] §1）。

### 6.2 成因（综合 [S2] + [WS] 验证）

1. **训练目标本身**：语言建模优化的是"下一个 token 的似然"，不是"真伪"。模型在拟合概率，而非事实（[WS]：LLMs optimize for next-token probability rather than truth）。
2. **训练数据噪声/冲突**：海量网络语料本身含错、含虚构（[S4] 摘要提到 GPT-3 在"训练于大规模网络语料"上存在方法学问题——some datasets where GPT-3 faces methodological issues related to training on large web corpora）。
3. **对齐压力（RLHF）的双刃剑**：RLHF 能降幻觉（21% vs 41%），但也可能引入**讨好性（sycophancy）**，即迎合用户错误前提（[WS]）。
4. **知识截止（knowledge cutoff）**：训练后世界变化，模型用旧知识答新问题（[S1] §2.3 GPT-4 Turbo 把知识源更新到 2023-04 正是为此）。
5. **长上下文遗忘**：上下文越长，"中间内容"越易被忽略（lost in the middle，[WS]）。

### 6.3 缓解策略（综合 [S2]/[S3]/[S1]/[WS]）

| 策略 | 机制 | 来源 |
|---|---|---|
| **RLHF/DPO 对齐** | 用人类偏好直接降低"编造"输出；InstructGPT 幻觉率 21% vs 41% | [S2] §1 |
| **RAG（检索增强）** | 引入外部权威知识作 grounding，事实性最常用方案 | [WS] |
| **低温度/确定性解码** | 降低采样随机性，减少"放飞" | [S1] §6.1 |
| **Self-consistency / 多采样投票** | 多次采样取一致答案 | [WS] |
| **Red teaming（红队）** | 主动构造恶意/边界 prompt 找漏洞，迭代修复 | [S1] §2.3 GPT-4 段 |
| **可预测扩展 + 安全奖励信号** | GPT-4 在 RLHF 训练中加额外安全 reward，6 个月迭代对齐 | [S1] §2.3 GPT-4 段 |
| **Self-awareness（让模型承认不知道）** | 训练模型识别自身知识边界 | [WS] ACL 工作 |

GPT-4 段（[S1] §2.3 原文）明确把幻觉列为需要 mitigate 的主要问题之一：
> "OpenAI has emphasized how to safely develop GPT-4 and applied a number of intervention strategies to mitigate the possible issues of LLMs, **such as hallucinations, privacy and over-reliance**. For example, they introduced the mechanism called **red teaming** to reduce the harm or toxic content generation."（[S1] §2.3）

【面试考点】
- "幻觉能不能完全消除？"——不能。根因是概率拟合目标，只能缓解（[S2]/[WS]）。
- "RLHF 降幻觉的实证？"——InstructGPT 21% vs GPT-3 41%（[S2] §1）。
- "工程上第一选择？"——RAG（grounding）+ 低温度 + 结构化 prompt。

【易错/陷阱】
- RAG 不能根治幻觉——**检索质量**决定上限，检索错则"带毒检索"（[WS]）。
- 低温度减少随机幻觉，但不减少"事实性幻觉"（模型本来就信错的）。

## 7. 涌现能力（Emergent Abilities）与 ICL（来自 [S1] §2.1 + [S4]）

### 7.1 涌现能力的定义（[S1] §2.1 原文）

> "When the parameter scale exceeds a certain level, these enlarged language models not only achieve a significant performance improvement but also show some **special abilities that are not present in small-scale language models**."（[S1] 摘要/§2.1）

具体（[S1] §2.1）：
- 小 PLM（如 330M BERT、1.5B GPT-2）没有、大模型才有的能力叫**涌现能力（emergent abilities [31]）**。
- 经典例：**GPT-3 能通过 ICL 解 few-shot 任务，而 GPT-2 做不到**（GPT-3 can solve few-shot tasks through in-context learning, whereas GPT-2 cannot do well）（[S1] §2.1）。
- 注意：**涌现不保证出现于所有 LLM**（脚注 1）（[S1] §2.1）。

### 7.2 ICL 的形式化（[S1] §6.2.1 原文公式）

> "ICL uses a formatted natural language prompt, consisting of the task description and/or a few task examples as demonstrations... Based on task demonstrations, LLMs can **recognize and perform a new task without explicit gradient update**."（[S1] §6.2.1）

形式化（抓取原文逐字，[S1] §6.2.1）：
- 示范集 `D_k = { f(x_1,y_1), …, f(x_k,y_k) }`，k 条样例，f(·) 把样例转成自然语言 prompt。
- 给定任务描述 I、示范 D_k、新查询 x_{k+1}，LLM 生成预测 ŷ_{k+1}。
- 流程：①从任务数据集选 k 条示范；②按特定顺序、模板组合成 prompt；③把测试实例接到示范后面，喂给 LLM 生成。

### 7.3 从"语言建模"到"任务求解"的范式跃迁（[S1] §2.1）

综述把语言模型演进分四代（[S1] §2.1）：
1. **统计语言模型**：辅助特定任务（检索/语音）。
2. **神经语言模型**：学任务无关表示，减少人工特征工程。
3. **预训练语言模型（PLM）**：学上下文相关表示，可按下游任务优化。
4. **LLM**：探索规模效应带来的能力，视为通用任务求解器。

关键论断（[S1] §2.1）：从 **language modeling 到 task solving** 是科学思维的重要飞跃，是理解语言模型研究史的关键。

【面试考点】
- "涌现能力是啥？举例。"——规模过阈值才出现、小模型没有的能力，如 ICL、CoT；GPT-3 能 ICL，GPT-2 不能（[S1] §2.1）。
- "ICL 公式化？"——见 §7.2，k 条示范拼成 prompt，无梯度更新（[S1] §6.2.1）。

【易错/陷阱】
- 涌现≠"线性外推"——它是**阈值后突现**，小模型外推预测不到。
- "参数大就一定涌现"错——综述明说可能不出现（[S1] §2.1 脚注 1）。

## 8. KV Cache（推理优化，来自 [WS] 验证）

### 8.1 为什么需要 KV Cache

自回归生成时，每生成一个新 token，注意力要让它"看到"**所有历史 token**。如果不缓存，每一步都要**重新计算**所有历史 token 的 Key/Value 投影——O(n²) 级冗余计算（[WS]：without caching, the model would recompute the Key and Value projections for all prior tokens at every single step）。

KV Cache 的做法（[WS]）：**存储并复用**已算出的历史 K/V，新 token 只需算自己的 K/V 并追加。

### 8.2 收益与代价（[WS]）

- **收益**：把每步注意力从"重算全序列"降到"只算新 token"，大幅降低计算与延迟（[WS]：dramatically reducing both computation and latency at inference time）。
- **代价**：**显存随序列长度线性增长**——这是长上下文显存大头（[WS]：memory usage grows linearly with sequence length）。
- **研究方向**：跨层共享（cross-layer attention, arXiv:2405.12981）、量化、淘汰策略（eviction）等压缩 KV Cache（[WS]）。

### 8.3 KV Cache 显存估算（[WS] 推导）

每生成一个 token，每一层、每个头都要存一份 K 和一份 V：
```
KV Cache 显存 ≈ 2 (K&V) × n_layers × n_heads × head_dim × seq_len × batch × bytes_per_elem
```
例（[WS] 口径）：Llama-2-70B（n_layers=80, n_heads=64, head_dim=128），seq_len=4096, batch=1, fp16(2 bytes)：
```
2 × 80 × 64 × 128 × 4096 × 1 × 2 ≈ 10.7 GB
```
可见**长上下文下 KV Cache 显存极可观**，是推理部署的主要瓶颈之一（[WS]）。

### 8.4 KV Cache 优化手段（[WS] 综述）

| 手段 | 思路 | 代表 |
|---|---|---|
| **MQA/GQA/MLA** | 减少共享 K/V 头数 | Llama GQA、DeepSeek MLA（见 §4.6） |
| **PagedAttention** | 类 OS 分页管理 KV，消除碎片 | vLLM |
| **Prefix sharing** | 相同 system prompt 共享 KV | 批处理优化 |
| **量化** | KV 用 8/4 bit 存 | KVQuant 等 |
| **淘汰/滑动窗口** | 丢弃早期 token 的 KV | StreamingLLM |

【面试考点】"为什么长上下文推理贵？"——KV Cache 随 seq_len 线性增长，显存与带宽成瓶颈（[WS]）。

【面试考点】
- "KV Cache 解决什么？"——自回归生成时避免重算历史 K/V，降低延迟；代价是显存随上下文线性增长（[WS]）。
- "为什么上下文窗口受显存限制？"——每个历史 token 在每层每头都要存 K/V，长上下文 KV Cache 占满显存（[WS]）。

【易错/陷阱】
- KV Cache 只在**推理（decode）阶段**有用，不影响训练前向/反向。
- 不同请求 KV Cache 不能混（除非做 prefix-sharing / paged-attention 类优化）。
- KV Cache 大小 ≈ `2 × n_layers × n_heads × head_dim × seq_len × batch × dtype_bytes`（K 和 V 各一）。

## 9.4 能力评估（Capacity Evaluation，来自 [S1] 引言/§7 框架）

- 综述把 **capacity evaluation（能力评估）** 列为 LLM 四大方面之一（pre-training / adaptation tuning / utilization / capacity evaluation）（[S1] 引言）。
- 两大评测范式（综合 [S1] §7 + [S2] §1）：

  2. **人类偏好评测**：让 labeler 对模型输出打分/排序——这正是 InstructGPT/ChatGPT 的核心评测方式（[S2] §1：we mainly evaluate our models by having our labelers rate the quality of model outputs）。
- 评测的关键警告（[S2] §1）：**公开 NLP 数据集不能反映真实使用**——只看 benchmark 会误判（[S2] §1 原文 public NLP datasets are not reflective of how our language models are used）。

【面试考点】"怎么评估一个 LLM？"——benchmark（客观能力）+ 人类偏好（主观对齐）双轨；警惕 benchmark 与真实使用脱节（[S1] §7, [S2] §1）。

【易错/陷阱】
- benchmark 分数高 ≠ 对齐好；InstructGPT 比 GPT-3 在公开 NLP 数据集上**可能略有回归**，但人类偏好大幅领先（[S2] §1）。
- RLHF 评估要看"真实性/有用性/无害性"三轴（HHH），不只看任务准确率（[S2] §1）。

## 10. 【面试考点】速查清单

1. **LLM 定义**：Transformer 语言模型，参数数百亿以上，海量文本训练；最小规模无共识，本综述以 >10B 为讨论范围（[S1] §2.1）。
2. **Scaling Law 两条**：KM（偏重增 N）vs Chinchilla（N、D 同比增长，`C≈6ND`，L=E+A/N^α+B/D^β，α=0.34,β=0.28）（[S1] §2.1）。
3. **训练三阶段**：预训练 → SFT → RLHF/DPO；SFT 在 InstructGPT 语境=RLHF 第一步（[S1] §2.3, [S2] Fig.2）。
4. **RLHF 三步**：SFT → RM → PPO（[S2] Fig.2）。
5. **InstructGPT 标志结论**：1.3B InstructGPT 优于 175B GPT-3（参数差 100×）（[S2] §1）。
6. **DPO**：闭式解把 RLHF 简化为分类损失，无需在线采样，稳定轻量（[S3] 摘要）。
7. **Tokenizer**：BPE（频率合并，GPT-2/LLaMA）vs WordPiece（似然增益合并，BERT）vs Unigram（[S1] §3.2）。
8. **few/one/zero-shot**：上下文窗口里塞 k 条示范；ICL 在前向传播内完成、无梯度更新（[S4] §2, Fig.1.1）。
9. **采样**：Temperature（t→0 贪心，t→∞ 均匀）、Top-k（固定候选数）、Top-p/nucleus（累计概率自适应）（[S1] §6.1）。
10. **幻觉**：根因是概率拟合而非真伪；InstructGPT 21% vs GPT-3 41%；缓解靠 RLHF/RAG/低温度/red teaming（[S2] §1, [S1] §2.3, [WS]）。
11. **涌现**：阈值后突现、小模型没有；GPT-3 能 ICL、GPT-2 不能（[S1] §2.1）。
12. **KV Cache**：复用历史 K/V 避免重算，降延迟；代价是显存随序列线性增长（[WS]）。

## 11. 【易错/陷阱】汇总

- **大 ≠ 对齐**：做大模型不自动等于遵循用户意图（[S2] 摘要）。
- **Scaling 不只看参数**：KM 偏重 N 会"参数过剩、数据不足"，Chinchilla 主张 N、D 同比（[S1] §2.1）。
- **SFT ≠ 传统 fine-tune**：SFT 用多指令格式示范，目标是泛化到未见指令；InstructGPT 里 SFT=RLHF 第一步（[S1] §2.3）。
- **DPO ≠ 无奖励**：仍有奖励概念，只是用策略参数化、闭式消掉显式 RM（[S3] 标题/§3）。
- **BPE 选频率，WordPiece 选似然增益**（[S1] §3.2）；tokenizer 词表训练前固定，跨模型不通用。
- **ICL ≠ fine-tuning**：无梯度更新，发生在单次前向（[S4] Fig.1.1）。
- **Temperature=0 是贪心不是无采样**；logits 是除以 t；API 约定方向看文档（[S1] §6.1）。
- **Top-p=1 仍采样**，不是贪心（[S1] §6.1）。
- **幻觉不可根除**：RLHF 减半（21% vs 41%）但不为 0；RAG 受检索质量制约（[S2]/[WS]）。
- **涌现不保证**：某些 LLM 可能不涌现（[S1] §2.1 脚注 1）。
- **KV Cache 只在 decode 有用**，且请求间不共享（除非 prefix-sharing/paged-attention）（[WS]）。
- **公开 NLP 数据集 ≠ 真实使用**：InstructGPT 明确指出二者不一致（[S2] §1）。

## 11.5 从源文献得到的三大"反直觉"结论（面试压轴用）

来自一手抓取原文，均颠覆直觉，适合作为面试压轴论点：

1. **大不等于好（[S2]）**：1.3B InstructGPT 输出优于 175B GPT-3（参数差 100×）。"把模型做大并不能让它更好地遵循用户意图"——对齐比堆参数更关键（[S2] 摘要/§1）。
2. **参数过剩、数据不足是常态（[S1] §2.1）**：Chinchilla 律证明 GPT-3/Gopher 等早期模型"参数远多于数据"，KM 律的"重参数轻数据"被纠正——业界转而**大幅提高 token 数**。
3. **策略即奖励（[S3]）**：DPO 证明在 KL 约束下，最优策略本身（相对参考模型的对数比）就等价于奖励函数，因此 RLHF 的显式 RM + RL 可被一个分类损失替代——这是"Your LM is Secretly a Reward Model"的数学含义（[S3] 标题/§3）。

附加反直觉点：
4. **涌现不可线性外推（[S1] §2.1）**：GPT-3 能 ICL、GPT-2 不能——能力是阈值后突现，小模型曲线外推预测不到。
5. **采样 t=0 不是"无采样"（[S1] §6.1）**：是贪心 argmax；Top-p=1 也仍采样。
6. **幻觉是概率拟合的副产物（[S2]/[WS]）**：RLHF 把幻觉率从 41% 降到 21%（减半）但不为 0，根因在训练目标优化"似然"而非"真伪"。
