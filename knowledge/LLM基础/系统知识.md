## 1. 什么是 LLM · 参数量 · Scaling Law

### 1.1 定义

**大语言模型（LLM）** 是以 Transformer（通常为 Decoder-only 架构）为骨架、用**自回归（Auto-Regressive）** 方式预测下一个 token 的概率模型。

- 输入：一段 token 序列 $x_{1..t}$
- 输出：下一个 token 的概率分布 $P(x_{t+1} \mid x_{1..t})$
- 本质：一个超大规模的条件概率估计器 $P_\theta(x)$

> 直觉：把"语言理解、世界知识、推理能力"全部压缩进参数 $\theta$。模型不"懂"，只是概率很高地"接得对"。

### 1.2 参数量与规模

| 规模档 | 参数量 | 代表 |
|---|---|---|
| 小 | < 1B | Qwen-0.5B, TinyLlama |
| 中 | 1B – 10B | Llama-3-8B, Qwen2.5-7B, Mistral-7B |
| 大 | 10B – 100B | Llama-3-70B, Qwen-72B, DeepSeek-V3 (671B MoE) |
| 超大 | 100B+ | GPT-4、Claude、Gemini、Grok（万亿级 MoE 猜测）|

**参数量粗略估算公式**（Decoder-only，隐藏维度 $d$、层数 $L$、词表 $V$）：

```
参数量 ≈ 12 * L * d²     （忽略 embedding 与 LM head 的大词表项）
更精确：N ≈ 12*L*d^2 + V*d + L*d   （其中 V 词表项可占数亿）
```

经验：FLOPs/token ≈ 6N（前向+反向 6 次，纯推理 2N）。

### 1.3 Scaling Law（Kaplan 2020 / Chinchilla 2022）

**核心结论**：在固定算力预算 $C$ 下，Loss 与 $N$（参数）、$D$（数据）、$C$（算力）呈幂律：

$$L(N) = \left(\frac{N_c}{N}\right)^{\alpha_N},\quad L(D) = \left(\frac{D_c}{D}\right)^{\alpha_D}$$

**Chinchilla 最优**（DeepMind 2022 修正）：当算力预算固定，最优策略是**参数量与数据量等比例增长**：

```
最优：D / N ≈ 20 tokens/参数
即：70B 模型最优训练量 ≈ 1.4 万亿 tokens
```

这是为什么 Llama 系列拼命堆数据（Llama-2 用 2T tokens，Llama-3 用 15T+ tokens）。

| 关键结论 | 含义 |
|---|---|
| 幂律可外推 | 小模型实验可预测大模型 Loss |
| 数据比模型更稀缺 | Kaplan 严重低估了数据需求（只算 ~1.7 tokens/参数），Chinchilla 修正为 20 |
| 涌现不可预测 | Loss 是平滑下降的，但下游能力是突跳的（见 §7） |

> 面试常考：**"Chinchilla Scaling Law 告诉我们什么？"** → 给定算力，模型与数据应按 ~1:20（参数:token）同步放大，单纯堆参数而数据不够是浪费。

## 2. 训练三阶段（核心 · 必考）

LLM 的训练是一条流水线：**Base Model → SFT → RLHF/DPO → Aligned Model**。

```
┌─────────────┐    ┌─────────┐    ┌──────────────┐
│  预训练 PT   │ -> │  SFT    │ -> │ RLHF / DPO   │
│ Next-Token  │    │ 指令微调 │    │  偏好对齐     │
│ 海量无标注   │    │ 少量指令 │    │ 偏好对数据    │
└─────────────┘    └─────────┘    └──────────────┘
   万亿 tokens       万~百万条        万~十万条对
   学"语言+知识"     学"听指令"        学"什么是好的"
```

### 2.1 预训练 Pre-Training

- **目标**：Next Token Prediction（NTP）/ Causal Language Modeling（CLM）
- **损失**：负对数似然

$$\mathcal{L}_{PT} = -\sum_{t=1}^{T} \log P_\theta(x_t \mid x_{<t})$$

- **数据**：万亿级 token，网页/Common Crawl、书籍、代码、论文，去重 + 质量过滤
- **特点**：自监督（不需要人工标注）、算力集中（万卡 GPU 训练数月）、产出"基座模型"
- 基座模型本身**不会对话**，只会"续写"。你输入"中国的首都是"，它续写"北京"；你输入"你好"，它可能续写一篇关于"你好"的百科。

### 2.2 监督微调 SFT（Supervised Fine-Tuning）

- **目标**：让模型学会"指令跟随"（Instruction Following）格式
- **数据**：`(指令, 期望回答)` 高质量人工范例，通常 1万~100万条
- **损失**：仍是 NTP，但**只在"回答"部分计算 loss**（prompt 部分不参与）

```
输入: <|user|>请把下面句子翻译成英文：今天天气很好<|end|
      <|assistant|>The weather is nice today.<|end|
                          ^^^^^^^^^^^^^^^^^^^^^^^^^^
                          只在这段算 loss
```

- **效果**：基座 → Chat（能对话、能听指令、会用工具格式）
- **关键**：数据质量 >> 数量。LIMA 论文证明 **1000 条高质量数据** 就能微调出不错的对话能力。

> 直觉：预训练让模型"懂语言和世界"，SFT 让模型"懂任务格式"。

### 2.3 RLHF（Reinforcement Learning from Human Feedback）

SFT 后模型能回答，但**好坏不分**（可能冗长、有毒、不安全）。RLHF 用人类偏好来对齐。

**三步法**（InstructGPT / ChatGPT 经典流程）：

**步骤 1：训练奖励模型 Reward Model（RM）**
- 收集偏好对：对同一 prompt，标注员给回答 A 排序高于 B（chosen vs rejected）
- RM 输入 `(prompt, response)`，输出标量奖励 $r(x, y)$
- 损失（Bradley-Terry）：

$$\mathcal{L}_{RM} = -\log \sigma(r(x, y_w) - r(x, y_l))$$

其中 $y_w$ 是更优回答，$y_l$ 是较差回答，$\sigma$ 是 sigmoid。

**步骤 2：用 PPO 优化策略（policy = SFT 模型）**
- 目标：最大化奖励，同时**不要漂离 SFT 太远**（防止 reward hacking）

$$\max_{\pi_\theta} \mathbb{E}_{x\sim D, y\sim \pi_\theta}[r_\phi(x,y)] - \beta \cdot \mathrm{KL}(\pi_\theta \| \pi_{SFT})$$

- **KL 散度惩罚** $\beta \cdot \mathrm{KL}$：核心防越界机制
- PPO 需要 4 个模型同时在显存：Policy、Reference（冻结的 SFT）、Reward Model、Value Model（Critic）—— **显存与工程成本极高**

**步骤 3：迭代**（可选）

> 优点：效果好、能持续优化、能处理细粒度偏好。
> 缺点：复杂、不稳定、4 模型显存爆、超参敏感、奖励黑客（reward hacking）。

### 2.4 DPO（Direct Preference Optimization）—— RLHF 的简化替代

**核心思想**（2023 NeurIPS Best Paper）：RLHF 的最优解有闭式形式，可以直接把"奖励"重参数化为"策略与参考策略的对数比"，于是**绕过显式 RM 和 RL 循环**，把对齐变成一个二分类问题。

**推导关键**：RLHF 的最优策略满足

$$r(x,y) = \beta \log \frac{\pi^*(y|x)}{\pi_{ref}(y|x)} + \beta \log Z(x)$$

把这个关系代入 Bradley-Terry 损失，$Z(x)$ 被消掉，得到 **DPO 损失**：

$$\mathcal{L}_{DPO} = -\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right)$$

直接用梯度下降优化，不需要 RM，不需要 PPO，不需要 Value 模型。

### 2.5 RLHF vs DPO 对比（必考 · 高频）

| 维度 | RLHF (PPO) | DPO |
|---|---|---|
| 是否需要 RM | ✅ 需要单独训练 | ❌ 不需要，隐式 |
| 是否 RL | ✅ PPO 在线采样 + Critic | ❌ 纯监督学习（梯度下降）|
| 显存 | 4 模型（Policy+Ref+RM+Value）| 2 模型（Policy+Ref）|
| 工程复杂度 | 高、不稳定 | 低、稳定 |
| 数据需求 | 偏好对 + 在线 rollout | 仅离线偏好对 |
| 在线/离线 | 在线（on-policy 采样）| 离线（off-policy）|
| 表现上限 | 通常更高（2024 研究显示 PPO 在很多场景仍优于 DPO）| 略低，但差距小 |
| 适用场景 | 大厂、追求极致效果、有充足资源 | 中小团队、快速迭代、资源有限 |
| 代表应用 | GPT-4、Claude 训练链 | Llama-3 后训练也部分用 DPO、Zephyr、大量开源模型 |

> **直觉记忆**：
> - RLHF = 先训练一个"裁判"（RM），再用强化学习让"选手"（policy）讨好裁判，但要看着参考选手别学坏（KL）。
> - DPO = 直接拿"选手 vs 参考选手"的回答概率差当奖励，用监督学习把偏好学进去，跳过裁判。

> **工程现实**：2024 年的共识是 DPO 是"够用且便宜"的默认选择；但顶级模型（GPT-4/Claude）仍以 RLHF/PPO 为主，因为其在线采样带来的探索能力上限更高。两者常组合使用（DPO 做粗对齐，PPO 做精修；或 RLHF 后再 DPO）。

### 2.6 其他对齐方法（了解）

- **ORPO**：把偏好 penalty 直接并入 SFT，一步到位
- **SimPO**：DPO 的无 reference 版本，用长度归一化奖励
- **KTO**：Kahneman-Tversky，只需"好/坏"二元标签，不需成对偏好
- **GRPO**：DeepSeek 提出的 PPO 变体，去掉 Value 模型，用组内相对优势，省显存（DeepSeek-R1 用它做推理强化）

## 3. Tokenizer：BPE / WordPiece / SentencePiece

### 3.1 为什么需要 Tokenizer

模型不认识字符，也不认识词，只认识 **token id（整数）**。Tokenizer 把字符串切成子词（subword），映射到词表 id。

- 太长（按词切）：词表爆炸，OOV（未登录词）多
- 太短（按字符切）：序列太长，丢失语义
- 子词（subword）是折中：常见词整词，罕见词拆成片段

### 3.2 三大算法

| 算法 | 原理 | 代表 | 训练方向 |
|---|---|---|---|
| **BPE**（Byte-Pair Encoding）| 从字符出发，反复合并最高频的相邻 pair，直到词表达到目标大小 | GPT-2/3/4、Llama、Qwen、Claude | 自底向上合并 |
| **WordPiece** | 类似 BPE，但合并判据是**似然增益最大化**（不是频率）| BERT、DistilBERT | 自底向上，似然判据 |
| **SentencePiece** | 直接在**原始字节流**上做 BPE/Unigram，不预设空格分词，**语言无关** | T5、Llama、Qwen、ALBERT | 含 BPE 和 Unigram 两种 |

**Unigram Language Model**（SentencePiece 可选）：从大词表开始，逐步删除对总似然贡献最小的 token，反向筛选。

> 关键区别：BPE/WordPiece 假设输入已按空格分词（适合英文）；SentencePiece 把空格也当普通字符 `▁`，**对中文/日文/韩文等无空格语言更友好**。

### 3.3 中英文分词差异（重要 · 影响成本）

| 语言 | 现象 | 后果 |
|---|---|---|
| 英文 | 一个 word ≈ 1~2 token | 1 token ≈ 0.75 个英文单词 |
| 中文 | 一个汉字 ≈ 1~3 token（视词表而定）| "你好"可能 = 2 token，也可能更多 |
| 代码/Emoji | 字节级展开 | 一个 emoji 可能 = 多个 token |

**实测经验**：同样一段内容，中文 token 数常是英文的 **1.5~3 倍**。这就是为什么中文场景下 token 计费和上下文消耗更高。

### 3.4 Token 计费（工程）

```
成本 = token 数 × 单价（输入/输出分别计价）
估算：1 token ≈ 4 个英文字符 ≈ 0.75 个英文单词
     中文：1 汉字 ≈ 1~2 token（GPT-3.5/4 时代），新模型（GPT-4o/Qwen）已优化到 ≈1
```

- **输入 token**（prompt）：用户给模型的内容
- **输出 token**（completion）：模型生成的内容，通常**单价更贵**
- **缓存 token**（prompt caching）：重复 prompt 可缓存，价格大幅降低

> 直觉：用 `tiktoken`（OpenAI）或各模型的 tokenizer 库本地算 token 数，**永远不要按字符数估算账单**。

## 4. Token · Embedding · 上下文窗口

### 4.1 三个层次

```
文本 ──Tokenizer──> token id ──Embedding──> 向量 ──Transformer 层──> 预测
       (字符串切分)    (整数查表)       (稠密表示)    (上下文建模)
```

### 4.2 Token & Embedding

- **Token**：离散的整数 id，范围 `[0, V)`，V 是词表大小（GPT-4 的 `cl100k_base` ≈ 100k，Qwen2 ≈ 152k）
- **Embedding**：把 id 映射为 $d$ 维稠密向量（d=4096 等），通过查表 `E[id]` 得到
- **位置编码 Position Encoding**：注入顺序信息，因为 Transformer 本身是置换不变的

**位置编码演进**：

| 方法 | 机制 | 代表 |
|---|---|---|
| 绝对位置（Sinusoidal）| 固定正弦/余弦 | 原始 Transformer |
| 可学习位置（Learned）| 词表一样查表 | GPT-2, BERT |
| **RoPE**（Rotary）| 在 Q/K 上做旋转，编码**相对位置** | Llama 全系、Qwen、DeepSeek |
| ALiBi | 在 attention 分数上加位置偏置 | BLOOM, MPT |

### 4.3 上下文窗口 Context Window

**定义**：模型一次前向能处理的最大 token 数 $L_{max}$（输入 + 输出合计）。

| 限制来源 | 说明 |
|---|---|
| 位置编码外推能力 | RoPE 训练长度外推差，需要 YaRN/NTK 等扩展 |
| 注意力复杂度 | 标准 Attention 是 $O(L^2)$，序列长 4 倍算力 16 倍 |
| KV Cache 显存 | 缓存量随 $L$ 线性增长，长上下文显存爆炸 |

**常见窗口规模（2025）**：
- 早期：2k/4k（GPT-3, Llama-2）
- 主流：32k~128k（GPT-4 Turbo 128k, Claude 200k, Llama-3.1 128k）
- 长文本：1M（Gemini 1.5 Pro）, 2M+（实验性 LongRoPE）

> 直觉：窗口越长，能"看"的资料越多，但**模型不等于能"用好"全部窗口**——存在"lost in the middle"现象（中间位置的信息容易被忽略）。

## 5. 采样策略（高频 · 必考）

模型每步输出的是**整个词表上的概率分布** $P(x_{t+1})$，从这个分布里"挑"出一个 token 的方法叫**采样策略**。

### 5.1 Greedy（贪心）

每步选概率最大的 token。
- 优点：确定、快
- 缺点：容易陷入重复、单调，长文本会循环（"我我我我我"）
- 适用：代码生成、数学、需确定性的任务

### 5.2 Beam Search（束搜索）

维护 $k$ 个候选序列（beam），每步扩展，保留累计概率最高的 $k$ 条，最后选总分最高。
- 优点：比 greedy 好，平衡局部与全局
- 缺点：仍偏确定性，生成文本"平庸"，长文重复；不适合开放生成
- 适用：机器翻译、摘要（目标明确的任务）

### 5.3 Temperature（温度）

在 softmax 前缩放 logits：

$$P_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}$$

| T 值 | 效果 |
|---|---|
| T → 0 | 趋近 greedy，确定性最强 |
| T = 1 | 原始分布 |
| T > 1 | 分布变平，更随机、更多样、更"发散" |

> 直觉：T 是"创造性旋钮"。代码/事实问答 T=0~0.3；创意写作 T=0.7~1.0。

### 5.4 Top-k

只在概率最高的 **k 个** token 里重新归一化采样。
- k=1 退化为 greedy
- k 太小：多样性不足；k 太大：可能采到噪声
- 典型：k=40

### 5.5 Top-p / Nucleus Sampling（核采样）

选概率累积和达到 **p** 的最小 token 集合（"核"），在该集合内重新归一化采样。

```
sorted_desc(P) = [0.5, 0.3, 0.1, 0.05, 0.05]
p=0.8 → 选 [0.5, 0.3]（累积 0.8）→ 在这两个里重采样
```

- **自适应**：分布尖锐时选少数 token，分布平坦时选多个
- 比固定 Top-k 更稳健，是**开放式生成的默认选择**
- 典型：p=0.9~0.95

### 5.6 采样策略对比

| 策略 | 多样性 | 质量 | 速度 | 适用 |
|---|---|---|---|---|
| Greedy | 低 | 易重复 | 快 | 代码/事实 |
| Beam | 低 | 平稳 | 慢 | 翻译/摘要 |
| Temperature | 中 | 取决 T | 快 | 通用 |
| Top-k | 中 | 稳 | 快 | 通用 |
| Top-p | 高 | 好 | 快 | 创意/对话 |

> **实战组合**：通常 `Top-p + Temperature` 同时调。例如 GPT API 默认 `temperature=1.0, top_p=1.0`；创意任务设 `T=0.8, top_p=0.9`。

### 5.7 其他采样参数

- **frequency_penalty / presence_penalty**：抑制已出现 token，减少重复
- **min_p**：用最低概率阈值（相对最高概率的比例）替代 Top-p，近年更流行
- **seed**：固定随机种子，使采样可复现

## 6. 幻觉 Hallucination（必考）

### 6.1 是什么

模型生成**语法流畅、语气自信，但与事实不符**的内容。例如编造论文、虚构 API、伪造人物履历。

### 6.2 成因（本质）

| 成因 | 说明 |
|---|---|
| 概率生成本质 | 模型是"最可能的下一个 token"，不是"最真实的"——它在**拟合分布**，不是在**检索事实** |
| 训练数据噪声 | 互联网数据本身有错、有矛盾、有过时信息 |
| 知识截止 | 训练后的新事件模型不知道 |
| 长尾知识稀疏 | 训练里只见过几次的事实，概率低，被高概率的"似是而非"覆盖 |
| 过度自信 | SFT/RLHF 让模型语气自信，掩盖了不确定性 |
| 上下文冲突 | 当 prompt 与训练知识冲突，模型可能两边都沾 |

> 直觉：模型不"撒谎"，它只是**不知道自己不知道**。它没有"事实核查"模块，只有"概率最高"。

### 6.3 缓解手段（三大方向）

**方向 1：RAG（Retrieval-Augmented Generation，检索增强）**
- 把外部知识库检索结果塞进 prompt，让模型"看着资料回答"
- 优点：可溯源、知识可更新、成本低
- 缺点：依赖检索质量、增加 token 消耗、长上下文仍可能丢失
- 适用：企业知识库、最新信息、文档问答

**方向 2：微调（Fine-tuning）**
- 把领域知识"烤进"参数（继续预训练 / 领域 SFT）
- 优点：内化知识、推理快
- 缺点：贵、不可更新、仍可能幻觉（甚至更自信）
- 适用：风格固定、领域术语、高频重复任务

**方向 3：约束解码 / 工程手段**
- **Constrained Decoding**：用 grammar/正则约束输出格式（如强制 JSON Schema、强制只能从词表选）
- **Grounding**：要求模型引用上下文，无依据时回答"我不知道"
- **多路校验**：self-consistency（多次采样取多数）、self-refine（让模型自我检查）、外部校验器
- **降低温度**、设置 system prompt 强调"不确定时如实说明"

### 6.4 评测幻觉

- **TruthfulQA**：测模型是否会模仿人类常见的错误观念
- **HaluEval / FactScore**：实体级事实性评分
- **LLM-as-Judge**：用更强模型当裁判

> **面试金句**：幻觉是 LLM 概率本质的副产品，**无法根除，只能缓解**。生产系统应：RAG 提供事实 + 约束保证格式 + 引用增强可溯源 + 让模型被允许说"不知道"。

## 7. 涌现能力 · ICL · CoT

### 7.1 涌现能力（Emergent Abilities）

**定义**（Wei et al. 2022）：某些能力在小模型上几乎不存在（接近随机），但当模型规模越过**某个阈值**后**突跳式**出现。

- 典型涌现：多步算术、词元操作（找出奇数位置的词）、指令理解、CoT
- 特征：**Loss 平滑下降，能力阶梯式出现**——这是为什么 Scaling Law 难以预测下游任务

> 2023 年有论文（Schaeffer）质疑"涌现"可能是评测指标的非线性造成的假象（用连续指标看就平滑了），但工程上大模型"突然好用"是真实体验。

### 7.2 In-Context Learning（ICL，上下文学习）

**定义**：不做任何参数更新，仅靠**在 prompt 里给几个示例**，模型就能学会新任务。

```
示例 prompt：
把"开心"翻译成英文 -> happy
把"伤心"翻译成英文 -> sad
把"愤怒"翻译成英文 ->

模型输出：angry
```

- **Zero-shot**：不给示例，直接下指令
- **One-shot**：给 1 个示例
- **Few-shot**：给几个示例（通常 3~8 个）

机制假说：ICL 是一种**隐式的梯度下降**——注意力机制在前向计算中"模拟"了学习过程。

### 7.3 思维链 Chain-of-Thought（CoT）

**定义**：在 prompt 里要求模型"一步步思考"（Let's think step by step），或在示例里展示推理过程，让模型输出中间步骤。

```
普通：  Q: 123 × 456 = ?   A: 56088      （可能错）
CoT：   Q: 123 × 456 = ?
        A: 先算 123×400=49200，再 123×50=6150，再 123×6=738，
           合计 49200+6150+738 = 56088.  （更准）
```

- 显著提升数学/逻辑/多步推理能力，**尤其在 ≥60B 模型上才显著（与涌现相关）**
- 衍生：
  - **Zero-shot CoT**："Let's think step by step"
  - **Self-Consistency**：多次采样 CoT，投票取多数
  - **Tree-of-Thoughts（ToT）**：树状探索多条推理路径
- 训练阶段（o1/R1 范式）：用 **RL 训练模型自发产生 CoT**（test-time compute scaling），把"思考"变成模型内在能力

## 9. KV Cache · 上下文长度扩展

### 9.1 KV Cache（推理优化核心）

**问题**：自回归生成时，每生成一个 token 都要对前面所有 token 做 attention。若不缓存，第 $t$ 步重复计算前 $t-1$ 个 token 的 K/V，复杂度 $O(L^2 \cdot d)$，浪费。

**做法**：把每个历史 token 算出的 **Key 和 Value 向量缓存**起来，新生成的 token 只需算自己的 Q，去 query 缓存里的 K/V。

```
第 t 步：Q_t · [K_1...K_t]^T  → attention → V_1...V_t 加权和
                  ^^^^^^^^^^^^
                  来自 KV Cache（已缓存，不重算）
```

**收益**：把生成的每步复杂度从 $O(t \cdot d)$ 降到 $O(d)$（注意力仍 $O(t)$ 但省掉了前向重算）。

**代价**：显存随序列长度线性增长。
- KV Cache 大小 ≈ $2 \cdot L \cdot d \cdot \text{layers} \cdot \text{precision}$
- 例：Llama-70B，batch=1，128k 上下文，FP16 → **数十 GB** 级别

**优化方向**：
| 技术 | 思路 |
|---|---|
| **PagedAttention**（vLLM）| 像 OS 分页一样管理 KV Cache，减少碎片 |
| **GQA / MQA** | 多个 Q 头共享一组 KV，减少 KV 头数（Llama-3 用 GQA）|
| **量化** | FP16→FP8/INT4，减半甚至更少 |
| **KV 压缩/驱逐** | 滑动窗口、H2O（保留重要 token）|

### 9.2 为什么需要上下文长度扩展

RoPE 在训练长度 $L_{train}$ 内表现好，但**外推到更长序列时** attention 分数衰减、性能崩。要让 4k 训练的模型支持 32k/128k，需要扩展方法。

### 9.3 RoPE 回顾

RoPE 把位置编码进 Q/K 的**旋转**：位置 $m$ 的向量乘以旋转矩阵 $R_m$。两个位置的注意力只依赖**相对位置** $m-n$，且高频维度对短距离敏感、低频维度对长距离敏感。

### 9.4 主流扩展方法

| 方法 | 思路 | 训练成本 |
|---|---|---|
| **位置插值 PI**（Position Interpolation）| 把位置 $m$ 缩放为 $m/s$（$s$=扩展倍数），把长序列"压缩"回训练范围 | 需少量微调 |
| **NTK-Aware** | 调整 RoPE 的 base 频率 $\theta$，高频不动、低频拉伸 | 免训练/少训练 |
| **YaRN** | **分段插值**：对 RoPE 不同频率维度区别对待（高频外推、低频插值、中频混合）| 极少 token（10x 少于 PI）|
| **LongRoPE** | 进化版，搜索最优二维缩放，支持 2M+ | 少量微调 |

**YaRN 关键洞察**：RoPE 不同维度频率不同，一刀切缩放会破坏高频维度的局部精度。YaRN 对：
- 高频维度（短距离）：**外推**（不缩放，保细节）
- 低频维度（长距离）：**插值**（缩放，扩范围）
- 中频：混合

> **KV Cache × RoPE 实现陷阱（YaRN 论文强调）**：应缓存**旋转之前**的 K（即先缓存再 RoPE），因为缩放因子 $s$ 一变，每个 token 的旋转都变；若缓存旋转后的 K，改 $s$ 时所有缓存作废。

> **直觉**：YaRN = 给 RoPE 装"变速齿轮"，近处看清楚（高频保真）、远处看得远（低频拉伸），用极少数据就教会模型用更长上下文。

## 面试考点（高频题 + 详细答案）

### Q1：完整描述 LLM 的训练流程，每个阶段在做什么？

**答**：三阶段。
1. **预训练**：海量无标注文本，目标 Next Token Prediction（CLM），损失是负对数似然。学的是"语言规律 + 世界知识"，产出 base model，不会对话。
2. **SFT（监督微调）**：用人工写的高质量 `(指令, 回答)` 对，仍用 NTP 损失但只在回答部分算 loss。让模型学会指令格式与对话方式。
3. **对齐（RLHF/DPO）**：用人类偏好数据让模型输出更安全、有用、符合价值观。RLHF 用奖励模型 + PPO；DPO 直接用偏好对做监督学习。

可选第 4 阶段：推理强化（如 o1/R1），用 RL 训练模型自发产生 CoT。

### Q2：RLHF 和 DPO 的核心区别是什么？什么时候用哪个？

**答**：
- **RLHF**：先训练一个奖励模型（RM），再用 PPO 在线采样优化策略，最大化奖励并加 KL 惩罚防止漂离 SFT。需要 Policy、Reference、RM、Value 四个模型同时在显存，工程复杂、不稳定，但**效果上限高**。
- **DPO**：数学上推导出 RLHF 最优策略的闭式解，把奖励重参数化为策略与参考策略的对数比，于是**绕过 RM 和 RL**，直接用偏好对做二分类式的监督学习。只需 2 个模型，简单稳定，但**表现略低于 PPO**。
- **选型**：资源有限、快速迭代、中小团队 → DPO；追求极致、有充足资源 → RLHF（PPO），顶级模型仍以 PPO 为主。两者可组合（DPO 粗调 + PPO 精修）。

### Q3：什么是 Next Token Prediction？为什么这个简单目标能学出这么多能力？

**答**：NTP 就是给定前文预测下一个 token 的概率分布，最小化负对数似然。它之所以强大，因为：要准确预测下一个 token，模型必须隐式地理解语法、语义、世界知识、常识、推理、甚至规划——这些是预测高质量续写的必要条件。海量数据 + 足够大的模型让这种"压缩"涌现出泛化能力。

### Q4：Temperature / Top-k / Top-p 各自作用？怎么调？

**答**：
- **Temperature**：缩放 logits，T↓ 趋向贪心确定，T↑ 更随机多样。事实/代码任务 T=0~0.3；创意任务 T=0.7~1.0。
- **Top-k**：只在概率最高的 k 个里采样，固定数量。k 太小缺多样性，太大引入噪声。
- **Top-p（核采样）**：选累积概率达 p 的最小集合，自适应——分布尖锐时选少数，平坦时选多个。开放生成默认 p=0.9~0.95。
- **组合**：通常 Top-p + Temperature 同时调，再配合 frequency_penalty 抑制重复。

### Q5：什么是幻觉？怎么缓解？

**答**：幻觉是模型生成流畅自信但与事实不符的内容。成因本质：模型在拟合概率分布，不是在检索事实；加上训练数据噪声、知识截止、长尾稀疏、过度自信。缓解三方向：
1. **RAG**：检索外部知识塞进 prompt，可溯源、可更新；
2. **微调**：领域知识烤进参数，但贵且仍可能幻觉；
3. **约束/工程**：约束解码（强制 JSON 等）、grounding（要求引用上下文）、允许回答"不知道"、self-consistency、降低温度、外部校验。幻觉无法根除，只能缓解，生产系统应组合使用。

### Q6：KV Cache 是什么？为什么重要？有什么代价？

**答**：自回归生成时，把每个历史 token 的 Key/Value 向量缓存，避免重复前向计算。新生成的 token 只需算自己的 Q 去 query 缓存。收益：每步生成从重复计算前文降到只算新 token，大幅加速。代价：显存随序列长度线性增长，长上下文下 KV Cache 可达数十 GB，是长文推理的主要显存瓶颈。优化：PagedAttention（vLLM）、GQA/MQA 减少 KV 头数、量化、压缩/驱逐。

### Q7：RoPE 是什么？YaRN 怎么扩展上下文？

**答**：RoPE 把位置信息编码进 Q/K 的旋转——位置 m 的向量乘旋转矩阵，注意力只依赖相对位置，高频维度对短距离敏感、低频维度对长距离敏感。RoPE 训练长度外推差，超长序列性能崩。YaRN 做分段扩展：高频维度外推（保细节）、低频维度插值（扩范围）、中频混合，用极少数据教会模型用更长上下文。注意实现上要缓存旋转之前的 K。

### Q8：什么是 Scaling Law？Chinchilla 给出什么结论？

**答**：Loss 随参数量 N、数据量 D、算力 C 呈幂律下降。Kaplan 2020 提出，但低估了数据需求。Chinchilla 2022 修正：给定算力预算，最优是**参数和数据等比例增长**，约 20 tokens/参数。结论：单纯堆参数而数据不够是浪费；这也解释了为什么 Llama 系列拼命堆数据（Llama-3 用 15T+ tokens）。

### Q9：什么是涌现能力？CoT 为什么对大模型有效？

**答**：涌现指某些能力在小模型接近随机，但规模过阈值后突跳出现（如多步算术、词元操作、CoT）。CoT（思维链）是在 prompt 里要求/示例逐步推理，让模型输出中间步骤，显著提升数学/逻辑推理。CoT 之所以对大模型有效：(1) 大模型涌现了"按步骤推理"的能力；(2) 中间步骤相当于增加了推理计算量（test-time compute），把复杂问题分解降低每步难度。self-consistency 多次采样投票能进一步提升。

### Q10：BPE 和 SentencePiece 有什么区别？为什么中文 token 更贵？

**答**：BPE 从字符出发反复合并最高频 pair，假设输入已按空格分词（适合英文）；WordPiece 类似但用似然增益判据（BERT）；SentencePiece 直接在字节流上工作，把空格当普通字符，语言无关，对中日韩等无空格语言更友好。中文 token 更贵是因为：英文一个词≈1~2 token，而一个汉字常被切成多个 token（尤其老模型），同样内容中文 token 数是英文的 1.5~3 倍，直接影响上下文消耗与计费。

### Q11：上下文窗口受什么限制？为什么不能无限长？

**答**：三重限制：(1) 位置编码外推能力（RoPE 超长会崩，需 YaRN 等）；(2) 标准 Attention 是 $O(L^2)$，序列长 4 倍算力 16 倍；(3) KV Cache 显存随 L 线性增长，长文显存爆炸。此外有"lost in the middle"现象——中间信息容易被忽略，所以即使支持 1M，也不等于能用满。

### Q12：怎么评估一个 LLM 的能力？看哪些 benchmark？

**答**：分维度看：通用知识 MMLU/MMLU-Pro，研究生科学 GPQA，代码 HumanEval/SWE-bench，数学 GSM8K/MATH，中文 C-Eval/CMMLU，对齐质量 MT-Bench/AlpacaEval，真实体验 LMSYS Arena Elo。方法上区分 zero/few-shot、loglikelihood/generate、Pass@k（代码）、LLM-as-Judge、人工评测。注意污染、刷榜、过拟合格式等陷阱——建议公开榜 + 任务专项 + 自有私有测试集三者结合。

## 易错点 / 陷阱清单

1. **"模型越大越聪明"是错的**。Chinchilla 证明数据同等重要；小模型 + 充分数据 + 蒸馏可超过大模型 + 数据不足。

2. **SFT 不是在"学知识"，是在"学格式/行为"**。知识主要来自预训练；SFT 主要是教模型"什么时候输出什么格式"。

3. **RLHF 不是"教模型新知识"，是"调偏好"**。它让模型更安全/有用/简洁，但不会让模型凭空学会预训练没有的领域。

4. **DPO 不需要 RM，但仍需 reference 模型**。很多人误以为 DPO 完全无参考——其实损失里就有 $\pi_{ref}$。

5. **Temperature=0 不等于完全确定性**。某些后端实现仍可能有微小数值波动；批量请求时 batch 内 padding 也可能影响。

6. **Top-p 不是"取前 p% 的 token"**，是"累积概率达到 p 的最小集合"。这是初学者最常见误解。

7. **降低温度不能消除幻觉**，只是让模型更"保守"。幻觉是概率本质的产物，工程上靠 RAG + grounding。

8. **上下文窗口 ≠ 模型能"用好"的长度**。"lost in the middle"会让中间信息被忽略；长上下文还有显著的"needle in haystack"问题。

9. **KV Cache 在多请求间不共享**（除非显式做 prefix caching）。每个请求独立维护一份，并发下显存是乘出来的。

10. **RoPE 缩放要重新处理缓存**。YaRN 改了缩放因子，所有历史 token 的旋转都变了，必须缓存旋转前的 K，否则缓存失效。

11. **benchmark 分数 ≠ 真实能力**。污染、刷榜、格式过拟合都会虚高；MMLU 高不代表对话好，代码强不代表推理强。

12. **GQA/MQA 减少的是 KV 头数，不是 Q 头数**。它们省的是 KV Cache 显存，不是计算量主体。

13. **BPE 词表对不同模型不同**，跨模型的 token 数不能直接比较。"你好"在 GPT-4 和 Qwen 里可能 token 数完全不同。

14. **预训练的 loss 看着还在降，但下游任务可能早就不涨了**。Loss 是平滑的，能力是阶梯的，别只用 loss 判断。

15. **"微调能消除过时知识"是错的**。模型的知识是分布式的，无法精确删除某条事实；继续预训练会有灾难性遗忘。

## 附：关键公式速查

| 名称 | 公式 |
|---|---|
| 预训练损失（NTP）| $\mathcal{L} = -\sum_t \log P(x_t \mid x_{<t})$ |
| RM 损失（Bradley-Terry）| $\mathcal{L} = -\log \sigma(r(x,y_w)-r(x,y_l))$ |
| RLHF 目标 | $\max \mathbb{E}[r(x,y)] - \beta \mathrm{KL}(\pi \| \pi_{ref})$ |
| DPO 损失 | $-\log \sigma\left(\beta \log \frac{\pi(y_w)}{\pi_{ref}(y_w)} - \beta \log \frac{\pi(y_l)}{\pi_{ref}(y_l)}\right)$ |
| Temperature softmax | $P_i = \mathrm{softmax}(z_i / T)$ |
| Attention | $\mathrm{Attn}(Q,K,V) = \mathrm{softmax}(QK^T/\sqrt{d_k})V$ |
| RoPE | $q_m = R_m q,\ k_n = R_n k$，注意力 $\propto \cos(\theta_{m-n})$ |
| KV Cache 大小 | $\approx 2 \cdot L \cdot d \cdot \text{layers} \cdot \text{bytes}$ |
| FLOPs/token（训练）| $\approx 6N$ |
