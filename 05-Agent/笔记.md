## 0. 本模块一句话总结

> **Agent = LLM + 记忆 + 规划 + 工具**，靠一个"模型循环（model loop）"驱动：模型读上下文 → 决定调哪个工具（或直接回答）→ 宿主执行 → 把结果塞回上下文 → 直到给出最终答案。

本模块三个 demo 把这条链路从最小到完整依次拆开：

| 文件 | 演示什么 | 对应理论 |
|------|---------|---------|
| `01-function_calling.py` | 单轮：模型决定调哪个工具 → 执行 → 回传 | Function Calling [S8] |
| `02-react_agent.py` | 多轮：Thought→Action→Observation→…→Finish 的完整 ReAct 循环 | ReAct [S1] |
| `03-多工具agent.py` | 多工具选择 + 多轮组合（搜索 / 计算 / 查库 串成一条链） | Function Calling + ReAct 综合 |

## 1. Agent 架构总览

### 1.1 四要素（经典公式，面试必背）

```
Agent = LLM（大脑）+ 记忆（Memory）+ 规划（Planning）+ 工具（Tools）
```

各要素在抓取论文里的对应来源：

- **LLM**：ReAct 用"冻结的 PaLM-540B / GPT-3"+ few-shot 提示 [S1]；Reflexion 让 LLM 同时扮演 Actor / Evaluator / Self-Reflection 三个角色 [S3]。
- **记忆**：短期 = 上下文窗口里的对话/轨迹（ReAct 的 Thought-Action-Observation 序列）；长期 = Reflexion 的"情景记忆缓冲区 `mem`"，把反思文本持久化 [S3]。
- **规划**：ReAct 的 thought 用于分解任务、跟踪进度、处理异常、调整计划 [S1]。
- **工具**：ReAct 设计了 `Search / Lookup / Finish` 三种 Wikipedia API [S1]；Toolformer 证明 LM 可自学调用计算器/QA/搜索/翻译/日历 [S2]。

### 1.2 运行时循环（这套代码里就是它）

```
        ┌─────────────────────────────────────────────────┐
        │                                                 │
        ▼                                                 │
  ┌──────────────┐    tool_call     ┌──────────────┐     │
  │   LLM 决策    │ ───────────────▶ │  宿主执行工具  │     │
  │ (读上下文)    │                  │ (沙箱外)      │     │
  └──────────────┘                  └──────┬───────┘     │
        ▲                                  │ observation │
        │         回传结果                  ▼             │
        └─────────────────────────────────────────────────┘
                          直到 LLM 输出 Finish / 最终答案
```

关键不变量：**LLM 永远不直接执行工具**。模型只输出"结构化的调用意图（函数名 + 参数）"，真正执行永远发生在宿主侧。这是安全与可控的基础（也是 MCP 把"工具描述不可信"写进规范的原因 [S5]）。

## 2. ReAct vs Function Calling（最高频对比题）

二者经常被混为一谈，但它们是**工具使用演化的两条不同路线**，必须分清。

### 2.1 本质区别

| 维度 | ReAct [S1] | Function Calling [S8] |
|------|-----------|----------------------|
| **怎么学会用工具** | 不训练，纯 prompt（few-shot 示例） | 厂商在训练时内置 |
| **谁决定调哪个工具** | 冻结模型 + 文本模板（`Thought/Action`） | 模型在推理时输出结构化 `tool_call` JSON |
| **工具描述形式** | prompt 里的自然语言说明 | JSON Schema（`name/description/parameters`） |
| **思考是否可见** | 是，`Thought` 显式写在轨迹里 | 通常不可见（黑盒决策） |
| **可控性** | 高（人类可"编辑 Thought"修正行为，论文图 5 [S1]） | 低（决策在模型内部） |
| **代表** | Princeton/Google 2022 | OpenAI / Anthropic 原生 |

### 2.2 ReAct 的核心创新（一句话）

> 把"思考（thought）"本身纳入动作空间 $A_t = \mathcal{A} \cup \mathcal{L}$（$\mathcal{L}$ 是语言空间），让**推理与行动交替（interleaved）**产生协同（synergy）[S1]。

论文原文（摘要）：
> "reasoning traces help the model induce, track, and update action plans ... while actions allow it to interface with and gather additional information from external sources."

### 2.3 ReAct 要解决的 CoT 两大缺陷

1. CoT 是**静态黑盒**——只用模型内部表示，不与外部绑定，无法更新知识；
2. 因此导致**事实幻觉 + 误差传播**。

ReAct 用"动作"接外部世界来锚定事实。论文还证明：纯 Act（不思考）也不行，不会"为 QA 而交互"——只有 Reason + Act 协同才能拿到时效性知识（附录 A.2 的酒店规模案例 [S1]）。

### 2.4 ReAct Prompt 模板骨架（论文附录 C）

```
Question: <问题>
Thought 1: <分解问题 / 决定搜什么>
Action 1:  Search[<实体>]
Observation 1: <结果>
Thought 2: <从观察提取信息 / 推理>
Action 2:  Lookup[<字符串>]   # 或 Search[新实体]
Observation 2: ...
Thought N: <综合信息得出结论>
Action N:  Finish[<最终答案>]
```

动作空间（论文特意设计得朴素）：
- `Search[entity]` 返回实体页前几句；
- `Lookup[string]` 模拟 Ctrl+F，返回页面里下一个含串的句子；
- `Finish[answer]` 结束。

`02-react_agent.py` 就是这套模板的可运行实现。

### 2.5 一张图区分三条路线（含 Toolformer）

| 路线 | 代表 | 学习方式 | 决策方 |
|------|------|---------|--------|
| **Function Calling** | OpenAI/Anthropic 原生 | 厂商训练时内置 | 模型推理时输出结构化 tool_call |
| **Toolformer** [S2] | Meta，自监督微调 | 模型自我标注 + 过滤 + 微调 | 微调后的模型自主决定 |
| **ReAct** [S1] | Princeton/Google，纯 prompt | few-shot 提示，不动权重 | 冻结模型 + 模板 |

## 3. MCP（Model Context Protocol）

### 3.1 一句话

> MCP 是连接 AI 应用与外部系统的**开源标准协议**，官方比喻：**"AI 应用的 USB-C 接口"** [S4]。目标："build once, integrate everywhere"——Claude / ChatGPT / VS Code / Cursor 都支持。

### 3.2 三层角色（面试必背，规范原文 [S5]）

```
┌──────────┐   JSON-RPC 2.0   ┌──────────┐   JSON-RPC 2.0   ┌──────────┐
│   Host    │ ◀──────────────▶ │  Client  │ ◀──────────────▶ │  Server  │
│ LLM 应用  │                  │ Host 内的 │                  │ 提供上下文 │
│ 发起连接  │                  │  连接器   │                  │  和能力   │
└──────────┘                  └──────────┘                  └──────────┘
```

- **Host**：发起连接的 LLM 应用（如 Claude Desktop）；
- **Client**：Host 内部的连接器（每个 Server 一个）；
- **Server**：提供上下文和能力的服务（数据源 / 工具 / 工作流）。

借鉴了 **LSP（Language Server Protocol）** 的思想 [S5]。

### 3.3 两大类原语（最高频易错点）

**Server 向 Client 提供（Server Features）**：

| 原语 | 给谁用 | 性质 |
|------|--------|------|
| **Resources** | 用户/模型**读** | 被动数据/上下文（如文件、DB 行） |
| **Prompts** | 用户**用** | 模板化消息/工作流 |
| **Tools** | 模型**执行** | 可执行函数 |

> 易错：**Resources ≠ Tools**。Resources 是被动数据（给模型读），Tools 是模型主动执行的函数。混淆是面试常见错误 [S5]。

**Client 向 Server 提供（Client Features，常被忽略）**：

| 原语 | 含义 |
|------|------|
| **Sampling** | Server 反过来请求 Host 的 LLM 帮它生成（递归 agent） |
| **Roots** | 告诉 Server 可操作的 URI/文件系统边界 |
| **Elicitation** | Server 运行中向用户征询额外信息 |

> 设计哲学：MCP 是**双向**的。Server 不仅"提供"能力，也能"消费"Host 的 LLM 推理/文件边界/用户输入。

### 3.4 MCP 安全（面试陷阱）

规范明确把安全当首要考虑 [S5]：

- **用户同意与控制**：所有数据访问/操作需用户明确同意；
- **数据隐私**：Host 暴露用户数据给 Server 前必须明确同意；
- **工具安全**：工具 = 任意代码执行，调用前需用户同意；
- **工具描述不可信（untrusted）**：恶意 Server 可在工具 description 里注入诱导危险操作的指令（"工具描述里的 prompt injection"），annotation 必须视为不可信，除非来自可信 Server。

### 3.5 MCP 与本模块代码的关系

`01/03` 里的 `TOOL_REGISTRY` / `TOOLS`（函数 + JSON Schema）就是 MCP 里 **Server 暴露给模型的 Tools 原语**的最小形态。真实环境：把这份注册表跑在一个 MCP Server 进程里，Claude/ChatGPT 作为 Host 通过 Client 连进来即可——这就是"build once, integrate everywhere"。

## 4. 函数调用关键工程细节 [S8]

- **parallel_tool_calls**：多数模型默认开启，单轮可并行调多个函数；`parallel_tool_calls: false` 可禁用。**注意：和 code_interpreter / file_search 等内置工具一起用时不能并行**。
- **Structured Outputs**：function schema 设 `strict: true`，模型生成的参数**保证**严格匹配 JSON Schema（保证结构，不保证语义）。
- **三层结构要分清**：JSON mode < Structured Outputs < Function Calling（能力递增）。

## 5. Agent 的常见陷阱 [S1][S3][S5]

1. **误差累积（error propagation）**：长轨迹中早期错误被放大——ReAct 摘要明确点名 CoT 此问题 [S1]。
2. **幻觉**：无外部锚定时模型编造事实 [S1]。
3. **上下文窗口限制**：超长轨迹塞不进窗口（Deep Agents 的"自动上下文压缩"正是为此 [S6]）。
4. **信用分配（credit assignment）**：稀疏奖励下难定位哪步出错（Reflexion 用语言化反思缓解 [S3]）。
5. **成本与延迟**：多轮工具调用导致 token 与延迟激增。
6. **反思污染**：Reflexion 的反思若归因错误，错误经验进记忆会持续误导（无形式化保证 [S3]）。
7. **工具描述注入**：MCP/Function Calling 攻击面最大——运行时任意 Server 提供描述，可诱导危险操作 [S5]。

## 6. 面试要点速记（高频问答）

**Q1：Agent 的四要素？**
LLM + 记忆 + 规划 + 工具。各举论文对应 [S1][S2][S3]。

**Q2：ReAct 的核心创新？**
把"思考"纳入动作空间 $A \cup \mathcal{L}$，让推理与行动交替产生协同 [S1]。

**Q3：ReAct vs CoT？**
CoT 是静态黑盒、不与外部绑定、易幻觉误差传播；ReAct 通过动作与外部环境交互获取信息 [S1]。

**Q4：写出 ReAct Prompt 模板骨架。**
`Thought N / Action N / Observation N` 循环，`Action N: Finish[答案]` 结束（见第 2.4 节）。

**Q5：Function Calling vs Toolformer vs ReAct？**
学习方式：厂商内置 / 自监督微调 / 纯 prompt；决策方：模型输出 tool_call / 微调后模型自主 / 冻结模型+模板（见第 2.5 节表）。

**Q6：MCP 三层角色？**
Host（发起连接的 LLM 应用）、Client（Host 内连接器）、Server（提供上下文和能力的服务）[S5]。

**Q7：MCP 服务端三大原语？区分？**
Resources（被动数据，给读）/ Prompts（模板工作流）/ Tools（模型执行的函数）[S5]。Resources ≠ Tools 是高频易错点。

**Q8：MCP 用什么消息格式？借鉴了什么？**
JSON-RPC 2.0；借鉴了 LSP（Language Server Protocol）[S5]。

**Q9：MCP 工具描述可不可信？**
不可信。规范明确警告 tool annotations 是 untrusted，存在工具描述内 prompt injection 风险；调用工具需用户明确同意 [S5]。

**Q10：Sampling 是什么方向的？**
是 **Client 给 Server** 的能力（Server 反过来请求 Host 的 LLM）。很多人误以为 Server 只"提供"能力——Client 还可提供 Sampling / Roots / Elicitation 三种能力 [S5]。

**Q11：Reflexion 和传统 RL 的区别？**
不更新权重，用语言化反思存入情景记忆作"语义梯度"；轻量、可解释、无需微调；但无成功的形式化保证 [S3]。

**Q12：Agent ≠ RAG？**
RAG = 一次性检索注入上下文；Agent = 循环决策（多轮工具调用 + 反思）。ReAct 是把"检索"变成"动作" [S1]。

## 8. 实战代码：多 Agent 协作 + 从零写 MCP server（`04` / `05`）

> 配套代码：`04-多Agent协作.py`（Supervisor + 专家 Workers）、`05-mcp_server.py`（从零 JSON-RPC MCP server）。两者零依赖、`uv run` 直接跑。

### 8.1 多 Agent vs 多工具（最易混的对比）
- `03` 是【单 Agent 调多工具】：一个 Agent 中心化决策"这一步调哪个工具"。
- `04` 是【多 Agent 协作】：Supervisor 把任务【拆解】，委派给多个【专家 Agent】（各自有角色/能力边界），各自处理后把结果回传【共享上下文（blackboard）】，再由 Writer 综合。
- 本质区别：把【决策与能力】分布到多个自治 Agent，而非一个 Agent 拿全部工具包打天下。
- 模式 = Orchestrator-Worker / Plan-and-Execute。
- 真实版要点：每个专家 = 一个带【各自 system prompt】的 LLM 调用——不同 prompt 正是"专家化"的落点（见 `04` 文末标注）。

### 8.2 从零写 MCP server（把协议跑出来给人看）
`05-mcp_server.py` 不用官方 SDK，从零用标准库实现一个真能通信的最小 server，内置 mock client 打印完整 JSON-RPC 往返。要点（面试能脱口而出）：
- **分帧**：JSON-RPC 2.0 over stdio，【按行分隔】（每行一个 JSON 对象，无内嵌换行）；【不用】 Content-Length（那是 LSP 的做法，混了会连不上）。
- **生命周期**：`initialize`（带 `protocolVersion` / `capabilities` 协商）→ `notifications/initialized`（通知，无 id，不回）→ 之后才能 `tools/list`、`tools/call`。
- **工具 schema 字段是 `inputSchema`**（不是 OpenAI function calling 的 `parameters`，两个生态别混）。
- **工具结果**：`{"content":[{"type":"text","text":...}],"isError":bool}`；调不存在的工具返回 JSON-RPC error（code -32602）。
- **工程铁律**：server 的 stdout 只能放 JSON-RPC，日志一律走 stderr，否则 client 解析乱套。
- 想直连 Claude Desktop / ChatGPT：把 `--serve` 注册成它们的 MCP server（规格同上即可）。
- 协议核对来源：[MCP 规范](https://modelcontextprotocol.io/specification)、[raw stdio 通信讲解](https://foojay.io/today/understanding-mcp-through-raw-stdio-communication/)。

### 8.3 跑通验证
- `uv run 05-Agent/04-多Agent协作.py` → 打印 Supervisor 拆解 → 三专家委派 → 共享上下文 → Writer 聚合。
- `uv run 05-Agent/05-mcp_server.py` → 打印完整 JSON-RPC 往返（① initialize ② initialized ③ tools/list ④⑤ tools/call ⑥ error）。
