## 1. Few-shot vs CoT：区别到底在哪（面试最常被追问）

**核心区别**：示例里给不给「推理过程」。

- **普通 Few-shot**：示例只写 `Q→A（最终答案）`。教会模型「输出长什么样」，**但教不会「怎么一步步想」**。
- **Few-shot CoT**：示例写出 `推理步骤 → 答案`。模型模仿分步推理，多步题正确率暴涨。

`01-few_shot与cot.py` 用一个「小模型 vs 大模型」的规则模拟器把这点可视化：
- 小模型两种 prompt 都错（CoT 对 <100B 模型不仅无效，还可能因抄错推理链越带越偏）。
- 大模型在普通 few-shot 下靠猜；换 CoT 后逐步算对。

> **最关键的面试点**：CoT 是**涌现能力**（emergent ability），只在 ≥~100B 参数模型上显著生效。对小模型上 CoT 可能**降低**性能。这是论文 [来源1] 明确警告的经典陷阱。

### Few-shot 设计要点（综合论文与 OpenAI 指南）

| 要点 | 说明 |
|---|---|
| 示例数量 | 1–8 个通常够（CoT 论文用 **8 个** 达到 SOTA） |
| 示例多样性 | 覆盖不同类型/难度，避免过拟合到表面模式 |
| 示例顺序 | 顺序影响结果（近因效应）；评测要多次 shuffle |
| 标签正确性 | 标签错会带偏；few-shot 教格式/过程，不教知识 |
| 格式一致 | 分隔符、标点、字段顺序严格统一 |

## 2. CoT 家族：CoT → Self-Consistency → ToT（层层递进）

```
CoT              单条推理链（贪婪解码）
   ↓ 升级解码
Self-Consistency  多条推理链（高温采样）→ 对最终答案投票
   ↓ 升级结构
Tree of Thoughts  推理树 + LM 自评估 + BFS/DFS 回溯/前瞻
```

### 为什么 CoT 有效（论文给的直觉）
1. **分解计算**：把多步难题拆成每步更简单的 next-token 预测。
2. **可解释**：推理链可读，便于人工/自动校验。
3. **额外算力**：生成更多 token = 给模型更多 forward pass 的「思考时间」。

### Self-Consistency 三个关键设计点（易错）
- 必须 **temperature > 0** 采样才有多样性；`temperature=0` 时多条链几乎一样，投票无意义。
- 投票的是 **最终答案**，不是中间步骤（中间步骤天然难对齐）。
- 只适用于 **有确定答案** 的推理任务（数学、多选、事实问答）；**不能**用于开放生成/创意写作（没有唯一正确答案可投票）。
- 代价：N 倍推理成本（N 通常 10–40）。

### ToT 三个核心组件
1. **Thought decomposition**：把问题拆成粒度适中、可评估、可组合的思维片段。
2. **Thought generator**：每节点生成 k 个候选下一步。
3. **State evaluator**：**让 LM 自己评估**每个候选状态（sure/likely/impossible）——精髓是评估器本身也是 prompt。
4. **Search**：BFS/DFS 系统搜索 + lookahead + backtracking。

> ToT 是「杀鸡用牛刀」：成本高，只在规划/搜索型难题才用；单轮问答用 CoT 足够。

## 3. ReAct：把「推理」和「行动」缝起来（Agent 的祖宗）

### 它修复了 CoT 的两个顽疾
- **CoT 的幻觉**：模型凭空编造事实。ReAct 用工具拿**真实信息**兜底。
- **CoT 的错误传播**：一步算错，后面全错。ReAct 每步基于 Observation **校正**。

### 循环结构
```
Thought 1: <分析问题，决定第一步>
Action 1: <ToolName>[<arg>]        ← 受限动作空间
Observation 1: <工具返回，不是模型生成>
Thought 2: <基于观察决定下一步>
Action 2: ...
... 直到 Action = Finish[<最终答案>]
```

`02-react_pattern.py` 用纯 Python 手写了这个循环，三个演示场景：
1. **单跳问答**：一次 Search 即答（最简形态）。
2. **多跳问答**：先查 A，再查 B，再 Calculate，最后 Finish（HotpotQA 缩影）。
3. **错误恢复**：Agent 误用非法动作 `Wiki[...]`，循环把「未知动作」回灌，Agent 自我纠偏——这正是论文说的「handle exceptions」。

### 手写循环的关键工程点
- **动作受限**：工具注册表 `{"Search": ..., "Calculate": ..., "Finish": ...}` 是 Agent 的能力边界。
- **Observation 不是模型生成**：由环境填回。
- **max_steps 兜底**：长轨迹会卡住/重复 Action，必须设上限（论文明确警告）。
- **非法动作处理**：模型可能输出不存在的动作名，循环要回灌提示而不是崩溃。

### ReAct vs CoT vs 纯 Act

| 范式 | 有推理 | 有工具 | 问题 |
|---|---|---|---|
| 纯 Act | ✗ | ✓ | 难规划、难纠错 |
| 纯 CoT | ✓ | ✗ | 幻觉、错误传播 |
| **ReAct** | ✓ | ✓ | 二者协同 |

## 4. 结构化输出：三档约束（由弱到强）

`03-结构化输出.py` 完整演示。

| 档位 | 手段 | 优点 | 缺点 |
|---|---|---|---|
| 1 Prompt 层 | 要求 JSON + 字段 + few-shot + 分隔符 | 零成本、跨厂商 | 模型小/输出长时不稳定 |
| 2 API 层 | Function Calling / JSON Mode / `response_format=json_schema` | 解码时强制合法 JSON | 各厂商 API 细节不同 |
| 3 文法层 | outlines / llama.cpp GBNF | 可做到 **100% 合规** | 仅本地推理可用，配置重 |

### 工程套路：解析失败 → 回灌 → 重试
```
模型输出 → 抠 JSON 片段(容忍 ```json 代码块和噪声) → json.loads
        → schema 校验 → 失败则把错误拼进 prompt 再调一次
```
- 关键字段务必在 Python 侧二次校验，**别盲信模型**。
- 长输出建议 **分片生成 + 校验 + 重试**（末尾更易破坏格式）。

### Pydantic 的价值
- 用类型注解 + `Field(...)` 声明 schema，解析失败自动抛 `ValidationError` 并给出**逐字段错误**。
- 比 prompt 硬、比手写校验省心。
- 生产里：`response_format=json_schema`（解码层）+ Pydantic（Python 层兜底）= 最稳。

### 分隔符的双重作用
- **隔离指令与数据**：用户输入用 `<data>...</data>` 包裹，防止输入内容被解释为指令。
- **同时是防 Prompt Injection 的手段**（见下）。

## 6. OpenAI 官方六策略（背下来）

1. **Write clear instructions** — 清晰指令：细节、角色、分隔符、输出长度、步骤
2. **Provide reference text** — 给参考资料（RAG/grounding）降幻觉
3. **Split complex tasks into simpler subtasks** — 任务分解、级联调用
4. **Give the model time to "think"** — 留思考时间（= CoT 的官方背书）
5. **Use external tools** — 用外部工具（ReAct 是其范式化实现）
6. **Test changes systematically** — 系统化评测，A/B + 回归，别凭感觉

## 7. Prompt Injection 与安全（OWASP LLM01，第一名风险）

### 两类
- **直接注入**：用户本人输入恶意 prompt（jailbreak）。
- **间接注入（更危险）**：外部数据源（网页/文档/邮件）里藏指令，模型读取后被劫持——用户完全无感。**这是 Agent/RAG 时代头号风险**：模型一旦有工具（发邮件、删文件、转账），间接注入就能转化为真实破坏。

### 防御（分层，无银弹）
| 层次 | 手段 |
|---|---|
| 输入层 | sanitize、限长、检测注入特征 |
| 架构层 | 分隔符+角色标记 `<untrusted>`、最小权限、高危操作 human-in-the-loop |
| 输出层 | 异常输出审计（外发 URL、密钥） |
| 评测层 | promptfoo / Garak 红队对抗 |
| 系统层 | 沙箱/容器隔离 |

> **关键认知**：截至 2026 **没有任何方法能 100% 防住 Prompt Injection**。所有手段都是「降低概率 + 限制爆炸半径」。面试答「分层防御 + 承认无银弹」，别说「我加了一句 system prompt 就防住了」。

## 8. 高频面试问答（含数据，带来源）

**Q：CoT 为什么有效？**
A：①把多步难题拆成单步简单预测；②给模型更多 forward pass 的思考时间；③推理链可解释、可校验。是涌现能力，只在 ≥~100B 才显著生效（小模型反而被带偏）。[来源1]

**Q：CoT 标志数据？**
A：PaLM 540B GSM8K 标准 18% → CoT 57%（8 个 exemplar，超越带 verifier 的微调 GPT-3）。[来源1]

**Q：Self-Consistency 和 CoT 的关系？**
A：CoT 的解码升级版：高温采样多条推理链，对**最终答案**多数表决。GSM8K 比 CoT 再 +17.9%。代价 N 倍成本。不能用于开放生成。[来源2]

**Q：ToT 比 CoT 多了什么？**
A：线性链 → 搜索树，引入 LM 自评估 + BFS/DFS 回溯。Game of 24: CoT 4% → ToT 74%。适用规划/搜索型难题。[来源4]

**Q：ReAct 解决 CoT 什么问题？**
A：用 Thought-Action-Observation 循环融合推理与工具，克服幻觉与错误传播。ALFWorld +34%、WebShop +10%，仅用 1–2 个示例。[来源3]

**Q：Zero-shot CoT 触发词？**
A：`Let's think step by step.`（Kojima et al.）。[来源6]

**Q：如何让模型稳定输出 JSON？**
A：prompt 明确字段+示例+分隔符（弱）→ Function Calling / `response_format=json_schema`（强）→ outlines/GBNF 文法（100%）。再加 Python 侧二次校验 + 失败重试。

**Q：Prompt Injection 两类？怎么防？**
A：直接（用户输入）vs 间接（外部数据藏指令，更危险）。分层防御：输入过滤+分隔符+最小权限+human-in-the-loop+输出审计+红队。**无银弹**。[来源7]

**Q：Prompt Engineering vs Context Engineering？**
A：PE 关注「一段静态 prompt 怎么写」；CE 是「动态系统，把正确信息+工具以正确格式在正确时机送进上下文」（just-in-time）。CE 是 PE 在 Agent/RAG 时代的泛化。[来源8]

## 9. 易错陷阱清单（12 条，面试反向题）

1. **CoT 对小模型有害**（<~100B 编错链）——先确认模型够大再上 CoT。[来源1]
2. **Self-Consistency 不能用于开放生成**（没唯一答案可投票）。[来源2]
3. **ToT 杀鸡用牛刀**（成本高，规划/搜索型才用）。[来源4]
4. **Few-shot 教格式不教知识**；盲目堆示例只增 token 不增效果。
5. **分隔符防注入不是万能**（只是提示，高危操作必须 human-in-the-loop）。[来源7]
6. **system prompt 优先级高 ≠ 防注入**（间接注入藏在数据里照样劫持）。[来源7]
7. **上下文不是越长越好**（lost-in-the-middle、成本、注入面恶化）。[来源8]
8. **ReAct 长轨迹会卡住/重复**——必须设 max_steps、去重、失败兜底。[来源3]
9. **Self-Consistency 必须高温**（temperature=0 多条链一样，投票无效）。[来源2]
10. **结构化输出别只靠 prompt**（小模型不稳，优先 Schema/Function Calling）。[来源5]
11. **示例顺序敏感**（近因效应，评测要 shuffle）。[来源6]
12. **CoT 可能「正确结论+错误过程」**——答案对 ≠ 推理对，需独立校验推理链。[来源1]
